# 最终交付页与图片降级策略

> 状态：设计说明 / 待实现约束  
> 主责：说明涛哥创作工作流最终交付物为什么要面向人类验收，并为非 Codex 环境的图片生成旁路预留兼容设计。  
> 边界：本文不实现外部图片 API，不保存 API key，不定义 Seedream 具体接口字段，不改变任何平台发布链路。

---

## 一、问题判断

当前 workflow 的后台产物以 Markdown 为主，适合 AI 追溯、恢复和复盘，但不适合作为人类最终验收入口。

人类验收最终需要的是：

```text
一眼知道这条内容为什么做。
能直接复制正式文案。
能看到实际画中画图片。
知道每张图插入到哪一句口播附近。
能直接复制各平台发布物料。
需要追溯时，能跳转到后台 Markdown。
```

因此，最终交付物不能只是一组 Markdown 文档。Markdown 是后台链路，HTML 才是前台验收页。

---

## 二、最终交付页定位

每轮内容在 `delivery_confirmed` 后，应生成：

```text
deliverables/final-delivery.html
```

它是人类验收入口，必须比后台文档更好读、更好复制、更好下载。

但它默认是 `project_local` 形态，只保证在当前 session 目录内可用。如果 HTML 被单独拿走，里面的图片和追溯链接可能失效。因此，最终交付页必须标记交付形态：

```text
delivery_page_mode: project_local / portable_bundle / standalone_html
```

推荐页面结构：

```text
1. 选题与切口
   - 选题是什么
   - 为什么做
   - 内容目标
   - 热点来源与时效
   - 内容定位

2. 正式文案
   - 推荐标题
   - 推荐 Hook
   - 完整口播
   - 一键复制

3. 画中画素材
   - 实际图片预览
   - 图片下载
   - 插入位置
   - 对应口播段落
   - 图片来源和生成状态

4. 发布物料
   - 抖音 / 快手 / 小红书 / 视频号
   - 封面标题、视频标题、描述、标签
   - 每个平台一键复制
   - 人工发布注意事项

5. 追溯链接
   - topic_card
   - content_brief
   - draft
   - visual_plan
   - quality_review
   - platform_package
   - content_delivery_record
```

---

## 三、后台 Markdown 的位置

Markdown 不取消，但角色要收窄：

| 类型 | 面向谁 | 主责 |
|---|---|---|
| `intermediate/*.md` | AI | 中间状态、推导链、字段交接、返工恢复 |
| `deliverables/*.md` | AI + 深度审阅者 | 最终文案、最终画面方案、最终平台包的结构化源材料 |
| `deliverables/final-delivery.html` | 人类验收者 | 阅读、复制、下载、确认、转交 |

最终 HTML 必须能反向链接到 Markdown，但不能让用户必须读 Markdown 才知道怎么用。

---

## 三点五、项目内页和可转交包

成熟 HTML 报告工具通常区分普通 HTML、带同级资源目录的 HTML 和自包含 HTML。本项目对应三种交付形态：

| 形态 | 位置 | 适合场景 | 风险 |
|---|---|---|---|
| `project_local` | `deliverables/final-delivery.html` | 项目内验收、AI 恢复 | 单独拿走会断图或断链接 |
| `portable_bundle` | `deliverables/export/{session_id}/` | 发给运营、客户、同事、网盘 | 需要复制资源和 sources |
| `standalone_html` | `deliverables/export/{session_id}/final-delivery-standalone.html` | 单文件转发 | 文件可能变大，后台材料只能摘要化 |

规则：

```text
项目内验收默认生成 project_local。
用户要转交时生成 portable_bundle。
用户要单文件时生成 standalone_html。
不得把 project_local HTML 单独当成可转交包。
```

## 四、图片生成主路径

当前 Codex 环境具备内置图片生成能力时，主路径是：

```text
visual_plan
-> image_asset_generation
-> assets/images/
-> quality_review
-> platform_package
-> content_delivery_record
-> final-delivery.html
```

规则：

```text
中间阶段可以只生成视觉方案和提示词。
进入最终交付前，必须尝试生成必要画中画图片。
生成图片必须保存到 assets/images/。
HTML 里必须展示图片预览和下载入口。
图片生成失败时，不得假装已有图片；必须标记 image_status = generation_failed，并进入降级策略。
```

建议图片命名：

```text
assets/images/B1-hook-price-vs-trust.png
assets/images/B3-customer-questions.png
assets/images/B6-public-interaction-review.png
```

## 五、降级策略预留

不是每个使用者都有 Codex，也不是每个环境都能直接调用内置 image 生成。因此要预留旁路，但暂不实现。

未来可选旁路包括：

```text
Seedream 4.0 / 5.0 等外部图片模型
用户手动把提示词复制到第三方工具
企业内部图片生成服务
人工设计师按提示词出图
```

本项目现在只沉淀兼容约束，不接入外部 API。

## 六、图片提示词兼容约束

为了未来兼容 Seedream 或其他外部模型，`visual_plan` 里的提示词不能只写给某一个模型看。

每个图片任务应同时保留：

```text
beat_id
use_case
insert_after_text
insert_before_text
aspect_ratio
quality_level
scene
subject
important_details
style
constraints
negative_prompt
allow_text_in_image
fallback_caption_text
provider_notes
```

其中：

| 字段 | 作用 |
|---|---|
| `beat_id` | 对应画中画插入点 |
| `insert_after_text` / `insert_before_text` | 让 HTML 和剪辑都知道图片放哪 |
| `aspect_ratio` | 统一 9:16、1:1、16:9 等出图规格 |
| `scene / subject / important_details` | 兼容五槽提示词，也方便转成其他模型入参 |
| `constraints` | 隐私、平台、产品边界 |
| `negative_prompt` | 避免真实手机号、微信号、车牌、平台后台误解 |
| `allow_text_in_image` | 控制是否让模型生成中文 |
| `fallback_caption_text` | 图片中文字不稳时，给剪辑层加字幕 |
| `provider_notes` | 记录某个模型的特殊注意事项，不能污染通用提示词 |

## 七、外部模型旁路的最小输入

未来如果接 Seedream 或其他外部模型，用户不应该重新填写内容策略。

理论上，用户最多只需要提供：

```text
选择图片服务商。
确认模型版本。
提供一次性 API key 或选择已配置环境变量。
确认成本 / 清晰度偏好。
确认是否允许图片含短中文。
```

除此之外，图片生成所需的内容输入都应从 `visual_plan` 和 `manifest.yaml` 自动取得。

禁止让用户重复输入：

```text
选题。
口播文案。
账号定位。
产品边界。
热点来源。
每张图的完整提示词。
```

## 八、降级结果记录

未来实现旁路时，每张图片都要记录生成状态：

```text
image_asset_id:
beat_id:
provider:
model:
source_prompt_id:
asset_path:
image_status: generated / pending_external / generation_failed / manual_required
failure_reason:
human_action_required:
```

最终交付页和导出包还要记录：

```text
delivery_page_mode:
final_delivery_status: html_ready / bundle_ready / standalone_ready / needs_export / blocked
export_status: not_requested / export_ready / export_needs_fix / export_blocked
export_manifest_path:
```

如果没有可用图片服务：

```text
image_status = pending_external
final-delivery.html 仍可生成，但图片区域必须显示“待外部生成”，并提供可复制提示词。
```

这属于降级可用，不属于完整交付。

## 九、当前不做

```text
不接 Seedream API。
不研究或锁定 Seedream 4.0 / 5.0 的具体入参。
不保存 API key。
不建设图片生成服务、队列、回调或上传链路。
不把第三方图片模型能力写成已实现功能。
```

后续如果要实现，必须先单独调研官方接口、成本、版权、失败重试、速率限制和图片保存策略，再进入工程设计。

## 十、可转交包资源清单

如果生成 `portable_bundle`，必须有 `export-manifest.json`。最小结构：

```json
{
  "export_bundle_id": "EXPYYYYMMDD-001",
  "session_id": "SYYYYMMDD-001",
  "delivery_page_mode": "portable_bundle",
  "created_at": "YYYY-MM-DD",
  "entrypoint": "final-delivery.html",
  "resources": [
    {
      "path": "final-delivery.html",
      "type": "html",
      "role": "human_delivery_entry",
      "required": true
    },
    {
      "path": "assets/images/B1-hook.png",
      "type": "image",
      "role": "pip_asset",
      "required": true,
      "source_beat_id": "B1"
    },
    {
      "path": "sources/final-script.md",
      "type": "markdown",
      "role": "source_trace",
      "required": false
    }
  ]
}
```

可选再生成 `manifest-sha256.txt`，用于检查文件是否缺失或被改动。
