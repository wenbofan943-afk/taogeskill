# 工作流问题包与产品设计草案

> 状态：产品设计草案  
> 日期：2026-07-06  
> 主责：沉淀真实跑 workflow 时暴露出的系统性问题，并把问题转成后续产品 / skill 工程化设计输入。  
> 边界：本文不改业务代码、不发版、不实现工具、不调用平台；只做问题包、根因分析和产品设计方向。

---

## 一、背景

本问题包来自 `示例行业观察号` 三篇内容生产测试，以及前一轮 `示例垂类经营号` 最终交付测试。

真实现象是：

```text
在 agent 扶着的情况下，workflow 可以跑通。
但跑通不等于 skill 成熟。
当前大量关键动作依赖 agent 临场判断、手工拆分、补字段、补索引、补图片、补 HTML 和补状态。
```

因此本问题包不把“能生成内容”视为主要验收，而是聚焦：

```text
workflow 能不能被另一个 AI 独立理解。
skill 能不能离开当前 agent 稳定复跑。
多账号、多选题、多画中画、多最终交付时是否会串台。
讨论稿里的方法论是否真正编译进 skill 的执行合同。
```

---

## 二、总体判断

当前 workflow 约为：

```text
L2：可复跑，但需要 agent 扶边界。
```

尚未达到：

```text
L3：可发布候选。
```

主要缺口不是文案能力，而是：

```text
方法论 -> skill -> 交接物 -> 状态机 -> validator -> 最终交付
```

这条编译链没有闭合。

## 三、问题总表

| 编号 | 问题 | 典型现象 | 根因 | 风险 |
|---|---|---|---|---|
| P01 | Skill 不是执行合同 | 讨论稿讲清楚了，但执行时仍可绕过 | 方法论没有变成必填字段、模板和阻断校验 | 换 agent 后跑偏 |
| P02 | 过度依赖 agent 扶跑 | 拆 session、补索引、修 ID、生成 HTML 都靠 agent | skill 缺少状态迁移和落盘器 | 成熟度被误判 |
| P03 | 多选题没有 fan-out / fan-in | 用户说“三篇都做”，agent 临时拆成三个 session | 缺父运行 / 子运行模型 | 多篇内容串台 |
| P04 | 旁支任务没有封锁 | 三篇共用调研，但图片和状态靠手工对应 | 缺 branch lock 和 side effects 记录 | A 图放到 B 文案 |
| P05 | 画中画逻辑没有正确执行 | 最终每篇只做 1 张主图 | final delivery 未反查 required_visuals | 画中画退化为配图 |
| P06 | 图片资产链路不健壮 | 按生成时间映射图片到作品 | 缺 image_asset_id / prompt_id / beat_id | 批量出图错配 |
| P07 | 中间产物和最终产物边界曾混乱 | 一堆 Markdown 被当成人类最终交付 | 未先定义 MD / HTML 角色 | 人看不友好，产品感弱 |
| P08 | 人类门禁设计不合理 | 选题后还要求“确认采用” | 平台包完成被误当人工门禁 | 用户被迫说无意义继续 |
| P09 | 状态同步容易残留旧说法 | `delivery_ready_waiting_human` 残留 | 缺统一状态迁移器 | 恢复时走回旧流程 |
| P10 | 根目录汇总表写入靠手工 | Brief 总表、质检记录、all_runs 都靠 patch | 缺 reducer 节点 | 漏写、重复写、顺序错 |
| P11 | ID 编号有撞车风险 | B/D/QR/DR 编号与其他账号撞 | 缺 ID 分配规则和唯一性检查 | AI 恢复时串内容 |
| P12 | 目录治理无自动约束 | 目录规范存在，但执行靠 agent 记忆 | 缺 scaffold / validator | 新账号或多轮后变散 |
| P13 | Execution trace 只是记录，不是检查器 | trace 写了 agent 扶跑，但不阻断 | trace 未绑定成熟度门槛 | 问题被记录但继续发生 |
| P14 | 讨论稿没有充分编译成 skill | 画中画目的清楚，但执行没按 beat 跑 | 知识层和执行层断开 | 方法论被弱化 |
| P15 | dbskill 式编译不足 | 有方法论和 skill，但缺触发 / 模板 / 不做什么 | 没把讨论稿拆成窄 skill 合同 | skill 难复用 |
| P16 | 最终交付生成不够产品化 | HTML 可用但模板由 agent 临时生成 | 缺固定模板、渲染器和校验器 | 下次交付风格不稳定 |
| P17 | 降级策略只是说明，不是链路 | Seedream 等只写说明，没有入参 / 状态 | fallback_provider 未进入资产字段 | 非 Codex 用户复现困难 |

## 四、问题分组

### A. Skill 编译问题

包含：

```text
P01 / P02 / P13 / P14 / P15
```

核心问题：

```text
讨论稿是知识。
skill 是说明。
但还不是执行合同。
```

表现：

```text
agent 看懂了，但 workflow 没强制执行。
字段缺了也能继续。
输出不完整也能进入下一环节。
trace 能记录错误，但不能阻断错误。
```

产品设计方向：

```text
引入 skill 编译包。
每个方法论必须被编译成：
1. 触发条件
2. 输入字段
3. 必填输出
4. 不做什么
5. 阻断条件
6. 下一步路由
7. validator 检查项
```

---

### B. 多分支与旁支封锁问题

包含：

```text
P03 / P04 / P10 / P11 / P12
```

核心问题：

```text
当前 workflow 默认一轮一条内容。
用户一旦多选题，就靠 agent 临时 fan-out。
```

表现：

```text
三篇内容要拆 session。
根目录索引要合并。
图片要对应到每篇。
状态要从子 run 汇总回父 run。
```

产品设计方向：

```text
引入 parent_run / content_run 模型。

parent_run：
  research_run_id
  selected_topic_ids
  child_run_ids
  fanout_status
  fanin_status

content_run：
  session_id
  branch_id
  topic_id
  input_snapshot
  output_manifest
  branch_status
  side_effects_log
```

硬规则：

```text
子 run 只能写自己的 session 目录。
根目录汇总只能由 fan-in / reducer 节点写。
每个 topic_id 只能绑定一个 active content_run。
```

---

### C. 画中画与图片资产问题

包含：

```text
P05 / P06 / P17
```

核心问题：

```text
画中画不是固定几张图。
画中画由留存风险和留人任务决定。
```

讨论稿已经明确：

```text
画中画不是装饰。
画中画的目的，是在观众本来会划走的地方，让他多停 2-5 秒。
```

正确产品逻辑：

```text
口播拆段
-> 找留存风险
-> 标记必须画中画 / 建议画中画 / 可不画 / 不建议画
-> 生成 required_visuals / optional_visuals / not_recommended_visuals
-> 最终交付必须满足 required_visuals
```

当前偏差：

```text
最终每篇只生成 1 张主图。
这只是最低交付图，不是完整画中画执行。
```

产品设计方向：

```text
visual_plan 必须输出：
beats_count
first_screen_visual_task
required_visuals
optional_visuals
not_recommended_visuals
image_prompts_count

每个 required_visual 必须输出：
beat_id
口播对应
留存风险
唯一留人任务
画面类型
为什么值得生成
不用这张图会损失什么
prompt_id
target_asset_path
image_status
```

最终交付规则：

```text
final-delivery-builder 必须反查 required_visuals。
已生成图片数量必须等于 required_visuals，或明确写 partial_generated / pending_external / generation_failed。
不能用 1 张主图冒充完整画中画资产。
```

---

### D. 人类门禁与状态问题

包含：

```text
P08 / P09
```

核心问题：

```text
人类应该判断业务取舍，不应该替 workflow 点“继续”。
```

新门禁原则：

```text
需要人判断：
1. 账号切换确认
2. 产品 / 活动对象缺失
3. 选题确认
4. 高风险阻断
5. 最终 HTML 验收
6. 发布结果回填

不需要人判断：
1. Brief 是否继续
2. 口播是否继续
3. 画中画方案是否继续，除非高风险或用户主动要改
4. 平台包装后是否确认采用
5. 是否生成最终 HTML
```

产品设计方向：

```text
选题确认 = 授权内容链路跑到底。
最终 HTML = 人类验收入口。
```

---

### E. 最终交付与人类阅读问题

包含：

```text
P07 / P16
```

核心问题：

```text
Markdown 是后台追溯资产。
HTML 是人类验收入口。
```

产品设计方向：

```text
final-delivery.html 必须稳定模板化。
必须包含：
1. 选题 + 切口 + 为什么做
2. 正式文案，可复制
3. 实际画中画图片，可预览 / 下载
4. 插入位置
5. 发布物料
6. 回链到中间 MD
```

后续需要：

```text
固定 HTML 模板。
固定渲染输入。
固定链接检查。
固定图片检查。
固定导出转交包规则。
```

## 五、与 dbskill 的差距

dbskill 的关键不是“讨论稿多”，而是把方法论拆成窄 skill。

它的编译方式是：

```text
方法论
-> 小 skill
-> 触发条件
-> 执行步骤
-> 固定输出格式
-> 不做什么
-> 任务后导航
```

本项目当前更像：

```text
方法论
-> 大 workflow 说明
-> 多个 skill 吸收一部分
-> agent 临场补执行
```

需要补的中间层是：

```text
skill 编译包。
```

## 六、建议的产品模块

### 1. Skill Compiler

作用：

```text
把讨论稿和方法论编译成 SKILL.md 可执行合同。
```

输入：

```text
方法论文档
现有 SKILL.md
字段词典
真实 execution trace
```

输出：

```text
trigger map
input contract
output contract
blocking rules
handoff rules
validator checklist
```

---

### 2. Run Orchestrator

作用：

```text
负责父运行 / 子运行 / 状态迁移 / 多选题 fan-out。
```

能力：

```text
创建 parent_run。
为每个 selected_topic_id 创建 content_run。
给每个 content_run 分配 branch_id 和 session_id。
监控 branch_status。
等待所有子 run 完成后 fan-in。
```

---

### 3. Branch Lock

作用：

```text
防止多篇内容、多张图片、多轮状态串台。
```

能力：

```text
每个 content_run 有独立 input_snapshot。
每个 side effect 必须记录归属 branch_id。
每个资产必须写 target_path。
不允许跨 branch 写文件，除非 reducer 节点执行。
```

---

### 4. Visual Asset Planner

作用：

```text
把画中画从“配图”升级为“留存任务驱动的视觉资产计划”。
```

能力：

```text
拆口播 beat。
判断留存风险。
生成 required_visuals。
绑定 prompt_id / beat_id / target_path。
生成 image_asset_set。
```

---

### 5. Final Delivery Builder

作用：

```text
稳定生成最终 HTML，而不是临场拼页面。
```

能力：

```text
读取 manifest。
读取 final_script / final_visual_plan / image_asset_set / platform_package。
渲染固定 HTML 模板。
检查链接、图片和复制入口。
生成 project_local / portable_bundle / standalone_html。
```

---

### 6. Validator Pack

作用：

```text
让 workflow 不靠 agent 肉眼扫。
```

必须检查：

```text
ID 唯一性。
manifest 路径存在。
topic_id 与 content_run 一一绑定。
research_run_id 全链路贯穿。
required_visuals 是否全部满足。
image_asset_set 图片是否存在且非空。
final-delivery.html 链接是否闭合。
根目录索引和 session manifest 状态是否一致。
是否残留旧状态。
```

## 七、后续修订优先级

### P0：先补执行合同

```text
把 17 个问题转成 skill 编译清单。
把画中画讨论稿转成 visual_plan 必填字段和阻断条件。
把 final-delivery-builder 改成反查 required_visuals。
```

### P1：补多分支模型

```text
定义 parent_run / child_run。
定义 branch_id。
定义 fan-out / fan-in。
定义 reducer 写根索引。
```

### P2：补 validator

```text
先做只读检查清单。
再决定是否做脚本化 validator。
```

### P3：补非 Codex 降级链路

```text
把 Seedream 等外部模型只作为 fallback_provider 字段和入参设计。
暂不接 API。
```

## 八、暂不做

```text
不改业务代码。
不实现前端 / 后端 / 数据库。
不做自动发布。
不接平台 API。
不接外部图片 API。
不把当前草案写成已验收产品。
```

## 九、下一步建议

建议下一轮只做产品设计，不做代码：

```text
1. 把本问题包拆成产品能力设计：Skill Compiler / Run Orchestrator / Branch Lock / Visual Asset Planner / Validator Pack。
2. 先从画中画开始做 skill 编译样例，因为它最能暴露“讨论稿没有进执行合同”的问题。
3. 再设计 parent_run / content_run 状态机，解决“三篇都做”的旁支封锁。
```

完成这三步后，再决定是否进入具体文档修订或工具实现。
