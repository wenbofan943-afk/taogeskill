> 检查范围：`R4-产品总览.md`、`R4-开源交付与净化规则.md`、`R4-产品确认清单.md`、路线图 Phase 5、README / PROJECT_MAP 索引。

结论：

```text
未发现硬冲突。
```

保留的正常重复：

| 重复项 | 判断 |
|---|---|
| `public_release/` 同时出现在总览、细则、确认清单 | 正常。总览定义对象，细则定义允许 / 排除 / 阻断，确认清单用于人类确认 |
| README / AGENTS / PROJECT_MAP 多次出现 | 正常。它们分别是人类入口、AI 入口和索引入口，是 R4 的核心三件套 |
| sample / examples 多次出现 | 正常。R4 的开源价值依赖样例，必须在总览、净化规则和确认项里重复强调 |
| License 多次出现 | 正常。R4 只要求必须有 License，不在产品层决定具体 License 类型 |

已确认不冲突：

| 表面差异 | 判断 |
|---|---|
| 产品阶段“不生成 public_release”，但确认项要求“必须先生成 public_release 候选包” | 不冲突。前者约束当前产品开发阶段，后者约束 R4 经确认后的编译 / 包装阶段 |
| `.github/workflows/` 进入允许清单，但 R4 产品层不实现 release workflow | 不冲突。R4 只把它定义为后续编译候选，不在产品层落实现 |
| R4 alpha 可无真实图片，但 R3 强调 generated 图片与 sidecar | 不冲突。R4 alpha 可用 pending_external 样例，但不得把它说成 generated 路径通过 |
| 产品文档可以出现工作母仓绝对路径，但 public_release 不得引用绝对路径作为依赖 | 不冲突。产品文档记录本地事实，公开包 manifest / README 不得依赖本机路径 |

当前 R4 自洽口径：

```text
R4 产品阶段：定义 public_release 结构、净化规则、确认项和发布门禁。
R4 编译阶段：生成模板、sample、public-manifest、release-checklist 和候选 public_release。
R4 发布阶段：由涛哥另行确认 License、远端仓库、tag 和是否真正发布。
```

## 5. R4 与原始方法论冲突检查

> 检查时间：2026-07-07  
> 对照来源：`AGENTS.md`、`docs/explanation/工作流问题包与产品设计草案-20260706.md`、`docs/explanation/最终交付页与图片降级策略.md`、`docs/explanation/工作流工程缺陷复盘与修订方案.md`。

结论：

```text
未发现硬冲突。
```

对齐关系：

| 原始方法论 / 边界 | R4 对齐方式 |
|---|---|
| 不改业务代码、不发版、不登录平台、不自动发布、不接外部图片 API | R4 只定义开源交付产品层；明确不推 GitHub、不实现自动发布、不接外部 API |
| 当前工作母仓不是可直接公开仓 | R4 强制 `public_release/` 候选包，不直接发布工作母仓 |
| 真实账号、真实内容、客户资料必须脱敏 | R4 把真实 runs、真实档案、真实图片、客户资料列为公开包阻断项 |
| Markdown 是后台追溯资产，HTML 是人类验收入口 | R4 要求 sample-final-delivery 和最终 HTML 链接闭合，但不把一堆 MD 当最终交付 |
| project_local 不能冒充可转交包 | R4 区分 sample、public_release、portable / standalone 交付边界，不把本地页面当公开包能力证明 |
| pending_external 是诚实降级，不是 generated | R4 alpha 可用 pending_external 样例，但必须明确没有验证 generated 图片和 sidecar |
| skill 能不能被另一个 AI 理解 | R4 增加 README / AGENTS / PROJECT_MAP / public-manifest 四入口，降低外部 AI 读晕概率 |
| trace / validator 不能只是记录 | R4 引入 release-checklist、scorecard、link / secret / privacy 检查作为开源前门禁 |

可接受差异：

| 差异 | 判断 |
|---|---|
| 原始方法论讨论的是内容生产 workflow，R4 讨论开源包工程 | 不冲突。R4 是把 workflow 的规则、样例、边界对外发布，不改变内容生产链路 |
| 原始最终交付强调 `final-delivery.html`，R4 强调 README / public-manifest / sample | 不冲突。前者是单条内容交付，后者是开源项目交付 |
| 原始旁路策略暂不做 Seedream，R4 仍保留 pending_external sample | 不冲突。R4 只展示降级边界，不接任何外部模型 |

当前 R4 判断：

```text
R4 是对原始问题包 P07、P13、P16、P17 的开源交付层补齐。
R4 不替代 R1-R3，也不把产品草案写成已验收发布能力。
```
