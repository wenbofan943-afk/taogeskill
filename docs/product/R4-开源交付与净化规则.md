# R4 开源交付与净化规则

> 状态：r4_environment_contract_h6_alpha4_candidate_validated
> 所属路线：R4 开源交付标准  
> 主责：把“本地工作母仓”转换为“公开 GitHub 候选包”的产品规则说清楚。  
> 边界：既有规则已用于 alpha.3；下一版本环境兼容与包完整性合同已完成 H1-H6 并生成 alpha.4 本地候选，不重写 alpha.3 已发布资产，不自动进入远端发布。

---

## 1. 工作母仓与公开包

本项目有两个概念：

| 概念 | 路径 | 作用 |
|---|---|---|
| 工作母仓 | `<PROJECT_ROOT>` | 本地创作、规则演进、真实账号验证；绝对路径由执行环境提供 |
| 公开候选包 | `public_release/` | 经净化后准备发布到 GitHub 的内容 |

硬规则：

```text
不得把工作母仓直接当 GitHub 公开仓。
不得把真实 accounts runs 直接复制进 public_release。
不得把“本地能跑”写成“开源可复现”，除非 sample 通过检查。
```

## 2. 净化动作

| 动作 | 输入 | 输出 | 失败处理 |
|---|---|---|---|
| collect_source_rules | README、AGENTS、docs、skills、objects、字段词典 | public_release 源规则 | 缺入口则 blocked |
| build_examples | tutorials / dry-run / 脱敏样本 | examples/sample-account、examples/sample-run | 发现真实信息则 blocked |
| rewrite_public_readme | 本地 README + R4 产品定义 | public README | 仍指向本机路径则 needs_fix |
| build_public_manifest | R1-R4 状态、样例、边界 | public-manifest.yaml | 缺成熟度或边界则 blocked |
| strip_private_data | accounts、外部资料、图片、日志 | 排除清单 | 发现敏感内容则 blocked |
| link_check | public_release 所有 md/html | 链接报告 | 断链则 needs_fix |
| secret_scan | public_release 文本文件 | 敏感信息报告 | 命中 key/token/cookie 则 blocked |
| package_scorecard | release-checklist | alpha/beta/blocked | P0 缺项则 blocked |

## 5. 样例净化标准

sample account 必须：

```text
使用虚构账号名。
不复用真实账号口头禅、具体经历、客户线索。
保留账号档案字段结构。
标记 sample_only: true。
```

sample run 必须：

```text
使用 SR* 或 SAMPLE* ID。
不包含真实热点截图、真实评论、真实客户信息。
manifest 路径闭合。
execution_trace 说明哪些能力未验证。
final-delivery.html 可本地打开。
图片若未生成，必须显示 pending_external。
```

## 6. 人类可读与 AI 可读

R4 不能只做“给人看的 README”，也不能只做“给 AI 的索引”。

必须双入口：

| 入口 | 主责 |
|---|---|
| README.md | 人先理解项目、快速开始、边界和样例 |
| AGENTS.md | AI 读规则、门禁、路径和不可做事项 |
| PROJECT_MAP.md | 人和 AI 找文件、理解索引关系 |
| release-checklist.md | 维护者判断是否能发布 |

## 8. 发布前阻断

任一命中即阻断：

```text
缺 License。
缺 README 快速开始。
缺 AGENTS 或 AI 入口。
缺 sample。
sample 含真实账号或真实客户信息。
public_release 内有 API key、Cookie、token。
public_release 内有真实 runs。
README 宣称未验证能力。
HTML / Markdown 链接断。
当前能力边界没有说明不自动发布、不登录平台、不调用外部 API。
public-manifest.yaml 缺失或与 README 成熟度口径冲突。
VERSION 与 CHANGELOG 首版条目冲突。
```

## 9. public-manifest 规则

`public-manifest.yaml` 是公开包的机器可读真源，解决“AI 下载后不知道当前能力边界”的问题。

最小字段：

```yaml
project_slug:
release_channel:
package_created_at:
source_commit:
workflow_maturity:
license_status:
entrypoints:
  human: README.md
  agent: AGENTS.md
  map: PROJECT_MAP.md
examples:
  sample_account:
  sample_run:
checks:
  link_check_status:
  secret_scan_status:
  privacy_scan_status:
  sample_trace_status:
unsupported_features:
  - auto_publish
  - platform_login
  - external_image_api
```

规则：

```text
README 与 public-manifest 冲突时，发布阻断。
public-manifest 不得引用工作母仓绝对路径。
sample_run_path 必须存在。
unsupported_features 必须包含自动发布、平台登录、外部 API。
```

## 11. R4 public_release candidate 前置确认记录

> 记录时间：2026-07-07  
> 触发条件：R3 generated 样本已通过、session-scope checker 已识别旧真实样本缺口，准备进入 `public_release/` 候选包生成前的人类闸门。  
> 边界：本记录只做 License 和公开样例范围确认，不生成真实 `public_release/`，不创建远端仓库，不推 GitHub。

参考成熟开源项目的常规做法，公开仓库必须先有明确许可证。GitHub 文档把 License 作为仓库授权入口；ChooseALicense 对 MIT 的定位是短、简单、宽松，适合低摩擦复用；Apache-2.0 同样宽松，但多了明确专利授权和 NOTICE / 变更声明要求。

本项目是 AI 内容工作流、skill 合集、模板和样例项目，不是核心算法库，也不开放业务服务端。因此当前推荐：

```yaml
license_recommendation: MIT
license_reason: "低摩擦、易理解、适合模板 / skill / 文档型开源项目复用"
license_alternative: Apache-2.0
license_alternative_reason: "如果后续更重视专利授权和企业法务口径，可改用 Apache-2.0"
license_status: confirmed_mit
```

公开样例范围推荐如下：

| 类别 | 是否进入 `public_release/` | 规则 |
|---|---|---|
| README / AGENTS / PROJECT_MAP | 是 | 改写为公开相对路径，不引用工作母仓绝对路径 |
| docs/reference、docs/product、docs/explanation | 是 | 保留方法论、规则和路线图，但不得包含真实客户、平台登录态、密钥 |
| skills、objects、templates | 是 | 作为 workflow 可读 / 可编译资产进入公开包 |
| `docs/tutorials/r3-generated-image-sample/` | 是，作为脱敏 generated 样本候选 | 必须保持 sample_only、sidecar、checksum、HTML 链接闭合 |
| examples/sample-account、examples/sample-run | 是 | 只能使用虚构账号和 SR* / SAMPLE* ID |
| 真实 `accounts/*/runs/` | 否 | 即使 checker 能读，也不能直接进公开包 |
| 真实账号档案 | 否 | 只能脱敏后转成 sample |
| `外部资料/` 原始资料包、cache | 否 | 只允许保留来源说明或 README，不带原始缓存 |
| 真实图片、视频、音频 | 否 | 只有明确人工构造、授权、低风险 sample 图片可进入 examples/assets |

候选包生成前必须由涛哥确认：

```text
是否采用 MIT。
是否采用上述公开样例范围。
是否允许把 R3 generated 脱敏样本作为公开 examples 的图片链路样本。
是否保持 release_channel: alpha、workflow_maturity: l2_8。
```

确认前状态必须保持：

```yaml
candidate_preconfirmation_status: confirmed
public_release_generated: false
github_remote_created: false
github_pushed: false
```

## 12. 线下测试包与公开候选包分离

真实大循环 `SAMPLE-SESSION-001` 后新增约束：

| 包类型 | 用途 | 是否可含真实 session | 是否可公开 GitHub |
|---|---|---:|---:|
| offline tester package | 发给少量测试者线下试用、验证 workflow 可读性 | 可以，但必须标注内部测试、不得公开传播 | 否 |
| public_release candidate | 净化后准备开源上线 | 否，只能含脱敏 sample / tutorial / template | 可以，经 release-checklist 通过后 |

线下测试包必须：

```text
有 README-TESTER.md。
有 manifest，说明这是 offline tester package，不是 public_release。
真实 session 如进入包内，必须只用于测试说明，不得对外公开。
包内 HTML 链接必须闭合。
图片必须复制进包内 assets，不依赖原 session 目录。
```

公开候选包必须继续遵守：

```text
不含真实 accounts runs。
不含真实账号档案。
不含真实媒体资产。
不含外部资料 cache。
不含 secrets / cookies / tokens。
```

## 13. 环境可安装性与包完整性

公开包只有“下载后能完整解压、能在声明宿主上运行 checker、失败时能给可行动原因”，才算可安装。以下任一项不能只降为提示：

```text
子进程参数因空格 / 中文被截断。
目标路径超过宿主预算却仍开始构建。
压缩 / 解压返回 0 但缺文件。
同一机器合同在 5.1 与 7 产生不同 BOM 或摘要。
checker 为了通过而联网安装用户模块。
执行策略 / MOTW 被误判为 workflow 缺陷。
Git checkout 因长路径、行尾或路径转义形成假脏工作区。
路径经 symlink / junction / reparse point 逃出允许根目录。
```

发布候选新增三层证据：

| 层 | 必须证明 |
|---|---|
| build preflight | 宿主版本、路径预算、写权限、依赖模式、执行策略可诊断 |
| archive manifest | 包内相对路径、文件类型、大小、SHA256、必需项；比较时使用规范化相对路径 |
| clean-room verify | 从全新短路径和带空格中文路径解压后，以 `-NoProfile`、无可选模块条件运行代表性 checker |

失败处理采用确定性边界：本地锁文件 / 防病毒瞬时占用可做有上限退避重试；路径超预算、非法文件名、策略阻断、摘要不一致和缺文件不得盲重试。测试不得自动修改系统注册表、Group Policy、全局 execution policy 或用户全局 Git 配置来制造通过。
