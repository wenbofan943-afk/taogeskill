# R2 产品总览

> 状态：r2_confirmed_runtime_compiled_dry_run_sampled  
> 所属路线：R2 没有稳定运行模型  
> 目标：把多选、旁支、状态、编号、索引、断流恢复这些运行问题，收敛成轻量 workflow 可执行的产品规则。  
> 边界：本文件记录 R2 产品定义与当前确认口径；已完成运行模型编译和 dry-run 样本，但不实现真正 workflow engine，不自动发布、不登录平台、不接外部 API。

---

## 1. R2 要解决什么

R2 解决的是：

```text
让内容 workflow 从“单篇能跑”升级到“多 session、多选题、旁支任务、断流恢复时不串台、不重跑、不污染”。
```

覆盖问题：

```text
P03：多选题会把后续流程跑废。
P04：旁支任务没有封锁。
P09：状态同步残留旧说法。
P10：根目录汇总表靠手工写，容易和 session 事实冲突。
P11：ID 编号有撞车风险。
P12：目录治理无自动约束。
R1-C18 延伸：真正 checkpoint / resume / retry / run lock 不属于 R1，进入 R2 产品设计。
```

R2 不解决：

```text
画中画数量合同和图片资产链。
Seedream / 外部模型降级入参。
portable_bundle / standalone_html。
GitHub 开源样例、LICENSE、CONTRIBUTING。
自动 validator 脚本。
真正并发执行器。
```

这些分别进入 R3 / R4 / 后续工程化。

## 2. 成熟 Workflow 借鉴

R2 借鉴成熟 workflow 的运行模型，但不照搬重型平台。

| 成熟做法 | 参考项目 | R2 吸收 | 本阶段不做 |
|---|---|---|---|
| Child Workflow | Temporal | 多选题拆成独立 child content_run，每条有自己的状态和产物 | 不实现 Temporal runtime |
| Dynamic Task Mapping | Airflow | 用户选择多个 topic 时，按 topic 列表生成多个独立运行单元 | 不做并发调度器 |
| State / Task Run | Prefect | 每个 stage 有独立状态，失败、完成、等待可区分 | 不做状态服务 |
| Checkpointer / Interrupt | LangGraph | 人类门禁和断流恢复必须保存状态，回来后从状态恢复 | 不实现图执行引擎 |

R2 的轻量取舍：

```text
用 manifest + run_state + execution_trace + indexes 实现可恢复证据。
用 fan-out / fan-in 规则解决多选题。
用 branch_lock 规则隔离产品开发和内容生产。
用状态字段和目录约束降低串台风险。
不用服务端任务队列、不做真正并发、不做自动重试脚本。
```

## 3. R2 核心原则

### 3.1 单篇隔离

R1 单 session 只承载一篇内容。R2 允许用户提出多篇需求，但必须拆开：

```text
一个 topic_id -> 一个 content_run_id -> 一个 session 目录
多个 topic_id -> 多个 content_run_id -> 多个 session 目录
```

禁止：

```text
在一个 session 里混三篇正文。
在一个 final-delivery.html 里塞多个独立选题的完整交付。
用一个 draft_id 写多篇内容。
```

### 3.2 Fan-out / Fan-in

用户说“三篇都做 / 都要 / 这几个都跑”时，R2 处理为：

```text
decision_type = branch_request
branch_mode = fan_out
parent_session_id = 当前选题池 session
child_sessions = 每个 topic_id 一个新 session
```

fan-out 后，每个 child session 独立跑：

```text
topic_card
-> content_brief
-> draft
-> visual_plan
-> quality_review
-> platform_package
-> final_delivery
```

fan-in 只做汇总，不合并正文：

```text
parent summary / indexes/all_runs.md / account index
```

### 3.3 Branch Lock

产品开发、内容生产、开源治理是不同分支，不能互相污染。

| 当前任务 | 允许改 | 不允许改 |
|---|---|---|
| 产品开发 | docs/product、docs/reference、路线图、状态记录 | 生产 session 正文、真实交付 HTML |
| 内容生产 | accounts/{账号}/runs/{session}/、根索引、工作流状态 | 产品路线图、skill 合同 |
| 开源治理 | README、PROJECT_MAP、docs/reference、examples 规划 | 真实账号隐私、生产内容正文 |

如果用户在产品开发中突然说“三篇都做”，R2 先判断为旁支请求，不直接启动内容生产，除非用户明确切换任务。

### 3.4 状态同源

R2 的事实源优先级：

```text
session manifest.yaml
-> session intermediate / deliverables
-> 工作流状态记录.md
-> accounts/{账号}/index.md
-> indexes/all_runs.md
-> 根目录汇总表
```

根目录汇总表和索引只能引用，不做事实源。

### 3.5 恢复边界

R2 要在产品层定义脚本级断点续跑的轻量近似，但不实现脚本：

```text
stage_status
resume_from
last_completed_stage
failed_stage
retry_policy_note
idempotency_policy
run_lock_status
```

这些字段先进入产品定义和 manifest 规则，后续是否脚本化进入 R4 / 工程化。

## 5. R2 完成口径

R2 产品层完成后，可以说：

```text
多选题、旁支任务、状态恢复、ID、索引、目录约束已有产品定义。
```

不能说：

```text
已经支持真正并发。
已经有自动 resume runner。
已经有 validator 脚本。
已经可以完整真实测试。
```

R2 确认后，才允许进入：

```text
运行模型和分支规则的 skill / reference 编译。
```

### 6.1 返修后新增颗粒度

这轮补强不是扩大 R2 范围，而是把编译时容易靠猜的部分压实：

```text
Parent Close Policy：父 session 完成、归档、取消、失败时 child 怎么办。
State Transition：每次状态变化都留 actor、action、from/to 和原因。
Checkpoint：阶段完成、人类停顿和断流恢复前有追加式快照。
Human Interrupt Payload：用户自然语言转成可执行决策包。
Branch Request Ledger：分支过程追加记录，不只覆盖 manifest 当前态。
Operation Contract：R2 关键动作必须有输入、输出、状态变化、失败处理和恢复提示。
Resume Read Order：断流恢复按固定顺序读状态和产物，不能凭聊天记忆恢复。
```
