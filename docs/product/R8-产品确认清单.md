## 产品二义性审核

| 问题 | 结论 |
|---|---|
| “拆 Skill”是否等于新增更多用户命令？ | 否。只有 router 内部两个人类门禁改变 owner，用户自然语言入口不变。 |
| references 是否每次都读？ | 否。每个 reference 必须有版本化或状态化条件。 |
| 行数下降是否自动等于效果提升？ | 否。必须通过旧版 / 新版 A/B。 |
| 是否重做热点产品逻辑？ | 否。R5 业务决定不重开。 |
| 是否把四个平台拆成四条 workflow？ | 否。仍形成一个 platform package。 |
| 是否改变历史 session？ | 否。历史 session 只按旧 baseline replay。 |
| 是否包含 Codex 自带 Skill？ | 否。只包含项目业务 Skill。 |
| 是否已经完成编译？ | H1 至 H4、H5R1 与 H5R2 已完成。H5 v0.1 的旧机器入口只保留历史失败证据；H5 v0.2 current 为 `inputs_and_snapshots_compiled_evaluator_pending`。H5R3-H5R5 尚未编译，H5 尚未关闭。 |
| 当前盲评包还能继续评吗？ | 不能。6 案只有 1 案双方均 produced，且机器 audit fail 时工具仍宣称 ready；应隔离为失败证据。 |
| “独立子代理”是否等于 clean-room？ | 否。当前只能证明独立任务和指令边界，不能证明共享文件系统没有越界读取。 |
| baseline / candidate 必须收到完全相同 JSON 吗？ | 不必须。共享业务语义必须相同，但不同合同版本应由各自 adapter 生成可直接消费的 typed input。 |
| 一边 produced、一边 blocked 能做质量盲评吗？ | 不能。它属于能力 / 回归处置，不是同类业务产物质量比较。 |
| 两条 actual_node_id 是否证明 Skill 路由错误？ | 否。两案均正确 fail-closed；错误来自 requested node 与 executed node 未分字段。 |
