## 2. 标准流程

```text
copywriting-draft-writer
-> talking-head-image-pip 编排 static-visual-director，原子生成 static_visual_director_plan / visual_need_analysis / visual_plan tasks / visual_text_plan
-> image-prompt-compiler 按 image_task_id 生成 image_prompt_set
-> image-asset-producer 执行 Codex 出图 / Seedream 降级 / 画中画确定性叠字，并生成 generation_record / metadata_sidecar / fallback_status
-> copywriting-quality-review(content_visual_review) 做正文 + 视觉联合质检
-> platform-packaging-adapter 生成 platform_package / cover_variant_set
-> cover-design-compiler 生成 cover_design_package / cover_composition
-> copywriting-quality-review(cover_review) 生成 cover_quality_gate
-> final-delivery-builder 嵌入实际图片或降级占位
```

自动推进口径：

```text
static_visual_director_status=director_plan_pass、visual_need_analysis_status=pass、visual_plan_status=visual_plan_pass、visual_text_plan_status=visual_text_plan_pass 且 accepted 任务一一对应 -> 进入 image-prompt-compiler。
prompt_integrity_check 全部 pass，且 accepted 图片任务已有真实资产或诚实 fallback -> 进入 content_visual_review / visual_text_quality_gate。
content_visual_review 通过 -> 进入 platform-packaging-adapter；随后才进入 cover-design-compiler 和 cover_review。
最终交付前 accepted image task 必须 generated 或有明确 fallback 状态。
图片缺失不阻断无图 HTML，但必须在 HTML 中清楚显示。
样本模式未通过前，不进入多篇批量出图。
静态视觉编导层未通过前，不得直接生成 image_prompt_set。
封面设计包缺失时，平台包装可以先给文字包装，但最终交付不得宣称“封面设计已完成”。
```

状态边界：

```text
`image_status` 只用于单张图片资产：generated / pending_external / generation_failed / manual_required / rejected。
`image_assets_status` 用于整组图片资产：all_generated / partially_generated / pending_external / generation_failed / manual_required / mixed / not_required。
```

## 2.1 静态图片边界

R3 当前只做静态图片资产：

```text
做：画中画图、证据图、解释图、情绪图、封面图、封面设计包。
不做：动态视频、分镜成片、剪辑时间线、Seedance / Veo / 视频生成 API、自动上传。
```

因此本层不使用完整影视 storyboard 作为主模型，而使用轻量静态视觉编导：

```text
draft
-> static_visual_director_plan
-> visual_plan / cover_design_package
-> image_prompt_set
-> image_asset_set
-> final_delivery
```

图片类型边界：

```yaml
image_asset_type:
  allowed_values:
    - picture_in_picture_image
    - cover_image
```

`picture_in_picture_image` 服务正文画中画；`cover_image` 服务平台入口封面。两者可以复用同一个文件，但必须分别通过对应质量门禁。

封面图片内部再区分：

```text
cover_background_asset：封面底图，可以无字，负责情绪、场景、主体和视觉冲突。
cover_composited_asset：最终成品封面，已经把封面标题、版式和平台比例合成进去，可直接上传。
platform_cover_asset：针对抖音 / 快手 / 小红书 / 视频号等平台裁切或标题策略不同的成品封面。
```

没有 `cover_composited_asset` 时，只能说“封面底图已生成 / 封面设计方案已生成”，不能说“成品封面已完成”。

图片生产路径：

```yaml
image_production_path:
  allowed_values:
    - codex_image2_render
    - seedream_prompt_delivery
    - manual_upload
```

`codex_image2_render` 表示 Codex 环境直接调用内置 image 生成，生成文件并整合进 HTML。
`seedream_prompt_delivery` 表示非 Codex 环境下不调用外部 API，只交付 Seedream 入参、完整 prompt 和外部生成说明。
`manual_upload` 表示用户或设计师人工补图。

三轨边界：

| 信息轨 | 负责 | 不负责 |
|---|---|---|
| `narration_track` | 观点、因果、叙事推进 | 不要求把每句话画成图 |
| `subtitle_track` | 忠实转写、无声观看、可访问性 | 不承担视觉隐喻或第二叙事 |
| `visual_text_layer` | 内心、角色、机制、比较轴、证据、语境、潜台词 | 不机械复写口播或字幕 |

决策算法：

```text
1. 先问图片不含字能否准确完成 viewer_job；能则优先 forbidden。
2. 再问文字是否提供口播和字幕没有提供的新信息；没有则 forbidden / needs_fix。
3. 再问缺字是否会造成角色、机制、比较或证据歧义；会则 required。
4. 再问文字是否涉及数据、引语、截图或事实证据；涉及则必须 source_bound。
5. 最后选择渲染方式：准确性优先时 deterministic_overlay / source_native_text；模型含字失败可降级。
```

当前静态图片链不新增字幕产物。`narration_source_type` 固定为 `draft_script_text`，`narration_source_path` 指向本 session 的 draft；字幕存在时记录 `subtitle_source_path`，不存在时写 `subtitle_source_status=not_available`。字幕不存在仍必须对 draft 口播文本完成重复检查。

### 2.2.2 原子规划事务与事实源

```text
1. talking-head-image-pip 读取 draft / brief，启动 static_visual_planning_bundle。
2. static-visual-director 先分配 static_visual_director_plan_id、visual_plan_id、beat_id、image_task_id、visual_text_plan_id。
3. 同一事务内写 visual_role_map、visual_need_analysis.candidates[]、visual_plan.accepted_visual_tasks[] 和 visual_text_plan.visual_text_tasks[]。
4. 每个 image_task_id 必须恰好对应一个 visual_text_task；forbidden 任务的 units 必须为空。
5. 三个对象全部通过后，image-prompt-compiler 才能按 task 编译 Codex / Seedream prompt。
6. 任一对象失败时保留草稿和 trace，但不得把 bundle 标记为 pass。
```

物理事实源固定为 `intermediate/05-visual-plan.md`，其中分区保存三个对象。第一版不再允许另建 `05-static-visual-director.md / 05-visual-text-plan.md` 的双路径；未来拆文件必须通过迁移版本升级，不能临场选择。

视觉角色默认矩阵：

| visual role | 默认决策 | 推荐文字角色 | 关键门禁 |
|---|---|---|---|
| `emotion_amplifier / scene_context / memory_symbol / metaphor` | `forbidden / optional` | `inner_voice / subtext / context_marker` | 不得冒充事实 |
| `hook_conflict / comparison` | `optional / required` | `role_label / comparison_label / inner_voice` | 角色和比较轴必须清楚 |
| `mechanism_explainer`（内容子型可为 flow） | `required`，除非关系已一眼可懂 | `mechanism_label / visual_narration` | 不可堆满解释文字 |
| `evidence_support`（内容子型可为 data / quote / proof_screenshot） | `required` | `source_native_text / data_point / source_note` | 必须来源绑定、原始资产和隐私可追溯 |

`effective_information_gain` 不等于 raw information density。高质量画中画应让观众在短时间内多理解一层，而不是多读一段字幕。

默认视觉文字预算：

| 文字角色 | 默认预算 |
|---|---|
| role / comparison / context label | 2-8 个汉字 |
| inner_voice / subtext / visual_narration | 4-14 个汉字 |
| mechanism_label | 每节点 2-6 个汉字，默认最多 4 个节点 |
| data / evidence label | 保留准确数字或原生短语，不为凑字数改写事实 |
| source_note | 6-24 个汉字或短来源名，必须手机可读 |

单张图默认 1-3 个文字单元；机制图最多 4 个节点，证据图可额外增加 1 个来源说明。超过预算必须写 `overflow_reason` 并进入认知负荷复检。

评分门槛：`semantic_delta_score / glance_comprehension_score / mobile_readability_score` 均为 1-10 分。optional / required 含字任务默认不得低于 7；5-6 为 needs_fix，1-4 为 blocked。forbidden 任务的 semantic delta 可为空。

`cover_text_render_strategy` 说明：

```text
deterministic_overlay：默认推荐。由工具 / HTML canvas / 设计软件把文字叠到封面底图上，文字最可控。
model_text_in_image：允许用于短字、海报感强的候选封面；必须保留 prompt 并质检文字准确性。
manual_design：交给设计师或用户手工排版。
prompt_only：非 Codex 或无法合成时，仅交付可复制提示词和版式说明。
```

平台变体规则：

```text
同一内容至少有主推荐 cover_composition。

抖音、快手可共用强冲突竖版封面，除非账号风格不同。
```

## 9.1 static_visual_quality_gate

静态图片质量门禁不只看“文件是否存在”，而是看这张图是否值得进入最终交付。

```yaml

  image_asset_id:
  static_visual_director_plan_id:
```

评分口径：

| 分数 | 含义 |

## 12. 样本模式与批量边界

R3 采用 dbskill 的“先样本、再批量”思想。即使用户要求多篇一起做，图片资产链也必须先在单篇样本上跑通。

```yaml
r3_sample_gate:
  sample_scope: one_content_minimum_asset_chain

    - visual_beat
    - prompt_card
    - generation_record
    - image_asset_or_honest_fallback
    - metadata_sidecar_if_generated
    - html_embed_manifest
  pass_condition:
    - no_missing_required_field
    - status_honest
    - local_path_valid_or_pending_clear
    - prompt_traceable
  batch_allowed: true / false
```

批量限制：

```text
样本不过，不开多篇出图。
样本可以固定验证 1 张 accepted 图的最小资产链；正式内容必须先完成 visual_need_analysis，再生成所有通过任务。样本的固定数量不得反推正式内容数量。
多篇任务必须按 R2 child session 隔离 image_asset_set。
fan-in 只汇总状态和路径，不混放不同账号或不同作品的图片。
```

## 14. R3 验收口径

R3 产品层通过标准：

```text
图片数量不是看心情或看时长，而是 visual_need_analysis 的派生结果；允许 0 到 N、不设上限。
图片生成不是内容语言直译，必须先有静态视觉编导方案。
图片资产先分画中画图片和封面图片，再决定生产路径。

每张图都有 visual_text_decision；含字图能说明 information_delta、文字角色和渲染策略。
视觉文字与字幕分轨；证据文字有来源绑定，生成情境图不冒充证据。
质量目标是有效信息增量，不是原始文字和元素数量。
每张 accepted 图都有 static_visual_quality_gate，且核心分数达标。
封面设计包不是封面标题，必须包含封面图来源、版式、安全区和平台差异。
封面设计包有 cover_quality_gate；重要内容有 cover_variant_set 和推荐理由。
重要封面变体具有实质视觉差异；封面设计合同、成品渲染和平台预览能互相核对。
generated / pending_external / generation_failed / manual_required / rejected 状态不会混用。
每次生成尝试都有 generation_record。
generated 图片有本地 asset_path 和 metadata_sidecar_path。
重做图片不会覆盖旧资产。
样本模式通过前不进入批量图片生产。
final_delivery.html 知道每张图怎么展示、下载和追溯。
发布后数据回流暂不开发；只作为 backlog，不阻断本轮 R3。
Codex 环境 final delivery 优先生成实际图片并嵌入 HTML；非 Codex 环境交付 Seedream 入参，不假装有图。
外部模型降级只保留兼容字段，不误称已实现。
```

## 16. P0-H7 最终交付修订模型

### 16.1 定位与真源

`final-delivery.html` 是创作者的发布执行工作台，不是新的业务事实源。事实与投影关系固定为：

```text
draft / visual_need_analysis / visual_plan / visual_text_plan
+ quality reviews / image_asset_set / platform_package / cover_composition
-> delivery_revision（绑定 source ID + digest）
-> official typed render input（当前 revision 的官方投影）
-> final-delivery.html / final-script.md / final-visual-plan.md / final-platform-package.md
```

`content_delivery_record` 必须登记当前 `delivery_revision_id` 和当前 source IDs。candidate 只允许作为待编译输入；不能因为 candidate 某个局部被替换，就让 final Markdown 和 content delivery record 留在旧 revision。

建议字段：

```yaml
delivery_revision:
  delivery_revision_id:
  session_id:
  revision_no:
  revision_status: current / superseded / blocked
  supersedes_delivery_revision_id:
  source_artifact_bindings:
    - artifact_type:
      artifact_id:
      sha256:
  generated_view_paths:
  semantic_gate_status:
```

### 16.2 平台交付单元

平台交付不再由“平台文字卡”和“封面图片卡”分别展示后让用户自己猜对应关系。每个平台形成一个 `platform_delivery_unit`：

```yaml
platform_delivery_unit:
  platform:
  cover_title:
  rendered_cover_text:
  cover_asset_id:
  cover_asset_path:
  cover_composition_id:
  cover_quality_gate_id:
  video_title:
  publish_description:
  hashtags:
  publish_readiness:
```

约束：

```text
cover_title 与 rendered_cover_text 规范化后必须一致。
cover_asset_path 必须存在且 hash 与 asset 记录一致。
共享一张封面成品时，共享平台必须使用相同 rendered_cover_text。
某个平台标题不同，就必须重新合成对应封面成品，不能只改文字卡。
任一项错配时 publish_readiness=blocked。
```

### 16.3 画中画执行卡

每张 generated / reused_verified PIP 的用户层最少展示：

```text
顺序。
插在“哪句话之后、哪句话之前”。
图片预览和下载。
这张图解决什么观看问题。
画面上真实可见的文字；无字图明确写无字，不展示空字段。
非阻断风险或使用注意。
```

折叠审计层展示：

```text
image_task_id / asset_id / source IDs。
完整 prompt 与 prompt digest。
generation record、metadata sidecar、来源绑定和 hash。
visual_text_role / information_delta / quality gate。
```

`trigger_text` 只用于识别语义节点，不能替代位置。`insert_after_text` 与 `insert_before_text` 是新交付的必填字段；只有真实音频或时间线存在时，才允许附加 `time_range`，不得凭预计时长伪造时间码。

### 16.4 Warning 聚合

最终交付 warning 不是字符串数组的最后一次覆盖，而是全链聚合对象：

```yaml
delivery_warning_item:
  warning_code:
  warning_category: research / copy / visual / cover / publishing / runtime_scope
  severity: blocking / non_blocking / known_scope
  user_message:
  impact:
  recommended_action:
  source_artifact_id:
  resolution_status: open / resolved / accepted
```

派生规则：

```text
任何 blocking warning -> delivery_readiness=blocked。
只有 non_blocking / known_scope -> ready_with_warnings。
resolved 不进入用户首屏，但保留审计记录。
下游重跑只能新增、关闭或确认 warning，不能用新的 warning_codes 数组覆盖上游开放项。
用户层只展示 user_message / impact / recommended_action；raw code 默认折叠。
```

### 16.5 时长与图片数量

时长和图片数量继续分离：

```text
图片数量：由 visual_need_analysis 派生，0 到 N，不设上限。
口播时长：用于用户拍摄和剪辑计划，不作为图片配额。
```

新交付建议字段：

```yaml
delivery_duration_estimate:
  duration_estimate_status: measured / derived_range / not_available
  measured_duration_seconds:
  estimated_duration_min_seconds:
  estimated_duration_max_seconds:
  speech_rate_profile:
  derivation_method:
  source_text_digest:
```

账号历史实测或已登记、可追溯的语速 profile 优先。没有校准来源时写 `duration_estimate_status=not_available` 和原因，不输出伪精确区间；禁止把 fixture 固定秒数或未经验证的全局语速常量带入真实内容。

### 16.6 业务语义门禁

最终交付 checker 必须从结构检查升级为跨产物语义检查，至少覆盖：

```text
所有派生视图属于同一 delivery revision。
content delivery record 使用当前 revision 的 draft / quality / image / cover / platform IDs。
平台文字与真实封面文字、平台、路径一一对应。
PIP 数量、顺序和 exact placement 与当前 visual plan 一致。
全链 warning 没有被覆盖或遗漏。
阶段标签来自 session context，不允许固定 H5 / H6 文案。
时长有测量或派生依据。
用户层不泄漏 raw enum、模板合同和无意义空分类。
```

P0-H7 编译完成后必须重建 `PRIVATE-REGRESSION-REDACTED`，旧 HTML 和旧 final Markdown 作为 superseded revision 保留或归档，不允许手工分别修到“看起来一致”。

### 16.7 物理提交点、版本与失败语义

多文件交付无法依赖普通文件系统提供跨文件事务，因此产品合同把 `delivery-revision.json` 定义为最后写入的 commit marker：

```text
1. candidate 使用 revision_status=preparing。
2. compiler 校验 source binding、warning union、platform binding 和 exact placement，官方 input 写 revision_status=compiled。
3. renderer 先生成并校验 HTML、三个 final Markdown 和 content delivery record。
4. 所有输出存在且 hash 闭合后，最后写 deliverables/p0/delivery-revision.json，revision_status=current。
5. commit marker 缺失、任一输出 hash 不符或语义门禁失败时，session 不得 completed；旧 current revision 继续有效。
```

不兼容必填字段升级固定使用：

```text
typed_components_v0.3
final-delivery-template-v0.3
final-delivery-renderer-v0.3
p0-contract-bundle-v0.3
```

v0.2 只允许历史 replay / 审计和原 renderer 复现；需要进入 H7 工作台时必须生成新的 v0.3 revision，不原地改写旧 receipt 或历史 event。

平台封面文字规范化只允许：统一换行分隔符、移除分隔符两侧空白、统一全半角空格。不得删除或改写汉字、数字、否定词和标点语义；规范化后仍不一致即 blocked。

warning 唯一键使用 `warning_code + source_artifact_id`。同一键只能更新 resolution 状态；不同来源的同码 warning 不得互相覆盖。

## 17. 画幅、槽位与封面派生合同（R3-C91 至 C110）

### 17.1 触发问题与产品归因

一次私有真实运行暴露了一个典型 false success：Image 2 基础图是 1672×941 的 16:9 横图，封面合成器用居中 `cover` 裁切生成 1080×1440 和 1080×1080 文件；按几何计算，竖封面只保留源图约 42.2% 的横向范围，方形封面只保留约 56.3%，但机器只核对尺寸、文件、标题和 hash，便把三个平台都记为 `pass`。实际 HTML 中竖封面出现构图截断、主体关系丢失和非意图黑区，用户人工验收不通过。真实账号名和 session 标识只保存在私有生产区。

因此产品归因不是“横图不能用于竖屏”，而是：

```text
系统把视频画布、画中画素材、画中画槽位和封面画布混成了一个比例概念；
把文件尺寸正确误当成业务视觉正确；
把正文画中画像素复用误当成封面已经设计；
没有记录焦点、保护区、裁切窗口、损失和真实平台预览。
```

历史 `cover_quality_gate=pass` 只保留为旧合同的机器语义结果，不再代表该真实交付可发布。当前交付应标记为 `human_review_rejected_cover / superseded_pending_recompile`，不得通过改文案掩盖封面失败。

### 17.2 成熟项目对标结论

成熟工具和研究没有把“居中裁切成功”当成最终质量：

| 对标 | 可复用做法 | 本项目取舍 |
|---|---|---|
| [TikTok Creative Best Practices](https://ads.tiktok.com/help/article/creative-best-practices?lang=en&redirected=1) | 原生 9:16、至少 720P、内容位于 UI safe zone | 抖音主发布画布默认绑定 9:16 和版本化安全区；平台相近不等于规格完全相同 |
| [Adobe Premiere Auto Reframe](https://helpx.adobe.com/premiere/desktop/add-video-effects/commonly-used-effects/auto-reframe-overview.html) | 先选目标比例，再跟踪主体并允许人工微调 | 先定目标画布 / 槽位，再决定生成或适配；复杂构图必须进入人工预览 |
| [Apple Final Cut Pro Smart Conform](https://support.apple.com/en-ie/guide/final-cut-pro/ver26664d93f/mac) | 为方形 / 竖形版本重新构图 | 平台 rendition 是独立交付物，不把一个源文件直接冒充全平台成品 |
| [Cloudinary Automatic Gravity](https://cloudinary.com/documentation/image_automatic_gravity) | 内容感知焦点、可用自定义坐标覆盖，并可返回建议裁切 | 自动焦点只做建议；人物、文字、图表节点等保护区必须可显式覆盖 |
| [Sharp resize](https://sharp.pixelplumbing.com/api-resize/) | `cover / contain / fill` 与 position、attention、entropy 分开配置 | 适配方式和焦点是合同字段；禁止把默认 center 当成产品判断 |
| [Adobe Research 自动裁图](https://research.adobe.com/publication/automatic-image-cropping-using-visual-composition-boundary-simplicity-and-content-preservation-models/) | 同时考虑构图、边界简洁和内容保留，避免切断对象 | 门禁必须检查内容保留和构图损失，不只检查输出尺寸 |
| [Instructor presence 系统综述](https://pmc.ncbi.nlm.nih.gov/articles/PMC9786415/) | 出镜者会改变注意分配，效果依知识类型、复杂度和呈现方式而异 | 画中画不是固定图片形状；是否保留主播、主播占位大小和素材位置是注意力编排决定 |

对本项目的核心推论：`PIP` 应从“图片资产类型”升级为“主视频中两个视觉信息源如何同时存在的展示决策”。解释图可能全屏替换主播，也可能与主播并列；新闻截图可能是证据卡片；情绪图可能只做背景。素材比例必须由展示任务和槽位反推，而不是每轮由 agent 临时写进 prompt。

### 17.3 四种画布和六种展示方式

必须分开记录：

```yaml
video_canvas:          # 最终视频画布，如 douyin-feed 9:16
visual_asset_canvas:   # Image 2 或截图产出的源素材画布
placement_slot:        # 素材在视频画布中的归一化矩形与安全区
cover_canvas:          # 平台封面 / 主页网格 / 分享卡等独立展示面
```

`visual_presentation_mode` 至少支持：

| 枚举 | 业务含义 | 比例原则 |
|---|---|---|
| `full_frame_replace` | 解释图 / 情绪图临时全屏替换主播 | 优先生成目标视频原生比例，本账号抖音主画布通常为 9:16 |
| `speaker_plus_visual` | 主播与解释图同时存在的真正画中画 | 按显式槽位比例生成；横图只有槽位能完整承载时才成立 |
| `split_screen` | 上下或左右并列对比 | 两侧保护区、字幕区和分割方式必须提前定义 |
| `floating_card` | 浮层卡片、数据点、步骤图 | 可使用 1:1 / 4:3 等卡片比例，但不能遮挡主播脸、手势和字幕 |
| `source_evidence_card` | 新闻网页 / 数据 / 引语截图证据卡 | 先裁出可读证据区域，再放入竖屏证据版式；不得把整页缩成看不清的小图 |
| `background_plate` | 情绪 / 场景背景，主播或文字在前景 | 重要区域服务于前景留白，不要求所有源画面都完整保留 |

`placement_slot` 以 0 到 1 归一化坐标记录 `x / y / width / height / z_order`，并绑定 `speaker_region / caption_safe_area / platform_ui_safe_area / protected_regions`。本项目当前只生成剪辑执行卡，不自动合成视频，但这些字段决定图片应生成什么比例，也让后续剪辑有明确依据。

### 17.5 封面决策顺序

```text
先确定平台展示面和点击承诺
-> 选择独立封面视觉任务
-> 判断能否复用母资产的“创意概念”
-> 为目标画布选择 reuse / crop / pad / outpaint / independent generation
-> 记录焦点、保护区和裁切窗口
-> 生成真实平台缩略图预览
-> 人工 / AI 视觉验收
-> 通过后才进入 H7 可发布状态
```

复用优先级不是“省调用”，而是“同一视觉概念能否在目标展示面保持点击承诺”。Codex 内置 Image 2 不设成本或调用次数门禁；当独立竖封面更能保证质量时，应直接生成独立竖封面。

### 17.7 可编译性反审计与消歧（R3-C111 至 C124）

R3-C91 至 C110 的方向正确，但首次反审计发现以下内容仍可能被不同 producer / checker 作出不同解释：

```text
PIP 是否包含全屏替换；
9:16 是字符串、比例还是实际像素；
槽位坐标从哪里起算；
中心裁切何时算破坏性裁切；
“平台预览”是本地模拟图还是真实 App 截图；
AI / 人工视觉审核由谁执行、机器能否自己写 pass；
主平台通过、次平台待审核时整个交付算什么状态；
同一封面能否跨平台复用；
旧 v0.3 如何回放、当前 revision 如何恢复。
```

这些问题不改变用户已经确认的业务方向，按质量优先、抖音主发、次平台同步、无成本门禁和不自动发布的既有口径消歧如下：

| 编号 | 无二义性合同 |
|---|---|
| R3-C111 | 用 `visual_insert` 统称插入主视频的静态视觉；只有 `speaker_plus_visual` 才是狭义 PIP。`full_frame_replace` 不能再以 `picture_in_picture` 名义交付。v0.3 的 `pip_card` 仅历史兼容。 |

| R3-C114 | `platform_surface_profile` 必须包含 profile ID、version、platform、surface role、target canvas、safe areas、priority 和规格证据状态。账号快照绑定 profile 版本，不从平台名临时猜尺寸。 |

## 18. 全文视觉覆盖合同（R3-C125 至 C139）

### 18.1 触发问题

现行 `visual_need_analysis` 能严格检查一个已提出候选是否具有注意力、理解、证据、演示、情绪或记忆价值，也允许一张图覆盖多个 `beat_id`。但它没有一个来自全文的权威 beat 集合，无法证明每段当前口播都被视觉导演处理过。

因此会出现一种新的 false success：候选本身、图片、比例、槽位和最终 HTML 全部合格，但 agent 只提出少数候选，其他重要段落从未进入 generate / reject 判断。图片质量正确，全文覆盖仍然不完整。

产品修复不是提高最低图片数，而是把“有没有逐段做出视觉处置”与“最终有几张图”分开：

```text
coverage completeness：全文是否全部处理
visual value：某个 generate 是否值得做
derived visual asset count：当前内容视觉任务去重后自然得到 0 到 N
```

### 18.2 上游与事实源

R3 不再自行重新切全文，而是只消费 R6-B01 定义、已绑定当前结构的 `content_beat_map(mapping_phase=structure_bound)`。直供稿用于结构诊断的 `semantic_only` revision 不得进入视觉规划：

```yaml
visual_coverage_ledger:
  ledger_id:
  content_beat_map_id:
  draft_id:
  draft_revision:
  draft_digest:
  coverage_records: []
  accepted_visual_task_ids: []
  coverage_completeness_status: complete / incomplete / stale
  visual_delivery_readiness: ready / ready_with_warnings / waiting_assets / waiting_authorization / blocked
  unresolved_beat_ids: []
  derived_visual_asset_count:
  materialized_visual_asset_count:
  provider_generation_task_count:
  provider_generation_attempt_count:
  source_capture_task_count:
  source_capture_attempt_count:
  visual_insert_occurrence_count:
```

`coverage_records[]` 与当前 `content_beat_map.beats[]` 必须集合相等。一个 record 必须选择一个主处置：

```text
generate_visual
create_deterministic_visual
use_source_evidence
use_existing_asset
reuse_visual_task
talking_head_intentional
evidence_blocked
manual_visual_required
```

记录还必须包含 viewer problem、decision reason、covered task、claim / source binding、expected viewer change、information delta、production path 和 recovery action 中适用的字段。确有不同观众任务时允许 `supplemental_visual_task_ids[]`，但每项必须有不同 visual job、独立插入锚点并通过冗余 / 认知负荷检查。不同条件路径分开校验，不把下游图像尺寸或 provider 输出反列为上游输入。

### 18.3 0 张与多节点共用图

0 张继续是合法产品结果，但证明方式从一个全文理由升级为逐节点处置：

- ready 的 0 张要求每个 current beat 均为 `talking_head_intentional` 且没有补充任务；`evidence_blocked / manual_visual_required` 虽然分析完整，但不能伪装成“无需图片”。
- `evidence_blocked / manual_visual_required` 可使 `coverage_completeness_status=complete`，但前者在主张未降级时 delivery blocked，后者为 waiting_assets。
- 历史已验证的非证据素材进入当前内容时使用 `use_existing_asset`：为本稿新建 current visual task，重新核对 asset hash / provenance / rights 与当前语义、画布适配，计入需要数和已成素材数，但 provider / source capture task 均为 0。
- 历史来源截图仍使用 `use_source_evidence(capture_mode=reuse_verified_capture)`，必须重验 claim / source binding、capture hash、captured_at 与 freshness；它不增加本轮 capture task / attempt，也不得改写成普通 existing asset 绕过证据门禁。
- 本次内容的一项 current visual task 可以覆盖多个相邻节点；在非相邻位置复用时使用 `reuse_visual_task` 并新增独立 `visual_insert_occurrence`，不重复计算图片或 provider 任务。
- 相邻节点若需要不同证据、不同展示时机或不同认知任务，不得为了少出图强行合并。

图片数量仍不受时长、fixture、账号默认值、成本或 provider 次数限制。Codex 内置 Image 2 对所有 `generate_visual` accepted task 直接生成；`create_deterministic_visual` 不创建 provider task；来源证据继续由 R6 source capture 路径生产。

### 18.4 结构门禁与质量门禁

确定性 checker 可以判断：

```text
beat 集合相等
每个节点恰有一个处置
决策枚举与条件字段合法
accepted task / covered beat 双向引用闭合
draft / beat map / ledger digest 未过期
derived_visual_asset_count 等于排除 reuse occurrence、平台 rendition 和封面后的唯一 current 内容视觉任务数
materialized / provider task / provider attempt / source capture task / capture attempt / occurrence 各自按独立口径闭合
当前视觉与最终 HTML lineage 闭合
```

确定性 checker 不得自行判断：

```text
主播表达是否真的比图片更好
图片是否真正帮助理解或留人
视觉节奏是否舒服
相邻画面是否造成审美疲劳
情绪与账号声音是否准确
```

这些结论由 `codex_visual_review / human_visual_review` 查看实际稿件与栅格资产后写入 `script_visual_alignment_review`，并给出可读理由。结构 pass 不得覆盖视觉质量 fail。

### 18.5 文案 revision 与失效

`visual_coverage_ledger` 绑定当前 draft revision。修改后按受影响范围恢复：

| 修改 | 视觉影响 |
|---|---|
| 排版、空白或可证明语义等价的错字修订 | 重新计算显示 digest；允许复用 beat / visual IDs。 |
| 局部节点的意思、顺序、事实强度或情绪作用变化 | 相关 coverage、视觉文字、来源 binding、asset acceptance 和 alignment 变为 stale。 |
| 核心承诺、受众或立场变化 | 全部视觉覆盖与对齐审查变为 stale。 |

恢复必须从最小受影响节点继续；旧资产保留审计，不得覆盖，也不得静默进入新 revision。

### 18.6 R3-C125 至 C139 产品合同

| 编号 | 产品合同 |
|---|---|
| R3-C125 | `visual_coverage_ledger` 必须覆盖 current `structure_bound content_beat_map` 的全部节点；`semantic_only` 诊断图或少数自选候选不得进入视觉规划。 |
| R3-C126 | 每个节点必须选择一个明确主处置并给出理由；只有承担不同视觉职责、具有独立插入锚点且通过冗余 / 认知负荷检查时，才可增加补充视觉任务。 |
| R3-C127 | 每项 current 视觉任务，不论 Image 2、确定性、来源、既有或人工路径，均需证明注意力、理解、证据、演示、情绪或记忆价值；完整覆盖不等于每段都出图。 |
| R3-C128 | 内容需要的图片数使用 `derived_visual_asset_count`：去重后的当前内容视觉任务数，包含新生成、来源证据、历史非证据素材、确定性与人工待补任务，排除 current task 复用引用、插入 occurrence、平台 rendition 和封面；保持 0 到 N、不设上限。 |
| R3-C129 | 图片任务数、已成素材数、provider / capture 任务与尝试数、插入次数分开；历史非证据素材用于本稿会新建 current task，历史来源截图仍走 verified evidence capture 复用；current task 非相邻再次出现只新增 `visual_insert_occurrence`。 |
| R3-C130 | 0 张是合法结果，但必须由每个节点的有效非生成决策共同证明，不能只写笼统全文理由。 |

### 18.7 可编译性消歧

| 容易混淆的口径 | 当前唯一语义 |
|---|---|
| `coverage complete` 与 `delivery ready` | 前者只证明每个 beat 已有合法决策；后者还要求证据、生成或人工素材已经满足。 |
| 图片需要数与已得到图片数 | `derived_visual_asset_count` 是内容需要；`materialized_visual_asset_count` 是当前已有。 |
| 图片数与插入数 | 一项任务 / 母资产可有多个 `visual_insert_occurrence`；重复插入不增加图片数。 |
| 图片数与 Image 2 调用数 | `provider_generation_task_count` 统计唯一基础任务，`provider_generation_attempt_count` 统计实际尝试；截图、确定性叠字、人工素材不计 Image 2。 |
| 历史素材复用与 current task 复用 | 历史非证据素材使用 `use_existing_asset`，计入本稿需要数和已成素材数但不调用 provider；历史来源截图仍走 `use_source_evidence(reuse_verified_capture)`；current task 复用使用 `reuse_visual_task`，只增加覆盖引用 / occurrence。 |
| 0 张与阻断 | ready 的 0 张只能来自全部 beat 的 intentional talking head；缺证据或等人工素材不是 0 张完成。 |
| supplemental task | 必须是不同 visual job、独立插入锚点并通过冗余 / 认知负荷检查；否则拒绝。 |
| `unresolved_beat_ids` | 只记录缺决策、非法决策或引用断裂；诚实的 evidence / manual 状态不算“未分析”。 |
| 质量 pass | checker 只判结构；实际画面与叙事 pass 必须有 Codex / human review 与观察理由。 |

`derived_visual_count / zero_visual_reason` 在 v0.3 仅历史兼容。v0.4 使用分离数量字段；`zero_visual_reason` 只可由逐 beat 决策派生为人类摘要，不能作为独立输入绕过 coverage。
