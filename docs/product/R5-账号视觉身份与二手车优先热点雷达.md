>
> 范围：为多账号内容工作流补齐“账号级视觉身份”和“账号策略驱动的热点雷达”。本文件是产品真源，不代表已经写入 Skill、Schema、runtime 或自动采集器。

> 文档关系：本文件承载字段、状态、研究依据和 fixture 细则；先读 [R5 产品总览](./R5-产品总览.md) 了解业务范围。R5-H1 至 H6 已完成编译并通过私有身份迁移 / 启动回归；实际热点研究仍须按 [R5 产品确认清单](./R5-产品确认清单.md) 的账号边界执行。

## 1. 已确认的业务结论

| ID | 已确认决策 |
|---|---|
| R5-C01 | 视觉一致性必须跟随账号，而不是跟随某一篇内容；每个账号拥有独立、轻量的视觉身份。 |
| R5-C02 | 账号视觉身份不等于所有图同色、同版式或强塞 Logo；它提供稳定的“证据感、语气、层级、禁忌”语法，单篇视觉仍由内容需求决定。 |
| R5-C03 | 账号级热点雷达以二手车直接相关为第一优先级。 |
| R5-C04 | 当且仅当本轮不足 3 条“事实可核验且二手车直接相关”的候选，才启用新车外溢池。 |
| R5-C05 | 新车候选必须写明对二手车的关联逻辑；没有残值、置换、车源、库存/现金流、二手新能源检测/流通或政策传导等证明时，不得因“新车很热”入选。 |
| R5-C06 | 该账号可同时服务买车消费者、卖车用户和车商经营者；每条候选必须标明主受众，不能把三类人强行揉成一句泛观点。 |
| R5-C07 | 热点发现需要浅层持久化：保存可复查的信号、查询版本、去重和生命周期，不把一次搜索结果当长期记忆。 |
| R5-C08 | 热度、事实和传播必须分开表达；单一 `heat_score` 不是“置信度”。 |
| R5-C09 | 初期不建设自动登录、自动发布或大规模爬虫；先把可复查的账号策略、候选结构和人工门禁编译进工作流。 |
| R5-C10 | 关键词扩展是雷达引擎能力：账号提供方向与硬边界，引擎可自由提出并使用可审计的扩词；长期偏好由这些词实际辅助发现、并最终被选择的候选计数决定。 |

以下三项来自本轮成熟项目 / 论文调研的补强决策，均已与 R5 一并确认：

| ID | 建议确认的业务决策 |
|---|---|
| R5-D10 | 已确认：雷达采用“信号 → 事件 → 候选 → 选题”四层模型；新闻标题、热词或平台观察不能跳过事件归并直接成为选题。 |
| R5-D11 | 已确认：第一次观察只能称“新发现”；只有同一事件至少两次可比快照后，才允许标为升温、持续或降温。 |
| R5-D12 | 已废止：扩词不采用“人工批准后才可使用”的白名单模式。 |
| R5-D13 | 已确认：扩词自由探索；每个扩词记录其辅助发现、形成候选和最终被选择的次数，以选择反馈形成账号偏好。 |

## 2. 当前问题与调研结论

现有工作流已有“账号档案 + 来源池 + 候选卡 + 选题卡”主链，但没有账号级查询词库、二手车优先/新车降级规则、候选生命周期和账号视觉合同。因此它更像“按本次提示词找热点”，不是可重复审计的账号雷达。

以“帅车值新山东代理商纠纷”为反例：不输入品牌名、仅用“二手车 + 车商 + 追款/不发车/资金链/代理”等泛化风险检索词时，现有流程不能可靠发现该事件；只有已知实体或偶然检索命中才可能发现。已核验的媒体报道应表述为“山东代理商被报道发生履约/资金链纠纷，品牌方称其为独立经营主体”，不能把对集团的法律责任或“暴雷”定性写成事实。[蓝鲸财经转载报道](https://finance.eastmoney.com/a/202607073795783895.html) [汽车之家报道](https://chejiahao.autohome.com.cn/info/24801834)

成熟产品的共同做法是把检索策略做成可版本化资产，而非一个关键词：Boolean 查询可保存同义词、排除词和邻近关系；[Brandwatch 的说明](https://www.brandwatch.com/social-media-glossary/boolean-search/)与 [Sprinklr 的 query-based listening](https://www.sprinklr.com/help/articles/topic-creation-setup/create-a-querybased-listening-topic/646372a830f12540268faa6e)均支持这一思路。趋势词的“相关查询”适合扩词，但只是信号，不是事实证明。[Google Trends](https://support.google.com/trends/answer/4355000?hl=en) 与 [百度指数帮助](https://index.baidu.com/Helper/?tpl=helpandword=)可作为扩词和趋势观察的参考。学术上，事件应看相对历史基线的突发变化，而不是某个绝对热度；[LABurst](https://arxiv.org/abs/1508.00488) 与概念漂移研究[《Learning under Concept Drift》](https://arxiv.org/abs/1911.05494)支持词库和阈值需要持续更新。

本地“电商图片设计加电商视频带货”项目的可借鉴点是：以品牌上下文包、语义视觉 token 和场景模板三层组织资产；同一品牌在不同页面保持可识别性，但不强迫每张图复制同一个版式。R5 采用其方法，不复制其电商视觉成品。

对“涛哥热点收集器”交付包的源码审计进一步确认：其中“帅车”事件位于后端的人工 `WATCHED_EVENTS` 清单，不是由泛化检索自动发现；行业事件的 `heatScore` 主要按时效和风险词计算，而非真实传播或事实强度。这不是该工具的缺陷——人工关注清单本身有价值——但产品必须诚实区分“人工监看”与“雷达发现”，否则无法检验 R5-F01，也会把内部排序分误当置信度。

本轮补充调研进一步表明，成熟趋势产品会把相近检索词聚为同一趋势簇，并保留开始时间、快照和组成查询；[Google Trends](https://support.google.com/trends/answer/3076011?hl=en-uk)的 Trending Now 即按相关查询簇呈现。国内的[百度指数](https://index.baidu.com/Helper/?word=)把搜索、资讯、媒体、前后相关词和人群拆开；[巨量算数](https://www.oceanengine.com/insight/juliang-suanshu-yidongduan)把热词趋势、关联内容和人群洞察拆开。学术研究同样不把“某词突然高”直接等同于真实事件：事件突发应比较多类活动流的短窗持续性，[Zhao 等](https://aclanthology.org/D12-1134/)据此区分噪声；跨文档事件共指研究则说明，不同报道中的相同事件必须归并而非仅靠标题去重。[Cremisini 与 Finlayson](https://aclanthology.org/2020.nuse-1.1/)

## 3. 产品模型

### 3.1 账号视觉身份：稳定语法，不是固定海报

```text
账号视觉身份（持久）
  -> 栏目视觉模板（按“二手车风险 / 市场解释 / 车商经营”等复用）
    -> 单篇视觉计划（由文案视觉需求产生 0 到 N 张）
```

本账号的首版身份方向是“可信行业观察 + 现场感/证据感 + 风险与机会的清晰判断”。实际色彩、字形、角标或 Logo 资产尚未由用户确认；在确认前只编译语义规则，不虚构品牌资产。

账号视觉身份至少包含：

| 字段 | 作用 |
|---|---|
| `identity_id` / `version` / `status` | 可追溯的身份版本与启用状态。 |
| `visual_promise` | 观众应感到的稳定印象，例如“讲证据，不煽动恐慌”。 |
| `evidence_visual_grammar` | 数据、来源、人物观点、风险提示怎样被区分呈现。 |
| `cover_hierarchy` | 封面先呈现事件/判断/受众收益，避免只堆情绪词。 |
| `tone_and_color_direction` | 语义色彩与对比方向，不等同固定 RGB。 |
| `negative_aesthetics` | 禁止的夸张豪车炫耀、无来源恐吓、伪官方背书等。 |
| `column_template_refs` | 栏目可复用模板及其适用条件。 |
| `override_reason` | 单篇因证据、情绪或平台要求偏离身份时的理由。 |

单篇图片数量仍完全由视觉需求分析决定，范围为 `0..N`。每张图必须证明注意力、理解、证据、演示、情绪或记忆价值；账号身份只约束“怎样表达”，不把数量、画中画位置或某次真实回归图片数写成常量。

机器字段固定为 `visual_count_policy=content_derived_by_r3_0_to_n`；账号视觉身份和栏目模板不得覆盖它。

### 3.2 信号 → 事件 → 候选 → 选题

```text
信号（报道 / 热词 / 平台可见观察 / 人工监看）
  -> 事件（主体 + 动作 + 时间 + 地点 + 业务链路）
  -> 候选（账号相关性、受众与内容角度）
  -> 选题（人类采用并进入 Brief）
```

四层的职责不得混用：

| 层 | 回答的问题 | 不应承担的结论 |
|---|---|---|
| `signal` | 在哪、何时、以什么查询或来源观察到了什么。 | 事件已成立、值得讲。 |
| `event` | 多条信号是否指向同一主体/动作/时地/业务链路。 | 已适合本账号、事实已定论。 |
| `candidate` | 该事件是否二手车优先、服务谁、能讲什么。 | 已被用户选中、可直接写稿。 |
| `topic` | 人工选定后的观点、边界和内容 Brief。 | 替代上游证据。 |

事件归并的最低 `event_fingerprint` 为：规范化主体、核心动作、事件时间窗、地点、车型/业务链路；标题相似只能作为辅助，不得成为唯一去重条件。同一事件的新报道应追加 `evidence_refs` 与状态，不重建一个候选。

### 3.3 二手车优先热点雷达

```text
信号采集
  -> 事件归并 / 去重
  -> 事实、传播、风险三态核验
  -> 二手车直接性判定
  -> 候选池（不足 3 条才开启新车外溢）
  -> 主受众与内容角度
  -> 人工选题门
```

候选必须分别记录以下维度：

| 维度 | 口径 |
|---|---|
| `used_car_directness` | `direct`、`spillover_proven`、`general_auto`、`none`。只有前两者可进入本账号候选池。 |
| `spillover_proof` | 新车进入时必填，说明它影响二手车的哪条链路及证据来源。 |
| `source_confidence` | 来源级别、原始链接、发布时间、抓取/记录时间和交叉来源数量。 |
| `fact_status` | `pending`、`verified`、`contested`、`unavailable`；媒体报道不自动等于法律定论。 |
| `propagation_status` | `unknown`、`observed`、`verified`、`blocked_external`；平台核验受阻不能伪装成“没有传播”。 |
| `risk_level` | 对名誉、金融、消费权益、未成年或医疗等的风险，单独决定措辞和人工门禁。 |
| `audience_primary` | `buyer`、`seller`、`dealer` 之一；可附次受众。 |
| `account_fit_reason` | 与账号母题、受众收益、可讲角度的明确关系。 |

评分只用于排序，必须展示组成项和来源。不得把“热度 94”或“机会分 90”解释成事实置信度。高热、低事实的事件可进入“核验队列”，不能直接进入口播文案。

### 3.4 发现来源与人工监看必须可区分

每条信号增加以下可审计字段：

| 字段 | 口径 |
|---|---|
| `discovery_mode` | `radar_discovered`、`manual_watchlist`、`manual_input`、`external_referral` 之一。 |
| `discovery_query_ref` | 自动发现时必填：词库版本、查询/来源批次与命中时间。 |
| `watchlist_reason` | 人工监看时必填：谁/为何加入、何时复核；人工条目不得计入“自动发现率”。 |
| `ranking_components` | 排序分的每个组成项、权重和原始观测值；不可只留最终分。 |
| `evidence_refs` | 原始来源、交叉来源、平台可见传播观察各自的引用。 |

`manual_watchlist` 可用于保证团队已知的重大事件不会遗漏，但不能通过 R5-F01“无品牌词发现”验收，也不能被用于证明词库或雷达已覆盖该类事件。

### 3.5 快照、趋势与词库学习

热点雷达需要保存最小快照，而不是从本次页面的绝对分数推断“正在升温”。

| 字段 | 口径 |
|---|---|
| `snapshot_id` / `observed_at` / `source_ref` | 一次来源观察的可追溯时间点和来源。 |
| `observed_metrics` | 该来源实际可见的排名、搜索量、报道数、视频数或互动观察；不可把缺失值补成 0。 |
| `comparison_basis` | 与哪一次同源、同口径、同时间窗的快照比较。 |
| `trend_status` | `new_observation`、`rising`、`sustained`、`cooling`、`not_comparable`。 |
| `burst_reason` | 相对基线的变化、持续时长、跨来源一致性；不是一个不可解释的总分。 |

首个快照只能是 `new_observation`；缺少可比来源或时间窗时写 `not_comparable`。只有至少两次同口径快照，且差异与持续性可复查，才可写升温/持续/降温。

词库的持久化还要增加：

```text
lexicon-candidates.yaml     # 新词、来源事件、建议包含/排除原因与探索状态
term-selection-ledger.jsonl # 每个词的使用、辅助发现、候选、被选择与拒绝计数
query-effectiveness.jsonl   # 查询批次命中、有效候选、误报、漏报复盘
```

词库候选可以由关联词、事件实体、人工复盘或用户输入提出，并可直接以 `exploration` 状态参与受边界约束的查询；不需要逐词人工批准。每个词需要记录 `run_count`、`signal_assist_count`、`candidate_assist_count`、`selected_candidate_assist_count`、`rejected_candidate_assist_count` 与最近一次命中时间。一个候选可能由多个词共同发现，因此 `selected_candidate_assist_count` 是“辅助选择”计数，不宣称单个词独占因果。

系统据此将词标记为 `exploration`、`preferred`、`deprioritized` 或 `blocked`：被选择的有效候选越多，后续排序权重越高；长期只带来误报或无候选的词自动降权。只有 `topic_exclusions`、安全/合规禁区或用户明确封禁才进入 `blocked`，不需要人工逐词审核。`query_effectiveness` 用于观察查询精确度、人工监看占比、已知验收事件覆盖和被采用候选率；这些是偏好学习依据，不是内容价值的机械总分。

### 3.6 浅层持久化边界

真实账号的轻量记忆属于私有生产区，建议后续放在：

```text
accounts/{账号名}/hotspot-memory/
  account-topic-policy.yaml       # 受众、优先级、阈值、禁区
  query-lexicon.yaml              # 实体、同义词、事件模式、排除词、版本
  signal-ledger.jsonl             # 每次发现的最小可复查信号
  candidate-lifecycle.yaml        # discovered -> verified/rejected/used/expired
  evergreen-problem-bank.md       # 可长期讲的买卖车/车商问题
```

只保存必要的标题、链接、时间、来源、摘要、查询版本、判定和去重键；不保存账号密码、私信、完整评论库或未授权平台数据。该目录依旧是本地私有区，不进入公开 Git 或发布包。

### 3.7 来源与扩词策略

优先建立四类信号，而不是只看热榜：

1. 二手车行业事实源：协会统计、监管/法院/公告、主流财经与汽车垂媒；中国汽车流通协会是行业数据的重要基础源。[CADA](https://www.cada.cn/index.html)
2. 交易风险事件模式：订金/尾款、未交付、追款、资金链、代理/加盟、库存、金融、维保、事故/调表、纠纷等模式词，与地域、主体、车型/品类交叉。
3. 消费者与车商问题：价格、保值、置换、验车、车况、库存、现金流、合规等问题词。
4. 传播信号：平台热榜、趋势词、相关搜索和媒体转载。它们用于发现与排序，不能替代事实核验。

词库必须包含实体词、别名、业务链路词、风险模式词、地域词和排除词，并保存 `lexicon_version`。新出现的车型、平台黑话和规避词要通过人工复盘加入，防止静态词表失效。

### 3.8 账号传参与受控关键词扩展

雷达引擎是全工作流共用能力，账号只提供“该找什么、为何找、不能找什么”的结构化策略；不可把整篇账号介绍作为自然语言提示词后由引擎临场猜测。

```text
account_profile
  -> radar_policy_ref
     -> account_topic_policy + query_lexicon + exclusions
        -> radar_engine.expand_keywords()
           -> lexicon_candidates
              -> bounded exploration queries
                 -> term-selection-ledger
                    -> preferred / deprioritized query_lexicon (next run)
```

账号档案 / 字段词典需要新增或明确以下合同字段：

| 字段 | 谁维护 | 引擎用途 |
|---|---|---|
| `radar_policy_ref` | 账号档案 | 解析本账号可执行的雷达策略，不让 runtime 依赖散文档案。 |
| `audience_priority` | 账号档案 | 为买车、卖车、车商等候选标主受众和排序方向。 |
| `topic_priority_rules` | 账号档案 | 定义母题、可讲业务链路、优先级。 |
| `used_car_priority_mode` / `new_car_spillover_threshold` | 账号档案 | 强制二手车优先与“不足 3 条再外溢”的降级规则。 |
| `new_car_spillover_proof_types` / `topic_exclusions` | 账号档案 | 限制新车外溢的证明类型，过滤不应讲的话题。 |
| `query_lexicon_ref` / `hotspot_memory_ref` | 私有账号区 | 提供已批准词库和可复查历史，禁止跨账号复用。 |
| `visual_identity_ref` / `column_visual_template_refs` | 私有账号区 | 传给视觉规划器，不参与热点相关性评分。 |
| `account_identity_id` / `account_technical_slug` | 账号档案绑定层 | 账号唯一机器身份；策略、词库、视觉资产和会话快照必须与其一致，不能用中文名或目录名代替。 |

#### 3.8.1 账号启动检查与按需补问

选择账号后，任何热点、选题或内容 Skill 都先读取结构化账号档案并执行 `account_startup_check`。它不是“再做一次建档表单”，而是只补齐当前动作无法可靠决定的最少信息：输出 `missing_fields`、`blocking_fields`、`non_blocking_fields`、`question_set_id`、`account_snapshot_ref`、`account_snapshot_status` 与 `next_skill`。`account_ready` 才可进入下游；缺少雷达策略时为 `account_policy_incomplete`，需要用户回答时为 `account_needs_input`，明确禁区 / 合规阻断时为 `account_blocked`。

补问遵循三条业务规则：

1. 一轮最多三个自然语言问题，只问当前任务相关的缺口；例如热点发现可不问 Logo，视觉交付才问视觉身份。
2. 平台与目标时长、买车 / 卖车 / 车商的当前优先级、以及高风险题材处理口径是本账号当前待补的首轮问题；它们不是由旧文章或其他账号自动猜出的默认值。
3. 用户确认后写成版本化的 session 账号快照；本次运行的雷达、选题与内容生产都读取同一快照，切换账号必须新建或重读该账号快照。

高风险题材默认采用 `verify_mechanism_only`：先做来源交叉核验，说明交易、履约、库存、现金流或消费者权益的行业机制，不对具体主体直接给出结论。只有用户明确提供或确认可核验来源后，才可形成带名称的事实陈述；任何情况下仍不得把媒体传播当最终定论。

#### 3.8.2 跨账号技术身份绑定与自动校验

账号目录是合法的人类可读私有目录；它不等于机器身份。多账号运行时必须建立版本化 `account_identity_binding`：以 `account_identity_id` 和 ASCII `account_technical_slug` 为不变主键，同时记录 `account_display_name`、私有目录键、账号档案引用和绑定 digest。雷达策略、词库、热点记忆、视觉身份、栏目模板及 session 快照只能引用同一绑定。

启动检查必须在读取下游资产前验证：所选目录与账号档案的绑定一致；每个引用仍在该账号私有根内；引用对象的技术 slug 一致；当前快照的 identity / digest 未过期。任一失败输出 `account_identity_inconsistent`，阻断而不补问、不猜测、不回退到前账号。历史账号必须用显式迁移记录补齐绑定，不能仅因目录名或展示名相同就视为同一账号。

扩词的三层口径：

| 层 | 产物 | 规则 |
|---|---|---|
| 种子词 | `seed_terms` | 来自账号策略，例如二手车、车商、收车、置换、保值、验车、库存、资金链。 |
| 引擎候选 | `lexicon_candidates` | 从关联搜索、事件实体、共现业务词、地域词、车型词和风险模式词提出；可直接按账号边界探索，必须记录来源与组合规则。 |
| 偏好词库 | `query_lexicon` | 根据 `term-selection-ledger` 的辅助选择计数自动升权或降权；只有硬禁区或用户明确封禁进入排除词。 |

例如“代理”可由引擎作为候选词提出，但单独使用会引入大量非汽车误报；它应先与二手车 / 汽车经销 / 车商 / 交付等业务链路组合探索。若这类组合持续带来被选择的候选，则自动升为偏好规则；若主要带来误报，则自动降权。扩词质量以有效候选率、已知事件覆盖、误报、漏报和辅助选择计数衡量，不用一个总分替代这些证据。

## 4. 可借鉴外部雷达，但不照搬

同事的“涛哥热点收集器”值得借鉴四点：把全国热榜和行业事件分池；显示来源、原链接与时间；把事实状态、平台传播状态和风险拆开；传播趋势不足快照时不宣称“上涨”。它的“帅车”事件说明行业事件池能够补足全国热榜的盲区。

它不能直接充当本工作流的决策引擎：重点事件存在人工 `WATCHED_EVENTS` 清单，页面未向使用者揭示其人工监看属性；同时也未暴露查询版本、采集来源谱系、去重规则或 `heatScore` 的计算证据。前端行业机会分对榜单名次依赖很强，不能判断“新车热点是否真影响二手车”。R5 只吸收其信号分层与人工核验思想，仍以本账号策略和可复查证据决定是否选题。

| R5-F06 | 人工把已知重大事件加入关注清单。 | 标为 `manual_watchlist`，保留加入理由和复核时间；可进入核验队列，但不得计为自动发现或词库命中。 |
| R5-F07 | 三篇标题不同的报道指向同一主体、同一履约纠纷和同一时间窗。 | 归并为一个 `event_id`，追加三条证据，而不是生成三条候选。 |

| R5-F12 | 热点任务已具备雷达策略、词库与受众优先级，但尚未配置视觉身份。 | `non_blocking_fields` 标记视觉身份，雷达可以进入发现阶段。 |
| R5-F13 | 视觉交付任务缺视觉身份或栏目模板。 | 仅补视觉相关缺口，不复问已确认的雷达策略；未补齐时不进入视觉规划。 |
| R5-F14 | 候选涉及代理履约 / 资金链等高风险事件，账号未确认风险口径。 | 停在核验与机制拆解，不能直接形成点名结论或口播文案。 |
| R5-F15 | 从账号 A 切到账号 B。 | B 的运行不读取 A 的 `account_snapshot_ref`、词库、候选或偏好计数。 |

## 6. 编译边界与下一步

若用户认可进入 R5 编译，必须把上述合同同步到账号档案、字段词典、热点 Skill/CONTRACT、机器 Schema 或确定性校验、正反 fixture、专项 checker、任务路由与状态记录；不能只改一篇产品说明。

首个编译批次建议按以下顺序进行：

1. `R5-H1`：账号视觉身份与栏目模板的字段合同，不生产具体品牌资产。
2. `R5-H2`：二手车优先的账号策略、字段词典传参、受控扩词、正式词库和浅层记忆合同。
3. `R5-H3`：信号 / 事件 / 候选对象、事件归并、快照趋势、三态核验、降级阈值与验收 fixture。
4. `R5-H4`：自由扩词的选择反馈账本、偏好升降权和查询效果回测，以及真实账号的人工辅助回归；不自动登录平台、不自动发布。
5. `R5-H5`：账号启动检查、最小补问、版本化 session 账号快照和字段按需阻断；不重新设计 R5-H1 至 H4 已确认的账号 / 雷达对象合同。
6. `R5-H6`：账号技术身份绑定、跨账号引用 / 快照自动校验、显式迁移和错绑阻断；重新闭合该新合同的 Schema、runtime、Skill、fixture、checker 与真实私有回归。

在用户明确授权“进入 R5 编译”前，R5 只停留在产品定义和调研结论，不修改现有 Skill、代码或真实账号私有资料。

### R5-H1 编译记录

`R5-H1` 已编译账号视觉身份和栏目模板的最小闭环：账号档案 v0.2 引用、视觉身份 / 栏目模板、机器 Schema、脱敏正反 fixture、专项 checker，以及 `account-onboarding -> static-visual-director` 的 producer / consumer 合同。它不创建真实账号视觉资产，不改图片数量规则，也不实现热点雷达。H1 通过后，下一批才可处理 R5-H2 的账号雷达策略和词库合同。

H6C 真实视觉启动补充发现：H6 跨账号校验要求视觉身份和栏目模板文件携带 `account_identity_id` / `account_technical_slug`，而历史 H1 v0.1 Schema 不允许这些字段，导致“符合视觉 Schema”和“通过账号隔离”无法同时成立。当前新增 v0.2 account-bound Schema / template / 正反 fixture；新建或启用的视觉身份使用 v0.2，v0.1 只保留历史 replay。账号语义身份仍不规定 Logo、固定 RGB 或图片数量。

同一 session 从热点研究进入视觉交付时必须生成独立阶段快照：热点快照与视觉快照可以共享账号技术身份，但不能共享 `snapshot_id`。视觉快照使用阶段化 ID，并保留自己的 binding digest、视觉身份引用和栏目模板引用；不得覆盖或伪装成上游热点快照 revision。

### R5-H5 编译记录

`R5-H5` 已把账号启动检查编译为 `propagation-router -> account_startup_check -> session account snapshot -> task-specific consumer`：新增确定性 resolver、可执行入口、账号快照模板和双 Schema；热点、选题、内容、视觉四类任务按字段相关性输出 `account_ready / account_needs_input / account_policy_incomplete / account_blocked`。每轮问题最多 3 个；热点任务缺视觉身份只记录为非阻断项，视觉任务才阻断；高风险默认 `verify_mechanism_only`；账号切换输出隔离证据。真实私有回归补充修复“平台已填但时长缺失时误报平台也缺失”的字段分组缺陷，新增单字段负例。R5-H5 fixture 7/7、入口 self-test、脱敏写快照 smoke、R5-H1 至 H4 回归、字段 / 路由 / 启动门禁均通过；未读取真实账号、未联网、未登录平台、未发布。

后续 R5-C23 已确认跨账号技术身份绑定是必做合同，故 H5 当前实现只保留为历史基础能力，身份隔离维度标记为 `superseded_pending_recompile`；在 R5-H6 完成前，不能用 H5 的快照隔离证据宣称跨账号自动校验已经完成。

### R5-H6 编译记录

`R5-H6` 新增版本化 `account_identity_binding`，把中文展示名、私有目录键和 ASCII 技术 slug 明确拆分。v0.2 启动检查先验证账号档案绑定、策略 / 词库 / 视觉资产引用、私有根目录、每份资产 SHA256、技术身份标记和 session binding digest；通过后才执行 H5 的按需补问。任一问题返回 `account_identity_inconsistent` / `snapshot_identity_inconsistent`，不向用户补问、不跨账号回退。新增显式绑定生成器、v0.2 双 Schema、模板、8 个身份正反 fixture 和 entry 写快照 smoke；fixture、身份绑定器 self-test、v0.2 启动器 self-test、账号启动门禁、runtime smoke、Windows runtime helper、字段 Schema 与文档治理均通过。H6C 真实集成又补齐 snapshot 时间合同：启动请求先物化含时区的 `requested_at`，v0.2 snapshot 逐字写成 `snapshot_at`；缺失 / 无时区负例必须阻断，R7 不得以 epoch 假时间生成热点请求。随后完成一份私有账号的显式绑定迁移与真实 H6 启动回归：绑定摘要重复生成稳定，结果为 `account_ready + identity_verified + snapshot_ready`；未联网、采集、登录、出图或发布。
