## 1. 这轮要解决的真实问题

现有主链默认从“热点雷达 -> 选题卡 -> Brief”进入。它适合发现型创作，却不适合以下两类已经高频出现的工作：

| 用户真实意图 | 现有链路的问题 | R6 的产品回答 |
|---|---|---|
| 用户交来一篇自己写好的口播稿，希望直接做成可拍 / 可审的交付物 | 若硬造热点研究和选题卡，会丢失“这是用户原作、没有跑热点”的事实；若直接拼 HTML，又会跳过质量、封面、交付和状态证据。 | 建立 `direct_content_intake`，把用户原稿、创作意图、可改范围和事实风险显式登记；它替代本次的热点选题入口，但仍进入 R1 / R3 / 封面 / 最终交付主链。 |
| 文案中有新闻、数据、引语或主体事实，适合用网页截图做画中画 | 现有 R3 已有 `source_bound` 字段，却缺少检索、核验、截图、逐句绑定、叠字和版权 / 隐私判断的生产闭环。 | 建立“主张 -> 证据 -> 捕获 -> 画中画”的证据链。截图是网页在某一时刻的可追溯展示，不自动等于主张为真。 |

这不是“多加一种图片”。它是两条不同的业务入口：**已有观点如何被负责任地生产**，以及**可核验事实如何被诚实地可视化**。

## 2. 能力一：用户直供文案的业务流程

### 2.1 入口分类

用户说“这是我写的，按 Skill 跑”时，路由不再假设其需要找热点。先创建一个 `direct_content_intake`，并由用户的口语指令决定运行模式：

| 模式 | 适用情况 | 是否检索热点 | 是否允许改写 |
|---|---|---:|---|
| `direct_delivery` | 用户要把原稿做成可拍交付物 | 否 | 只做必要的合规、事实和口播修订；所有改动可追溯。 |
| `direct_polish` | 用户要优化表达、Hook、结构或平台包装 | 否，除非原稿包含待核验事实 | 可按明确目标改写。 |
| `direct_evidence_enrichment` | 用户希望给原稿补新闻 / 数据 / 引语证据画中画 | 只检索被选中的事实主张，不跑热点候选池 | 正文观点不因检索而自动改写；证据不足时降级或标注。 |
| `direct_to_radar` | 用户明确说“看看这篇能不能结合热点重做” | 是 | 重新进入 R5 发现 / 选题流程。 |

本次用户直供的车商行业文案属于 `direct_delivery`：用户明确要求不按热点逻辑运行。它的选题意图已经由用户给出，不能再要求从候选池中选题。

### 2.2 正式主链

```text
选择账号
  -> account_startup_check + session account snapshot
  -> direct_content_intake（原稿、目标、可改范围、风险主张清单）
  -> direct_content_card（本次内容对象；不是伪造的 hotspot topic_card）
  -> content_brief（继承用户意图与原稿边界）
  -> draft revision / preserve decision
  -> claim map（区分观点、可核验事实、引用、数据、不可核验判断）
  -> R3 visual need analysis
      -> 需要证据时走新闻证据画中画
      -> 不需要时走 Image 2 情境图 / 确定性信息卡 / 零图
  -> quality review
  -> platform package input -> package
  -> cover design -> cover review
  -> final delivery -> finalize / human publish decision
```

`direct_content_card` 的意义是把“用户已经决定要讲什么”变成可计算的内容来源，而不是丢掉 R1 所需的 `content_goal`、受众、风险边界、平台和血缘。

### 2.3 必须记录的业务事实

| 对象 / 字段 | 业务作用 |
|---|---|
| `content_origin=user_supplied_draft` | 明确不是雷达发现，也不能拿它计算雷达覆盖或词库效果。 |
| `original_draft_artifact_id` / `original_draft_digest` | 保护用户原稿；每次改写可追溯。 |
| `revision_policy=preserve_voice / polish_allowed / rewrite_allowed` | 防止 AI 擅自把用户的强观点改成另一篇稿。 |
| `direct_intent` | 记录是交付、润色、补证据还是转热点，决定后续路由。 |
| `claim_map` | 每一句关键陈述标为 `opinion`、`experience`、`factual_claim`、`quote`、`statistic` 或 `prediction`。 |
| `claim_evidence_status` | `not_required / supported / refuted / not_enough_info / contested / not_checked`；没有来源不能写成已核验。 |
| `topic_origin=direct_user_input` | 让 R1 consumer 知道跳过的是“热点候选选择”，不是跳过整个生产链。 |

用户观点和个人经验可保留为观点，但不应因为情绪强烈就被自动升级为事实。反过来，系统也不应把“没有找到足够来源”强行改成“事实错误”；它只能给出 `not_enough_info`、建议改为观点表达或放弃证据画中画。

## 3. 直供文案的门禁、恢复与完成定义

| 节点 | 自动推进 | 何时停下 | 恢复语义 |
|---|---|---|---|
| 账号启动 | 是 | 账号身份错绑、当前任务必需档案缺失 | 补当前任务所需字段后新建 / 更新快照。 |
| 内容入口 | 是 | 用户没说是否允许改写，且原稿存在会改变发布含义的风险修订 | 只问一个人话问题：“这条是尽量保留原话，还是允许我重写？” |
| 主张地图 | 是 | 高风险事实、主体指控、数字、引语无法判断其语义 | 先将该句标为待核验，不直接进入证据卡。 |
| 视觉选择 | 是 | 无；0 张、信息卡、Image 2 情境图、来源截图都可能是正确输出 | 每张都必须证明注意力、理解、证据、演示、情绪或记忆价值。 |
| 最终交付 | 是 | 只在最终发布强度、主体点名、高风险事实或重大改写影响原意时给人类审阅项 | `html_ready` 不等于已确认发布；用户后续可局部返工或记录人工发布。 |

完成条件不是“生成了 HTML”，而是：原稿 / 修订稿 / 主张地图 / 视觉决策 / 质检 / 平台包装 / 封面 / final delivery 均可追溯，并有 execution trace、checkpoint 和单调状态收口。

## 4. 能力二：新闻证据画中画

### 4.1 产品定义

新闻证据画中画是：**用公开网页在某个捕获时点的局部截图，配合来源、日期和创作者解读，帮助观众理解一个已被精确绑定的主张。**

它不是：

```text
“网上有图”就证明事实为真。
把媒体标题、转述或热度当裁决。
用 Image 2 生成“新闻截图”。
整页搬运、遮掉来源，或用“来自网络”代替可追溯来源。
```

来源页面、截图资产和创作者观点必须分层展示：

```text
来源事实：页面可见的、被精确选中的原文 / 数据 / 引语。
来源元数据：发布者、URL、发布时间（若可得）、捕获时间、截图 hash。
涛哥解读：基于该事实的行业解释、推论或观点；不得伪装成来源原话。
```

### 4.2 生产流程

```text
draft / direct_content_card 中的主张
  -> claim extraction（只提出候选，不判真）
  -> evidence retrieval plan（实体、动作、时间窗、需证明的最小句子）
  -> source evaluation（来源类型、独立性、时间、冲突、风险）
  -> claim verdict（supported / refuted / not_enough_info / contested / not_checked）
  -> source capture request
  -> browser capture + visible text selector + source metadata + screenshot hash
  -> evidence_claim_binding（某张截图只支持哪一句）
  -> evidence_screenshot_pip（确定性裁切 / 叠字；不调用 Image 2）
  -> R3 visual_text_quality_gate + claim / copyright / privacy review
  -> final delivery
```

截图生成是一个可能有外部副作用的步骤：先持久化 `capture_attempt`、结果和可复用输出，再裁切、叠字或嵌入 HTML；中断时先 reconcile 已有截图，不能盲目重新抓取或反复访问来源。

### 4.3 四层证据状态

| 层 | 回答的问题 | 不能替代什么 |
|---|---|---|
| `source_access_status` | 页面是否在当时可打开 / 可捕获 | 不能证明页面内容真实。 |
| `capture_integrity_status` | 截图、URL、捕获时间、所选片段和 hash 是否同源 | 不能证明来源可靠或主张正确。 |
| `claim_evidence_status` | 该来源是否支持、反驳、无法充分判断或存在冲突 | 不等于司法 / 行政结论。 |
| `publish_risk_status` | 能否以当前表达方式放入短视频 | 不替代用户的最终发布判断。 |

这对应事实核验研究中的“支持 / 反驳 / 信息不足 + 证据”结构，而不是一个黑箱置信度。FEVER 与中文 CFEVER 都把主张标签和证据关联作为独立对象；本项目借鉴这个分层，不把论文数据集当成新闻真实性引擎。[FEVER](https://aclanthology.org/N18-1074/) [CFEVER](https://ojs.aaai.org/index.php/AAAI/article/view/29825)

### 4.4 最小对象合同（产品层）

| 对象 | 最小字段 | consumer |
|---|---|---|
| `claim_card` | `claim_id`、原句范围、主张类型、主体 / 时间、风险、是否需要来源 | 检索计划、quality review。 |
| `source_record` | `source_id`、publisher、canonical_url、title、published_at、source_type、accessed_at | 来源评估、HTML 来源表。 |
| `source_capture_record` | `capture_id`、source_id、capture_at、viewport、selected_target、screenshot_path、sha256、capture_status、attempt_number、attempt_history、archive_ref（可选） | 证据截图 renderer、失败审计与恢复。 |
| `evidence_claim_binding` | `binding_id`、claim_id、source_id、capture_id、selector / quote、relation、status、rationale | visual plan、quality review。 |
| `evidence_screenshot_pip` | `pip_id`、binding_id、裁切范围、叠字内容、来源条、解读条、asset_path、asset_hash、render_status | final delivery。 |

`selector / quote` 不能只写“某新闻报道”。要至少能指出网页中的哪段、表格哪行或页面哪块区域支撑当前主张。W3C Web Annotation 的 body / target / motivation 模型和片段 selector 是适合借用的结构：注释必须连接“我说什么”与“网页的哪一部分”，并说明其用途。[W3C Web Annotation Data Model](https://www.w3.org/TR/annotation-model/)

### 4.5 截图的呈现合同

每张证据画中画固定包含但不喧宾夺主的四部分：

```text
1. 画面主体：来源页面的必要局部，不用整页当壁纸。
2. 来源条：发布者 | 页面标题 / 资料名 | 发布日期（若有） | 捕获日期。
3. 解读条：明确标为“涛哥解读 / 行业含义”，与来源原文视觉区分。
4. 追溯入口：交付页保存 URL、截图 hash 和绑定主张；视频内可用短来源标签，不必塞长 URL。
```

如果资料只是传播线索、标题党、二手转述、无法确定时间，或只能支持“有人这样说”而不能支持文案里的事实，则它可以作为 `context_signal`，不能成为 `evidence_support`。

### 4.6 合规与权利边界

“信息来自网络”不是充分的来源说明，也不是自动获得使用授权。产品默认采用最小必要裁切、保留发布者 / 作品名 / 链接和捕获信息、不移除原有署名、不对付费墙 / 登录内容自动截图、不抓取个人隐私和评论区头像。中国著作权法第二十四条对为评论、说明问题而适当引用已发表作品规定了条件，同时要求署名且不得不合理损害权利人利益；这不是本项目能够自动判定的“可用许可证”。[中国人大网：著作权法第二十四条](https://www.npc.gov.cn/c2/c30834/202011/t20201119_308796.html)

因此，`copyright_review_status` 和 `privacy_review_status` 不通过时，系统应当给出“只保留文字来源卡 / 不使用截图 / 请求用户提供授权材料”的降级，不以“加免责声明”绕过。

## 5. 不调用 Image 2 的边界

| 视觉任务 | 正确生产方式 |
|---|---|
| 真实新闻网页、数据表、公告、原话作为证据 | 浏览器捕获 + 确定性裁切 / 叠字，`image_production_path=source_capture`；**禁止 Image 2**。 |
| 用图解释库存、供需、现金流等机制 | Image 2 情境 / 示意图或确定性图表；标为解释，不标为来源证据。 |
| 情绪渲染、角色隐喻、注意力重置 | Image 2 可直接生成，但不得暗示真实现场、真实新闻或真实数据。 |
| 无新增信息 | 拒绝该图；不是每条新闻都需要截图。 |

来源截图和生成图可以在同一条内容里并存，但资产角色、来源绑定、图中文字和 `information_delta` 必须逐图独立判断。

## 6. 成熟做法带来的取舍

| 参考 | 可借鉴的原则 | 本项目的取舍 |
|---|---|---|
| C2PA Content Credentials | 将来源 / 编辑历史与内容绑定；但 provenance 不对内容“好坏 / 真伪”做价值判断。 | 先做本地 `source_capture_record + hash + binding`，不承诺对网页截图签发 C2PA 或宣称事实已认证。[C2PA Specifications](https://spec.c2pa.org/) |
| W3C Web Annotation | 明确 body、target、动机与可定位的资源片段。 | `evidence_claim_binding` 必须定位到主张和网页片段，而不是只附一个 URL。[W3C Web Annotation Data Model](https://www.w3.org/TR/annotation-model/) |
| FEVER / CFEVER | 主张分类与证据检索 / 标注分离，允许“信息不足”。 | `not_enough_info` 是正常可交付结果：降级为观点或不做证据图，不硬凑来源。[FEVER](https://aclanthology.org/N18-1074/) [CFEVER](https://ikmlab.github.io/CFEVER/) |
| Browsertrix / Webrecorder | 真实网页归档有 capture、QA 和可分享归档的独立工作流。 | 首期只保存截图、可见片段、URL、时间与 hash；WARC / replay archive 作为高风险或长期引用的后续可选能力，不能偷偷变成大采集器。[Browsertrix Docs](https://docs.browsertrix.com/) |

## 7. 与现有工作流的对齐与差距

| 现有能力 | 可复用 | R6 缺口 |
|---|---|---|
| R5 `manual_input` / source refs / 事实、传播、风险分层 | 用户手工提供的新闻线索可诚实标记；高风险仍先核验机制。 | R5 以“发现热点”为起点，不能表示用户自写文案的来源 / 改写边界。 |
| R1 Brief、draft、quality、platform package | 直供内容仍应进入这些环节。 | 需要 `direct_content_intake` / `direct_content_card` 作为合法上游，而非伪造 `topic_card`。 |
| R3 `evidence_requirement=source_bound`、`screenshot_asset`、逐图文字门禁 | 现有字段可迁移为 consumer 要求；已明确生成情境图不能冒充证据。 | 缺 `claim_card`、source capture producer、逐句 binding、确定性证据截图 renderer、正反 fixture。 |
| P0 event / lineage / reconcile-first | 截图是外部副作用，应该沿用 attempt / output / hash / reconcile 的可靠性语义。 | 需要为网页 capture 定义不登录、不自动采集、失败分类、重复调用和 archive 可选语义。 |
| 账号视觉身份 | 来源事实、涛哥解读、风险提示的视觉语法应随账号走。 | 必须扩展 `evidence_visual_grammar` 的实际 consumer，不只留描述字段。 |

结论：R6 是 R1 的第二个合法入口、R3 的一个新来源型资产路径、R5 的可选新闻来源 consumer；它不替代 R5 雷达，也不建设热点抓取 SaaS。

## 8. 已确认的业务决策

用户已于 2026-07-14 认可 R6-C01 至 C19；以下口径成为本轮编译真源：

1. 用户直供文案是否一律跳过热点选题，但仍必须经过完整 Brief、质检、平台、封面、最终交付和执行证据链？
2. 对“用户没有说明能否改写”的原稿，默认应当**保留声音，只做必要风险 / 事实提示**，还是默认允许润色？
3. 新闻证据画中画是否采用“只对被选中的 factual claim 检索”，而不是每条文案自动全网搜？
4. 证据状态是否采用 `supported / refuted / not_enough_info / contested / not_checked`，并明确截图只证明捕获和来源，不直接证明真实？
5. 是否确认：来源截图一律不调用 Image 2；来源事实与“涛哥解读”必须视觉分层；无法给出 publisher、URL、时间或片段定位时不能作为 `evidence_support`？
6. 首期是否只保存截图 + URL + 时间 + hash + 片段定位，WARC 网页归档只留作高风险 / 长期引用的后续能力？
7. 版权 / 隐私风险时是否确认默认降级为文字来源卡或不使用截图，而非用“内容来自网络”免责？

完整逐项口径见 [R6 产品确认清单](./R6-产品确认清单.md)。

## 9. 口播脚本与视觉协同编排

### R6-B01 跨入口写作质检与内容设计编排增强

```yaml
backlog_status: compiled_local_validated_v0.5
scope:
  - user_supplied_draft
  - hotspot_generated_draft
not_in_current_compile: false
blocks_r6_c01_c19: false
compiled_contracts:
  - R6-C20-C50
```

## 12. 新闻证据锚点与语义一致性返修（R6-C51 至 C60）

### 12.1 触发事实与产品定位

真实热点交付中，来源网页、证据包、截图和工程图层都指向 2026 年，但人工编写的资产摘要把年份写成 2025 年，candidate 与 HTML 随后忠实放大了错误。现有合同能证明“截图来自哪里”，却不能证明“口播讲的这一句在截图哪里、后工程文字是否仍与原始证据一致”。

因此，新闻 / 数据 / 公告 / 引语画中画必须增加 `evidence_anchor_annotation`。它是原始来源截图的确定性派生产物，不是新事实，也不是 Image 2 生成任务：

```text
spoken claim
  -> visible quote in source capture
  -> normalized anchor region(s)
  -> deterministic emphasis / explanatory overlay
  -> annotated evidence asset
```

原始捕获保持不可变和可哈希；下划线只是句子类锚点的一种样式，不是所有证据的统一模板。数字更适合框选、圈选或局部放大，表格更适合行列高亮。目标信息在当前截图中已经足够显眼时可以不加视觉强调，但必须记录明确理由，不能以“截图存在”代替锚定。

### 12.2 对象与完成语义

`evidence_anchor_annotation` 至少包含：

| 字段组 | 产品语义 |
|---|---|
| `claim_ref / beat_ref / spoken_claim_text` | 当前口播主张及插入位置。 |
| `capture_asset_ref / capture_sha256 / captured_at` | 不可变原始截图及其时点。 |
| `visible_quote / source_identity / source_published_at` | 观众在截图中实际能看到、可支撑该主张的来源文字。 |
| `anchor_regions[]` | 0–1 归一化坐标、页码 / viewport、目标类型和样式；每个区域都绑定具体可见文本。 |
| `annotation_reason / no_annotation_reason` | 为什么这样强调；若不强调，为什么原图已足够明确。 |
| `creator_interpretation` | 涛哥的行业解读，必须与来源事实分层。 |
| `overlay_spec_digest / annotated_asset_ref / annotated_asset_sha256` | 确定性后工程规则和派生资产证据。 |
| `semantic_parity_result` | 主体、时间、数字、单位、来源身份和口径是否与口播 / 可见引文一致。 |

一致性门禁不得只对两段自由文本做模糊相似度。每个关键事实要物化为 `semantic_fact_bindings[]`：`fact_id / fact_type(entity|date|number|unit|source_identity|quote) / normalized_claim_value / normalized_visible_value / normalized_overlay_value / normalized_summary_value / normalization_basis / comparison_result`。`visible_quote` 同时记录 `extraction_method=dom_text|ocr|visual_verified`、原文和 `original_capture_normalized` 坐标；`visual_verified` 必须记录 `verification_actor=codex_visual_review|human_review`、review ref 与实际观察，不允许无证据手填。

`comparison_result` 只允许 `match / mismatch / not_assessed`。规范化只能使用版本化 `semantic_normalization_registry` 中声明的实体别名、ISO 日期、十进制数字和单位规则；不得用语义相近、约等于或自由改写把不同数字判为相同。DOM 精确文本可以直接比较；OCR 识别出的关键实体、日期、数字和单位在没有视觉复核时固定为 `not_assessed`，不设置无依据的置信度阈值。

每个 required binding 都先计算单项结果，再按唯一优先级派生整体 `semantic_parity_result`：任一 `mismatch` 则整体 `mismatch`；否则任一 `not_assessed` 则整体 `not_assessed`；全部为 `match` 才整体 `match`。前两种状态都阻断 candidate，并分别路由到修正文 / 修资产，或重新捕获 / 视觉复核 / 缩小主张；不得交给最终人审兜底。

一张截图只支持其中一部分口播时，只能绑定被支持的部分；其余内容必须增加第二张证据卡、缩小口播主张，或诚实标记证据不足。禁止用一个局部截图对多项数字给出“全部已支持”的视觉暗示。

### 12.3 产品合同

| 编号 | 产品合同 |
|---|---|
| R6-C51 | 新闻、数据、公告和引语等 `source_bound_evidence` 必须使用真实来源截图与确定性后工程；Image 2 不得生成或补造证据底图。 |
| R6-C52 | 每项证据视觉必须把 current 口播主张、截图可见引文、来源身份和坐标区域逐项绑定；只有 URL 或整页截图不算闭合。 |
| R6-C53 | 强调样式由证据目标决定：句子可下划线 / 高亮，数字可框选 / 圈选 / 放大，表格可高亮行列；没有强调时必须证明目标在当前画面中已足够显眼。 |
| R6-C54 | 截图只覆盖部分主张时，必须拆卡、补截图或缩小表达；不得把局部证据投影为整句 / 整段已证实。 |
| R6-C55 | 来源事实与“涛哥解读”使用不同图层和标签；创作者解读不能被排成来源原话。 |
| R6-C56 | 原始 capture 不可变；裁切、遮罩、下划线、框选、放大和说明文字均产生带父引用与 overlay digest 的派生资产。 |
| R6-C57 | 口播主张、visible quote、overlay text、alt text、visual summary 和最终 HTML 必须通过 typed `semantic_fact_bindings[]` 对主体、时间、数字、单位、来源身份执行机器可判的一致性检查；结果只允许 `match / mismatch / not_assessed`，不得只靠自由文本相似度猜测。 |
| R6-C58 | 整体结果按 `mismatch > not_assessed > match` 唯一派生；只有全部 required binding 为 `match` 才能继续。关键不一致或无法判断都属于 blocker，不能以 warning、OCR 分数或人工最终会看为由继续生成 candidate。 |
| R6-C59 | 浏览器捕获和后工程都先落 `attempt / outcome / output ref` 再派生下游；中断恢复先 reconcile 已有 capture / annotation，禁止盲目重抓。 |
| R6-C60 | 最终 HTML 必须能看出“口播说什么、截图哪里支持、来源事实是什么、涛哥如何解读、当前性如何”；无法闭合时不得显示证据完整。 |
