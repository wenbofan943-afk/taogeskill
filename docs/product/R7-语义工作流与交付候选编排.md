## 1. 本轮实测证明了什么

该私有回归最终产出了可用的 `final-delivery.html`，v0.5 compiler / renderer、lineage、receipt 和响应式页面均通过；但过程证据同时说明：

```yaml
result: pass_with_warnings
skill_autonomous_completion_count: 0
deterministic_tool_completion_count: 15
agent_assist_level_observed: high
image_provider_invocation_count: 0
network_used: false
publishing_invoked: false
```

业务链里以下工作仍由 Codex 临场编排：

- 建 Brief、baseline draft、16 个内容节点、五阶段结构、口播审查与继续决策。
- 逐节点视觉覆盖、两项视觉任务、供需机制图、复用资产和三平台包装。
- 把多个 Skill 的产物手工编入 `final-delivery-render-candidate.json`。
- candidate 首次把一份通用复用说明误作三张封面的逐 rendition review，编译失败。
- candidate 第二次使用未注册动作 `regenerate_visual`，再次编译失败。
- 原始 Chrome CLI 截图受到 device scale 与异步落盘影响，产生过假裁切 / 假缺文件信号；改用 Playwright 固定 CSS viewport 并直接测量布局后才得到可信结论。
- `direct-content-intake/SKILL.md` 仍保留“R6-B01 未编译”的历史语句，说明执行合同与状态真源之间存在文档漂移。

因此本轮不是要重做文案、视觉或 HTML 产品，而是补上它们之间缺失的产品层：

```text
版本化业务蓝图
-> 当前步骤任务单
-> Skill 按任务单产出语义结果
-> 确定性校验与 revision 提交
-> event / projection 推进
-> candidate 自动编译
-> renderer
-> 真实视口验收
```

## 2. 递归根因

### 2.1 表层问题与父因

| 实测现象 | 直接原因 | 产品父因 |
|---|---|---|
| Skill 合同很多，但自主完成数为 0 | Codex 仍决定步骤、输入文件和产物拼法 | 没有一份可执行的跨 Skill 业务蓝图和统一 coordinator |
| candidate 需要手工组装 | renderer 只接收完整 typed candidate，上游没有 candidate compiler | “内容判断”和“交付数据编译”职责混在 Agent 手里 |
| 通用 review 被当作逐封面 review | 复用摘要、资产 review、rendition review 没有在 candidate 前分层校验 | 资产 binding 门禁太晚，只在最终 runtime 才暴露 |
| `regenerate_visual` 被拒绝 | Agent 根据自然语言猜 enum | action registry 没有成为提交前的选择源 |
| 修 candidate 后还要手工同步 digest / event | candidate、输入摘要与运行证据分开维护 | 编译操作没有独占派生值和证据更新 |
| Chrome CLI 给出假信号 | 把截图文件 / 退出状态当布局事实 | 浏览器调用成功、页面加载成功、布局通过没有分层 |
| Skill 仍写“未编译” | 状态散落在 prose | 当前合同状态没有机器登记和文档漂移门禁 |

### 2.2 不能采用的修法

以下做法只会把同类问题延后：

- 再写一份更长的“请 Codex 按顺序做”说明，但不生成机器任务单。
- 给最终 runtime 增加更多晚期报错，却不在 producer / candidate 提交前预检。
- 在 candidate 模板里写死本次的 16 个 beat、2 张 PIP 或 3 张封面。
- 允许 Codex 遇到错误后直接手改已提交 candidate、event 或 current pointer。
- 把“Codex 做了语义判断”本身算作扶跑；真正要消除的是 Codex 临场发明流程、字段、枚举和恢复方式。
- 因本机有 Playwright 就把它写成所有公开环境的无条件依赖，或在缺浏览器时仍宣称响应式通过。

## 3. 成熟项目取舍

R7 延续 P0 已采用的 Temporal / Prefect / Dagster 原则，并补充 LLM 工作流和浏览器验收的当前官方做法：

| 成熟做法 | 本项目吸收 | 本项目不照搬 |
|---|---|---|
| Temporal 将持久执行、Workflow 与外部 Activity 分开 | plan / event / projection 仍是恢复真源；外部副作用单独记 attempt / outcome | 不部署 Temporal Server、数据库或 worker 集群 |
| LangGraph 将工作拆成单一职责节点，checkpoint 保存状态，interrupt 作为一等等待；恢复时节点可能重跑，副作用必须幂等 | 每个语义 producer 都有稳定 task envelope；等待人类 / 外部输入是显式状态；提交与外部动作有幂等键 | 不引入 LangGraph 运行依赖，不把聊天历史当 checkpoint |
| Dagster 区分资产定义、materialization 与 asset check | 文件存在、已物化、逐资产 review、交付资格继续分开 | 不建设资产平台或 daemon |
| Playwright 可固定 viewport、在页面上下文测量 DOM，并控制截图使用 CSS pixel / device pixel | 响应式验收直接记录 clientWidth、scrollWidth、越界元素和图片加载；截图只是辅助证据 | 不再用原始 Chrome CLI 文件是否出现推断页面布局通过 |

参考：

- [Temporal Documentation](https://docs.temporal.io/)
- [LangGraph overview](https://docs.langchain.com/oss/python/langgraph/overview)
- [LangGraph persistence](https://docs.langchain.com/oss/python/langgraph/persistence)
- [LangGraph interrupts](https://docs.langchain.com/oss/python/langgraph/interrupts)
- [Dagster assets](https://docs.dagster.io/guides/build/assets)
- [Playwright Page API](https://playwright.dev/docs/api/class-page)
- [Playwright emulation options](https://playwright.dev/docs/test-use-options)

产品取舍固定为：不引入重型框架；在现有文件、P0 event / projection、Skill 和确定性工具之上，实现轻量 coordinator、typed submission、candidate compiler 与 viewport acceptance harness。

## 4. R7 产品定位与边界

### 4.1 一句话定位

R7 是“单篇内容的语义工作流总控与交付编译层”：Codex 负责合同内的语义判断，确定性工具负责选输入、校验、提交、推进、组 candidate 和验收；任何一方都不越权替代另一方。

### 4.2 首期范围

```yaml
in_scope:
  - 单账号、单 session、单篇内容
  - direct_delivery 直供稿完整链
  - hotspot_to_delivery 热点稿同一收敛链
  - versioned workflow blueprint
  - semantic task envelope
  - typed semantic submission
  - deterministic artifact commit
  - delivery candidate compiler
  - action registry
  - per-asset / per-rendition review binding preflight
  - artifact-level execution contribution
  - deterministic viewport acceptance report
out_of_scope:
  - 多篇自动并行与 R2 fan-out 执行器
  - 自动发布、登录、评论、私信或后台数据采集
  - 外部 LLM API、模型路由或 Codex 前端模型自动切换
  - 数据库、队列、常驻服务、Temporal / LangGraph 部署
  - 用 runner 自动替用户改变立场、事实强度或高风险主张
```

### 4.3 “自主”的业务含义

R7 不把“没有 AI 参与”定义为自主。内容业务本来就需要 AI 做语义判断。

R7 所说的自主完成是：

```text
用户给出业务输入和必要决策后，
Codex 只按已注册 Skill / task envelope 做语义工作，
不临场发明步骤、字段、枚举、目录、恢复规则或 candidate 拼法，
确定性 coordinator 能验证并推进到最终交付或一个诚实等待点。
```

人类正常确认、Codex 合同内判断、Image 2 真实生成和浏览器真实验收都可以是完整链的一部分；只有“未登记的临场补流程 / 补规则 / 手改机器对象”才破坏自主完成结论。

## 6. 统一运行链

### 6.1 Blueprint 编译

首期登记两条蓝图：

```text
direct_delivery_single_v0.1
hotspot_to_delivery_single_v0.1
```

蓝图每个 node 固定：

```yaml
node_id:
skill_ref:
step_kind: deterministic_tool / semantic_skill / human_gate / external_side_effect
input_selectors: []
required_contract_versions: []
output_artifact_type:
output_schema_ref:
allowed_result_statuses: []
allowed_action_registry_ref:
success_route:
warning_route:
failure_route:
stale_scope:
retry_policy:
```

`semantic_skill` 是 P0 旧 `agent_required` 的细化产品身份；event 层仍可兼容记录为 agent-required activity，但必须带 `skill_ref / task_contract_version / task_envelope_id`。

### 6.2 单步执行循环

```text
1. coordinator 从 projection 找到唯一 next node。
2. 按 input_selectors 解析 current artifacts，并验证 ID / digest / status。
3. 生成 immutable semantic_task_envelope。
4. Codex 读取 envelope 指定的 Skill 和最小输入，产出 typed semantic submission。
5. deterministic submitter 先做 schema、enum、cross-ref、digest 和状态预检。
6. 通过后写 immutable revision、人类投影、lineage，最后替换 current pointer。
7. event writer 记录成功，projection 重建，coordinator 进入下一 node。
8. 人类门禁或外部副作用则生成 interrupt / request，停在可恢复状态。
```

coordinator 每次只能给一个 `next_node_id`。同一 projection 下出现两个未声明并行的 next node，属于 blueprint defect；不由 Codex任选一个继续。

### 6.3 语义任务单

`semantic_task_envelope` 最小字段：

```text
task_envelope_id
session_id
plan_id
node_id
skill_ref
task_contract_version
input_artifact_bindings[]：artifact_id / type / path / sha256 / status
business_objective
decision_boundaries[]
required_output_schema_ref
allowed_statuses[]
allowed_actions[]
output_commit_policy
idempotency_key
resume_context
```

`business_objective` 和 `decision_boundaries` 用业务语言告诉 Codex这一步要判断什么、什么不能擅自改变；路径、枚举和 schema 不让 Codex 猜。

### 6.4 Typed submission 与提交

Codex 产出 `semantic_artifact_submission`，包含业务对象 payload 和来源引用，但不直接写 current pointer、event、projection 或最终 candidate。

确定性 submitter 独占：

- artifact ID / revision 路径规范化。
- Schema 和条件必填校验。
- allowed status / action enum 解析。
- current input digest 与 cross-reference 校验。
- immutable revision、可读投影和 lineage 写入。
- current pointer 最后提交。
- event、projection 和 resume 更新。

任何提交失败都保留 submission 与 compile report；不得直接编辑旧 revision 后重试。

## 7. 新增产品对象

### 7.1 `workflow_blueprint`

版本化定义一条内容路径的节点、依赖、状态和失败路由。它是 tracked 产品配置，不保存真实账号和正文。

### 7.2 `contract_status_registry`

机器登记每个合同的 `contract_id / active_version / lifecycle_status / superseded_by / compiled_layers / effective_from`。Skill、blueprint、状态页和漂移 checker 读取同一 registry；普通 prose 可以解释状态，但不能独立宣称“已编译 / 未编译”。

### 7.3 `semantic_task_envelope`

某一 session 当前一步的不可变任务单。它只引用已经 materialized 的输入，不能引用“预计稍后会有”的路径。

### 7.4 `semantic_artifact_submission`

Codex 按 envelope 产生的待提交业务对象。submission 通过不等于 current；只有 deterministic submitter 成功后才 materialized / current。

### 7.5 `delivery_candidate_compile_report`

记录 candidate compiler 读取了哪些 current pointers、资产、review、action registry 和 warning，输出 candidate 的 ID / hash，以及每张卡的 source bindings。它是 candidate 可追溯性的直接证据。

### 7.6 `viewport_acceptance_report`

记录浏览器、viewport、CSS pixel、页面加载、图片加载、client / scroll 宽度、越界元素、截图与结论；不能只保存一张截图后写 pass。

### 7.7 派生执行贡献

不再新建第二本手工台账。`artifact_execution_contribution` 从 task envelope、submission、commit event 和 check records 派生：

```text
artifact_id
skill_ref
execution_source
contract_coverage: complete / partial / missing
semantic_executor: codex / human / deterministic_tool / external_provider
ad_hoc_intervention_count
manual_patch_detected
producer_event_id
check_ids[]
```

## 8. 最终 Candidate 编译

### 8.1 Candidate 不再由 Agent 手工拼

R7 计划中的 v0.6 变更：

```text
v0.5：上游 Agent 生成 final-delivery-render-candidate，runtime 晚期校验。
v0.6：delivery candidate compiler 从 current pointers / assets / reviews / registries 确定性组装 candidate。
```

Codex 继续负责上游语义对象和真实视觉 review；但卡片字段映射、排序、ID、warning 合并、action enum、digest 和 source bindings 由 compiler 独占。

### 8.2 六级预检

| 级别 | 检查 | 失败后回哪 |
|---|---|---|
| P1 task | envelope 的 Skill、输入、schema、allowed enums 是否存在 | blueprint / registry |
| P2 producer | submission 字段、状态、current digest 是否有效 | 对应 Skill producer |
| P3 cross-object | structure / beat / review / coverage / alignment 引用是否闭合 | 最小 stale producer |
| P4 asset | 文件、hash、materialization、quality、逐 asset / rendition review 是否闭合 | 对应 asset / cover producer |
| P5 candidate | 卡片、平台、warning、action、source map 是否可编译 | candidate compiler / registry |
| P6 render | render input、template、HTML、同步视图、receipt、revision marker | renderer / template |

错误必须尽可能在 P1-P4 暴露，不能把所有错误推到 P5/P6。

### 8.3 封面 Review 规则

- `carried_forward_asset_summary` 只说明复用了哪些资产，不能替代质量 review。
- 每个 materialized `cover_rendition_id + rendition_revision + output_sha256 + preview_sha256` 必须绑定唯一 current visual review。
- review 必须证明实际观察了对应 raster；通用说明、另一平台 review、旧 hash review 都不满足。
- compiler 在生成 platform delivery unit 前检查 review；缺失时返回 `asset_review_binding_error`，不写半成品 candidate。

### 8.4 Action Registry

- action card 只能从版本化 registry 选择 `action_code`，人类文案单独使用 `label / user_message`。
- Skill 与 compiler 都读取同一 registry；不得各自维护 enum 副本。
- 未注册自然语言先返回可用 action 列表，不自动猜最接近值。
- action registry 版本进入 task envelope 和 candidate compile digest。

### 8.5 Digest 与事件

candidate compiler 一次操作内派生：

```text
candidate payload
candidate sha256
source binding digest
render input compile request
compile report
success / failure event payload
```

Agent 不再手工改 candidate 后再追改 input digest、event 或 lineage。任何业务输入变化先形成新 upstream revision，再重新编译 candidate。

## 9. HTML 真实视口验收

### 9.1 三层事实

```text
browser_invocation_status：浏览器是否成功启动并完成页面加载
layout_measurement_status：是否取得可信 DOM / CSS viewport 测量
visual_acceptance_status：页面是否通过结构与实际视觉验收
```

三者不得合并成一个 `screenshot_status=pass`。

### 9.2 标准 profile

当前最低业务 profile：

```yaml
desktop:
  viewport_css_px: 1440x1000
mobile:
  viewport_css_px: 390x844
required_measurements:
  - documentElement.clientWidth
  - documentElement.scrollWidth
  - body.scrollWidth
  - overflow_offender_count
  - failed_image_count
  - unloaded_font_or_timeout_status
screenshot_scale: css
```

viewport 是 CSS 像素，不从输出 PNG 像素倒推。页面必须等待 DOM、图片和字体达到规定状态后测量；异步 screenshot 文件尚未落盘属于 `browser_invocation_error / evidence_pending`，不是 workflow fail。

### 9.3 通过与缺能力

- 每次新 final HTML 在浏览器能力可用时都执行 desktop + mobile；测量结果和截图 hash 绑定当前 HTML hash。
- `scrollWidth > clientWidth` 或越界元素存在时为 `visual_acceptance_fail`，不能靠截图肉眼忽略。
- 截图用于人工查看，不替代 DOM 测量；DOM 无溢出也不替代对实际封面、PIP、文字层级的视觉观察。
- 本地内容运行缺 Playwright / 可用浏览器时记录 `viewport_acceptance=not_tested`，交付最多 `ready_with_warnings`。
- template / renderer 变更和 public release 缺真实 viewport gate 时直接阻断，不允许沿用旧绿灯。

## 10. 执行透明度与自主完成计量

### 10.1 四类计数

```text
semantic_skill_step_completed_count
deterministic_tool_step_completed_count
human_gate_completed_count
external_side_effect_step_completed_count
```

### 10.2 单步来源

| 情况 | execution_source | 是否计入 Skill 合同完成 |
|---|---|---|
| Codex 按完整 envelope 与 Skill 产出，submission 一次或按声明 revision 路由通过 | `skill_defined` | 是 |
| Skill 方向存在，但 Codex 仍需猜输入、enum、路径或状态 | `skill_inferred` | 部分，产生 warning |
| Codex 自己选择未登记步骤、手拼 candidate、手改 event / pointer | `agent_orchestrated` | 否 |
| 运行中新增临时规则才继续 | `agent_created_rule` | 否，阻断自主结论 |
| 工具按 schema 独立完成 | `deterministic_tool` | 计入工具，不冒充语义 Skill |

### 10.3 End-to-end 自主完成

新增 `workflow_autonomous_completion_count`；一次 session 计 1 必须同时满足：

```text
blueprint / plan / event / projection 闭合。
所有 semantic node 均为 skill_defined，无 agent_orchestrated / agent_created_rule。
所有 current artifact 通过 typed submission + deterministic commit 产生。
candidate 由 compiler 产生，无手工 patch。
人类门禁均在允许节点；正常人类决定不扣分。
外部副作用有授权、attempt / outcome 和 reconcile 证据；未调用时状态诚实。
render / lineage / receipt / delivery revision 闭合。
viewport gate 为 pass，或在内容运行缺能力时诚实 ready_with_warnings；后者不能用于 release autonomy 证明。
```

现有 `skill_autonomous_completion_count` 保留历史读取；v0.6 后迁移为上述分步计数与 `workflow_autonomous_completion_count`，不得把 15 个 deterministic tool 步骤相加成 15 个 Skill 自主完成。

## 11. 失败、恢复与禁止手改

| failure_category | 含义 | 恢复动作 |
|---|---|---|
| `blueprint_contract_error` | node、依赖、route 或版本缺失 | 修 blueprint 后新 plan version；不执行业务 step |
| `task_envelope_error` | 输入选择、schema 或 allowed enum 无法编译 | 回 coordinator / registry |
| `semantic_submission_error` | Codex 结果不满足当前任务合同 | 保留 submission，按同 node 新 submission revision |
| `cross_artifact_binding_error` | ID、digest、current ref、stage / beat / task 引用不闭合 | 只回最小 stale producer |
| `asset_review_binding_error` | 资产物化但缺 hash-bound review | 回对应 asset / rendition review |
| `candidate_integration_error` | current 对象合法但无法映射卡片 | 修 candidate mapping，不改业务对象 |
| `enum_registry_error` | action / status 未登记或版本漂移 | 修 registry / producer；禁止自然语言猜值 |
| `browser_invocation_error` | 浏览器、页面加载、证据写入失败 | 可重试有上限；不判业务 HTML 失败 |
| `visual_acceptance_fail` | 真实测量或观察发现溢出、错图、不可读 | 回 template / renderer / asset producer |
| `documentation_drift` | current 状态与 Skill / 文档声明冲突 | 阻断相关成熟度宣称，修状态引用 |

恢复规则：

- pending submission、已提交 revision、current pointer、event 和 projection 是不同阶段，不能用文件存在代替状态。
- 恢复先读 projection 和最后 event；存在 pending submission 时先验证 / 提交或明确废弃，不再重复生成语义结果。
- 非幂等外部动作继续遵守 reconcile-first；R7 不扩大 provider / network / publish 权限。
- 禁止直接编辑 current revision、candidate、event、projection 或 delivery revision marker 来“修到通过”。
- 修订必须从 owning producer 或 compiler 新建 revision，并让受影响下游按 digest 自动 stale。

### 15.2 已确认的业务取舍

本产品推荐以下三项，不要求用户理解技术实现：

1. **自主完成口径**：允许 Codex 做合同内语义判断；只要它不临场发明流程和手改机器产物，就算 Skill 按合同自主跑完。
2. **浏览器缺失口径**：普通内容交付可 `ready_with_warnings`；改模板、改 renderer 或发版时必须有真实桌面 / 移动视口证据。
3. **编译顺序**：先把直供稿全链稳定，再接热点稿入口；两种入口各自保持正确的稿前顺序，在 current draft、current structure plan、current structure-bound beat map 三件套闭合后共用下游，不做两套 runner。

### 15.5 H3 状态语义与 producer adapter 反查

产品收口为：

```text

  direct_content_intake
  -> content_brief
  -> direct_baseline_draft(materialize_user_baseline)
  -> semantic_beat_map(mapping_phase=semantic_only, revision=1)
  -> direct_structure_plan(diagnose_existing_draft, only materialized refs)
  -> content_beat_map(mapping_phase=structure_bound, revision=2)

```

热点入口不套用该前置顺序，仍为 `structure_plan(design_before_draft) -> draft(generate_from_structure) -> content_beat_map(structure_bound)`。两条入口在 current draft、current structure plan、current structure-bound beat map 三件套闭合后才真正汇合。

| 风险 | 编译口径 |

### 15.9 H5B 单次视觉跨多 beat 的物理归属

H5B 当时的新直供 v0.2 session 第一次进入 deterministic candidate compile 时，真实的“四能力图”暴露了此前单 beat fixture 没覆盖的合同缺口：同一张图只出现一次，却连续覆盖“框架揭示、获客、销转、交付、裂变”多个 beat。旧 compiler 按 visual task 反查 occurrence，于是把同一个 occurrence ID 投影到每个复用 beat；typed candidate 正确以 `beat_card_v05_occurrence_duplicate` 阻断。本段是历史问题记录，不声明当前直供版本。

该问题不需要新的业务选择。R3 / R6 已确认“一项 task 可覆盖多个相邻节点、一次物理出现只算一个 occurrence”，本批只把该口径补进 candidate：

```text
visual task       = 这张图承担什么视觉工作
coverage beats    = 这张图持续服务哪些相邻口播节点
occurrence        = 这张图实际出现一次
owner beat        = occurrence.covered_beat_ids 中按 current beat map 最早的 beat
```

确定性投影规则：

1. 所有被覆盖 beat 都保留 `visual_task_ids[]`，证明语义覆盖没有丢失。
2. 只有 owner beat 的 `occurrence_ids[]` 写入该 occurrence；同一 ID 在 candidate 全局只出现一次。
3. visual insert card 的 trigger 直接取 owner beat，不再从 task 的任意 coverage record 猜测。
4. occurrence 的 covered beats 必须非空、存在、去重、连续，且属于对应 task 的 `covered_beat_ids[]`；非相邻再次出现必须新建 occurrence。
5. 真实失败 session 保留在 deterministic compile failure，修复后新建 session 重跑，不补写 candidate。

### 15.10 H6 热点入口二次产品定义

#### 15.10.1 为什么不能直接写 adapter

H6 反查现有 R5 / R6 / R7 合同时，确认“热点研究 -> 用户选题 -> Brief”之间仍有四个产品空洞：

1. `topic_human_gate` 被写成“产出 topic card”，会让人类选择动作看起来像重新生产研究内容；真实动作应是对既有 topic revision 做选择。
2. `content_brief` 的 current source selector 只认识直供稿；热点入口若继续靠文件扫描或“找到第一个 topic”运行，会重新引入 Agent 临场猜输入。
3. 热点事实会在选题后变化；现有合同没有区分“只多了一篇同口径报道”与“核心事实反转”，也没有定义应刷新展示还是使下游 stale。
4. R5 的高风险核验、R6 的主张证据和 R7 的最终人类门禁都已存在，但尚未明确“允许围绕高风险事件写稿”不等于“允许把未核实指控写成事实”。

因此 H6 不再定义为“给现有 v0.1 蓝图补两个 producer”，而是建立不兼容的 `hotspot_to_delivery_single_v0.2`。v0.1 保留历史审计，不原地改义。

#### 15.10.2 热点入口的对象链

```text
account startup / identity verified
  -> hotspot_research_request（确定性冻结本轮账号、策略和研究 mode）
  -> hotspot_research_set（单一提交，内含 R5 signal -> event -> candidate -> topic options）
  -> topic_panel_projection（确定性生成 topic_selection_panel）
  -> topic_human_gate
  -> topic_selection_decision（只记录选择、重找、放宽、人工来源、多选或停止）
  -> selected_topic_source（本次内容唯一热点来源绑定）
  -> content_brief
  -> structure_plan(design_before_draft)
  -> draft(generate_from_structure)
  -> content_beat_map(structure_bound)
  -> shared script / visual / platform / candidate / HTML chain
```

`topic_selection_decision` 不能复制或改写 topic 内容，只能绑定：

```yaml
decision_code: select_one / rerun_same_policy / broaden_within_account_policy / add_manual_source / select_multiple / stop
action_code: select_topic / rerun_hotspot_research / broaden_hotspot_scope / attach_manual_hotspot_source / branch_selected_topics / archive_session
research_set_ref:
selection_panel_ref:
selected_topic_refs: []
scope_delta:
manual_source_input_set_ref: null
decided_at:
human_instruction_summary:
```

`select_one` 生成一个 `selected_topic_source`；`select_multiple` 不把多篇塞进单篇稿，只生成交给 R2 的 branch handoff。rerun / broaden / manual source 均必须新建 research request、research set 和 panel revision，不能覆盖旧面板；`stop` 可审计结束，不伪造后续稿件。精确条件必填与 action route 以 15.11.4 为准。

`selected_topic_source` 是轻量、不可变的引用包，不复制整份雷达输出：

```yaml
content_origin: hotspot_selected_topic
account_snapshot_ref:
radar_policy_ref:
research_set_ref:
selection_decision_ref:
topic_option_ref:          # component_ref
event_ref:                 # component_ref
candidate_ref:             # component_ref
topic_evidence_packet_ref: # component_ref
freshness_policy:            # selected source 内嵌版本化 policy，不使用悬空 ref
selected_at:
content_semantic_digest:
monitoring_digest:
```

其中 `content_semantic_digest` 只覆盖影响写作含义的事实、证据关系、受众、风险和可说边界；`monitoring_digest` 覆盖最新检查时间、新增同口径来源等监测事实。这样“多了一篇没有改变结论的报道”不会把全文无意义打回重写，而真正改变主张的事实仍能使下游 stale。

#### 15.10.3 证据包与写作边界

`topic_evidence_packet` 不是来源链接列表，也不是把整份研究报告交给写作节点。它只携带本题写作需要的最小事实与边界：

```text
event / topic identity
允许表达的事实主张及其 source bindings
只可归因转述的说法
contested / not_enough_info / not_checked 主张
source publisher / URL / published_at / observed_at / independence
fact status / propagation status / risk level（继续分层）
对买车用户、车商、卖车用户的影响链
source freshness 与最后复核结果
```

传播热不等于事实真，内部排序分也不进入事实字段。检索失败、页面失效或没有足够证据是合法研究结果；系统应缩小主张、改成机制解释、取消证据截图或回到选题门禁，不能用 Image 2、标题摘要或“网络上都在说”填补证据。

高风险 topic 可以进入面板，也可以在用户明确选择后进入结构和草稿；这次选择只代表“允许围绕该事件开发内容”，不把未核实指控升级成事实。草稿可按来源归因点名、明确不确定性、区分已证实事实与传播说法，并把最终发布强度留给 final human gate；没有证据支持的肯定式主体结论仍须阻断或降级。该口径取代“高风险题材一律只能讲机制”的过窄解释，但不降低证据标准。

#### 15.10.4 时效复核与失效范围

首期不发明统一的“几小时过期”常量。selected source 内的 `freshness_policy` 由事件类型、风险、来源更新时间和账号策略给出；没有产品或实测依据的小时阈值时固定采用 `always_revalidate_before_delivery`，不能跳过检查，也不能把 fixture 时间写进真实合同。

至少在以下时点检查：

1. 用户提交 `select_one` 前，确认选中的 topic revision 仍是面板 current revision。
2. 进入最终 candidate compile 前，复核 topic 的关键事实、争议状态和公开来源可用性。
3. 长时间中断恢复、来源状态变化或人工触发复核时，追加检查；不把每个节点都变成一次网络检索。

复核完成后只允许四种变化结果；外部读取失败另写 execution status，不冒充变化判断：

| 变化级别 | 例子 | 运行结果 |
|---|---|---|
| `no_material_change` | 没有新事实或新来源，只完成本次复核 | 新建监测 revision、更新检查时间；不使 Brief / 稿件 stale。 |
| `observation_refresh` | 新增同口径来源、只更新 checked_at，允许主张未变 | 新增监测 revision、更新 `monitoring_digest` 与 HTML 当前性；不使 Brief / 稿件 stale。 |
| `material_fact_update` | 数字、时间、主体动作、证据关系或受影响链变化，但仍是同一 topic | 新建 evidence / selected source revision，更新 `content_semantic_digest`；Brief 及依赖它的下游全部 stale，从 Brief 恢复。 |
| `topic_reversal_or_identity_change` | 核心事实反转、主体 / 事件不是同一件事、关键来源被撤回且无替代 | 标记 `topic_revalidation_required`，新建 research run / panel 并要求重选；不得返回旧 panel 或只加 warning 继续。 |

旧 revision、当时选择和旧 HTML 均保留审计，不原地覆盖。来源截图继续遵守 R6：只有视觉需求判断它能承担证据价值时才捕获；捕获失败不把研究伪装成功，已有截图也必须重新检查 claim binding、hash、时间和 freshness。

#### 15.10.5 蓝图、汇合点与失败恢复

`hotspot_to_delivery_single_v0.2` 固定前置顺序：

```text
hotspot_research_request_commit
-> hotspot_research
-> topic_panel_projection
-> topic_human_gate
-> selected_topic_source_commit
-> hotspot_content_brief
-> hotspot_structure_plan(design_before_draft; draft refs empty)
-> hotspot_draft(generate_from_structure)
-> content_beat_map(structure_bound)
-> script_review
-> shared visual / platform / cover downstream
-> delivery_topic_freshness_review
-> delivery_topic_freshness_apply
-> delivery_candidate_compile / render / viewport / final gate
```

直供稿与热点稿只在 current draft、current structure plan、current structure-bound beat map 三件套闭合后汇合。Brief 可共用业务合同，但输入必须是显式 tagged union：`direct_content_source` 或 `selected_topic_source`；不能回退到目录扫描，也不能把 panel、event、candidate 任意一个单独当成内容来源。

失败恢复采用以下业务语言：

| 场景 | 结果与下一步 |
|---|---|
| 二手车直接候选少于 3 条 | 按 R5 打开有影响链证明的新车外溢池；H6 只消费结果，不重写阈值。 |
| 两个池仍无可核验候选 | `research_ready_no_recommendation`；让用户选择重找、在账号边界内放宽、提供人工来源或结束，不生成空 topic。 |
| 候选只有 1-2 条但有效 | 正常展示并可选择；“3 条”是启动外溢的阈值，不是面板最小交付数。 |
| 选择时 panel / topic 已换 revision | `topic_selection_stale`；刷新面板后重新选择，不猜用户仍选哪一条。 |
| 来源不可访问 | 保留 attempt 和来源状态；能由其他来源支撑则继续，否则缩小主张、转机制或回门禁。 |
| 选择多个 topic | 进入 R2 多分支；首期仍是一分支一 session，不在单篇蓝图内自动并行。 |
| 账号 binding / snapshot 不一致 | `account_identity_inconsistent`；先修账号绑定，不继承其他账号词库、候选或视觉身份。 |

#### 15.10.6 最终 HTML、透明度与自主计量

热点稿 HTML 在现有五个同步视图外，至少能让用户看见：选题来自哪次 research run、选择了哪个 topic revision、关键事实最后复核时间、事实 / 传播 / 风险三层状态、来源清单与证据截图当前性。它不得把“传播热度高”“用户选了”或“有截图”渲染成事实已证实。因为这些是新增必填对象与交付语义，热点稿不得静默塞进 v0.6：H6 必须升级 `typed_components / renderer / template` 到 v0.7，并把 v0.6 作为直供及历史 session 的兼容输入登记；新 renderer 通过 tagged source variant 区分 direct / hotspot，不能要求直供稿伪造 research run。

热点全链只有同时满足以下条件才计一次新的自主完成样本：

```text
账号身份与 session snapshot 绑定
research run / panel / topic / decision / selected source 均有 current revision 与 digest
热点结构先于生成稿，且无 future artifact reference
事实变化按 freshness contract 恢复
candidate / HTML / viewport 由现有确定性链生成
无 Agent 自创步骤、手改 current pointer、手拼 candidate 或补写 evidence verdict
正常 Topic Gate 与 final human gate 不扣分
```

网络检索、公开页面捕获和资产复用分别记录为外部 / 既有资产步骤，不冒充 semantic Skill。首次真实热点回归可联网检索公开来源；证据画面按视觉分析需要使用真实 source capture，非证据画面优先复用语义、权利和画布均适配的已验证素材，不自动调用新 Image 2。是否生成新图由后续用户另行授权，不影响 H6 编排合同验收。

### 15.11 H6 编译前二义性审核

#### 15.11.1 本次审核发现的编译阻点

R7-C47-C70 的业务方向成立，但直接对照现有 Schema / registry / runtime 后，仍有九处会逼迫编译者临场决定：

1. `semantic_artifact_submission.v0.2` 与 plan step 都只允许一个 `output_artifact_type`，而旧热点 Skill 一次声称产出 research run、candidate、panel、topic card 四类对象。
2. Topic Gate 若不改写 topic card，就必须明确“已选中”由哪个对象证明；旧 `topic_selected_for_brief` 不能继续既当 topic 状态又当人类决定。
3. “重找、放宽、人工来源、多选、停止”写在文案里，却没有稳定 action code、条件必填和 status route。
4. `content-brief.v0.3` 用 `hotspot_topic`，结构计划用 `hotspot_selected_topic`，而 Brief 的 `next_skill` 又固定成 writer；这与“热点先结构后成稿”直接冲突。
5. `session-execution-plan.v0.7` 把 blueprint、renderer 和 template 写死成直供 v0.2 / v0.6，不能承载热点蓝图或 v0.7 页面。
6. freshness 没有生产者、输出对象、失败状态和 blueprint 节点；“从 Brief 重跑”也没有 plan revision / restart marker。
7. `not_available` 的小时阈值没有定义下一步，可能被实现成跳过复核或永久阻断。
8. 证据包没有把 R5 的 event fact status、R6 的逐 claim evidence relation 和表达许可分开，writer 仍可能自行判断“能不能当事实说”。
9. v0.6 candidate 以 `source_map >= 12` 做直供基线；热点新增来源若继续按固定数量检查，会把一次真实 run 的对象数编译成产品常量。

以下口径用于消除这些阻点；它们细化 C47-C70，不改变“二手车优先、先结构后成稿、用户最终判断、0..N 视觉”的业务方向。

#### 15.11.2 研究请求、单产物提交与对象生产者

H6 延续现行 R7 的硬约束：**一个 plan node、一个 submission、一个 current artifact type**。热点研究不得要求 submitter 跨四类文件做伪原子提交。

用户在 Topic Gate 增加一个或多个链接、文件或引用线索时，先由 human input recorder 提交单一 `hotspot_manual_source_input_set-v0.1`，再提交引用它的 decision。该对象只是“请研究这些线索”，不是证据 verdict，也避免同一 artifact type 的多个 current pointer：

```yaml
manual_source_input_set_id:
manual_source_input_set_revision:
account_identity_ref:          # artifact_ref
inputs:                        # minItems=1；input_id 唯一
  - input_id:
    input_kind: public_url / local_document / citation_lead
    public_url: null | string
    local_document_ref: null | artifact_ref
    citation_lead: null | string
submitted_at:
input_status: recorded_unverified
```

每个 input 的三个 locator 严格三选一。输入 recorder 成功而 decision 中断时保留该 current set，恢复时复用，不重复生成；research 后续仍须逐条建立 source record / evidence status，不能把 `recorded_unverified` 当 supported。

热点研究必须先有版本化 `hotspot_research_request`，不能让 research Skill 从聊天原句、decision 摘要或目录现状临场拼查询范围：

```yaml
research_request_id:
research_request_revision:
supersedes_request_ref: null | artifact_ref
account_identity_ref:          # artifact_ref
account_snapshot_ref:          # artifact_ref
radar_policy_ref:              # artifact_ref
triggering_decision_ref: null | artifact_ref
triggering_freshness_review_ref: null | artifact_ref
prior_research_set_ref: null | artifact_ref
prior_panel_ref: null | artifact_ref
request_mode: initial / same_policy_rerun / broaden_within_account_policy / manual_source_refresh / revalidation_after_reversal
scope_delta: null | {}
manual_source_input_set_ref: null | artifact_ref  # 不得直接放 URL 字符串
requested_at:
request_status: ready
```

初次进入 H6 时，request compiler 从 current account identity / snapshot / radar policy 生成新 request ID，`requested_at` 必须逐字复制账号启动请求已物化到 current snapshot 的 `snapshot_at`；缺失、无时区或格式非法时阻断，不得用 epoch、当前机器时间或聊天时间补位。decision / freshness / prior / manual refs 为 null。重找、放宽、加入人工来源或事实反转后重验时，每次都生成新的 request ID（revision 从 1 开始）并用 `supersedes_request_ref` 连接上一请求；只有修正同一请求的机器字段才增加该 ID 的 revision。`same_policy_rerun` 不得携带 scope / manual source；`broaden_within_account_policy` 必填合法 `scope_delta`；`manual_source_refresh` 必填一个 current manual source input set；前三者必填 triggering decision、不得填 freshness review。`revalidation_after_reversal` 必填 triggering freshness review 与 prior research / panel，不得携带 scope / manual source。research Skill 只消费 current、`ready` 的 request。

request compiler 的值复制也固定：decision=`rerun_same_policy / broaden_within_account_policy / add_manual_source` 必须一一映射到同名 request mode，request 的 scope / manual ref 必须与 triggering decision 完全一致；reversal request 的 freshness ref 必须等于刚提交的 current review。任何非 initial request 的 supersedes ref 必须等于提交前的 current request，不能跨过中间请求或取目录中最新时间文件。

新增 `hotspot_research_set` 作为一次研究的机器提交单元。它不是第二套业务状态，而是 R5 多个业务组件的版本化聚合包：

```yaml
research_set_id:
research_set_revision:
account_identity_ref:        # artifact_ref
account_snapshot_ref:        # artifact_ref
radar_policy_ref:            # artifact_ref
research_request_ref:        # artifact_ref
research_run_record: {}
signals: []
events: []
candidates: []
topic_options: []
topic_evidence_packets: []
panel_model: {}
source_records: []
ledger_write_refs: []
component_digest_map: {}
researched_at:
research_set_status:
```

`research_set_status` 只允许：

```text
ready_for_panel          # 至少 1 个可选 topic option
ready_no_recommendation  # 研究已完成，但 0 个可选 topic
```

来源读取尚未完成、网络不可用或合同失败时不提交 current research set，只保留 attempt / pending submission / failure evidence，分别返回 `waiting_external / blocked`。不能提交一个半成品 set 后再让 panel 猜哪些 component 可用。

`hotspot-topic-research` 只向 R7 提交这一种 artifact。set 内每个 component 仍保留 R5 的稳定 ID 和独立 digest，signal / event / candidate / topic 关系没有被压扁；人类 Markdown、词库账本和查询效果账本是该 set 或独立 append-only operation 的投影 / side record，不成为另一个 current truth。

`panel_model` 只允许保存已经由 R5 排序完成的 `ordered_topic_option_refs`、`recommended_topic_ref | null` 和可解释的 recommendation reason。deterministic panel projector 只能校验、排版和补交互状态，不能重新评分、换序、增删 topic 或生成新的推荐；因此 research set 是排序真源，panel 是人类选择投影，不形成第二套推荐结论。

`topic_selection_panel-v0.2` 的机器 payload 固定为：

```yaml
panel_id:
panel_revision:
research_set_ref:              # artifact_ref
ordered_topic_option_refs: []  # 完整 component_ref；必须与 panel_model 同序、同数量、同 digest
recommended_topic_ref: null | component_ref
recommendation_reason: string       # 始终非空；推荐或不推荐都要说明
panel_status: panel_ready_waiting_human / panel_no_recommendation
projected_at:
```

research set 为 `ready_for_panel` 时 panel 至少 1 个 option，status 必须 `panel_ready_waiting_human`；recommended ref 可以为 null，表示有选项但没有单一主推荐。只有 research set=`ready_no_recommendation`、option 数为 0 时才能使用 `panel_no_recommendation`。recommended 非 null 时必须出现在 ordered refs 中。投影时间不改变 research 排序或 component digest。

对象生产者固定为：

| artifact | producer | step_kind | consumer |
|---|---|---|---|
| `hotspot_manual_source_input_set` | human input recorder | human_gate support operation | Topic Gate decision / request compiler |
| `hotspot_research_request` | deterministic research request compiler | deterministic_tool | hotspot research |
| `hotspot_research_set` | `hotspot-topic-research` | semantic_skill | panel projector / freshness / audit |
| `topic_selection_panel` | deterministic panel projector | deterministic_tool | Topic Gate |
| `topic_selection_decision` | human decision recorder | human_gate | selected source compiler / rerun routing |
| `selected_topic_source` | deterministic selected source compiler | deterministic_tool | hotspot Brief / freshness |
| `topic_freshness_review` | 新的 `hotspot-topic-freshness-review` | semantic_skill with external reads recorded | freshness apply |
| `selected_topic_source` 新 revision | deterministic freshness apply | deterministic_tool | candidate 或 replan |

node result status 与是否写 current pointer 固定为：

| node | result status | current pointer |
|---|---|---|
| research request compiler | `request_ready` | 写 research request；输入条件失败为 `blocked`、不写 |
| hotspot research | `research_ready_for_panel / research_ready_no_recommendation` | 写 research set；`waiting_external / blocked` 不写 |
| panel projection | `panel_projected / panel_no_recommendation` | 写 panel；`blocked` 不写 |
| Topic Gate recorder | `decision_committed` | 写 decision 后按 decision code 路由；尚未回答为 `waiting_human`、不写 |
| selected source compiler | `selected_source_ready` | 只对合法 select-one 写 source；`topic_selection_stale / blocked` 不写 |
| freshness review | `review_complete` | complete 才提交 current review；`waiting_external / blocked` 只保存 attempt / failure revision evidence，不写 current、不推进 candidate |
| freshness apply | `monitoring_applied / semantic_update_replan / topic_revalidation_replan` | 只写 selected source revision；前者继续 candidate，后两者交 coordinator 执行独立 replan 事务 |

payload 的业务 status 与 node result status 分开映射，统一登记在 artifact-commit / status-route registry；Skill 不得返回 payload 原生状态后让 runner做字符串相似匹配。

固定物理路径沿用 R7 既有约定：

```text
intermediate/r7/revisions/{artifact_type}/{artifact_id}.json
intermediate/r7/current/{artifact_type}.json
intermediate/r7/commits/{submission_id}.json
```

revision 文件先写、commit marker 通过后才替换 current pointer；panel Markdown / HTML 投影失败时不得推进 Topic Gate。

#### 15.11.3 引用类型与摘要口径

H6 只允许两种引用，不能用一个字符串同时冒充文件、component 和 artifact：

```yaml
artifact_ref:
  artifact_id:
  revision:
  sha256: sha256:<64hex>

component_ref:
  container_artifact_ref: { artifact_id, revision, sha256 }
  component_type: event / candidate / topic_option / topic_evidence_packet / source_record
  component_id:
  component_sha256: sha256:<64hex>
```

`container_artifact_ref` 首期只允许 `hotspot_research_set` 或 `topic_freshness_review`。初次选择的 event / candidate / topic / evidence packet 全部来自 research set；material update 的 replacement evidence packet 来自 freshness review。不能把一个后来生成的 evidence packet 伪装成旧 research set component。

编译消歧：container 自身 payload 内不得嵌入带“自身最终 SHA”的完整 component ref，否则形成不可求值的自哈希循环。`hotspot_research_set.panel_model` 内部因此使用 `local_component_ref={component_type, component_id, component_sha256}`；research set 完成 pointer-last commit 后，panel projector 才把 current research set artifact ref 补成跨对象使用的完整 `component_ref`。这不改变排序、推荐或证据结论，只固定物理可提交顺序。

新业务对象内的 digest 全部使用 `sha256:<64hex>`。`semantic_task_envelope / submission` 现有的 bare 64 位 `input_binding_digest` 只作为该既有机器协议的内部字段保留，不能复制到 artifact/component ref。

`hotspot_research_set.component_digest_map` 必须覆盖全部被 panel、decision、selected source 或 evidence packet 引用的 component；component 内容变化必须新建 research set revision，不能只改 map。

#### 15.11.4 Topic Gate 的动作、条件必填与状态

H6 使用 `r7-action-registry-v0.2`，新增五个 action，并复用已有 `archive_session`：

| decision_code | action_code | 条件必填 | 路由 |
|---|---|---|---|
| `select_one` | `select_topic` | 恰好 1 个 current topic ref；scope=null、manual=null | selected source compiler |
| `rerun_same_policy` | `rerun_hotspot_research` | selected=[]、scope=null、manual=null、current panel ref | request compiler 生成 `same_policy_rerun`，新 plan revision 从 research 重启 |
| `broaden_within_account_policy` | `broaden_hotspot_scope` | selected=[]、合法非空 `scope_delta`、manual=null | request compiler 生成 `broaden_within_account_policy`，新 plan revision从 research 重启 |
| `add_manual_source` | `attach_manual_hotspot_source` | selected=[]、scope=null、1 个 current manual source input set ref | request compiler 生成 `manual_source_refresh`，新 plan revision 从 research 重启 |
| `select_multiple` | `branch_selected_topics` | 至少 2 个 current topic refs；scope=null、manual=null | 调用 R2 `detect_branch_request`，状态进入 `branch_request_requested` |
| `stop` | `archive_session` | selected=[]、scope=null、manual=null | 当前 session 审计结束 |

`topic_selection_decision` 必填：

```yaml
decision_id:
decision_revision:
panel_ref:                 # artifact_ref
research_set_ref:          # artifact_ref
decision_code:
action_code:
selected_topic_refs: []    # component_ref
scope_delta: null
manual_source_input_set_ref: null | artifact_ref
decided_at:
decision_status: committed
human_instruction_summary:
```

旧 topic option 保持 `topic_pending_human_choice`，不会因人类选择被原地改成 selected；`topic_selected_for_brief` 在 v0.2 只作为 v0.1 历史兼容状态。当前是否选中由 current decision + selected source 共同证明，panel 也不从 `panel_ready_waiting_human` 原地改写成 `panel_selected`。

decision commit 后的路由是串行事务，不能由一个 human gate submission 同时写 decision、request 和 plan：`select_one` 进入 selected source compiler；rerun / broaden / manual source 先提交新 `hotspot_research_request`，其 commit marker 成功后再由 coordinator 创建并激活 plan revision；`select_multiple` 把 current decision、账号引用和所选 topic refs 交给既有 R2 `detect_branch_request`，由 R2 生成 `human_decision_payload / branch_request`，H6 不新发明 handoff artifact；`stop` 进入既有 archive action。manual source 是唯一前置例外：先独立提交 source input set artifact，decision 只能引用 current set，仍不发生多输出 submission。

`panel_no_recommendation` 必须 `ordered_topic_option_refs=[]`、`recommended_topic_ref=null`；此时只允许 rerun / broaden / manual source / stop，禁止 `select_one`。多选只形成 R2 branch request，H6 不负责自动创建或并行执行 child session，也不计入 F51 单篇完整链。

#### 15.11.5 selected source 与证据表达合同

`selected_topic_source` 只由 deterministic compiler 生成。它使用 `artifact_ref` 绑定账号 / policy / research / decision，使用 `component_ref` 绑定 event / candidate / topic / evidence packet：

```yaml
selected_topic_source_id:
selected_topic_source_revision:
content_origin: hotspot_selected_topic
account_snapshot_ref:
radar_policy_ref:
research_set_ref:
selection_panel_ref:
selection_decision_ref:
event_ref:
candidate_ref:
topic_option_ref:
topic_evidence_packet_ref:
monitoring_source_record_refs: []  # component_ref；初始来自 research set，复核后可来自 freshness review
latest_freshness_review_ref: null | artifact_ref
freshness_policy:
  policy_mode: duration_bound / event_triggered / always_revalidate_before_delivery
  max_age_hours: number | null
  threshold_basis: string | null
  policy_basis_refs: []
  event_triggers: []
  fallback_reason: null / no_trusted_duration_or_trigger_policy
selection_freshness_status: current  # const；stale selection 不产出 source artifact
digest_basis_version: selected-topic-source-digest-v0.1
content_semantic_digest: sha256:<64hex>
monitoring_digest: sha256:<64hex>
selected_source_status: ready_for_brief / ready_for_delivery / revalidation_required
selected_at:
```

只有 `select_one` 且 panel、research set、topic component、账号 binding、radar policy digest、selection freshness 全部 current 时才创建 revision 1。每个新的 select-one decision 必须生成新的 selected source ID + revision 1；只有 deterministic freshness apply 可以在同一 source ID 下递增 revision，重找 / 反转后重新选择不得复用旧 source ID。`freshness_policy` 直接嵌入 selected source，不保留一个没有 producer 的悬空 ref：账号或研究对象有带依据的版本化 duration / trigger policy 时按其生成；否则固定生成 `always_revalidate_before_delivery`。三种 mode 的互斥字段以 15.11.8 为准。选择已 stale 时返回 node result `topic_selection_stale` 且不创建 source artifact，直接要求新 research set / panel 后重新选择，避免在 payload 中保留一个永远不可消费的 refresh-required 状态。

status 的消费者固定为：初次选择与 material update apply 生成 `ready_for_brief`，只有 hotspot Brief 可消费；no-material-change / observation-refresh apply 生成 `ready_for_delivery`，只有 candidate compile 可消费；reversal / identity change 生成 `revalidation_required`，Brief 和 candidate 都不得消费。不能用“仍可交付”“当前可用”一类模糊判断替代枚举匹配。

revision 1 的 `latest_freshness_review_ref=null`，monitoring source refs 只包含 evidence packet 已引用的 research set source records。每次 freshness apply 都必须把 latest review ref 指到刚完成的 current review，并把该 review 的 source record delta refs 按稳定 ID 合并到 monitoring refs；`ready_for_delivery / revalidation_required` 必须 review ref 非 null。candidate 的 `freshness_review_ref` 必须与 selected source 的 latest review ref 完全一致，不能分别取两个“最新”文件。

热点 Brief v0.4 同时保存选择时的 `selected_topic_source_ref` 和 `selected_topic_content_semantic_digest`。后续 freshness apply 产生同 ID 的新 source revision 时，candidate checker 只在以下全部成立时允许 Brief 继续有效：新 revision 由注册的 freshness apply 产生、旧新 `content_semantic_digest` 完全一致、freshness review complete、`selected_source_status=ready_for_delivery`。该例外只适用于 selected source 的监测 revision；其他 current artifact 仍按现行 ID + revision + hash 严格绑定。semantic digest 改变时，旧 Brief 必须 stale，不能靠“同一 source ID”继续。

两个摘要的输入集合也固定，不能由编译者自行挑字段：

| digest | 必须纳入 | 明确排除 |
|---|---|---|
| `content_semantic_digest` | topic identity、event 主体 / 动作 / 时间边界、event fact status、受众与影响链、逐 claim type / evidence / support basis / allowed expression、risk level、账号内容角度 | checked_at、抓取 attempt、同口径新增来源 URL、传播数量、截图文件路径、plan / revision 时间戳 |
| `monitoring_digest` | freshness policy、checked_at、source identity / published_at / observed_at、同口径来源增减、传播观察、capture availability、attempt outcome | plan ID、HTML 路径、无业务含义的对象写入时间 |

`digest_basis_version` 固定 canonical input 形状。对象键按字典序；claims、source refs、basis refs、attempt refs、source record deltas 等集合按稳定 ID 排序；只有 `panel_model.ordered_topic_option_refs`、影响链 steps 及显式声明 `sequence_no / display_order` 的数组保留业务顺序。artifact / component revision 和写入时间不直接进入 semantic digest，实际解引用后的 semantic 字段才进入。以 UTF-8 无 BOM 的 canonical JSON 计算 SHA256 并加 `sha256:` 前缀。只改变 JSON 属性或集合输入顺序不得改变 digest；改变上述 semantic 字段必须改变 content digest。

每个 `topic_evidence_packet` 必须分开记录：

```yaml
event_fact_status: pending / verified / contested / unavailable
propagation_status: unknown / observed / verified / blocked_external
risk_level: low / medium / high / not_assessed
claims:
  - claim_id:
    claim_type: opinion / experience / factual_claim / quote / statistic / prediction
    claim_evidence_status: not_required / supported / refuted / not_enough_info / contested / not_checked
    source_refs: []
    source_independence_status: independent_sources / same_origin_republication / single_source / not_assessed
    source_support_basis: authoritative_primary / two_independent_eligible_secondary / single_eligible_secondary / same_origin_republication / unsupported / not_assessed
    allowed_expression: assert_as_fact / attribute_to_source / state_uncertainty / express_as_opinion / express_as_first_person_experience / mechanism_only / omit
```

`risk_level` 是发布 / 名誉 / 权益风险，不是事实置信度；`high / not_assessed` 必须进入最终人类审阅。`authoritative_primary` 指官方文件、原始数据发布者或该陈述的原始主体；`two_independent_eligible_secondary` 必须是两个通过 R5 来源资格判断、且不是同源转载的二级来源。

`assert_as_fact` 只允许 `factual_claim / statistic` 同时满足 event fact verified、claim supported、risk low / medium，且 support basis 为 `authoritative_primary` 或 `two_independent_eligible_secondary`。单一二级来源或同源转载最多 `attribute_to_source / state_uncertainty`；quote 必须 `attribute_to_source`；prediction 必须 `state_uncertainty` 或归因；opinion / experience 分别使用显式 opinion / first-person expression，不能伪装成事实。高风险 topic 即使被用户选中，也只能按当前 claim 的 `allowed_expression` 写，用户选择不改写 evidence status。来源网页记录与 source capture 继续是两个对象：有 source URL 不等于已经截图，有截图也不等于 supported。

#### 15.11.6 Brief、结构和 Draft 的版本闭合

热点入口不再复用当前 `content_brief` node 的直供 success route，固定使用入口专属节点：

```text
hotspot_content_brief
hotspot_structure_plan
hotspot_draft
```

合同升级如下：

| 合同 | 新版本 | 必须修正 |

#### 15.11.8 freshness 的可执行状态机

`freshness_policy` 不靠一个可空小时数运行：

```yaml
policy_mode: duration_bound / event_triggered / always_revalidate_before_delivery
max_age_hours: number | null
threshold_basis: string | null
policy_basis_refs: []
event_triggers: []
fallback_reason: null / no_trusted_duration_or_trigger_policy
```

条件关系严格互斥：`duration_bound` 必填正数 `max_age_hours + threshold_basis + policy_basis_refs`，triggers 为空、fallback null；`event_triggered` 必填至少一个 trigger 和 basis refs，hours / threshold / fallback 为 null；`always_revalidate_before_delivery` 的 hours / threshold / basis / triggers 全为空并必填 fallback reason。没有可信小时或 trigger 依据时使用最后一种，不是跳过，也不是拿 fixture 时间代替。

`hotspot-topic-freshness-review` 是独立、聚焦的语义 Skill，只复核已选 topic，不重新扩词、排名或偷换选题。输出：

```yaml
freshness_review_id:
selected_topic_source_ref:
review_status: complete / waiting_external / blocked
change_class: no_material_change / observation_refresh / material_fact_update / topic_reversal_or_identity_change / not_assessed
topic_identity_status: same_topic / identity_changed / not_assessed
checked_at:
source_attempt_refs: []        # artifact_ref
source_record_deltas: []       # 完整 source_record component；新来源不能只留悬空 ref
replacement_evidence_packet: null | topic_evidence_packet  # 必含新的 packet ID 与完整 claims
replacement_evidence_packet_digest: sha256:<64hex> | null
component_digest_map: {}       # 覆盖全部 source_record_deltas 与 replacement packet
review_reason:
```

条件关系：

- `complete` 必须使用前四个 change class、identity 不得 `not_assessed`，且至少有一个可审计 source attempt；`waiting_external / blocked` 必须 `change_class=not_assessed + topic_identity_status=not_assessed`，保存 attempt / preflight failure 后停在 freshness node，不进入 candidate。
- `material_fact_update` 只允许 `topic_identity_status=same_topic`；`topic_reversal_or_identity_change` 在主体 / 事件身份变化时必须 `identity_changed`，在事实反转但身份未变时可 `same_topic`。
- `material_fact_update / topic_reversal_or_identity_change` 必填带新 packet ID 的完整 `replacement_evidence_packet + replacement_evidence_packet_digest`，后者必须等于 review component 的 canonical digest；前两类 change 必须两者都为 null。Codex 不直接写 selected source 或 current pointer。
- `no_material_change / observation_refresh` 不得改变 `content_semantic_digest`。
- `observation_refresh` 必须至少有一个可校验的 source record delta；没有来源变化只完成复核时必须归 `no_material_change`。`component_digest_map` 必须覆盖 review 内每个 component，供后续 component_ref 校验。

material update 保留原 event / candidate / topic option identity refs，只把 evidence packet ref 切到 freshness review 内的 replacement component，并按新证据重算 semantic digest；如果主体、事件指纹或选题身份已经不是同一对象，必须归入 reversal / identity change，不能为了少重跑而伪装成 material update。freshness policy 由 deterministic apply 按新 evidence / risk 与账号政策重新派生；policy 本身不单独决定 semantic digest，但造成 policy 变化的 risk / evidence 变化必须已体现在 semantic digest。

deterministic freshness apply 每次 review complete 后生成一个新的 immutable selected source revision：

| change_class | semantic digest | monitoring digest | 路由 |
|---|---|---|---|
| no material change | 保持 | 更新复核时间摘要 | selected source=`ready_for_delivery`；semantic binding 相同，继续 candidate |
| observation refresh | 保持 | 纳入同口径新来源 | selected source=`ready_for_delivery`；semantic binding 相同，继续 candidate |
| material fact update | 重算 | 重算 | selected source=`ready_for_brief`；apply commit 后由 coordinator 新建 plan revision，从 `hotspot_content_brief` 重启 |
| reversal / identity change | 重算并标 revalidation required | 重算 | selected source=`revalidation_required`；request compiler 新建 `revalidation_after_reversal` request，coordinator 从 `hotspot_research` 重启，旧 panel / decision 不复用 |

freshness apply 和 workflow replan 是两个可恢复事务，不伪装成跨文件原子操作：

1. apply 只提交 selected source revision、commit marker 和 current pointer；提交成功后追加 `workflow.replan.requested.v1`。
2. coordinator 以 `old_plan_id + selected_source_ref + restart_from_node_id` 形成稳定 idempotency key，把 v0.8 新 plan 写入 `intermediate/p0/plan-revisions/{plan_id}.json`，再写 `intermediate/p0/plan-commits/{plan_id}.json`。
3. commit 校验通过后，最后原子替换 `intermediate/p0/session-execution-plan.json` 作为 plan activation；新 plan 内的 `supersedes_plan_id` 就是旧 plan 的物理 superseded marker，不另写一个可能与 current 分叉的真源文件。
4. activation 后追加唯一 `workflow.replanned.v1` event 并重建 projection；新 plan 在 restart 之前的有效依赖投影为 `carried_forward`，旧 plan 被失效的 pending / completed step 投影为 skipped with `stale_replanned`，新 plan 的 restart node 才能变成 next。

中断恢复时：selected source 已提交但 plan 未激活，按同一 idempotency key补完 plan；plan 已激活但 event / projection 未完成，只补 event / rebuild，不再新建 selected source、request 或 plan。已完成 artifact / asset 不删除。只有新视觉任务认定旧资产语义不再适配时才重做 provider / capture，不因 replan 盲目重调。

#### 15.14.1 触发事实

#### 15.14.2 返工对象与路由

新增 typed `delivery_revision_request`，至少绑定：当前 delivery / candidate / source map、用户原始指令、规范化 action、target artifact / asset / rendition、change class、owning producer、restart node、request digest 和创建时 plan ref。

一次人类退回可以包含 1 到 N 个修改点，但只能物化一个 request。request 使用 `change_items[]`，每项固定 `change_item_id / target_ref / target_sha256 / change_class / normalized_instruction / owning_producer / candidate_source_map_ref`；同一 `target_ref + change_class` 在一份 request 中必须唯一，目标不属于 current candidate 或无法映射 owning producer 时整份 request 阻断，不能只执行其中一半。

`restart_node_id` 不是人工输入，而是 coordinator 在 active blueprint 中取所有 change item owning nodes 的最早节点；`invalidated_refs` 是各修改项依赖闭包的并集，`carried_forward_refs` 是 current plan 减去该并集后的仍有效对象。一个 request 只创建一个 plan revision、一次下游重建和一个新 delivery revision；所有 change items 闭合前不得推进 candidate。首期同一 delivery 同时只允许一个 active revision request，防止两条 plan revision 竞争 current pointer。

| 返工内容 | 最早重启节点 |
|---|---|
| 是否需要这张图、来源类或视觉职责改变 | `visual_need_analysis` |
| base image、证据标注、叠字、alt / summary 或素材文字错误 | `visual_production` |
| 封面底图、标题、版式或平台 rendition | `cover_design` |

### 15.16 真实直供稿暴露的交付页误放行与产品修订（R7-C113 至 C132）

#### 15.16.1 问题定位

私有直供稿运行别名 `PRIVATE-H7-REJECTED-RUN` 生成了 HTML，也取得 desktop / mobile 的零溢出、零坏图和零失败请求结果；但人工打开页面后确认它不符合业务交付：主页面被结构、beat、trace 和执行计数淹没，标题是内部核心承诺而非交付标题，平台封面重复展示，视觉插入仍指向未叠字的 base image，暂定平台规格却被表述为“全部通过、可以发布”。

因此，现行 `viewport_acceptance=pass` 只能证明页面可加载和几何稳定，不能证明页面好找、好读、好验收，更不能证明视觉成品与发布就绪。上一轮为继续运行而在 candidate consumer 临时补全 compact ledger、layout review 和 digest 形态，会把 producer 缺字段伪装成成功；该临时源码补丁已撤回，本次 HTML 保留为缺陷证据，不作为可发布交付或自主样本。

#### 15.16.2 交付页的两层信息架构

最终 HTML 的默认读者是要验收、复制、下载和剪辑内容的人，不是排查 runtime 的开发者。页面固定分为：

1. **交付主层**：交付标题与真实状态、正式口播、画中画成品与插入位置、各平台发布包、发布前提醒、可执行返工动作。
2. **折叠审计层**：结构策略、逐 beat 覆盖、内部 ID、来源路由、lineage、执行透明度和机器报告。

审计信息必须完整保留，但默认折叠，不参与主交付阅读长度。主层不得出现 `STAGE-* / BEAT-* / R7 v* / generated_context` 等内部语言；需要解释时翻译为“为什么这样讲、哪里放图、素材来自哪里”。

主层顺序固定为：

```text
交付标题与状态
-> 正式口播
-> 画中画成品与剪辑位置
-> 抖音 / 小红书 / 视频号发布物料
-> 风险提醒
-> 返工 / 发布动作
-> 折叠审计详情
```

平台卡只展示一次可下载封面成品。只有存在与成品不同、且按 surface profile 真实生成并绑定 hash 的预览资产时，才额外展示“平台预览”；同一文件不得复制两遍冒充成品与预览。相同行为的多张画中画返工入口可合并为一个入口，或必须以人类可读名称区分目标，不能重复出现两个完全相同的“重做指定画中画”。

#### 15.16.3 验收语义拆分

`viewport_acceptance` 改为纯技术门禁，只回答：浏览器是否启动、资源是否加载、是否横向溢出、字体是否稳定、截图和 measurement 是否绑定当前 HTML hash。新增 `business_delivery_acceptance`，至少包含：

| 维度 | 通过条件 |
|---|---|
| `information_hierarchy` | 主层按业务顺序展示，内部对象默认折叠，移动端无需穿过审计卡才能到达正式口播和视觉成品。 |
| `delivery_title_quality` | 使用显式 `delivery_title`；不得直接拿 core promise、诊断语句或内部策略名当页面标题。 |
| `final_asset_binding` | HTML 展示和下载的每张画中画、封面均绑定 current accepted final rendition，而不是 base / prompt / 临时输出。 |
| `readiness_truthfulness` | banner 与 warning、平台 profile、事实核验、视觉 review 状态一致；暂定规格不能渲染为无条件可发布。 |
| `visual_human_review` | reviewer 实际查看当前 hash 对应的 desktop/mobile 截图与关键图片，并记录结论；DOM 测量不能自动生成视觉 pass。 |
| `action_usability` | 发布、改文案、改视觉动作可区分目标且无无意义重复。 |

最终状态按最弱维度派生：技术 viewport pass 但业务验收未执行时为 `rendered_waiting_business_review`；任一 final asset、标题、层级或 readiness 维度失败时为 `business_delivery_rejected`；只有两类门禁均通过才可进入 final human gate。

### 15.17 L3 三个缺口的递归产品化（R7-C133 至 C160）

#### 15.17.1 触发事实：三个表象其实是一个证据模型缺口

Alpha.8 已经证明确定性 runtime 能完成 task envelope、event、projection、pointer、candidate、renderer、viewport、返修重开和最终 HTML；但还不能把整项目升级为 L3：

```text
缺口 A：直供稿只有一条复用既有素材的 autonomous sample，未证明真实素材生产路径。
缺口 B：热点稿完成过真实全链，但语义 payload、视觉判断、prompt 和一次封面实现仍有实质扶持。
缺口 C：视觉需求、来源路由、prompt、栅格目检和返修结果虽有合同碎片，尚未形成完整的合同内语义工作包。
```

递归后的共同父因不是“Codex 参与太多”，而是现有 `workflow_autonomous_completion_count` 同时承担了三种不同结论：

```text
一次 session 是否按合同闭合；
一条业务入口是否可重复自治；
整个项目是否覆盖了它对外声称的关键能力。
```

这会导致两个方向的误判：一次闭合被过度解释成项目 L3，或者一条内容恰好不需要图片时被错误要求强行调用 provider。新产品必须把“按内容正确决策”和“项目能力已经被真实证明”分开计量。

#### 15.17.2 成熟工作流的可借鉴原则

本轮只吸收轻量原则，不引入新的调度平台：

| 成熟做法 | 本项目吸收方式 |
|---|---|
| LangGraph 把每步状态保存为 checkpoint，失败后从成功节点恢复；API / 非确定性调用放进可持久化、幂等 task | 继续使用 plan / event / projection；Image 2、网页捕获等外部步骤必须有 task、attempt、outcome 和 reconcile，不把重跑当恢复 |
| LangGraph 的持久化把单次 thread、checkpoint 历史和跨 thread store 分开 | 把 `session_autonomy_evidence`、`route_autonomy_evidence`、`project_maturity_evidence` 分开，不再由一个 session 计数越级推导 |
| OpenAI Agents SDK 把 tool call、handoff、guardrail 和自定义事件纳入 trace，并把人工批准保存为可恢复 run state | 人类 Topic Gate / 最终验收保留为合法节点；每个语义任务、工具调用、人工门禁和外部副作用分别留证 |
| OpenAI tool guardrail 在调用前后验证工具输入输出；未注册工具不能靠自然语言猜测 | 引入版本化 capability registry 和 run-start snapshot；运行中出现未注册 helper / sidecar / producer 即记干预，不得计自治 |
| 可恢复工作流升级需要保存版本标记，避免旧 pending state 被新代码静默解释 | L3 样本绑定 maturity baseline digest；影响自治合同的版本变化后，新基线重新取证，旧样本只作历史证据 |

`autonomous_waiting` 表示系统按合同诚实停在缺授权、缺来源、缺素材或缺通用能力的等待点。它证明恢复语义正确，但不等于已经交付 HTML，因此不能增加 `consecutive_autonomous_delivery_count`。

#### 15.17.4 可观测干预与 capability freeze

运行前物化 `run_capability_snapshot`：

```yaml
maturity_baseline_digest:
workflow_blueprint_digest:
schema_bundle_digest:
skill_contract_digest:
runtime_entry_digest:
registered_capabilities:
  - capability_id
  - kind: semantic_skill / deterministic_tool / external_provider / browser_capture / human_gate
  - producer_entry
  - version
  - digest_or_provider_identity
```

运行结束由机器从 task、receipt、event、artifact commit、provider/capture attempt 和文件变更证据派生 `intervention_ledger`。`manual_patch_detected=false` 不能再只是一项自报布尔值；至少必须证明：

```text
每个 current artifact 都能回到 registered producer 和 receipt；
每个 executable producer 在 run-start snapshot 中已存在；
candidate / HTML / pointer / event 没有 producer 外写入；
运行中新增的 helper、临时转换脚本、伪 review 或兼容 sidecar 被识别；
外部 provider / capture 的调用次数可从逐 task evidence 汇总；
人类动作只发生在 blueprint 已登记门禁。
```

业务运行中发现缺少通用后工程能力时，正确结果是 `autonomous_waiting + waiting_capability`。可另起 `skill_compile` 开发通用能力，但原 session 不能因为临时 helper 后来跑通而改口为 L3 样本；新能力完成编译后用新 session 取证。

#### 15.17.5 直供稿入口的 L3 证明

直供稿从用户原始输入开始，必须经过 baseline、结构、口播、全文视觉覆盖、资产路由、平台包装、candidate、HTML、双 viewport 和业务验收。图片数量仍由内容决定：

```text
0 张是合法业务结果；
复用合法既有素材是合法生产结果；
需要 Image 2 时必须真实生成；
需要来源证据时必须真实捕获；
不能为了“证明 provider”给不需要图片的稿子强行配图。
```

因此，一条只复用资产的 session 可以成为 `session autonomous_delivery`，但只证明 existing-asset 分支；它不能单独证明 Image 2 或 source capture 能力。直供 route L3 要求两条不同输入指纹的连续新 session 都完成自主交付，并且直供样本组至少覆盖一次真实 scoped revision / resume；图片 provider 和 source capture 的项目级证明由能力覆盖矩阵统一判断，不强塞给每篇直供稿。

#### 15.17.6 热点入口的 L3 证明

热点 route 样本从 current account strategy snapshot 和 materialized research request 开始，不允许 Codex 先手工拼好 future research set、source list 或 selected topic 再倒填蓝图。

```text
账号策略生成检索任务
-> 公开网络检索 attempt/outcome
-> signal/event/candidate/topic 研究对象
-> 人类 Topic Gate
-> selected source / freshness / risk expression
-> Brief / structure / draft / beat
-> 全文视觉决策与资产生产
-> candidate / HTML / viewport / 业务验收
```

用户给 URL 可以作为 typed `user_seed_source`，但不能替代“雷达按账号策略自主发现”的项目能力证明。热点 route 的两条连续样本必须来自不同 event / source fingerprint，其中至少一条从账号策略生成检索而不是依赖用户给链接。Topic Gate 和最终人审是预期人工判断，不扣自治。

来源不足、网页失效、事实争议或 freshness reversal 必须路由到缩小主张、改机制解释、重新检索、重选题或诚实等待；不得用 Image 2、旧截图或自由改写把缺证据伪装成已完成。只有最终交付闭合的热点样本增加 route 连续成功数。

#### 15.17.7 项目能力覆盖矩阵

项目 L3 不要求每个 session 都调用所有能力，但同一 `maturity_baseline_digest` 的证据组必须真实覆盖：

| 能力分支 | 项目级真实证据 |
|---|---|
| `direct_delivery` | 两条不同输入的连续 autonomous delivery |
| `hotspot_to_delivery` | 两条不同事件 / 来源的连续 autonomous delivery |
| `generated_context` | 至少一个内容确实需要的新视觉任务完成真实 Image 2 base generation、后工程、finalize 和看图 review |
| `source_bound_evidence` | 至少一个内容确实需要的来源完成真实检索 / capture、锚点后工程、语义一致性和 final binding |
| `explicit_existing_asset` | 至少一次精确授权、hash 绑定和无 provider 调用的合法复用 |
| `deterministic_postprocess` | 至少一次叠字 / 标注 / 裁切 / rendition 从 base 到 delivery asset 闭合 |
| `revision_resume` | 至少一次 scoped revision 或中断 reconcile 后重新闭合，不重复外部副作用 |
| `honest_waiting` | 至少一个负例证明缺来源 / 缺能力时停住，而不是临场手搓通过 |

这些是能力覆盖，不是图片数量、调用次数或内容时长配额。视觉任务数、Image 2 次数和 capture 次数仍从内容及逐 task evidence 派生。

#### 15.17.9 连续样本与基线的可编译性消歧

为防止事后只挑绿灯，新增 `autonomy_certification_cohort`。认证运行必须在开始前绑定 cohort、route 和 baseline；启动后无论成功、扶持或失败都进入有序 ledger，不能从报告中删除：

```yaml
cohort_id:
maturity_baseline_digest:
opened_at:
route_ledgers:
  direct_delivery: []
  hotspot_to_delivery: []
capability_coverage:
current_contract_blockers: []
```

顺序使用上游物化、带时区的 `run_started_at`，同一时刻再以 session ID 稳定排序。状态语义固定为：

| 认证运行状态 | 连续交付计数 |
|---|---|
| `autonomous_delivery` | +1 |
| `autonomous_waiting` 且仍可恢复 | 不增加、不清零；恢复后仍是同一 session |
| `assisted_delivery` | 清零 |
| `failed` / 被放弃的 waiting | 清零 |
| 非认证普通内容运行 | 不进入 cohort，不影响认证 ledger |

“不同输入”不是文件名不同：

```text
direct_input_fingerprint
  = account identity digest
  + original normalized body digest
  + intake mode

hotspot_input_fingerprint
  = account identity digest
  + event cluster digest
  + selected source set digest
```

同一正文只改 session ID、同一热点只换标题、同一 source set 改排序都不能算第二个样本。热点 route 的两条样本还必须使用不同 event cluster。

`current_contract_blockers` 只统计会影响当前直供 / 热点自治、视觉能力或交付闭合的 blocker；OneDrive、non-NTFS、外部 tester 等尚未声称为 L3 支持面的环境范围不混入业务自治 blocker，但公开发布仍按各自 release gate 管理。

两条连续样本是 L3“可发布候选”的最小重复证明，不是传播效果统计，也不代表 L4 跨账号稳定。L4 仍需跨账号、跨 session、更多异常样本和迁移证据。

