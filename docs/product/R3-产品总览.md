# R3 产品总览

> 状态：r3_visual_need_auto_dispatch_compiled
> 所属路线：R3 画中画与图片资产模型  
> 主责：把静态视觉编导、画中画、封面设计、提示词、图片生成、图片状态、HTML 嵌入和外部模型降级编成稳定资产链。
> 边界：R3 只解决静态图片资产，不做短视频生成、剪辑、视频镜头 storyboard、Seedance 视频 API、外部图片 API、自动发布或平台登录。

---

## 1. R3 要解决什么

R3 解决路线图中的 P05、P06、P17，并和 P07、P16 交界：

| 问题 | R3 处理方式 |
|---|---|
| P05 画中画逻辑不稳定 | 逐语义节点判断注意力、理解、证据、演示、情绪或记忆需求；图片数量由通过的视觉任务自然派生 |
| P06 图片资产链不健壮 | 定义 `visual_plan -> image_prompt_set -> image_generation_record -> image_asset_set -> html_embed_manifest` 的资产链 |
| P18 静态图片全是内容语言 | 增加 `static_visual_director_plan`，先把口播翻译成静态画面编导语言，再进入 prompt |
| P19 只有封面标题，没有封面设计 | 增加 `cover_design_package`，把封面标题、封面图、版式、安全区、平台差异和封面质检分开 |
| P20 画中画文字与字幕混为一层 | 增加 `visual_text_plan`，把口播、字幕、视觉文字拆成三条信息轨 |
| P21 图片追求堆信息且生成图可能冒充证据 | 用有效信息增量、认知负荷和来源绑定替代原始信息密度；生成情境图不得标为证据 |
| P22 封面候选只换标题，设计稿与成品缺少一致性证据 | 增加实质视觉变体、设计-渲染一致性和平台预览门禁 |
| P17 降级策略只是说明 | 定义 provider / input_schema / fallback_note，不实现 API |
| P07 / P16 最终交付产品化 | 给 final-delivery-builder 提供可预览、可下载、可追溯的图片资产 |

R3 不解决：

```text
单篇主链路自动推进：R1。
多篇 fan-out / fan-in 和断点恢复：R2。
GitHub 开源包、示例仓库、贡献规范：R4。
短视频生成、剪辑、动态镜头 storyboard、视频 API 调用：不在当前 taogeskill 范围。
```

## 2. R3 核心原则

```text
画中画不是装饰，是留存资产。
静态图片不是文案的插图翻译，而是静态视觉编导后的传播资产。
封面不是封面标题，也不是一张无字背景图；封面是“底图 + 标题文字 + 版式 + 平台适配”合成后的点击入口资产。
图片类型先分清：picture_in_picture_image 服务正文留存和理解，cover_image 服务点击和平台入口。
图片生产路径先分清：Codex 环境直接生成并嵌入 HTML；非 Codex 环境交付 Seedream 入参和可复制提示词，不假装有图。
Codex image / gpt-image-2 可以尝试直接生成含字封面，但稳定交付不能只赌模型文字渲染；封面成品必须有可检查的 `cover_composition`。
提示词不是图片资产，只是图片资产的来源之一。
最终 HTML 不能展示不存在的图片。
每张图必须知道为什么要做、放在哪、谁生成、是否成功、失败后怎么办。
图片数量是视觉需求分析的结果，不是按时长预分配的额度；允许 0 到 N 张，不设最低值和最高值。
Codex 内置 Image 2 对所有通过视觉需求门禁的任务直接生成，不设置成本或调用次数门禁。
```

## 3.1 图片类型与生产路径

R3 的静态图片先按用途分两类：

| image_asset_type | 中文 | 主要任务 | 进入位置 |
|---|---|---|---|
| `picture_in_picture_image` | 画中画图片 | 服务口播正文里的留存、解释、证据、情绪、信任 | `visual_plan` / `image_asset_set` / final HTML 的画中画区 |
| `cover_image` | 封面图片 | 服务平台点击、推荐流停留、搜索/转发第一眼理解 | `cover_design_package` / final HTML 的封面设计区 |

同一张图片可以被复用，但必须明确角色：

```text
一张画中画图可以作为 cover_image_source。
但它只有进入 cover_design_package 并通过 cover_quality_gate 后，才算封面设计完成。
```

R3 的图片生产路径分两类：

| environment_path | 适用环境 | 产物口径 | HTML 展示 |
|---|---|---|---|
| `codex_image2_render` | Codex 环境可调用内置 image 生成 | 直接生成 gpt-image-2 图片，写入 assets/images，生成 generation_record / metadata，并嵌入 HTML | 展示图片预览、下载、追溯链接 |
| `seedream_prompt_delivery` | 非 Codex 环境或不具备直接出图能力 | 不调用外部 API，按 Seedream 入参逻辑交付 prompt、负面词、画幅、模型建议、参考图说明 | 展示占位、可复制 prompt、外部生成任务说明 |

对应字段：

```yaml
image_generation_decision: render_now / deliver_prompt_only / manual_required
prompt_delivery_mode: html_copyable_prompt / prompt_card_md / external_model_payload
provider_mode: codex_builtin / external_api / manual_upload / not_available
provider: gpt-image-2 / seedream-4.0 / seedream-4.5 / seedream-5.0 / manual
```

硬规则：

```text
Codex 环境进入最终 HTML 时，required 图片优先 render_now，并落本地 asset。
非 Codex 环境不得要求用户重填选题、账号、文案或产品边界；必须直接交付 Seedream 可用入参。
deliver_prompt_only 不是失败，而是非 Codex 环境的正常降级交付。
pending_external 不能展示成 generated。
```

## 3.2 静态视觉编导层

本项目当前只做到静态图片交付，不生成短视频、不接视频 API、不做剪辑。因此本轮补的是“静态视觉编导层”，不是完整视频编导系统。

```text
文案语言：说什么、怎么说。
静态视觉编导语言：哪些静态图值得做、每张图承担什么传播任务、画面如何组织。
视觉语言：画面长什么样、风格是否统一。
模型语言：prompt / provider 入参怎么写。
平台语言：封面、标题、描述、标签怎么适配。
```

`static_visual_director_plan` 必须先回答：

```text
哪一张图服务首屏留存？
哪一张图服务解释抽象机制？
哪一张图服务证据 / 信任？
哪一张图可作为封面候选？
每张图的视觉角色是冲突图、证据图、解释图、情绪图、机制图，还是封面图？
这一组图是否像同一条内容里的资产，而不是三张无关素材？
```

最小字段：

```yaml
static_visual_director_plan:
  static_visual_director_plan_id:
  draft_id:
  brief_id:
  source_research_run_id:
  content_visual_mode: talking_head_static_assets
  non_goal:
    - video_generation
    - video_editing
    - seedance_api_call
  visual_role_map:
    - segment_id:
      script_range:
      visual_role: hook_conflict / comparison / mechanism_explainer / evidence_support / emotion_amplifier / scene_context / trust_builder / transition / memory_symbol / metaphor / cover_click / not_needed
      viewer_job:
      retention_risk:
      static_image_reason:
  visual_concept:
  composition_strategy:
  style_anchor:
  visual_text_plan_id:
  cover_candidate_policy:
  visual_consistency_rule:
  asset_task_list:
  director_plan_status: director_plan_pass / director_plan_needs_fix / director_plan_blocked
```

静态视觉编导层不直接写 prompt。它把“内容意思”转成“静态画面任务”，再交给 image_prompt_set。

### 3.2.1 产品草图：三条信息轨

```text
口播轨 narration_track
  负责观点、因果和叙事推进
          |
          +---- 字幕轨 subtitle_track
          |     负责忠实转写、无声观看和可访问性
          |
          +---- 视觉文字轨 visual_text_layer
                负责口播没有直接说出的角色、机制、证据、内心、语境和潜台词
```

三条轨道允许协同，但不能互相冒充：

```text
字幕不是画中画文字；字幕默认忠实复写口播。
视觉文字不是字幕；它必须产生可说明的 information_delta。
图片单独已经能准确完成任务时，visual_text_decision=forbidden。
缺少文字会让角色、关系、证据或比较轴含糊时，visual_text_decision=required。
```

`visual_text_plan` 的产品对象：

```yaml

  static_visual_director_plan_id:
  visual_plan_id:

  narration_source_path:

      visual_text_units:
        - visual_text_unit_id:

          visual_text_role: inner_voice / role_label / mechanism_label / comparison_label / evidence_label / data_point / context_marker / visual_narration / subtext / source_note

      task_information_delta_summary:

  next_skill:
```

这里不采用“文字越多、信息密度越高越好”。产品目标是：在用户可用的注意时间内，以尽量低的认知成本获得尽量高的有效信息增量。

视觉角色决策矩阵：

| 画面角色 | 默认文字决策 | 可用视觉文字 | 来源要求 |
|---|---|---|---|
| 情绪、场景、记忆、隐喻 | `forbidden / optional` | 内心、潜台词、语境标记；不得写长段解释 | 通常不要求；不得冒充事实 |
| 冲突、对比 | `optional / required` | 角色标签、对比轴、内心立场 | 出现事实数字时必须绑定来源 |
| 机制、流程 | `required`，除非图形关系已一眼可懂 | 节点、箭头、机制标签 | 涉及外部事实时绑定来源 |
| 数据、证据、引语 | `required` | 原生文字、数据点、来源说明、必要标注 | 必须 `source_bound` |
| 截图、凭证 | 保留 `source_native_text`，只做高亮和隐私处理 | 证据标签、来源说明 | 必须绑定原始资产，禁止生成替代 |

生成的车场、人物、账本等情境插图只能承担 `scene_context / metaphor / emotion_amplifier`；没有原始来源时不得标成 `evidence_support`。

对象生成顺序不是三个 Skill 互相等待，而是一个原子规划事务：先为 `visual_plan` 分配稳定的 `beat_id / image_task_id`，再按这些 ID 生成 `visual_text_tasks[]`，最后一起写入同一 session。任一对象写入失败，三者都不得标记 pass。

## 3.3 封面设计包

封面设计不能只交付 `cover_title`。成熟平台包装至少要区分：

```text
封面标题：封面上的短文字。
视频标题：平台标题区的完整标题。
封面底图：无字或少字的视觉背景，可由 Codex image 生成、视频帧截取、人工上传或复用画中画。
封面成品：底图 + 封面标题 + 版式 + 平台比例合成后的可上传图片。
封面设计：策略、图、字、版式、安全区、平台尺寸和质检的组合。
```

调研结论：

```text
YouTube 官方缩略图建议把图片、品牌和描述性文字组合起来，并保证字体易读。
小红书 / 笔记封面语境强调“封面要总结内容并吸引点击”，不是只放背景图。
视频号封面存在竖版 / 横版尺寸差异，跨平台不能假设一张图通用。
OpenAI image 生成模型可以按文本提示生成图片，gpt-image-2 在 Codex / API 中可用于复杂视觉任务和更强文字渲染；但含字封面的最终交付仍需小屏可读和文字准确质检。
```

`cover_design_package` 最小字段：

```yaml
cover_design_package:
  cover_design_package_id:
  source_package_id:
  source_visual_plan_id:
  platforms:
    - platform:
      cover_title:
      video_title:
      cover_image_source: existing_image_asset / generated_cover / video_frame / manual_required
      source_image_asset_id:
      image_asset_type: cover_image
      cover_visual_concept:
      cover_text_render_strategy: model_text_in_image / deterministic_overlay / manual_design / prompt_only
      cover_background_asset_id:
      cover_composited_asset_id:
      cover_composition_status: composition_ready / composition_needs_fix / composition_not_generated / prompt_only
      layout_spec:
        subject_position:
        text_position:
        subtitle_safe_area:
        face_safe_area:
        logo_safe_area:
      aspect_ratio:
      typography_note:
      text_overlay_max_chars:
      cover_prompt_id:
      cover_composition_prompt_id:
      platform_specific_reason:
      cover_quality_gate:
        small_screen_readability:
        title_visual_alignment:
        no_misleading_claim:
        no_privacy_risk:
```

第一版允许复用画中画图作为封面图，但必须明确：

```text
是否只是封面底图，还是已经是可上传封面成品。
是否需要二次裁切 / 加字。
是否需要单独生成封面图。
不同平台是否共用封面，还是平台差异化。
```

封面生产的默认产品逻辑：

```text
1. 先定 cover_job：停住划走 / 制造疑问 / 给结果 / 立观点。
2. 再定 cover_title：短、强、小屏可读，通常 6-14 个汉字为主，可分两行。
3. 再定 cover_background：用 Codex image 生成或复用图片，但尽量不要把长中文交给模型一次性画死。
4. 再定 cover_composition：用可控叠字、排版和裁切生成平台成品图。
5. 再进 cover_quality_gate：小屏可读、标题画面一致、无误导、无隐私风险、平台安全区通过。
```

关于 image 2 含字：

```text
允许：短字、强视觉、风格化海报、平台要求不高的封面候选。
不建议：长中文、多行复杂排版、必须逐字准确的最终上传封面。
默认：Codex image 生成 cover_background；最终成品用 deterministic_overlay / manual_design 合成。
如果使用 model_text_in_image，必须做文字准确和小屏可读检查；失败时不得进入 cover_design_pass。
```

关于平台适配：

```text
final-delivery.html 必须展示封面区，并区分“封面底图 / 成品封面 / 平台变体”。
如果只有封面底图，没有成品封面，HTML 必须标 `cover_composition_status=composition_not_generated`，不能说封面完成。
抖音 / 快手可以优先共用竖版强冲突封面。
小红书封面更偏图文笔记入口，可用解释型标题和更清楚的信息层级。
视频号可能需要不同裁切比例和更稳的标题，不能默认原封不动复用竖版封面。
跨平台共用封面必须给理由；平台入口差异明显时必须输出 platform_cover_asset。
```

封面资产角色统一进入 `image_asset_set.assets[]`，不创建第二套资产事实源：

```yaml
cover_asset_role: cover_background_asset / cover_composited_asset / platform_cover_asset
platform_cover_strategy: reuse / crop / retitle / independent_composition / prompt_only
```

职责边界：

| 对象 | 负责 | 不负责 |
|---|---|---|
| `cover_design_package` | 点击任务、视觉概念、标题、版式意图、平台策略 | 不冒充已渲染图片 |
| `cover_composition` | 叠字、裁切、安全区、导出和成品状态 | 不重新发明选题或视觉概念 |
| `image_asset_set` | 保存底图、成品图、平台图的资产事实和追溯 | 不决定传播策略 |
| `cover_quality_gate` | 文字准确、小屏可读、画面一致、平台适配和风险 | 不替代人工审美偏好 |
| `final-delivery.html` | 预览、复制、下载、状态与追溯 | 不成为图片事实源 |

统一字段规则：

```text
只保留 cover_text_render_strategy，不再使用 render_strategy。
只保留 cover_composition_status，不再使用 composition_status。
output_asset_id 必须指向 image_asset_set 中的 cover_composited_asset 或 platform_cover_asset。
只有 composition_ready + asset exists + quality gate pass 才能说“成品封面可上传”。
```

链路和 skill 所有权：

```text
platform-packaging-adapter -> platform_package / cover_variant_set
```

`cover-design-compiler` 是本轮新增推荐 skill。它插在 `platform_package` 之后、`final_delivery` 之前；封面专项返工不默认回滚热点、Brief 或口播。

## 4. 内容驱动的视觉需求模型

R3 不按时长给图片配额，也不把成本当成图片数量门禁。正式产品规则是：

```text
图片数量是文案视觉需求分析的结果。
允许 0 到 N 张，不设最低数量和最高数量。
visual_count_policy 固定为 content_derived_unbounded。
generation_policy 固定为 generate_all_accepted。
accepted_task_dispatch_policy 固定为 auto_continue_all_accepted_without_human_confirmation。
human_confirmation_required 固定为 false。
每张图必须证明其注意力、理解、证据、演示、情绪或记忆价值。
Codex 内置 Image 2 对所有通过任务直接生成，不设置成本或调用次数门禁。
```

`content_duration_estimate` 仍可用于时间轴定位和检查口播节奏，但不得映射成 `default_required_min / max`、`default_optional_min / max` 或任何图片数量包络。最终数量只能这样产生：

```text
draft
-> semantic_beats[]
-> visual_need_candidates[]
-> 逐候选 visual_need_gate
-> accepted_visual_tasks[] / rejected_visual_candidates[]
-> derived_visual_count = accepted_visual_tasks.length
-> pass 后自动进入 image-prompt-compiler -> image-asset-producer
```

这里的“通过”是机器 visual need gate 的结果，不是等待用户点击确认。H6A 和 H6B 只为事件记录、失败恢复和审计而拆成两个步骤；H6A pass 后必须无人工停顿进入 H6B。审美偏好在首轮出图后走版本化返工，不能阻止 accepted task 首次生成。

分析必须同时读取账号受众、受众既有知识、平台观看情境和文案表达方式。同一篇文案面对新手和专业用户，或者用于陌生流量 feed 与熟人 / 粉丝场景，可能得到不同的视觉需求数量；这种差异必须由 audience / context 证据解释，不能由账号预设图片配额解释。

### 4.1 视觉任务类型

每个候选任务必须有一个 `primary_visual_job`，允许再有多个 `supporting_visual_jobs[]`：

| visual_job | 业务含义 | 合法触发条件 |
|---|---|---|
| `attention_reset` | 重新聚焦、防止划走 | 当前段存在具体注意力流失风险，画面变化后能把注意力带回当前信息；不得仅因“已经几秒没换图”触发 |
| `hook_amplification` | 放大首屏冲突、悬念或反常识 | 视觉能立即建立正文真实兑现的冲突，不制造标题党 |
| `concept_explanation` | 帮助理解抽象、专业或关系复杂的内容 | 删除图片后，目标用户很难在口播时长内形成正确心智模型 |
| `evidence_support` | 建立证据和信任 | 使用真实数据、截图、凭证、来源或可验证对象；生成情境图不得冒充证据 |
| `process_demonstration` | 展示动作、流程、结构或前后变化 | “展示”明显比继续口述更清楚、更快或更可信 |
| `emotion_amplification` | 强化故事已有的情绪、处境和氛围 | 情绪与文案事件一致，不用无关震撼图替代真实叙事 |
| `memory_anchor` | 形成核心结论的记忆锚点 | 视觉符号能帮助观众记住关键判断，且不是机械复写字幕 |

一个任务可以同时承担多个作用，但不能用多个空泛标签代替一个明确主任务。

### 4.2 逐候选证明

`visual_need_candidate` 最小产品字段：

```yaml
visual_need_candidate_id:
beat_id:
covered_beat_ids: []
trigger_text:
insert_after_text:
insert_before_text:
viewer_problem_without_visual:
attention_risk_without_visual: low / medium / high
comprehension_risk_without_visual: low / medium / high
primary_visual_job:
supporting_visual_jobs: []
expected_viewer_change:
information_added:
why_image_is_better_than_talking_head:
emotion_congruence_status: aligned / not_applicable / misaligned
evidence_requirement: source_bound / generated_context_only / not_applicable
redundancy_status: unique / duplicates_narration / duplicates_other_visual
cognitive_load_risk: low / medium / high
misleading_risk: low / medium / high
visual_need_decision: generate / reject
decision_reason:
```

`beat_id` 是主语义节点，`covered_beat_ids[]` 允许一张图覆盖相邻的多个节点，防止为每句话重复出图。若全部候选均被拒绝、`derived_visual_count=0`，分析仍可通过，但必须填写 `zero_visual_reason`，证明 talking head、人物表情或口播本身已经完成当前传播任务。

核心反事实问题固定为：

```text
如果删掉这张图，观众具体会损失什么？
```

回答必须落到注意力、理解、证据、演示、情绪或记忆中的至少一项；只写“丰富画面”“更好看”“增加信息量”不得通过。

### 4.3 生成与拒绝门禁

`visual_need_decision=generate` 必须同时满足：

```text
存在明确 trigger_text 和插入窗口。
至少一个合法 visual_job，且 primary_visual_job 唯一。
viewer_problem_without_visual 能描述缺图的具体损失。
expected_viewer_change 能描述加图后的具体改善。
图片相对口播 / 字幕有信息、注意力、证据、演示、情绪或记忆增量。
不与其他图片重复，不造成 high cognitive_load_risk。
misleading_risk 不是 high；证据任务已 source_bound。
```

以下情况必须 `reject`：

```text
纯装饰或只为凑数量。
仅因为视频时长、固定秒数或已有画面持续了一段时间。
机械复写字幕或重复另一张图。
视觉复杂度会抢走当前口播的主要注意力。
生成情境图被当作真实证据。
情绪与文案不一致，或靠夸张画面制造正文没有的承诺。
无法说明删图后观众会损失什么。
```

通过门禁的任务不再分 `required / optional`。在 Codex 内置 Image 2 可用时，所有 `generate` 任务都进入 prompt 编译和真实生成；未通过任务保留拒绝证据，但不进入 provider。

### 4.4 数量、provider 与封面边界

```text
derived_visual_count = accepted_visual_tasks.length
actual_provider_execution_count = 实际发生的 Image 2 基础生成尝试数
```

两者都属于运行结果，不是规划上限。不得再使用 `expected_provider_call_count`、单次 / 总成本或最大调用次数阻断 Codex 内置 Image 2。provider 不可用、调用失败、结果未知和图片质量不通过仍按运行 / 质量门禁处理；这些是能力与质量问题，不是成本配额。

封面继续独立于画中画：封面解决点击入口，画中画解决正文观看过程。封面任务不得为了“凑总数”混入 `derived_visual_count`。

### 4.5 调研依据与适用边界

本模型吸收以下成熟研究，但不把教学视频或广告实验直接伪装成短视频平台的确定因果：

- Carney / Levin 的图片功能分类显示，解释、组织、再现和记忆型图片有明确功能，纯装饰图片几乎没有学习收益：https://doi.org/10.1023/A:1013176309260
- Mayer 的多媒体原则强调相关图文、coherence、signaling、temporal contiguity 和 segmenting，反对无关视觉与机械冗余：https://doi.org/10.1016/j.jarmac.2021.03.007
- seductive details 元分析显示，有趣但无关的视觉可能通过分散注意、增加工作记忆负担和破坏连贯性损害理解：https://doi.org/10.1016/j.edurev.2012.05.003
- 事件分段与连续性剪辑研究表明，观众按有意义的动作 / 事件边界理解叙事，而不是按固定秒数理解：https://pubmed.ncbi.nlm.nih.gov/21972849/
- 2026 年逐帧眼动研究表明场景切换可短时重新聚焦注意力，但切换收益递减，复杂画面也会分散注意：https://doi.org/10.1007/s11747-025-01137-x
- narrative transportation 元分析支持注意力、心理意象和情绪卷入共同影响叙事说服：https://papers.ssrn.com/abstract=2033192
- YouTube retention 的 peak / dip 是发布后诊断证据，peak 也可能代表没看懂而重看，不能反推固定图片配额：https://support.google.com/youtube/answer/9314415

发布后表现可以用于校准哪些 `visual_job` 有效，但不能沉淀成“每 N 秒必须一张图”的新魔法规则。

## 5. 图片资产状态

`image_status` 只能使用：

```text
generated
pending_external
generation_failed
manual_required
rejected
```

状态含义：

| 状态 | 含义 | 是否能进 HTML |
|---|---|---|
| `generated` | 已有实际图片文件，路径可读 | 能展示和下载 |
| `pending_external` | 已有完整 prompt，等外部工具生成 | 可展示占位和 prompt，不可假装有图 |
| `generation_failed` | 已尝试生成但失败 | 可展示失败原因和重试建议 |
| `manual_required` | 需要人工设计或人工上传 | 可展示任务说明 |
| `rejected` | 图片质检失败 | 不进入最终展示，只保留追溯 |

## 8. 静态视觉质量系统

R3 返修后，质量层必须从“资产存在检查”升级为“静态视觉质量系统”。质量系统分四个 gate：

| gate | 主责 | 解决的问题 |
|---|---|---|
| `static_visual_quality_gate` | 判断每张静态图是否完成视觉角色、留存任务、构图和一致性 | 防止图片只是内容解释图 |
| `visual_text_quality_gate` | 判断每张图的文字决策、信息增量、认知负荷、文字准确和来源绑定 | 防止字幕复写、堆字和伪证据 |
| `cover_quality_gate` | 判断封面是否可点、可读、不误导、适配平台 | 防止只有封面标题，没有封面设计 |
| `asset_trace_quality_gate` | 判断文件、prompt、生成记录、metadata、HTML 引用是否可追溯 | 防止有图但不可复盘 |

### 8.1 static_visual_quality_gate

最小评分：

```yaml
static_visual_quality_gate:
  image_asset_id:
  visual_role_fit_score: 1-10
  retention_task_score: 1-10
  composition_score: 1-10
  mobile_readability_score: 1-10
  style_consistency_score: 1-10
  misleading_risk: low / medium / high
  privacy_risk: low / medium / high
  quality_gate_status: pass / needs_fix / blocked
```

通过规则：

```text
核心 required 图片的 visual_role_fit_score、retention_task_score、mobile_readability_score 均不得低于 7。
misleading_risk 或 privacy_risk 为 high 时必须 blocked。
style_consistency_score 低于 7 时，不能作为成熟样本证据；可进入人工验收，但必须标 warning。
```

### 8.2 visual_text_quality_gate

逐图检查：

```text
是否存在一一对应的 visual_text_task。
forbidden 是否保持无字，required 是否实际落字。
文字是否相对口播 / 字幕产生信息增量。
手机扫读、认知负荷和文字准确是否通过。
证据文字单元是否逐项 source_bound。
生成情境图是否被错误标成 evidence_support。
```

对象内部使用 `pass / needs_fix / blocked`；跨 Skill 写入 `quality_review.visual_text_quality_gate_status` 时映射为 `pass / fail / not_applicable / not_run`。

### 8.3 cover_quality_gate

封面质量不只看标题。最小判断：

```text
封面标题是否短、强、可读。
封面主体是否一眼看懂。
文字是否遮挡主体、人脸或平台 UI 安全区。
封面和视频标题是否互补，而不是重复或冲突。
封面是否承诺正文没有讲到的价格、内幕、收益或渠道。
手机小图是否还能看清。
```

### 8.4 cover_variant_set

重要内容默认至少给主推荐；当内容有明显多种入口时，输出封面候选：

```yaml
cover_variant_set:
  cover_variant_set_id:
  variants:
    - variant_id:
      cover_visual_entry_type: conflict / evidence / emotion / method / trust / result
      cover_variant_difference_type: title_only / crop / visual_concept / subject / composition / evidence / emotion
      cover_title:
      cover_image_source:
      recommended_platforms:
      expected_click_reason:
      risk_note:
  recommended_variant_id:
  recommend_reason:
  materially_distinct_variant_count:
```

第一版不做自动 A/B 测试，但要把候选和推荐理由交付出来。

成熟变体补充规则：

```text
只换 cover_title 属于 title_only，不算新的视觉假设。
强观点、信息密度高或跨平台内容，至少交付 2 个非 title_only 的视觉入口。
视觉入口使用 cover_visual_entry_type：conflict / evidence / emotion / method / trust / result。
cover_quality_gate 必须检查设计合同是否真实渲染、手机缩略图是否可读、平台 UI 预览是否遮挡。
```

### 8.5 发布后数据回流待办

发布后数据回流不进入本轮 R3 产品开发和 skill 编译。

本轮边界：

```text
R3 到发布前交付结束。
不定义 post_publish_feedback_gate 为本轮必做对象。
不要求最终交付后再收集平台表现。
```

## 9. 成熟项目与 dbskill 对标结论

本轮对标后，R3 不应只定义“出几张图”，而要定义“图片作为可追溯资产如何被生产、记录、替换、嵌入和降级”。

| 参考对象 | 可借鉴点 | R3 吸收方式 |
|---|---|---|
| ComfyUI | 生成图可带 workflow metadata；也可另存人类可读 JSON，便于版本、归档和分享 | R3 增加 `metadata_sidecar_path`，不依赖图片文件一定保留元数据 |
| Gradio Flagging | 输入、输出和媒体文件分开保存，日志里记录文件路径 | R3 要求 `image_generation_record` 保存入参、输出、状态和文件路径 |
| MLflow Tracking | run 记录参数、时间、输出文件和 artifacts，方便追溯与复现 | R3 增加 `generation_run_id` 和 `generation_attempts[]` |
| Diffusers | 图片生成是明确的 pipeline input/output，不是随口一句 prompt | R3 强化 provider input schema、负面词、尺寸、seed / reference 等兼容字段 |
| dbskill | 结构先于规模、原始素材不改写、样本模式先跑通、交付的是能继续运转的系统 | R3 增加样本模式、资产不可覆盖、版本化和 source-of-truth 规则 |
| YouTube / B 站创作者资料 | 缩略图 / 封面和标题共同影响点击，且封面要小屏可读、主体明确、文字少而大 | R3 增加 `cover_design_package`，不再只交付封面标题 |
| Netflix Artwork / discovery assets | 视觉资产要表达内容本质、情绪和识别点，并按场景形成多版本资产 | R3 增加 `static_visual_director_plan` 和视觉一致性检查 |
| YouTube Test & Compare | 标题 / 缩略图可多版本测试，最终看观看表现 | R3 只吸收发布前 `cover_variant_set`；发布后表现回流进入 backlog，不进本轮开发 |
| 多媒体学习的 coherence / signaling / redundancy 原则 | 删除无关信息、突出必要结构、避免相同语言信息在多个通道机械重复 | R3 用 `information_delta`、`cognitive_load_status` 和三轨分工替代原始信息堆叠 |

R3 返修后的新增原则：

```text
HTML 不是图片事实源，只是交付展示层。
generated 图片必须有 sidecar 元数据；pending / failed / manual 图片必须有 generation_record 或人工任务说明，确保图片文件缺失时仍能知道它从哪里来。
图片资产不可覆盖；重做是一张新资产，旧图标记 superseded / rejected。
批量出图前先跑样本模式；样本不过，不进入多篇或多图批量。
不热链外部图片；最终 HTML 只引用 session 本地资产或诚实占位。
失败和待外部生成不是异常噪音，而是资产链必须记录的状态。
内容语言不能直接越级生成图片 prompt；必须先经过静态视觉编导，形成视觉角色、构图策略和封面候选。
图片质量不是审美评价，而是视觉角色、留存任务、小屏可读、平台适配、误导风险和可追溯性的综合门禁。
```
