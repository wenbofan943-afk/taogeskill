## 2. R3 核心原则

```text
画中画不是装饰，是留存资产。
提示词不是图片资产，只是图片资产的来源之一。
最终 HTML 不能展示不存在的图片。
每张图必须知道为什么要做、放在哪、谁生成、是否成功、失败后怎么办。
```

## 3. R3 标准资产链

```text
draft
-> visual_plan
-> image_prompt_set
-> image_generation_record
-> image_asset_set
-> image_quality_gate
-> html_embed_manifest
-> final_delivery
```

含义：

| 资产 | 说明 | 事实源 |
|---|---|---|
| `visual_plan` | 画中画策略、beat、留存任务、插入位置 | `intermediate/05-visual-plan.md` |
| `image_prompt_set` | 每张图的完整 prompt 卡和 provider 兼容字段 | `intermediate/05-visual-plan.md` |
| `image_generation_record` | 每次生成尝试、入参、provider、输出、失败原因 | `assets/images/generation-records/` |
| `image_asset_set` | 实际图片或降级状态、路径、provider、prompt_used | `assets/images/image-assets.md` |
| `image_quality_gate` | 图片是否能进最终交付，是否误导、是否可读 | `intermediate/06-quality-review.md` 或 `assets/images/image-assets.md` |
| `html_embed_manifest` | HTML 里每张图展示、下载、追溯方式 | `deliverables/content-delivery-record.md` / `final-delivery.html` |

## 4. 默认图片数量规则

R3 不允许“看心情出图”。每篇短视频口播默认视觉预算如下：

| 内容长度 / 结构 | 默认 required_visuals | optional_visuals | 说明 |
|---|---:|---:|---|
| 30 秒以内，单观点 | 1 | 1 | 优先首屏图，必要时补一个证据 / 转场图 |
| 30-60 秒，标准口播 | 2 | 1-2 | 首屏图 + 正文关键解释 / 证据图 |
| 60-90 秒，信息密度较高 | 3 | 1-2 | 首屏图 + 核心论证 + 产品 / 证据图 |
| 超 90 秒或多段结构 | 3-4 | 2 | 超出部分必须证明留存任务，不机械加图 |

必须增加图片的情况：

```text
前 5 秒需要停住划走。
口播出现抽象概念，缺图会听不懂。
需要证据或可信度支撑。
产品能力需要演示但不能虚构界面。
转场信息密度下降，需要节奏打断。
```

必须减少或删除图片的情况：

```text
图片只是好看但不承担留存任务。
图片会抢口播注意力。
图片暗示未实现能力。
图片和热点桥接太硬。
同一类视觉重复出现。
```

## 5. 图片资产状态

`image_status` 只能使用：

```text
generated
pending_external
generation_failed
manual_required
rejected
```

状态含义：

| 状态 | 含义 | 是否能进 HTML |
|---|---|---|
| `generated` | 已有实际图片文件，路径可读 | 能展示和下载 |
| `pending_external` | 已有完整 prompt，等外部工具生成 | 可展示占位和 prompt，不可假装有图 |
| `generation_failed` | 已尝试生成但失败 | 可展示失败原因和重试建议 |
| `manual_required` | 需要人工设计或人工上传 | 可展示任务说明 |
| `rejected` | 图片质检失败 | 不进入最终展示，只保留追溯 |

## 7. R3 成功标准

```text
一篇内容能解释为什么是 1/2/3/4 张图。
```

## 8. 成熟项目与 dbskill 对标结论

本轮对标后，R3 不应只定义“出几张图”，而要定义“图片作为可追溯资产如何被生产、记录、替换、嵌入和降级”。

| 参考对象 | 可借鉴点 | R3 吸收方式 |
|---|---|---|
| ComfyUI | 生成图可带 workflow metadata；也可另存人类可读 JSON，便于版本、归档和分享 | R3 增加 `metadata_sidecar_path`，不依赖图片文件一定保留元数据 |
| Gradio Flagging | 输入、输出和媒体文件分开保存，日志里记录文件路径 | R3 要求 `image_generation_record` 保存入参、输出、状态和文件路径 |
| MLflow Tracking | run 记录参数、时间、输出文件和 artifacts，方便追溯与复现 | R3 增加 `generation_run_id` 和 `generation_attempts[]` |
| Diffusers | 图片生成是明确的 pipeline input/output，不是随口一句 prompt | R3 强化 provider input schema、负面词、尺寸、seed / reference 等兼容字段 |
| dbskill | 结构先于规模、原始素材不改写、样本模式先跑通、交付的是能继续运转的系统 | R3 增加样本模式、资产不可覆盖、版本化和 source-of-truth 规则 |

R3 返修后的新增原则：

```text
HTML 不是图片事实源，只是交付展示层。
generated 图片必须有 sidecar 元数据；pending / failed / manual 图片必须有 generation_record 或人工任务说明，确保图片文件缺失时仍能知道它从哪里来。
图片资产不可覆盖；重做是一张新资产，旧图标记 superseded / rejected。
批量出图前先跑样本模式；样本不过，不进入多篇或多图批量。
不热链外部图片；最终 HTML 只引用 session 本地资产或诚实占位。
失败和待外部生成不是异常噪音，而是资产链必须记录的状态。
```
