# R3 产品确认清单

> 状态：r3_visual_text_confirmed_and_compiled
> 所属路线：R3 画中画与图片资产模型  
> 主责：保留 R3 进入规则 / skill 编译前的确认依据，并记录静态视觉编导、封面设计和封面成品合成的确认项。
> 边界：R3-C01 到 R3-C70 已完成确认和编译；当前实现只覆盖静态视觉规划、图片提示词、图片资产、视觉文字、封面与发布前 HTML，不扩展到视频生成、外部图片 API 或自动发布。

---

## R3-C01 范围确认

R3 只解决画中画和图片资产链，不解决 R1 单篇主链路、R2 运行模型或 R4 开源交付包。

确认口径：

```text
认可 R3 只做 visual_plan、image_prompt_set、image_generation_record、image_asset_set、metadata_sidecar、html_embed_manifest 和 fallback provider 兼容。
```

---

## R3-C02 图片数量规则

每篇短视频口播必须有默认视觉预算：

```text
30 秒以内：1 required + 1 optional。
30-60 秒：2 required + 1-2 optional。
60-90 秒：3 required + 1-2 optional。
90 秒以上：3-4 required，超出必须说明留存任务。
```

---

## R3-C03 首屏优先

首屏图优先服务前 5 秒，不允许平均分配视觉预算后忽视开头留存。

---

## R3-C04 留存任务

每张 required 图片必须有且只有一个主 `retention_task`。

---

## R3-C05 插入位置

每张图必须有 `insert_after_text` / `insert_before_text`，让 HTML 和剪辑都能定位。

---

## R3-C06 完整 prompt 卡

每个 `image_prompt` 必须保留完整 prompt 卡，不得缩水成短关键词。

---

## R3-C07 Provider 兼容

每个图片任务保留 provider 兼容字段，但当前不实现 Seedream 或任何外部 API。

---

## R3-C08 图片资产状态

单张图片状态必须区分：

```text
generated / pending_external / generation_failed / manual_required / rejected
```

整组图片资产状态必须和单图状态分开：

```text
all_generated / partially_generated / pending_external / generation_failed / manual_required / mixed / not_required
```

---

## R3-C09 generated 诚实性

`image_status=generated` 时必须有实际文件路径；否则不得写 generated。

---

## R3-C10 pending_external 展示

`pending_external` 可以进入 HTML，但只能显示占位、prompt 和插入位置，不能假装有图。

---

## R3-C11 图片质检

图片进入最终交付前必须检查小屏可读、语义贴合、产品边界、隐私、prompt 追溯、生成记录、metadata sidecar、本地资源和状态诚实。

---

## R3-C12 HTML 嵌入

final-delivery.html 必须能根据图片状态决定展示图片、占位、prompt 卡或隐藏 rejected。

---

## R3-C13 追溯链

最终 HTML 中每张图必须能追溯到 beat、prompt、generation_record、provider、asset_path、metadata_sidecar 和 quality_gate。

---

## R3-C14 降级不等于实现

Seedream / 外部模型只做兼容字段，不写成已接入能力。

---

## R3-C15 用户输入最小化

外部模型旁路未来实现时，用户最多只补 provider、model、API key / 手工上传、成本质量偏好；不重复输入选题、口播和提示词。

---

## R3-C16 执行透明度

图片生成、失败、外部待生成、人工上传必须写入 execution_trace，不能算作 skill 本身天然能力。

---

## R3-C17 编译前边界

R3 经用户确认前，不进入 `talking-head-image-pip`、`final-delivery-builder` 或字段词典编译。

---

## R3-C18 生成记录

每次图片生成、外部待生成、失败或人工上传，都必须形成 `image_generation_record`。

确认口径：

```text
prompt 卡说明“想生成什么”，generation_record 说明“实际尝试了什么”。
```

---

## R3-C19 sidecar 元数据

`generated` 图片必须有 `metadata_sidecar_path`，记录 prompt、provider、generation_record、checksum 和追溯路径。

---

## R3-C20 图片不可覆盖

图片重做、换模型、改 prompt、换尺寸或人工替换，都必须生成新的 `image_asset_id`；旧图只改状态，不删除、不覆盖。

---

## R3-C21 样本模式先行

R3 图片资产链先跑单篇最小图片资产链样本。样本可以只验证 1 张 required 图，但正式内容仍按默认视觉预算生成 required_visuals。样本未通过，不进入多篇批量出图或多图批量处理。

---

## R3-C22 本地资源优先

最终 HTML 只展示 session 本地图片资产。外部链接图片只能作为 reference_note，不得当成最终 generated 图片。

---

## R3-C23 HTML 不是事实源

`final-delivery.html` 是人类验收展示层，不是图片资产事实源。图片事实源是 `visual_plan`、`image_prompt_set`、`image_generation_record`、`image_asset_set` 和 `metadata_sidecar`。

---

## R3-C24 失败可恢复

`generation_failed` 必须写明失败原因、重试建议和是否需要人工动作；不能只留空图或空占位。

---

## R3-C25 多篇隔离

多篇内容并行时，每个 R2 child session 必须有独立 `image_asset_set`，fan-in 只汇总状态和路径，不混放不同作品图片。

---

## R3-C26 静态图片边界

R3 当前只做静态图片资产，不做短视频生成、剪辑、动态 storyboard、Seedance / Veo 等视频 API。

确认口径：

```text
认可 R3 补的是静态视觉编导层，不升级成视频生产系统。
```

---

## R3-C27 静态视觉编导方案

每篇内容在生成图片 prompt 前，必须先有 `static_visual_director_plan`，回答每张静态图的视觉角色、传播任务、构图策略、风格锚点和封面候选。

确认口径：

```text
不允许从文案句意直接跳到图片 prompt。
```

---

## R3-C28 视觉角色枚举

每张图必须有视觉角色：

```text
hook_conflict / mechanism_explainer / evidence_support / emotion_amplifier / trust_builder / cover_candidate / not_needed
```

`not_needed` 不生成图片，除非用户明确追加。

---

## R3-C29 视觉语言一致性

同一条内容的多张图片必须有统一 `style_anchor`，包括 medium、lighting、palette、texture、realism_level、people_policy 和 text_policy。

确认口径：

```text
不能生成三张互不相干的“素材库图片”。
```

---

## R3-C30 封面设计包

多平台包装必须输出 `cover_design_package`，不能只有 `cover_title_options`。

至少包含：

```text
cover_title
video_title
cover_image_source
source_image_asset_id
cover_visual_concept
layout_spec
aspect_ratio
typography_note
platform_specific_reason
cover_quality_gate
```

---

## R3-C31 封面来源诚实

封面图来源必须清楚：

```text
existing_image_asset / generated_cover / video_frame / manual_required
```

如果只给了封面标题，没有图和版式，不能写成封面设计完成。

---

## R3-C32 平台封面差异

不同平台可以共用封面图，但必须说明原因；如果平台目标不同，必须输出不同封面设计建议。

---

## R3-C33 封面质量门禁

封面进入最终 HTML 前必须检查：

```text
small_screen_readability
title_visual_alignment
no_misleading_claim
no_privacy_risk
platform_safe_area
```

任一阻断时，封面设计包不得标记为 pass。

---

## R3-C34 最终 HTML 展示封面设计

最终 HTML 必须展示封面设计包，而不只是平台标题文案。至少展示封面图来源、封面标题、视频标题、版式说明和追溯链接。

---

## R3-C35 skill 拆分边界

本轮产品层建议后续 skill 编译时优先拆分或重组：

```text
static-visual-director：静态视觉编导方案。
image-prompt-compiler：把静态视觉任务编译为 prompt。
image-asset-recorder：生成记录、sidecar、asset 状态。
cover-design-compiler：封面设计包。
```

第一版可以保持现有 skill 名称兼容，但必须在合同中分清职责，不再让 `talking-head-image-pip` 同时承担全部逻辑。

---

## R3-C36 静态视觉质量门禁

每张 required 图片必须有 `static_visual_quality_gate`，不能只检查文件是否存在。

最小评分：

```text
visual_role_fit_score
retention_task_score
composition_score
mobile_readability_score
style_consistency_score
prompt_alignment_score
misleading_risk
privacy_risk
```

---

## R3-C37 required 图片最低分

required 图片的 `visual_role_fit_score`、`retention_task_score`、`mobile_readability_score` 均不得低于 7。低于 7 时，不得作为成熟样本证据；严重时必须返工。

---

## R3-C38 封面候选集

重要内容或跨平台发布时，应输出 `cover_variant_set`。候选不是多写几个标题，而是不同视觉入口：

```text
conflict / evidence / emotion / method / trust
```

必须给 `recommended_variant_id` 和推荐理由。

---

## R3-C39 资产追溯质量门禁

`asset_trace_quality_gate` 必须检查 prompt、generation_record、metadata_sidecar、checksum、HTML embed 是否闭合。

确认口径：

```text
有图但不可追溯，不算成熟交付。
```

---

## R3-C40 发布后数据回流待办

本轮 R3 产品边界到发布前交付结束，不做发布后数据回流开发。

边界要求：

- 不定义 `post_publish_feedback_gate` 为本轮必做对象。
- 不进入字段词典编译。
- 不进入 skill 编译。
- 不自动登录平台。
- 不自动抓取后台数据。
- 不要求用户在本轮回填发布后表现。
- 仅在路线图 backlog 保留“发布后反馈 / 数据回流”增强项。

发布后数据回流未来可作为独立增强模块处理，用于人工记录封面点击、标题匹配、图片理解、平台表现和下一轮修订建议。

---

## R3-C41 图片类型二分

R3 图片必须先区分：

```text
picture_in_picture_image：画中画图片，服务正文留存、解释、证据、情绪和信任。
cover_image：封面图片，服务平台点击、推荐流停留、搜索 / 转发第一眼理解。
```

同一张图片可以复用，但只有进入 `cover_design_package` 并通过 `cover_quality_gate` 后，才算封面设计完成。

---

## R3-C42 图片生产路径二分

R3 图片生产路径必须先判断环境：

```text
codex_image2_render：Codex 环境可调用内置 image 生成，直接生成图片、写入 assets/images、整合进 HTML。
seedream_prompt_delivery：非 Codex 环境或无法出图时，按 Seedream 入参逻辑交付提示词和外部生成 payload。
```

该判断必须写入 `image_production_path`。

---

## R3-C43 Codex 环境交付口径

当 `image_production_path=codex_image2_render` 且进入最终 HTML 时：

```text
image_generation_decision=render_now
provider=gpt-image-2
provider_mode=codex_builtin
image_status=generated 或 generation_failed
generated 时必须有本地 asset_path、generation_record、metadata_sidecar 和 HTML 展示 / 下载。
```

---

## R3-C44 非 Codex 环境交付口径

当 `image_production_path=seedream_prompt_delivery` 时：

```text
image_generation_decision=deliver_prompt_only
provider=seedream-4.0 / seedream-4.5 / seedream-5.0
provider_mode=not_available 或 external_api_pending
prompt_delivery_mode=html_copyable_prompt / external_model_payload
image_status=pending_external / manual_required
```

HTML 必须展示可复制 prompt、负面词、画幅、模型建议、参考图说明和外部生成任务，不得展示成 generated 图片。

---

## R3-C45 用户输入最小化

非 Codex 环境交付 Seedream 入参时，用户不需要重复输入账号、选题、口播、产品边界或完整提示词。用户最多只需要选择：

```text
使用哪个 Seedream 版本。
是否手工上传生成结果。
是否接受成本 / 质量偏好。
```

---

## R3-C46 封面资产角色

封面底图、成品封面和平台封面不是三个新的顶层交接物，而是 `image_asset_set.assets[]` 中三种可追溯的封面资产角色：

```text
cover_background_asset：只负责主体、场景、情绪和视觉冲突，不等于可上传成品。
cover_composited_asset：底图、封面标题、版式和目标比例已经合成，可直接上传。
platform_cover_asset：针对某个平台独立裁切、改标题或重新合成的成品封面。
```

每个资产仍使用唯一 `image_asset_id`、generation record、metadata sidecar 和质量门禁，不另建平行资产事实源。

---

## R3-C47 创意与合成分工

封面合成不负责发明创意，也不能把封面降级为固定模板套字。产品链路必须分开：

```text
cover_design_package：决定点击任务、视觉概念、画面冲突、标题和平台策略。
cover_background_asset：承载创意画面。
cover_composition：执行文字、版式、裁切、安全区和导出。
cover_quality_gate：验证成品是否可读、准确、无误导并适配平台。
```

默认采用“生成模型探索视觉创意 + 可控合成保证交付稳定”的双层方案。

---

## R3-C48 含字封面策略

统一使用 `cover_text_render_strategy`，不得再并存 `render_strategy`。枚举为：

```text
model_text_in_image / deterministic_overlay / manual_design / prompt_only
```

默认正式交付走 `deterministic_overlay`；`model_text_in_image` 可用于短字、海报感候选，但必须检查逐字准确和手机小图可读。失败时自动降级到 `deterministic_overlay` 或 `manual_design`，不得把错误文字图标成成品。

---

## R3-C49 平台封面策略

跨平台交付必须对每个平台记录 `platform_cover_strategy`：

```text
reuse / crop / retitle / independent_composition / prompt_only
```

允许多个平台共用一张成品封面，但必须说明共用理由；裁切会破坏主体、安全区或标题层级时，必须生成独立 `platform_cover_asset`。平台差异是规则输入，不承诺算法效果。

---

## R3-C50 最终 HTML 封面合同

`final-delivery.html` 的封面区必须按平台展示：

```text
推荐封面成品预览和下载。
封面标题与视频标题分开。
平台封面策略及适配理由。
底图、成品图和平台变体的状态与追溯链接。
非 Codex / 无法合成时的可复制 prompt、版式和人工动作。
```

只有 `cover_composition_status=composition_ready` 且成品资产存在、质量门禁通过时，才能显示“成品封面可上传”。

---

## R3-C51 编译门禁

本轮产品确认后，按以下顺序进入编译：

```text
1. 交接物字段词典：注册资产角色、统一字段和状态枚举。
2. CONTRACT：明确 producer / consumer、输入、输出、失败和降级。
3. SKILL：编译环境判断、合成动作、平台变体和返工路由。
4. final-delivery 模板：展示封面成品、平台策略、下载和 prompt 降级。
5. checker / sample：验证字段一致、文字准确、小屏可读、状态诚实和本地链接。
```

产品确认不等于 skill 已实现；五层完成并通过静态检查后，才允许写 `cover_composition_compiled`。

---

## R3-C52 链路插入顺序

封面成品链不能和平台包装形成循环依赖。标准顺序固定为：

```text
quality_review（文案 / 初始静态视觉）
-> platform_package（标题、平台目标、封面候选策略）
-> cover_design_package（正式封面设计合同）
-> cover_composition（成品渲染或 prompt_only）
-> cover_quality_gate（封面专项复检）
-> final_delivery（汇总展示和下载）
```

封面专项复检只返工封面相关对象，不默认重跑文案、热点或整套平台包装。

---

## R3-C53 Skill 所有权

新增独立 `cover-design-compiler` 作为本轮推荐编译方案，避免继续扩大 `talking-head-image-pip` 和 `platform-packaging-adapter`：

```text
platform-packaging-adapter：产 platform_package 和 cover_variant_set，负责标题与平台策略。
talking-head-image-pip：产 static_visual_director_plan、visual_plan、prompt、generation record、image_asset_set 和封面底图资产。
cover-design-compiler：产 cover_design_package、cover_composition、合成成品和 prompt_only 降级包。
copywriting-quality-review：增加 cover_review 模式，产 cover_quality_gate，不承担合成。
final-delivery-builder：只展示、复制、下载和追溯，不生产或修图。
```

为兼容旧样例，`platform-packaging-adapter` 旧版 `cover_design_package` 输出可以读取，但新合同中不再作为该对象的长期 producer。

---

## R3-C54 三条信息轨

口播、字幕、视觉文字必须分轨。字幕负责忠实转写；视觉文字负责口播和字幕未表达的内心、角色、机制、比较、证据、语境或潜台词，不得把字幕截图式铺进画中画。

## R3-C55 visual_text_plan 所有权

`talking-head-image-pip` 继续作为用户入口和编排门面；`static-visual-director` 是 `static_visual_director_plan / visual_plan.required_visuals[] / visual_plan.optional_visuals[] / visual_text_plan` 的内部 producer。三者在同一原子规划事务中写入 `intermediate/05-visual-plan.md` 固定分区，并由稳定 ID 互相引用。第一版不允许运行时任选多个物理路径。

## R3-C56 文字决策枚举

`visual_plan.required_visuals[] / optional_visuals[]` 中每个实际图片任务必须恰好对应一个 `visual_text_tasks[]`，并逐图明确 `visual_text_decision=forbidden / optional / required`。`forbidden` 的文字单元必须为空；没有决策的图片不得进入生成或降级提示词。

## R3-C57 视觉文字角色

视觉文字只允许使用字段词典中的角色枚举；至少要说明文字内容、对象、位置、字数、渲染方式和是否需要来源。

## R3-C58 角色决策矩阵

情绪、场景、记忆和隐喻默认少字或无字；冲突与比较在角色含糊时加短标签；机制在关系无法一眼理解时加节点文字；证据、数据、引语和截图必须保留原生信息并绑定来源。

## R3-C59 信息增量与重复门禁

含字图必须写 `information_delta`。视觉文字只复写口播或字幕时，`visual_text_redundancy_status` 必须为 `needs_fix / blocked`，优先删字或重写为第二信息层。

## R3-C60 有效信息增量

R3 不以原始信息密度为质量目标。应综合 `semantic_delta_score`、`effective_information_gain`、`cognitive_load_status` 和小屏理解速度，防止用堆元素、堆文字制造“信息丰富”的假象。

单图默认 1-3 个文字单元，机制图最多 4 个节点，证据图可额外增加 1 个来源说明；核心增量、扫读和手机可读分默认不得低于 7，超预算必须写 `overflow_reason` 并复检。

## R3-C61 证据与来源完整性

数据、引语、截图、凭证和事实标签必须在对应文字单元内写 `is_source_required=true`、`evidence_source_type`、`evidence_source_id`、`evidence_source_path` 和 `source_binding_status=source_bound`。模型生成的情境图不能标为 `evidence_support`；应降级为 `scene_context / metaphor / emotion_amplifier` 或阻断。

## R3-C62 视觉文字渲染与降级

准确性优先时使用 `deterministic_overlay / source_native_text`；短字创意候选可用 `model_text_in_image`，但必须质检；非 Codex 环境使用同一 `visual_text_plan` 编译 Seedream prompt 和叠字说明，不要求用户重新输入业务信息。

## R3-C63 visual_text_quality_gate

新增视觉文字专项门禁，检查信息增量、口播重复、字幕重复、扫读理解、手机可读、认知负荷、来源绑定和文字准确。required 缺字、forbidden 出字、证据无来源或生成图冒充证据均阻断。

门禁对象内部使用 `pass / needs_fix / blocked`；写入 `quality_review.visual_text_quality_gate_status` 时固定映射为 `pass / fail / not_applicable / not_run`，确保现有路由可消费。

## R3-C64 封面视觉入口

每个封面候选必须声明 `cover_visual_entry_type=conflict / evidence / emotion / method / trust / result`，使封面创意不只依赖标题文案。旧 `variant_role` 只读迁移，不与 `cover_visual_entry_type` 并存为长期真源。

## R3-C65 实质封面变体

`title_only` 只表示换标题，不算新的视觉假设。强观点、信息密度高或跨平台内容至少应提供 2 个非 `title_only` 视觉入口；普通内容允许只有一个主推荐。

## R3-C66 设计与渲染一致性

`cover_contract_render_alignment_status` 检查设计包中的主体、色彩、版式、文字层级和视觉入口是否真实出现在成品里，不能只验证字段存在。

## R3-C67 平台预览证据

重要封面应记录 `platform_preview_status` 和可选 `platform_preview_evidence_path`，检查缩略图、裁切和平台 UI 遮挡；工具不可用时必须诚实标记 `unavailable / not_checked`。

## R3-C68 兼容迁移

旧视觉角色短名只读迁移到字段词典标准名；旧 `text_overlay_strategy / text_policy` 只能作为输入提示；旧 `variant_role` 迁移到 `cover_visual_entry_type`；历史样例中由 `talking-head-image-pip` 直接生产 generation record / image asset 仍可读取，新链路长期 producer 迁移为 `image-asset-producer`；新产物必须输出 `visual_text_plan_id` 和逐图 `visual_text_tasks[]`。

## R3-C69 Skill 编译影响

确认后按顺序编译：字段 schema / CONTRACT -> `talking-head-image-pip` 编排门面 -> `static-visual-director` 内部规划能力 -> `image-prompt-compiler` -> `image-asset-producer` -> `copywriting-quality-review` -> `platform-packaging-adapter` / `cover-design-compiler` -> `final-delivery-builder` -> 模板 / checker / 脱敏 sample。不得只改提示语而漏掉消费者和门禁。

Skill 边界固定为：

```text
talking-head-image-pip：用户入口、环境判断、阶段编排，不继续吞入详细视觉文字和 prompt 工艺。
static-visual-director：视觉角色、图片任务、逐图文字决策和来源要求。
image-prompt-compiler：把已确认任务编译为 Codex image / Seedream prompt，不重新决定内容策略。
image-asset-producer：执行环境判断、Codex 出图、Seedream prompt_only、画中画确定性叠字、生成记录、sidecar 和资产版本；不重写视觉策略。
copywriting-quality-review：分别执行 content_visual_review、visual_text_quality_gate、cover_review。
cover-design-compiler：只做封面设计合同、合成和平台预览证据。
```

编译前必须准备九个脱敏产品 fixture：

| fixture | 预期结果 |
|---|---|
| 纯场景图 | `forbidden`，无文字单元，仍可生成图片 |
| 内心想法图 | `optional`，使用 `inner_voice / subtext`，不得复写口播 |
| 冲突对比图 | 角色含糊时 `required`，使用 role / comparison label |
| 机制解释图 | 关系不清时 `required`，使用最小节点文字 |
| 来源证据图 | `required + source_bound`，保留原生证据与来源 |
| 生成图冒充证据 | `blocked`，降级视觉角色或更换真实来源 |
| 字幕不可用 | `subtitle_source_status=not_available`，仍完成 draft 重复检查 |
| 模型文字失败 | 从 `model_text_in_image` 降级 `deterministic_overlay / prompt_only` |
| 封面只换标题 | 标记 `title_only`，不计入 materially distinct variant |
