# R2 运行模型与分支封锁规则

> 状态：confirmed_compiled_dry_run_sampled  
> 所属路线：R2 没有稳定运行模型  
> 目标：定义 fan-out / fan-in、branch_lock、状态恢复、ID、索引和目录约束，让多篇内容和旁支任务不串台。  
> 边界：本文件是 R2 产品真源和编译后静态检查承载；不实现并发执行器、不写 validator 脚本、不做完整真实测试。

---

## 1. 核心对象

R2 引入轻量运行对象，不替代现有 session，而是补清关系。

| 对象 | 含义 | 典型路径 |
|---|---|---|
| `workflow_session` | 一次工作流对话或生产上下文 | `accounts/{account}/runs/{session_id}/` |
| `content_run` | 一篇独立内容的生产链路 | 默认与一个 session 一一对应 |
| `branch_request` | 用户提出多篇、旁支或跨任务请求 | 记录在 parent manifest / 工作流状态记录 |
| `parent_run` | 发起多选题或批处理的父运行 | 可是选题池 session |
| `child_run` | fan-out 后的单篇内容运行 | 每个 topic 一个 child session |
| `run_index` | fan-in 汇总索引 | `accounts/{account}/index.md`、`indexes/all_runs.md` |

R2 默认规则：

```text
content_run_id 可以先等于 session_id。
如果后续一个 session 承载多个轻量任务，再单独引入 content_run_id。
R2 产品层先定义字段，不强制改历史目录。
```

## 2. Fan-out 规则

### 2.1 触发

用户出现以下表达时，标记为 fan-out 候选：

```text
三篇都做。
都要。
这几个都跑。
T001/T002/T003 全做。
每个账号都来一条。
```

R2 必须先判断当前任务类型。

| 当前任务 | 处理 |
|---|---|
| 内容生产 / 选题池已生成 | 进入 fan-out 规划 |
| 产品开发 / 文档治理 | 标记 branch_request，先不启动内容生产 |
| 开源治理 | 标记 branch_request，要求用户明确切换任务 |

### 2.2 Fan-out 规划字段

父 session 必须记录：

```yaml
branch_request:
  branch_request_id:
  branch_mode: fan_out
  task_context_type:
  parent_session_id:
  source_artifact:
  requested_items:
    - topic_id:
      target_account:
      planned_child_session_id:
      child_run_status: run_planned / run_active / run_completed / run_blocked / run_archived
  branch_request_status: branch_request_none / branch_request_requested / branch_request_confirmed / branch_request_deferred / branch_request_cancelled / branch_request_completed
  fan_out_status: fan_out_none / fan_out_planned / fan_out_running / fan_out_completed / fan_out_blocked
  fan_in_artifact:
```

同时必须追加写入父 session 的分支台账：

```text
accounts/{account}/runs/{parent_session_id}/intermediate/branch-request-ledger.md
```

`manifest.yaml` 保存当前可恢复状态；`branch-request-ledger.md` 保存历史过程。两者冲突时，先读 manifest，再用台账核对状态变更来源。

### 2.3 Child session

每个 child session 必须：

```text
有独立 manifest.yaml。
有独立 intermediate/ 和 deliverables/。
只绑定一个 topic_id。
只生成一篇 draft / visual_plan / final_delivery。
记录 parent_session_id 和 branch_request_id。
```

禁止：

```text
child session 共用 draft_id。
child session 把正文写回 parent session。
parent session 合并多个完整 final-delivery.html。
```

### 2.4 Fan-out 边界情况

| 情况 | 状态 | 动作 |
|---|---|---|
| 用户多选为空或无法识别 topic_id | `branch_request_deferred` + `blocked_reason=missing_topic_id` | 不创建 child，输出可选 topic 列表 |
| 用户重复选择同一 topic_id | `branch_request_deferred` + `blocked_reason=needs_dedupe` | 去重后再规划，并在台账记录去重 |
| 用户跨账号多选 | `branch_request_deferred` + `blocked_reason=needs_account_confirm` | 每个账号必须先完成账号档案确认 |
| child session_id 分配失败 | `fan_out_blocked` | 不继续生成正文，记录失败原因 |
| 某个 topic_card 字段不完整 | `run_blocked` + `blocked_reason=topic_card_incomplete` | 只阻断该 child，不影响其他已规划 child |
| 用户在产品开发任务中说“三篇都做” | `branch_request_deferred` | 只登记旁支请求，不启动内容生产 |

## 3. Fan-in 规则

fan-in 只汇总，不合并正文。

允许生成：

```text
父 session 的 intermediate/branch-summary.md。
accounts/{account}/index.md 的多篇索引。
indexes/all_runs.md 的跨账号索引。
```

fan-in 摘要字段：

```yaml
fan_in_summary:
  branch_request_id:
  parent_session_id:
  child_count:
  completed_count:
  blocked_count:
  archived_count:
  child_runs:
    - child_session_id:
      topic_id:
      final_delivery_path:
      child_run_status:
```

禁止：

```text
把多个 child 的正文合成一个最终交付页。
把一个 child 的质量结论套到另一个 child。
把一个账号的素材挪到另一个账号。
```

## 4. Branch Lock 规则

### 4.1 任务类型

每轮必须有 `task_context_type`：

```text
product_development
content_production
workflow_governance
opensource_preparation
status_recovery
```

`task_context_type` 必须写入本轮 manifest 或 `工作流状态记录.md`。它不是装饰字段，而是决定本轮允许改哪些文件、遇到跨任务指令时是否要先生成 branch_request。

### 4.2 允许写入范围

| task_context_type | 允许写入 | 禁止写入 |
|---|---|---|
| `product_development` | `docs/product/`、必要时 `docs/reference/`、`STATUS.md`、`工作流状态记录.md` | `accounts/*/runs/*/intermediate/04-draft.md` 等生产正文 |
| `content_production` | 当前账号 session、账号 index、全局索引、工作流状态 | R1-R4 产品路线图、skill 合同 |
| `workflow_governance` | `AGENTS.md`、`PROJECT_MAP.md`、`docs/reference/`、状态记录 | 真实内容正文，除非是修索引 |
| `opensource_preparation` | README、PROJECT_MAP、examples 规划、开源清单 | 真实账号隐私和生产素材 |
| `status_recovery` | manifest、execution_trace、状态记录中的恢复字段 | 新内容正文和产品规则 |

如果用户指令跨任务，必须先形成 branch_request，而不是同时改两边。

## 5. 状态模型

### 5.1 Run 状态

R2 必须规范以下 run 状态：

```text
run_planned
run_active
run_waiting_human
run_completed
run_blocked
run_archived
run_discarded
```

R2 还必须补充分支状态：

```text
branch_request_requested
branch_request_confirmed
branch_request_deferred
branch_request_cancelled
branch_request_completed
fan_out_none
fan_out_planned
fan_out_running
fan_out_completed
fan_out_blocked
fan_in_none
fan_in_waiting_children
fan_in_ready
fan_in_completed
fan_in_blocked
```

状态命名规则：

```text
run_*：只描述单个 session / child_run 的运行状态。
branch_request_*：只描述用户多分支请求的处理状态。
fan_out_*：只描述 parent 拆 child 的展开状态。
fan_in_*：只描述 parent 汇总 child 的收口状态。
blocked_reason：用于解释阻断原因，不再新增临时状态名。
```

### 5.2 Stage 状态

每个主要阶段必须能记录：

```yaml
stages:
  research_run:
    status: not_started / active / completed / failed / skipped
    artifact_path:
    updated_at:
  topic_card:
    status:
    artifact_path:
  content_brief:
    status:
    artifact_path:
  draft:
    status:
    artifact_path:
  visual_plan:
    status:
    artifact_path:
  quality_review:
    status:
    artifact_path:
  platform_package:
    status:
    artifact_path:
  final_delivery:
    status:
    artifact_path:
```

R2 产品层要求定义 checkpoint 文件结构和写入时机，但不实现自动 checkpoint runner。

### 5.3 恢复字段

```yaml
recovery:
  last_completed_stage:
  current_stage:
  failed_stage:
  resume_from:
  resume_action:
  idempotency_policy:
  retry_policy_note:
  run_lock_status:
  updated_at:
```

`resume_action` 允许值：

```text
continue_next_stage
rerun_current_stage
manual_fix_required
postcheck_only
do_not_resume_archived
```

### 5.4 状态变更日志

每次改变 run / stage / branch 状态，必须追加 `state_transition`，不得只覆盖当前状态。

```yaml
state_transitions:
  - transition_id:
    at:
    actor_type: user / skill / agent / environment
    action:
    from_status:
    to_status:
    reason:
    input_artifact:
    output_artifact:
```

最低要求：

```text
选题确认、fan-out 规划、child 创建、child 完成、child 阻断、fan-in 汇总、断流恢复、归档，必须有 state_transition。
```

### 5.5 Checkpoint 快照

R2 产品层引入轻量 checkpoint，但不实现自动 runner。

路径：

```text
accounts/{account}/runs/{session_id}/intermediate/checkpoints/
```

命名：

```text
CKPT-{session_id}-{stage}-{YYYYMMDD-HHMMSS}.md
```

checkpoint 必须包含：

```yaml
checkpoint_id:
session_id:
content_run_id:
stage:
checkpoint_reason: stage_completed / human_interrupt / branch_created / error_before_retry / manual_snapshot
input_artifacts:
output_artifacts:
last_completed_stage:
current_stage:
resume_from:
resume_action:
state_transition_id:
created_at:
```

规则：

```text
每个主要 stage 完成后，必须生成 checkpoint。
每个人类停顿点和断流恢复前，必须生成 checkpoint。
checkpoint 是追加式快照，不覆盖旧快照。
恢复时优先读 manifest，再读最新 checkpoint，再读 state_transitions。
```

### 5.6 Human Interrupt Payload

用户决策必须落成统一输入包，不能只停留在聊天里。

```yaml
human_interrupt:
  interrupt_id:
  session_id:
  current_artifact:
  prompt_summary:
  allowed_decisions:
    - decision_type:
      example_reply:
      resulting_action:
  default_recommendation:
  created_at:
```

用户回复后写：

```yaml
human_decision_payload:
  decision_id:
  interrupt_id:
  raw_user_reply:
  decision_type:
  selected_items:
  rejected_items:
  target_artifact:
  resulting_action:
  state_updates:
  handled_at:
```

R2 负责的 `decision_type`：

```text
branch_request
branch_confirm
branch_cancel
resume
archive_parent_keep_children
archive_parent_archive_children
```

R1 已有的 `select / approve / edit / reject / archive / export / publish_record / cancel` 继续沿用；R2 只补多分支和恢复相关决策。

## 6. 幂等和重跑策略

R2 先定义产品口径，不实现脚本。

| 阶段 | 重跑策略 |
|---|---|
| research_run | 默认新建 research_run_id，不覆盖旧调研 |
| topic_card | 可追加候选，不覆盖已选择 topic |
| content_brief | 可生成新版本，但必须保留 source_topic_id |
| draft | 返工生成新 draft 版本，不覆盖已确认最终稿 |
| visual_plan | 可重做指定 beat，不影响其他 beat |
| quality_review | 可重跑，必须指向具体 draft_id + visual_plan_id |
| platform_package | 可重跑，必须指向具体 review_id |
| final_delivery | 可重建 HTML，但不得改变上游事实 |

## 7. Run Lock

R2 先做轻量 run lock 字段：

```yaml
run_lock:
  run_lock: run_lock_free / run_lock_acquired / run_lock_conflict / run_lock_released
  lock_owner:
  lock_owner_type: skill / agent / human / environment
  lock_reason:
  locked_at:
  stale_after:
  expected_release_action:
```

规则：

```text
同一 session 被内容生产占用时，产品开发不得改它的正文。
同一产品路线图正在修订时，内容生产不得反写产品规则。
发现 run_lock_conflict 时，不自动删除；先记录恢复建议。
```

锁状态动作：

| 输入 | 当前锁 | 动作 | 输出状态 |
|---|---|---|---|
| 新任务要写 session | `run_lock_free` | 写入 `run_lock_acquired` | `run_active` |
| 同任务继续写 | `run_lock_acquired` 且 owner 一致 | 允许继续 | 状态不变 |
| 不同任务要写 | `run_lock_acquired` 且 owner 不一致 | 阻断并生成 branch_request 或恢复建议 | `run_blocked` |
| 超过 stale_after | `run_lock_conflict` | 不自动删锁，输出人工恢复建议 | `run_blocked` |
| 任务正常完成 | 任意有效锁 | 释放为 `run_lock_free` 并写 state_transition | `run_completed / run_waiting_human` |

## 8. ID 规则

R2 继续沿用现有 ID，但补层级关系：

```text
session_id：一次运行目录。
branch_request_id：一次多分支请求。
content_run_id：一篇内容链路，默认等于 child session_id。
topic_id：选题。
artifact_id：brief / draft / visual / review / package / delivery。
```

多选 fan-out 时：

```text
parent_session_id = 选题池 session
child_session_id = SYYYYMMDD-{递增号}
content_run_id = child_session_id
```

禁止复用：

```text
同一 draft_id 对应多个 topic_id。
同一 final_delivery_id 对应多个 content_run_id。
同一 image_asset_id 跨账号复用为不同含义。
```

## 9. 索引规则

事实源：

```text
session manifest.yaml
```

索引只引用：

```text
accounts/{account}/index.md
indexes/all_runs.md
根目录汇总表
```

索引必须记录：

```text
session_id
content_run_id
account
topic_id
current_stage
final_delivery_path
run_status
updated_at
```

索引冲突处理：

```text
先读 session manifest。
以 session manifest 修正账号 index。
再修正 indexes/all_runs.md。
最后修正根目录汇总表。
```

### 9.1 索引重建动作

R2 产品层定义索引重建动作，不实现脚本。

| 动作 | 输入 | 输出 | 状态 |
|---|---|---|---|
| `rebuild_account_index` | 账号下所有 session manifest | `accounts/{account}/index.md` | `index_rebuilt` |
| `rebuild_all_runs_index` | 所有账号 index 或 session manifest | `indexes/all_runs.md` | `index_rebuilt` |
| `repair_root_summary` | all_runs + 根目录汇总表 | 根目录汇总表修正 | `summary_repaired` |

索引重建必须记录：

```yaml
index_rebuild_record:
  rebuild_id:
  scope: account / all_runs / root_summary
  input_count:
  changed_count:
  skipped_count:
  conflict_count:
  output_artifact:
  created_at:
```

## 10. Parent / Child 生命周期

R2 借鉴成熟 workflow 的 parent / child 关系，但只做轻量规则。

### 10.1 Parent Close Policy

父 session 关闭、归档或失败时，必须明确 child 怎么办。

```yaml
parent_close_policy:
  on_parent_completed: keep_children
  on_parent_archived: ask_human / keep_children / archive_children
  on_parent_cancelled: ask_human / cancel_planned_children / keep_active_children
  on_parent_failed: keep_children_and_mark_parent_blocked
```

默认策略：

| 父状态变化 | child 默认动作 | 说明 |
|---|---|---|
| parent completed | keep_children | 父完成只代表分支规划完成，不代表 child 都完成 |
| parent archived | ask_human | 归档父任务前必须问是否保留 child |
| parent cancelled | cancel_planned_children | 未启动 child 可取消，已启动 child 需人工决定 |
| parent failed | keep_children_and_mark_parent_blocked | 不因父任务失败自动删除 child |

禁止：

```text
父 session 归档时静默删除 child。
父 session 完成时把 child 当成全部完成。
child 失败时把 parent 直接标记为全部失败。
```

### 10.2 Child 完成对 Parent 的影响

| child 结果 | parent 更新 |
|---|---|
| child completed | `completed_count + 1` |
| child blocked | `blocked_count + 1`，parent 进入 `fan_out_running` |
| child archived | `archived_count + 1` |
| 所有 child completed / archived / blocked | parent 可进入 `fan_in_ready` |
| fan-in summary 生成 | parent 进入 `fan_in_completed` |

说明：

```text
child completed / blocked / archived 是人类可读描述。
正式状态字段必须写 `run_completed / run_blocked / run_archived`。
```

## 11. R2 操作合同

R2 编译时必须按操作合同走，不得靠临场判断。

| 操作 | 必需输入 | 必产输出 | 状态变化 | 下一动作 |
|---|---|---|---|---|
| `detect_branch_request` | 用户原话、当前 `task_context_type`、当前 artifact | `human_decision_payload` 或 `branch_request` | `branch_request_requested` | 判断是否可 fan-out |
| `plan_fan_out` | parent manifest、topic_id 列表、账号确认状态 | parent manifest + branch ledger | `fan_out_planned` | 创建 child session |
| `create_child_session` | branch_request、单个 topic_card、账号档案、对象档案 | child session 目录 + child manifest | `run_planned` | 进入单篇生产链 |
| `start_child_run` | child manifest、topic_card 完整性检查 | run_lock + checkpoint | `run_active` | 调用 R1 单篇链路 |
| `complete_child_run` | child final_delivery、quality_review、manifest | child state_transition + parent 计数更新 | `run_completed` | 判断 fan-in 是否 ready |
| `block_child_run` | 失败阶段、blocked_reason、checkpoint | child manifest + parent blocked_count | `run_blocked` | 给局部恢复建议 |
| `build_fan_in_summary` | 所有 child manifest | parent `intermediate/branch-summary.md` + indexes | `fan_in_completed` | 等人工验收或归档 |
| `resume_run` | manifest、最新 checkpoint、state_transitions、run_lock | resume plan | `run_active / run_blocked` | 从恢复点继续或停给人，原因写入 blocked_reason |
| `release_lock` | run_lock、state_transition | 更新 manifest | `run_lock_released` | 进入下一 stage 或等待 |

每个操作都必须回答：

```text
input_artifacts：吃了哪些文件。
output_artifacts：写了哪些文件。
state_transition：状态从什么到什么。
failure_mode：失败时停在哪里。
resume_hint：下次怎么接。
```

## 12. Branch Request Ledger

分支台账是追加式运行记录，解决“manifest 只看当前态，看不到过程”的问题。

路径：

```text
accounts/{account}/runs/{parent_session_id}/intermediate/branch-request-ledger.md
```

每条记录模板：

```yaml
ledger_entry:
  entry_id:
  branch_request_id:
  at:
  actor_type:
  action:
  input:
  output:
  from_status:
  to_status:
  affected_children:
  note:
```

必须登记的动作：

```text
branch_request_requested
fan_out_planned
child_session_created
child_started
child_completed
child_blocked
child_archived
fan_in_ready
fan_in_completed
parent_archived
resume_attempted
```

## 13. 恢复读取顺序

断流、接续或“活了吗”时，R2 恢复读取顺序：

```text
STATUS.md
-> 工作流状态记录.md
-> current_artifact
-> session manifest.yaml
-> 最新 checkpoint
-> state_transitions
-> branch-request-ledger.md
-> execution_trace
-> 实际 intermediate / deliverables 文件
```

恢复输出必须包含：

```yaml
resume_report:
  session_id:
  content_run_id:
  current_stage:
  last_completed_stage:
  latest_checkpoint:
  pending_human_interrupt:
  run_lock_status:
  recommended_resume_action:
  do_not_rerun:
  safe_to_rerun:
  blocked_reason:
```

禁止：

```text
只凭聊天记忆恢复。
重新跑已完成 child。
跳过 run_lock 直接改正文。
把无 checkpoint 的恢复说成脚本级断点续跑。
```

## 14. R2 编译前验收口径

R2 产品层进入编译前，必须满足：

```text
R2-C01 到 R2-C20 全部确认。
每个 R2 关键动作都有输入、输出、状态变化和失败处理。

```

R2 编译后只允许做：

```text

```

仍不允许：

```text

自动 resume runner。

```

## 15. R2 验收口径

R2 产品层通过标准：

```text
用户说“三篇都做”时，规则明确拆成多个 child session。
产品开发和内容生产有 branch_lock，不互相污染。
manifest 能表达 last_completed_stage、resume_from、failed_stage。
索引只做引用，不当事实源。
ID 层级能防止多篇串台。
状态字段能区分 active / waiting_human / completed / blocked / archived。
分支台账能解释 parent / child 每次状态变化。
checkpoint 能说明断流后从哪里恢复。
human_decision_payload 能把用户自然语言转成可执行动作。
```

R2 仍不要求：

```text
自动并发。
自动脚本恢复。
自动 validator。
真实多篇内容测试。
```

### 17.2 Dry-run 最小清单

Dry-run 使用假 topic / sample manifest，不生成真实内容正文，不调用图片生成。

|---|---|---|---|---|
| R2DR-001 | 内容生产多选 | parent session 有 T001 / T002 / T003，用户说“三篇都做” | `branch_request_requested -> branch_request_confirmed`，`fan_out_planned`，3 个 child 规划 | parent 不写正文；ledger 有记录 |
| R2DR-002 | 产品开发旁支封锁 | 当前 `task_context_type=product_development`，用户说“三篇都做” | `branch_request_deferred` | 不创建 child，不写内容正文 |

| R2DR-006 | run_lock 冲突 | 当前 session 已 `run_lock_acquired` 且 owner 不同 | `run_blocked + blocked_reason=lock_conflict` | 不继续写入 |

### 17.3 成熟产品对标

参考对象和取舍：

| 成熟产品 | 可借鉴点 | R2 已吸收 | R2 不照搬 |
|---|---|---|---|
| Temporal | 消息处理要考虑当前 workflow state、初始化、并发交错、幂等和 handler 完成；Update Validator 可在执行前拒绝不合法输入 | R2 引入 `human_decision_payload`、`run_lock`、`blocked_reason`、`resume_report` | 不做真正 worker、队列、SDK、exactly-once 执行 |
| Airflow | Dag Run / Task Instance 有明确状态；重跑任务会保留尝试历史；失败任务可以局部重跑 | R2 引入 stage 状态、state_transition、checkpoint、局部 child 阻断 | 不做调度器、DAG UI、任务清除命令 |
| Prefect | 运行态是 flow / task run 的属性，状态历史可用于恢复；强调成功、失败、重试和恢复 | R2 引入 manifest + state_transitions + latest_checkpoint | 不做结果缓存、自动 retry 引擎 |
| Dagster | partition / backfill 支持对多个分区或子集发起运行，适合批处理和局部补齐 | R2 将多个 topic 拆 child session，fan-in 只汇总索引 | 不做数据资产系统、分区调度、并发 backfill |
| dbskill | `/dbs` 双模式：任务前路由，任务后根据结果推荐 2-3 个下一步；接续靠 save / restore | `propagation-router` 承担总控，R2 恢复读 manifest / checkpoint / ledger | dbskill 偏导航和内容诊断，不承担本项目的 parent / child 运行模型 |

| 等级 | 问题 | 原因 | 建议 |
|---|---|---|---|

### 17.5 本轮结论

```text
R2 方向是对的：把“三篇都做”、断流恢复、分支封锁、最终交付收口，从 agent 临场编排变成运行模型。
这一轮最大价值：明确 parent / child、branch ledger、checkpoint、state_transition、run_lock、resume_report。
这一轮最大不足：还缺 R2 样本实物，不能证明另一个 AI 离开当前 agent 也能稳定执行。
下一步不建议直接进完整真实测试；建议先做 R2 dry-run 样本包。
```
