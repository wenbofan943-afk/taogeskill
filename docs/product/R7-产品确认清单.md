| R7-C23 | 普通内容运行缺浏览器能力时为 `not_tested + ready_with_warnings`；template / renderer 变更或 public release 缺真实视口证据时阻断。 |

| R7-C28 | 编译按 H1 blueprint / registry、H2 coordinator / submitter、H3 直供 producer adapters、H4 candidate compiler、H5 viewport / transparency、H6 热点入口与真实回归六批推进。 |
| R7-C29 | 直供原稿不能从 Brief 直接进入结构诊断；必须先确定性物化 `baseline draft`，再产生 `semantic_only content_beat_map`，结构诊断只能读取这两个已存在对象。 |
| R7-C30 | 结构选定后必须新建 `structure_bound content_beat_map` revision；不得在 `semantic_only` revision 上原地补 `structure_plan_ref / stage_id`。视觉、口播质检和最终交付只消费 current `structure_bound` revision。 |
| R7-C31 | task envelope、结构计划和 submission 只能引用生成任务时已经 materialized 且 hash 可验证的对象；“预计下一步会产生”的 draft / beat ID 属于 `future_artifact_reference`，必须在提交前阻断。 |
| R7-C32 | `direct_delivery_single_v0.1` 的错误顺序保留为历史合同，不原地改义；修复后新建不兼容 `direct_delivery_single_v0.2`，新 session 默认使用 v0.2，旧未完成 session 不自动迁移。 |
| R7-C33 | 同一 artifact type 的 current pointer 可由新 artifact / revision 前进，但 `output_revision` 必须由 artifact commit registry 指定的 payload revision 字段派生；不得固定为 1，也不得覆盖同 revision 的不同 submission。 |
| R7-C34 | producer adapter 必须声明直供阶段约束：`direct_baseline_draft.draft_mode=materialize_user_baseline`、`semantic_beat_map.mapping_phase=semantic_only`、正式 `content_beat_map.mapping_phase=structure_bound`；错相位在 submission build 前阻断。 |
| R7-C35 | 直供稿与热点稿的真正汇合点是“current draft + current structure plan + current structure_bound beat map”之后，不再宣称 Brief 之后顺序完全相同；两种入口可使用不同的前置节点。 |
| R7-C36 | 单线 blueprint 的实际成功链以版本化有序 `node_refs` 编译进 plan；同一 blueprint 内相邻节点、输入选择器和 task guidance 必须闭合，checker 不允许只看节点存在。 |
| R7-C37 | 一次真实 session 若因合同矛盾需要未来 ID、固定 revision 或手工 current pointer 才能继续，应记录为 `product_contract_defect` 并停止，不得计入自主完成；修复后必须新建 session 重跑。 |
| R7-C38 | H6 热点入口仍保持“结构先于生成稿”的顺序；若复用节点会与直供顺序冲突，必须使用入口专属前置节点，不能为了复用而改变 R6 的业务顺序。 |
| R7-C39 | `visual_insert_occurrence` 表示一次物理画面出现，不等于“被该画面覆盖的每一个 beat”；同一 occurrence ID 在最终 candidate 中只能登记一次。 |
| R7-C40 | 一次 occurrence 可以覆盖多个相邻 beat；其确定性 owner 是 `covered_beat_ids[]` 中按 current beat map 顺序最早的 beat。只有 owner beat 的 `occurrence_ids[]` 登记该 ID，其余覆盖 beat 只保留 visual task 引用。 |
| R7-C41 | 同一 current visual task 在相邻 beat 间持续展示不新增 occurrence；若在非相邻位置再次出现，必须拆成新的 occurrence，不能靠重复投影同一 ID 表示两次出现。 |
| R7-C42 | candidate compiler 必须验证 occurrence ID 唯一、task 存在、covered beats 非空且存在、无重复、按 beat map 连续，并且都属于该 task 的覆盖范围；失败归为 `candidate_occurrence_contract_error`。 |
| R7-C43 | visual insert card 的触发文本和 owner 绑定必须从 occurrence 自己的首个 covered beat 派生，不得从“该 task 的任意第一条 coverage record”猜测。 |
| R7-C44 | `content_beat_cards[].visual_task_ids[]` 表达语义覆盖关系，`occurrence_ids[]` 表达物理出现归属；两者不可再按 task ID 全量互推。 |
| R7-C45 | fixture 必须覆盖相邻多 beat 共用一次 occurrence 的合法路径，以及缺失 beat、非连续 beat、越出 task coverage 的拒绝路径；单 beat fixture 不能代替。 |
| R7-C46 | 真实 session 若因 occurrence 重复投影导致 deterministic candidate compile 失败，保留失败证据并修产品 / compiler；不得删除重复字段、手工拼 candidate 或在原 session 冒充成功，修复后新建 session 重跑。 |
