# R4 产品总览

> 状态：r4_environment_contract_confirmed_h2_compiled
> 所属路线：R4 开源交付标准  
> 主责：定义本项目 GitHub 开源上线前，怎样从本地工作母仓整理成“人能读、AI 能接、维护者能管、隐私不泄露”的开源包。  
> 边界：本文记录 R4 产品定义与当前确认口径；既有开源规则 / 包装已编译并发布 alpha.3，本轮只补环境兼容产品合同与后续编译门禁，不移动已发布 tag、不修改 Release、不直接修源码。

---

## 1. R4 解决什么

R4 只解决开源交付问题：

```text
别人下载后能不能知道这是什么。
人能不能按 README 找到入口。
AI 能不能按索引读到执行规则。
维护者能不能判断哪些文件能改、哪些不能改。
示例能不能复现关键链路。
真实账号、真实内容、图片和隐私会不会被误发到 GitHub。
```

R4 不解决：

```text
R1：方法论到 skill 合同。
R2：多分支运行模型。
R3：图片资产链。
完整真实内容测试。
自动发布、平台登录、外部图片 API。
```

---

## 2. 开源交付对象

本项目公开 GitHub 时，不直接发布当前工作母仓。

公开发布必须经过净化包：

```text
public_release/
```

`public_release/` 是发布候选，不是工作目录镜像。

---

## 3. 目标用户

| 用户 | 需要看到什么 |
|---|---|
| 创作者 / 运营者 | 这是一个 AI 内容工作流，怎么开始，能做什么，不能做什么 |
| 使用 Codex / AI agent 的人 | 入口文件、AGENTS、skill 合同、样例和状态恢复规则 |
| 维护者 | 文件边界、贡献规则、版本策略、发布检查、敏感内容排除 |
| 安全审阅者 | 不含真实账号隐私、API key、平台登录态、未授权素材 |
| 二次开发者 | skill 拆分关系、字段词典、sample run、validator / dry-run 口径 |

## 5. 必须排除

公开包不得包含：

```text
真实账号 runs。
真实账号未脱敏档案。
真实内容最终交付物。
真实图片、视频、音频。
平台 Cookie、登录态、API key、环境变量。
客户资料、聊天记录、录音、表单、电话号码、微信号、车牌、身份证。
外部资料 zip、缓存包、未授权第三方素材。
本机绝对路径作为发布依赖。
```

公开包可以包含：

```text
脱敏 sample account。
脱敏 sample run。
pending_external 图片占位样本。
少量人工构造的示例 HTML。
不含真实素材的 prompt / manifest / trace。
```

## 6. R4 资产分层

| 层级 | 作用 | 是否进公开包 |
|---|---|---|
| source_rules | README、AGENTS、PROJECT_MAP、字段词典、reference、skills | 是 |
| product_docs | R1-R4 产品定义、确认清单、路线图 | 是，但标清状态 |
| examples | sample account、sample run、dry-run 结果 | 是，必须脱敏 |
| templates | manifest、trace、final HTML、release checklist 模板 | 是 |
| local_working_data | 真实 accounts runs、真实图片、真实交付 HTML | 否 |
| external_cache | 外部资料包、zip、未授权素材 | 否 |

## 7. README 目标结构

公开 README 不应只堆文件索引，必须先回答：

```text
这是什么。
适合谁。
能做什么。
不能做什么。
最快怎么跑一个样例。
目录怎么读。
AI agent 应该先读什么。
当前成熟度。
如何贡献。
License 和安全边界。
```

推荐结构：

```text
1. 项目一句话
2. 能力边界
3. 快速开始
4. 示例：sample account / sample run
5. Workflow 概览
6. 目录结构
7. Skill 合集
8. 状态与成熟度
9. 安全与隐私
10. 贡献方式
```

---

## 8. 示例策略

开源包至少需要三个样例层：

| 样例 | 目的 | 来源 |
|---|---|---|
| sample-account | 展示账号档案应该长什么样 | 脱敏构造 |
| sample-run-minimal | 展示单篇内容链路的最小目录 | R1 / R2 / R3 dry-run 整理 |
| sample-final-delivery | 展示最终 HTML 怎么给人看 | 脱敏构造或教程样本 |

样例必须满足：

```text
能被 README 链接到。
能被 AI 按 PROJECT_MAP 找到。
不能包含真实账号、真实客户、真实图片或平台隐私。
manifest、trace、final HTML 链接闭合。
样例状态不能冒充完整真实测试。
```

## 9. 发布检查

R4 发布前必须有检查清单：

```text
README 入口完整。

docs / skills / examples 均被索引。
public_release 无真实账号 runs。
public_release 无 API key / Cookie / token。
public_release 无本机绝对路径依赖。
sample 链接闭合。

```

## 10. 成熟度目标

| 阶段 | R4 要求 |
|---|---|
| alpha | README、AGENTS、PROJECT_MAP、License、贡献说明、安全边界、脱敏 sample、发布检查清单 |
| beta | public_release 自动或半自动生成，示例可复现，链接和敏感信息检查可执行 |

R4 产品确认后，只能进入开源规则 / 包装编译；不得直接推 GitHub。

## 11. Windows 环境能力合同

环境兼容是公开产品能力的一部分，不是安装文档里的备注。公开版本必须按“能力轴”声明和验证，不能只写笼统的“支持 Windows”。

| 能力轴 | 产品合同 |
|---|---|
| PowerShell | Windows PowerShell 5.1 保留短路径兼容；PowerShell 7 为推荐宿主。两者必须分别实测，不能互相代替。 |
| 路径 | 空格与中文必须支持；5.1 安装根目录建议不超过 90 个字符。超过预算时 preflight 先阻断，不允许写到一半才失败。 |
| 进程调用 | 子进程参数必须逐项保真；含空格、中文、引号和空参数的 fixture 必须通过。不得依赖 `Start-Process -ArgumentList` 的数组外观判断安全。 |
| 编码 | 机器合同、JSON/JSONL、摘要输入和可哈希产物统一 UTF-8 无 BOM；脚本源码单独按宿主兼容策略处理。不得让 5.1/7 的默认编码改变语义或哈希。 |
| 压缩包 | 压缩 / 解压命令返回 0 只表示工具未报告错误；还必须核对 manifest、必需文件、数量和摘要，防止静默少文件。 |
| 依赖 | 校验默认 `-NoProfile`、离线、无用户模块前提；可选 YAML 模块缺失时走内置 fallback，不得在 checker 中静默 `Install-Module`。 |
| 执行策略 | 检测 Mark-of-the-Web、执行策略和 Group Policy；只给当前包的安全处理指引，不自动修改全局策略、注册表或系统安全设置。 |
| Git | `.gitattributes` 固定文本 / 二进制规则；机器解析 Git 路径优先 NUL 分隔；长路径能力按仓库 preflight，不修改用户全局 Git 配置。 |
| 文件系统 | 检测保留名、尾随空格 / 点、大小写碰撞、只读 / 锁文件和 reparse point 越界；路径必须归一化后仍在允许根目录。 |
| 地区与时间 | 机器字段使用 ISO 8601、显式时区和 invariant 格式；不得依赖系统区域设置解析数值、日期或排序。 |
| 入口与临时区 | 入口从 `PSScriptRoot` / 显式 root 定位，不依赖调用者 cwd；同卷临时文件、原子替换、可写性、空间与残留清理在副作用前检查。 |
| 平台变体 | OS build / edition、CPU architecture、filesystem 是支持轴；当前只验证 AMD64 Windows + NTFS，ARM64、Windows Server、非 NTFS 暂不认证。 |

### 11.1 当前支持口径

本轮推荐采用：

```yaml
windows_support_tier:
  recommended: powershell_7_short_or_medium_path
  compatible: windows_powershell_5_1_short_path
  path_root_recommendation_max_chars: 90
  whitespace_and_unicode_paths: required
  deep_path_without_preflight: unsupported
  network_share_and_onedrive_sync_root: not_certified
```

这里的“PowerShell 7 推荐”不是取消 5.1；是把 5.1 限定为经过短路径矩阵验证的兼容档。网络盘、OneDrive 同步目录、大小写敏感 NTFS、受企业 Group Policy 管理的主机，在有专项 fixture 前只能标记 `not_certified`，不得写成支持或不支持。

### 11.2 完成定义

环境兼容完成必须同时满足：

```text
source clean clone 与发布 zip 解压目录分别跑通。
PowerShell 5.1 与 7 分别有证据。
短路径、带空格中文路径、中等 / 超预算路径分别有预期结果。
无 powershell-yaml、-NoProfile、LongPathsEnabled=0 的基础环境可得到诚实结果。
压缩包解压后 manifest / required files / count / hash 完整。
失败按 invocation / path_budget / archive_integrity / encoding / policy / permission / dependency 分类。
报告明确 not_tested 轴，不能把未覆盖环境合并成 pass。
```

### 11.3 机器可读兼容信息

下一版 `public-manifest.yaml` 与环境报告至少投影同一个兼容对象：

```yaml
compatibility:
  contract_version:
  verified_at:
  evidence_ids: []
  recommended_host:
  compatible_hosts:
    - name:
      version_range:
      path_root_max_chars:
      status: verified / limited / not_certified
  path_features:
    whitespace:
    unicode:
    deep_path:
    unc:
  platform_axes:
    os_build:
    edition:
    architecture:
    filesystem:
  clean_room:
    no_profile:
    offline:
    optional_modules_absent:
  archive_integrity:
    manifest_required:
    extracted_hash_check:
  known_limits: []
  not_tested_axes: []
```

公开 README / INSTALL、public manifest、CI matrix 与 release gate 对同一字段有冲突时阻断发布。环境报告可以记录更多机器细节，但不得把用户名、绝对私有路径、token 或真实账号内容写进公开 manifest。

## 11. 成熟项目与 dbskill 对标返修

本轮 R4 第一版后，对标了 GitHub 社区健康文件、OpenSSF Scorecard、The Turing Way reproducible project template、Hugging Face model card / metadata，以及本地 dbskill 发布结构。

### 11.1 对标结论

| 参考对象 | 成熟做法 | R4 吸收 |
|---|---|---|
| GitHub community profile | README、CODE_OF_CONDUCT、LICENSE、CONTRIBUTING、SECURITY 等社区健康文件会影响贡献者判断 | R4 把这些文件设为 alpha 必备，而不是可选美化项 |
| OpenSSF Scorecard | 用自动检查给开源安全姿态打分 | R4 增加 release scorecard，至少先做人工 / 半自动检查 |
| The Turing Way reproducible project template | 用固定目录承载可复现项目、协作和研究对象 | R4 要求 sample、templates、manifest 和 source rules 分层，避免只给说明文档 |
| Hugging Face model card | README 兼具文本说明和机器可发现 metadata | R4 增加 `public-manifest.yaml`，公开包必须有机器可读能力边界和样例索引 |
| dbskill | 根 README 讲清安装、更新、工具箱、知识库、License；发布 zip 由 build 脚本按 skill 分组打包 | R4 增加 VERSION、build-public-release、skill 分组、主入口说明和发布产物边界 |
| dbs-content-system | scaffold/root、docs/quickstart、docs/acceptance、templates、tools 自包含 | R4 要求 examples / templates / tools 不只存在，还要能说明如何验证和交付到什么程度 |

### 11.2 返修后的 R4 重点

R4 不再只定义“开源包里放哪些文件”，而是定义五个可检查对象：

```text
public_manifest：机器可读的能力、边界、版本、样例索引。
release_checklist：人类发布前门禁。
public_readme：人类入口。
agent_entry：AI 入口，主要是 AGENTS + PROJECT_MAP。
build_public_release：后续编译阶段的净化包构建器。
```


### 11.4 dbskill 对 R4 的直接提醒

dbskill 的强项不是“文件多”，而是：

```text
根 README 先解决安装和使用。
主入口 skill 能导航。
单个重型 skill 自包含 scaffold / docs / templates / tools。
release workflow 不直接打散文件，而是构建可安装 zip。
VERSION 与 tag 一致。
```

本项目吸收为：

```text
R4 编译阶段应产出 public_release，不把工作母仓直接发布。
skills/ 可以保持源码形态，但 release 包需要说明不同宿主如何安装或复制。
示例要能告诉用户“做到什么算通过”，不能只给样例文件。
VERSION 和 CHANGELOG 必须进入 R4 alpha。
```
