## 1. 问题定位

本轮由一次真实直供稿运行暴露两个相互关联的缺口：

1. 口播稿已有 Hook、信息密度、核心机制等检查，但缺少完整的观点推进、证据位置、信息释放、情绪变化、口语可说性和逐处修改理由。
2. 视觉分析能判断“被提出的候选图是否值得做”，却不能证明整篇稿的所有重要语义都被分析过；因此可能只分析两处、生成一张合格图片，同时漏掉其他本可帮助理解或留人的段落。

根因不是 Image 2 能力，也不是应该规定“一篇必须几张图”，而是写作与视觉各自从全文临时抽取重点，没有共同、完整、可追溯的内容结构。

本轮产品定位为：

```text
先把全文拆清楚
-> 再判断怎么讲
-> 再判断哪里需要视觉、哪里故意不需要
-> 最后检查当前文案与当前视觉是否仍然一致
```

## 2. 调研结论与本项目取舍

| 成熟做法 / 研究 | 可借鉴原则 | 本项目取舍 |
|---|---|---|
| [Descript Scenes](https://help.descript.com/hc/en-us/articles/10248939749517-Scenes-overview) | 以脚本场景为单位组织画面、文字和素材层，并允许局部编辑。 | 建立稳定内容节点；视觉不再从全文自由跳读。 |
| [Adobe Premiere Text-Based Editing](https://helpx.adobe.com/uk/premiere/desktop/edit-projects/edit-video-using-text-based-editing/overview-of-text-based-editing.html) | 文本、时间码与视频片段保持绑定，文本编辑能映射到具体片段。 | 拍摄前先绑定精确原文范围；已有录音 / 成片时再追加时间码，不把时间码强加给纯文案。 |
| [CapCut AI Storyboard](https://www.capcut.com/tools/ai-storyboard-generator) | 脚本先转成可编辑的场景 / 镜头计划，而不是直接黑盒出片。 | 最终交付展示“怎么讲、哪里用图、为什么不用”的可编辑工作台。 |
| [TeViS](https://arxiv.org/abs/2301.00135) | 按语义紧凑单元选择数量可变的关键帧，同时考虑相关性和连贯性。 | 图片数保持 0 到 N；一张图可覆盖多个相邻节点，不按句子机械配图。 |
| [VideoDirectorGPT](https://arxiv.org/abs/2309.15091) | 先形成明确视频计划，记录场景、实体、布局、背景和一致性关系。 | 视觉任务先有传播职责和展示方式，再派生 prompt、比例和构图。 |
| [MCSC-Bench](https://arxiv.org/abs/2604.15127) | 可执行脚本需要筛选相关材料、补足叙事缺口，并提供镜头对齐的口播与拍摄说明。 | 文案和视觉共用语义节点，但各自保留独立 producer、门禁和返工责任。 |
| [PaperTok](https://arxiv.org/abs/2601.18218) | 生成脚本与视听 storyboard 后仍需人的领域判断、叙事声音和细粒度控制。 | 系统给具体建议，不替创作者无授权重写立场；重大改意必须交人。 |
| [视觉叙事评估研究](https://aclanthology.org/2024.findings-emnlp.679/) | grounding、连贯性和重复不能由单一自动分数可靠覆盖。 | 确定性 checker 只判结构闭合；审美、叙事和视觉价值由 Codex / 人实际审阅并留理由。 |

对中文短视频的补充判断：叙事聚焦、受众关系、情绪共鸣和账号类型之间存在条件关系，不能把某种高情绪、高密度或固定结构当成所有账号的万能公式。因此本项目只定义必须回答的问题，不规定固定三幕式、固定时长、固定情绪曲线或统一合格分。

## 3. 共同事实源：`content_beat_map`

`content_beat_map` 是口播设计、事实核验、视觉分析和最终交付共同读取的全文结构真源。它不是分镜，也不是简单按段落切块。

### 3.1 内容节点的业务定义

`content_beat_map` 顶层合同：

```yaml
content_beat_map:
  beat_map_id:
  beat_map_revision:
  mapping_phase: semantic_only / structure_bound
  draft_id:
  draft_revision:
  draft_digest:
  normalized_body_sha256:
  structure_plan_id:       # structure_bound 必填
  structure_plan_revision: # structure_bound 必填
  beats: []
  full_coverage_status: complete / incomplete
  unresolved_ranges: []
  mapping_status: ready / blocked / stale
  next_skill:
```

一个 `content_beat` 表示“观众在这里新增理解、感受、判断或行动意愿的一步”。每个节点至少回答：

```yaml
beat_id:
source_text_anchor:
  normalized_body_sha256:
  start_utf8_byte_offset:   # inclusive
  end_utf8_byte_offset:     # exclusive
  exact_text_digest:
  exact_text_preview:
beat_role: hook / claim / mechanism / evidence / example / counterargument / transition / payoff / action / other
structure_stage_id:
audience_state_before:
audience_state_after:
information_delta:
narrative_dependency_ids: []
claim_ids: []
evidence_status:
stance_and_certainty:
emotion_function:
spoken_delivery_intent:
importance: core / supporting / connective
```

`beat_role` 是描述当前功能，不是强制模板。一篇内容可以没有反方、案例或行动号召，但缺少的角色若会使核心承诺无法兑现，审查必须指出；不需要时写 `not_applicable + reason`，不能硬补套路。

### 3.2 全文覆盖规则

- `normalized_body` 只做 UTF-8 无 BOM、换行统一为 LF；保留正文其他字符，不改全半角、标点、数字或措辞。
- 当前稿件所有非空正文必须按顺序落入一个且仅一个节点；不得出现未解释的空白区或交叉覆盖。
- source anchor 使用 normalized body 的 UTF-8 byte 半开区间 `[start, end)`；相邻 beat 允许只留下空白字节间隙，所有非空白字节必须且只能被覆盖一次。
- checker 通过 byte range 提取原文并复算 `exact_text_digest`；文本预览只供人阅读，不能作为机器定位主键。
- 标题备选、平台描述、来源引用和创作备注不混入正文节点；它们用独立对象引用节点。
- 节点可以包含一到多句，不能仅按标点机械拆分。
- 节点图绑定 `draft_id + draft_revision + draft_digest`。正文发生语义变化后，旧节点图立即为 `stale`。
- 只改空白、换行或确定性排版时可以保持节点 ID；改动主张、顺序、立场或信息增量时必须新建节点 revision。

### 3.3 与现有 `segment_map / semantic_beats` 的关系

```text
content_beat_map：全文内容语义真源
segment_map：口播写作层对节点的组织与表达视图
visual_need_analysis.semantic_beats：视觉层对同一节点的消费视图
```

编译后不允许三套对象各自重新切全文。历史 `segment_map / semantic_beats` 只按兼容矩阵 replay；新运行必须引用相同 `beat_id` 集合。

为避免直供稿形成“先有结构才能建 beat、先有 beat 才能诊断结构”的循环，`content_beat_map.mapping_phase` 分两态：

```text
semantic_only：只还原原稿语义，structure_plan_id / structure_stage_id 为空；仅供 diagnose_existing_draft 消费。
structure_bound：绑定已选 structure plan；每个 beat 恰有一个 structure_stage_id，才可进入口播质检、视觉和最终交付。
```

热点 / 新写稿在结构已选后直接产生 `structure_bound` beat map。直供稿先产生 `semantic_only` revision，结构选择完成后再产生新的 `structure_bound` revision；不得原地补字段冒充同一 revision。

### 3.4 对象所有权与固定路径

| 对象 | 计划 producer | 主要 consumer | 机器真源固定路径 | 人类可读投影 |
|---|---|---|---|---|
| `short_video_structure_plan` | 新 `short-video-structure-planner` | draft writer、beat mapper、spoken script review、final delivery | `intermediate/contracts/short-video-structure-plan.current.json` | `intermediate/03a-short-video-structure.md` |
| `content_beat_map` | 新 `content-beat-mapper` | structure planner、spoken script review、R3 static visual director、claim / evidence、final delivery | `intermediate/contracts/content-beat-map.current.json` | `intermediate/04a-content-beat-map.md` |
| `script_design_review` | 新 `spoken-script-review` | decision policy / human gate、R3 static visual director、final delivery | `intermediate/contracts/script-design-review.current.json` | `intermediate/04b-script-design-review.md` |
| `content_revision_decision` | policy / Codex content review / 人类 evidence command | script readiness projection、draft writer、R3、final delivery | `intermediate/contracts/content-revision-decision.current.json` | `intermediate/04b-script-design-review.md` 决策分区 |
| `visual_coverage_ledger` | `static-visual-director` | prompt / evidence producer、joint quality review、final delivery | `intermediate/contracts/visual-coverage-ledger.current.json` | `intermediate/05-visual-plan.md` 固定分区 |
| `script_visual_alignment_review` | `copywriting-quality-review` 的联合审查模式 | platform package、cover、final delivery | `intermediate/contracts/script-visual-alignment-review.current.json` | `intermediate/06-quality-review.md` 固定分区 |

每个机器对象的 immutable revision 写入 `intermediate/contracts/revisions/{object_type}/{artifact_id}.json`；对应 `.current.json` 只保存当前 artifact ID、revision path、hash、status、source draft digest 和更新时间。producer 先写 revision、校验并生成 Markdown 投影，最后原子替换 current pointer；下游只读取 pointer 指向且 hash / digest 闭合的 `ready / ready_with_warnings` revision。Markdown 是可读投影，不能反向覆盖 JSON 真源。跨对象不宣称文件系统事务，consumer 以 IDs、digest 和 current pointer 逐项验证。

## 4. 短视频结构设计：`short_video_structure_plan`

### 4.1 产品定义

短视频结构不是固定模板，也不是把全文切成更多小段。它回答的是：为了让当前受众从进入内容时的状态，走到内容结束时的目标状态，这篇内容应按什么宏观顺序推进。

本对象只负责内容 / 叙事结构，不等于镜头脚本、拍摄 shot list、动态 storyboard、剪辑时间线、字幕时间轴或视频生成参数；这些能力仍不在当前 taogeskill 范围。已有录音 / 视频时可以把时间码作为节点补充证据，但不能反向把产品扩成自动剪辑系统。

```text
short_video_structure_plan：宏观叙事路线与阶段义务
content_beat_map：当前稿件中每一步具体的观众变化
segment_map：口播表达层怎样组织句段
visual_coverage_ledger：每个具体节点怎样视觉呈现
```

结构计划属于写稿前 / 改稿前的目标合同；节点图属于当前稿件写完后的现实映射。两者必须通过 `script_design_review.structure_realization` 对照，不能互相冒充。

### 4.2 两种入口模式

| `plan_mode` | 适用入口 | 正确顺序 | 不能做什么 |
|---|---|---|---|
| `design_before_draft` | 当前为热点选题生成稿；未来其他无原稿入口须先注册合法 `content_origin` | Brief -> 结构计划 -> draft -> beat map -> 实现审查 | 不能先写完再反向伪造“原本就是这样设计”，也不能临时发明 content origin。 |
| `diagnose_existing_draft` | 所有用户直供原稿，不论是否允许重写 | 原稿 -> beat map -> 现有结构诊断 -> 保留 / 局修 / 重排 / 重构建议 | 不能因为已获润色授权就跳过原稿结构还原。 |

直供稿诊断必须始终保留 `keep_current` 候选。若推荐动作只影响局部表达，可以形成可追溯候选 revision；若涉及重排、重构、事实强度、立场或行动主张，必须先取得当前 revision 的人类授权。

### 4.3 结构策略不是封闭模板

结构通过版本化 `structure_strategy_ref` 引用可扩展策略注册表。首期内置策略只作为可复用起点：

| 策略 ID | 典型推进 | 适合材料 |
|---|---|---|
| `judgment_mechanism_action` | 判断 -> 反方 / 疑问 -> 底层机制 -> 现实表现 -> 建议 | 强观点、行业判断 |
| `event_fact_impact_mechanism_stance` | 事件 -> 可核验事实 -> 影响对象 -> 行业机制 -> 创作者判断 | 热点、新闻、风险事件 |
| `misconception_mechanism_example_method` | 常见误解 -> 正确机制 -> 例子 -> 判断方法 | 知识解释、避坑 |
| `experience_conflict_turn_insight` | 场景 -> 冲突 -> 转折 -> 得失 -> 可迁移经验 | 亲历、案例、复盘 |
| `question_answer_reason_action` | 问题 -> 直接回答 -> 理由 -> 边界 -> 行动 | 高频问答、结论型内容 |
| `comparison_criteria_tradeoff_recommendation` | 选择问题 -> 标准 -> 对比 -> 取舍 -> 建议 | 车型、方案、方法比较 |
| `risk_signal_evidence_mechanism_response` | 风险信号 -> 证据 -> 机制 -> 影响 -> 应对 | 暴雷、合规、经营风险 |
| `custom` | 由当前内容定义 | 内置策略均不适配时 |

注册表可新增、弃用和版本化，不能把首期列表编成永远封闭的 enum。`custom` 必须给出自定义阶段、选择理由和与现有策略不适配的原因，不能成为无结构的逃生口。

### 4.4 结构计划对象

```yaml
short_video_structure_plan:
  structure_plan_id:
  structure_plan_revision:
  plan_mode: design_before_draft / diagnose_existing_draft
  brief_id:
  content_source_id:
  content_origin:
  account_snapshot_ref:
  target_audience:
  audience_entry_state:
  audience_exit_state:
  core_promise:
  core_point:
  material_inventory:
  evidence_availability:
  account_voice_constraints:
  platform_viewing_context:
  depth_reasoning:
  strategy_registry_version:
  source_draft_ref:       # 仅 diagnose_existing_draft 必填
  source_beat_map_ref:    # 仅 diagnose_existing_draft 必填
  alternatives_considered:
    - candidate_id:
      strategy_ref:
      transformation_level: keep_current / local_repair / reorder / rebuild
      stage_summary: []
      expected_benefit:
      tradeoffs:
      semantic_impact: none / local_expression / order / claim_strength / stance / core_structure
      requires_human_confirmation:
  recommended_candidate_id:
  selected_candidate_id:
  selected_strategy_ref:
  transformation_level: keep_current / local_repair / reorder / rebuild
  stages:
    - stage_id:
      order:
      stage_function:
      stage_purpose:
      audience_state_before:
      audience_state_after:
      content_obligation:
      claim_or_evidence_requirements: []
      transition_contract:
      necessity: required / optional_by_design
  selection_reason:
  selection_decision_id:
  authorization_ref:
  selection_status: auto_selected / human_selected / waiting_human / blocked
  plan_status: pending_selection / ready / ready_with_warnings / blocked / stale
  next_skill:
```

`stage_function` 使用版本化注册表，可包含 entry、orientation、claim、problem、evidence、mechanism、contrast、example、transition、payoff、action 等；一篇内容不要求全部出现。`necessity=required` 表示对当前选定路线不可缺，不是全项目硬模板。

`alternatives_considered[]` 和 `stages[]` 都是内容派生的 1 到 N，不规定固定 3 个候选或固定阶段数，也不设产品上限。直供模式的 alternatives 至少包含 `keep_current`；每个 selected stage 的 `order` 是从 1 开始、无重复且连续的整数，`content_obligation` 不得为空。一阶段直接回答可以合法，复杂内容也可以有更多阶段；数量本身不构成质量结论。

### 4.5 结构选择依据

推荐结构必须同时说明：

- 当前受众进入时知道什么、误解什么、在意什么。
- 内容结束时希望受众新增什么判断、理解、情绪或行动依据。
- 现有材料能否支撑事实、机制、案例和结论，缺材料时是否应换结构。
- 账号声音适合强判断、解释、故事还是方法，不因模板破坏账号定位。
- 当前内容需要讲深还是讲浅，以及原因；时长只是成稿后的派生估计，不是结构配额。
- 主平台观看情境如何影响信息前置和表达方式；平台包装不得改变结构的核心语义。

禁止仅凭“这种结构容易爆”“前三秒必须冲突”或固定秒数选择路线。没有真实账号传播数据时，结构选择只能写业务理由，不能伪造效果预测。

### 4.6 结构实现审查与人类门禁

`script_design_review.structure_realization` 将选定结构的每个 `stage_id` 映射到当前 `beat_ids[]`：

```yaml
structure_realization:
  structure_plan_id:
  structure_plan_revision:
  draft_id:
  draft_revision:
  stage_realizations:
    - stage_id:
      beat_ids: []
      realization_status: aligned / partial / missing / diverged
      finding:
  overall_status: aligned / aligned_with_warnings / needs_revision / blocked
```

`design_before_draft` 默认由系统选择有充分理由的推荐结构并自动写稿，不增加例行确认。只有直供稿发生重排 / 重构 / 立场或事实强度变化，或用户明确要求在多种结构间选择时才停人类门禁。`selection_status=waiting_human` 时 `plan_status=pending_selection`，不能写成业务失败；用户接受当前结构 warning 后可以继续，但 warning、原稿和结构差距必须保留。

直供稿的人类门禁读取 `direct_content_intake.revision_policy`，不重复索要已经给过的授权：

```text
preserve_exact：自动选择 keep_current；只给建议，不改 current draft。
allow_local_polish：semantic_impact=none / local_expression 的 local_repair 可自动形成并选择新 revision。
allow_structure_rewrite：已构成当前稿重排 / 重构授权；仍需保留 diff、理由和语义影响，不再二次泛问。
未覆盖的 claim strength / stance / action 变化：selection_status=waiting_human。
```

若用户拒绝推荐重排 / 重构，不判任务失败；回到 `keep_current + ready_with_warnings`，继续保留原稿和结构差距。

## 5. 正式业务主链

```text
热点 / 新写：账号快照 + Brief + claim map
  -> short_video_structure_plan(design_before_draft)
  -> draft
  -> content_beat_map
  -> script_design_review（含结构实现审查）
  -> 当前可用稿件 revision
  -> visual_coverage_ledger

用户直供：账号快照 + direct_content_card + Brief + 原稿
  -> baseline draft revision（原稿语义不变，建立 draft ID / revision / digest）
  -> content_beat_map(semantic_only)
  -> short_video_structure_plan(diagnose_existing_draft)
  -> keep / local repair / human-authorized reorder or rebuild
  -> 当前 draft revision + content_beat_map(structure_bound)
  -> script_design_review（含结构实现审查）
  -> visual_coverage_ledger

共同下游：accepted visual tasks（0 到 N）
  -> 图片 / 来源截图 / 确定性卡片生产
  -> script_visual_alignment_review
  -> 平台包装、独立封面与最终 HTML
```

热点稿与用户直供稿顺序不同，但最终必须汇合到同一组 current structure plan、current draft、current beat map、script review 和 visual coverage 合同。直供稿继续遵守原稿保护；热点稿继续绑定 topic、event 和来源证据。

直供 baseline 由 `copywriting-draft-writer(mode=materialize_user_baseline)` 生产，只允许确定性换行规范化和元数据封装；正文 normalized digest 必须等于 original draft normalized digest。它不是“提前润色”，也不算一次用户授权改写。后续 local repair / reorder / rebuild 必须创建新的 draft revision。

### 5.1 条件路由与 `next_skill`

| 当前对象 / 条件 | `next_skill` | 说明 |
|---|---|---|
| Brief pass + `content_origin=hotspot_selected_topic` | `short-video-structure-planner(design_before_draft)` | 先结构后写稿；未来新增来源必须先注册 origin。 |
| Brief pass + `content_origin=user_supplied_draft` | `copywriting-draft-writer(materialize_user_baseline)` | 先建立语义不变 baseline。 |
| baseline ready | `content-beat-mapper(semantic_only)` | 只还原当前原稿语义。 |
| semantic-only beat map ready | `short-video-structure-planner(diagnose_existing_draft)` | 产生 keep / repair / reorder / rebuild 选择。 |
| structure waiting_human | `human_structure_gate` | 不写稿、不出图；记录唯一待授权问题。 |
| structure ready + keep current | `content-beat-mapper(structure_bound)` | 原稿不改，只绑定 selected stages。 |
| structure ready + local repair / reorder / rebuild | `copywriting-draft-writer(revision)` | 依据 authorization scope 生成新 draft revision。 |
| design-before-draft structure ready | `copywriting-draft-writer(generated)` | 按 selected stages 写稿。 |
| generated / revised draft ready | `content-beat-mapper(structure_bound)` | 映射当前 draft，不沿用旧 beat。 |
| structure-bound beat map ready | `spoken-script-review` | 同时检查结构实现和口播质量。 |
| script readiness ready / ready_with_warnings | `static-visual-director` | 进入全文视觉覆盖。 |
| script decision request_rewrite | `copywriting-draft-writer(revision)` | 改稿后重建 structure-bound beat map 与 review。 |
| script waiting_authorization / blocked | 对应 human gate / upstream producer | 不派发图片任务。 |
| visual coverage complete + accepted tasks 尚未物化 | 对应 Image 2 / source capture / deterministic / existing asset / manual producer | 只执行账本中已接受且依赖满足的 current tasks。 |
| visual delivery ready / ready_with_warnings | `copywriting-quality-review(script_visual_alignment)` | 必须读取 current draft、beat、coverage、asset revisions。 |
| alignment pass / pass_with_warnings | `platform-packaging-adapter` | 才可进入平台包装、独立封面和最终交付。 |
| alignment needs script / visual revision | 对应 writer / static visual director | 只失效受影响 revision，不继续 final delivery。 |

同一 current object 只能有一个 `next_skill`；人类门禁是显式状态，不允许同时写“等待用户”和“已自动进入下游”。

## 6. 口播脚本设计与质检

### 6.1 四层质检

| 层级 | 必须回答 | 不做什么 |
|---|---|---|
| 受众与价值 | 这条给谁看；看完多了什么判断、理解或行动依据；开头承诺在哪里兑现。 | 不用泛化“提高留存”代替具体受众改变。 |
| 观点与叙事 | 核心判断如何被机制、证据、例子、反方和结论推进；哪里跳步、重复、先后失当。 | 不强制三段式、黄金三秒或固定反转。 |
| 口语可说性 | 主语是否清楚、指代是否明确、句子是否能自然说完、术语和数字是否过载、转折是否口语化。 | 不把书面通顺等同于口播可说。 |
| 视听协同 | 哪些信息仅靠口播足够；哪些需要证据、解释、演示、情绪或记忆视觉；视觉是否打断主要理解。 | 不把“有图”当成质量通过。 |

职责分界固定为：`spoken-script-review` 在图片生产前检查结构与口播，决定是否允许进入视觉；现有 `copywriting-quality-review` 在图片生产后检查 current 稿件与 current 视觉的对齐、素材质量和交付就绪。后者发现脚本缺陷可以路由回前者 / writer，但不重新发明另一套口播评分，也不直接改稿。

### 6.2 审查结果

`script_design_review` 必须同时给出全文结论与逐节点结论；质检报告本身保持只读：

```yaml
review_status: pass / needs_revision / blocked
audience_value_status:
narrative_progression_status:
claim_evidence_status:
information_release_status:
oral_speakability_status:
emotion_and_stance_status:
promise_payoff_status:
visual_handoff_status:
issues:
  - issue_id:
    beat_ids: []
    exact_text_anchor:
    issue_type:
    why_it_matters_to_viewer:
    recommended_edit_operation: keep / cut / move / split / merge / clarify / substantiate / soften / strengthen / rewrite
    suggested_text:
    semantic_impact: none / local_expression / claim_strength / stance / structure
    issue_gate: advisory / authorization_required / hard_boundary
    requires_human_confirmation:
```

是否继续由独立、append-only 的 `content_revision_decision` 记录：

```yaml
content_revision_decision:
  decision_id:
  draft_id:
  draft_revision:
  script_design_review_id:
  decision_actor_type: policy / codex_content_review / human
  decision_action: accept_current / apply_local_revision / request_rewrite / waiting_authorization / block
  accepted_warning_ids: []
  authorization_scope:
  authorization_ref:
  decided_at:
```

`script_readiness` 是 `script_design_review + content_revision_decision + current draft digest` 的确定性投影，只允许 `ready / ready_with_warnings / not_ready`，不反写质检报告。

数值指标可以作为诊断信号，但不再以 `hook_score >= 7` 或 `body_information_density_score >= 7` 单独决定通过。没有真实传播数据时，不输出“预计完播率”“必然留人”等伪预测。

`review_status` 与能否继续分开：

- `pass + accept_current` 派生 `script_readiness=ready`。
- `needs_revision + accept_current` 只有在全部未解决 issue 均为 `issue_gate=advisory`、且 `accepted_warning_ids[]` 完整覆盖时，才能派生 `ready_with_warnings`；系统不得把审美或观点分歧擅自升级为硬阻断。
- 存在 `authorization_required` 时必须由 decision 写 `waiting_authorization`；存在 `hard_boundary` 时 decision 只能 `block` 或先回上游消除边界。`accept_current` 不能覆盖这两类 issue。
- `review_status=blocked` 只用于正文 / 账号 / 受众等必填输入缺失、节点或 digest 断链，或当前表达在来源 / 权利硬边界下无法被诚实审查。改写会改变意思但尚未授权时，review 仍记录真实的 `needs_revision`，由 decision 写 `waiting_authorization`；不得把正常的人类门禁伪装成质检失败。
- 用户明确“按当前稿继续”时，只新增当前 draft revision 的 decision；不删除 warning，不改变事实证据状态，也不成为以后内容的默认授权。
- `decision_action=request_rewrite / waiting_authorization / block` 时不派发外部图片任务；只有 `script_readiness=ready / ready_with_warnings` 才进入视觉覆盖和生产，防止先出图再因改稿全部过期。

### 6.3 原稿保护

- `direct_delivery` 默认保留用户声音；审查先给问题和局部建议，不自动整篇改写。
- 不改变意思的口语整理可以形成候选 revision，但必须保留 original / current / diff / reason。
- 改变事实强度、立场、对象指向、核心结构或行动主张时必须等待用户确认。
- 用户可以明确授权“允许结构性重写”；授权只对当前稿件 revision 生效，不成为账号永久默认。

## 7. 视觉覆盖账本

`visual_coverage_ledger` 负责证明：全文每一个内容节点都已经被视觉导演看过，而不是证明每个节点都要配图。

### 7.1 每个节点必须有一个明确主处置

| 决策 | 业务含义 | 必填理由 |
|---|---|---|
| `generate_visual` | 需要由 Codex 内置 Image 2 新生成情境、解释、演示、情绪或记忆视觉。 | 不用图时观众会失去什么；图带来什么变化；为什么模型生成更合适。 |
| `create_deterministic_visual` | 用确定性 SVG / HTML / overlay 生产结构图、清单、比较卡或数据排版。 | 信息单元、排版规则、文字准确性与为什么确定性渲染更合适。 |
| `use_source_evidence` | 使用新闻、数据、公告或引语的来源截图 / 证据卡；允许新捕获或复用仍有效的 verified capture。 | 对应 claim、来源状态、capture mode、最小证据范围、freshness 和可读性。 |
| `use_existing_asset` | 把当前账号或已验证历史 run 中的非证据素材作为本次内容的一项视觉任务。 | 既有 asset ID / hash / provenance / rights、对当前语义与目标画布仍然适配的理由。 |
| `reuse_visual_task` | 当前节点由本次内容中另一项已接受的 current visual task 共同覆盖。 | 被复用的 current task、覆盖关系和为什么不会造成语义错位。 |
| `talking_head_intentional` | 主播表情、语气、停顿和口播本身已经是最佳呈现。 | 为什么加图会重复、分心、削弱信任或破坏情绪。 |
| `evidence_blocked` | 需要证据视觉，但来源不足、冲突、不可访问或存在权利 / 隐私风险。 | 阻断原因、正文如何降级、是否改用非证据解释图。 |
| `manual_visual_required` | 需要实拍、录屏、产品演示或人工素材，当前静态生成 / 截图不能完成。 | 所需素材、责任人和后续动作。 |

禁止使用无理由的 `not_applicable`、`skip` 或“这段不重要”。连接节点也必须说明是被相邻视觉覆盖，还是保留主播连续表达。一个节点可以增加 `supplemental_visual_task_ids[]`，但每个补充任务必须承担与主任务不同的视觉职责、拥有独立插入锚点，并通过冗余与认知负荷检查；例如同一数字主张可先展示来源证据，再用机制图解释含义。补充任务不能替代主处置，也不能成为按节点堆图的后门。

### 7.2 数量与覆盖关系

- `derived_visual_asset_count` 等于去重后的 current accepted visual tasks / 母资产数量，不等于节点数量。
- 一张图可以覆盖多个相邻节点；必须记录 `covered_beat_ids[]` 和每个节点的消费方式。
- 同一节点只有一个主处置；确有不同观众任务时可以有补充视觉任务，全部进入唯一 accepted task 计数，不能靠同义重复堆数量。
- `use_existing_asset` 会为本次内容建立一项新的 current visual task，并计入图片需要数和已成素材数，但 provider / source capture task 均为 0；`reuse_visual_task` 只复用本次内容已有 task，只增加 occurrence。
- 0 张仍然是合法结果，但必须是所有节点逐项得到有效的非生成决策，不再只写一个全文 `zero_visual_reason`。
- 图片数量继续 0 到 N、不设上限；所有通过且 `production_path=image_generation` 的 `generate_visual` 任务都直接调用 Codex 内置 Image 2，不设置成本或调用次数门禁。source capture、deterministic、existing asset 和 manual 路径各走自己的 producer，不能为了“统一”伪装成 provider 调用。

### 7.3 视觉职责与呈现顺序

每个 accepted task 仍必须证明注意力、理解、证据、演示、情绪或记忆价值。决策顺序固定为：

```text
观众问题
-> 视觉职责
-> 信息增量 / 预期观众变化
-> 生产路径（Image 2 / source capture / deterministic / manual）
-> presentation mode
-> slot / canvas / ratio / prompt
```

不得先看到一张横图或竖图，再倒推它适合什么内容。

## 8. 文案与视觉的对齐门禁

`script_visual_alignment_review` 只检查当前稿件 revision 与当前视觉 revision：

```yaml
alignment_status: pass / pass_with_warnings / needs_script_revision / needs_visual_revision / blocked / stale
coverage_completeness_status:
draft_binding_status:
claim_visual_consistency_status:
information_redundancy_status:
visual_sequence_continuity_status:
attention_competition_status:
evidence_presentation_status:
findings: []
```

至少检查：

- 每个正文节点都有且只有一个当前视觉处置。
- 视觉文字、来源截图和画面概念服务当前原句，不服务旧稿。
- 图片没有把观点包装成事实，也没有与口播相矛盾。
- 相邻图片不会连续重复同一种信息，或让观众同时读两套高负荷内容。
- 结构完整不等于质量通过：checker 可证明无漏项、ID 与 digest 闭合；`codex_visual_review / human_visual_review` 才能判断实际叙事和画面价值。

`pass_with_warnings` 只容纳不改变主张、不造成证据误导、且不破坏移动端可读性的观察项；主张冲突、证据冒充、关键文字不可读或 current revision 错绑必须进入对应 revision / blocked / stale，不能被 warning 吸收。只有 `pass / pass_with_warnings` 可以进入平台包装。

## 9. 修改、失效与恢复

| 变化 | 必须失效 | 可保留 |
|---|---|---|
| 只改错别字 / 空白且语义不变 | 对应显示 digest 重新计算 | beat ID、视觉决策可在等价性检查后保留 |
| 修改某个节点的意思、事实强度或顺序 | 该节点 revision、脚本审查、覆盖决策、关联视觉与对齐审查 | 未受影响节点与其资产 |
| 修改核心承诺、受众或立场 | 全篇脚本审查与视觉覆盖 | 原始稿、历史 revision 和生成资产审计记录 |
| 来源证据状态变化 | 对应 claim、节点、证据视觉与最终 warning | 不依赖该来源的其他节点 |
| 视觉生产失败 | 当前视觉任务和相关对齐状态 | 已完成节点、脚本审查和其他有效资产 |

旧资产不自动删除或覆盖。恢复时从 `stale / needs_revision / blocked` 的最小受影响节点继续；不得为了省事重跑整篇，也不得让旧图静默留在新稿交付页。

### 9.1 结构与脚本恢复路由

| 失败 / 等待 | 归因 | 恢复到哪里 |
|---|---|---|
| Brief 缺受众、核心承诺或材料边界 | upstream input gap | `content-brief-compiler` 补最小字段，不临时猜结构。 |
| 直供 `semantic_only` beat map 漏正文 / 锚点冲突 | product artifact defect | `content-beat-mapper` 重建诊断 revision。 |
| 内置结构均不适配 | normal product branch | 有充分理由时生成 `custom`；材料本身不足则回 Brief。 |
| strategy registry / version 不存在 | contract / implementation defect | 阻断 planner，修注册表或版本引用；不能临时硬编码。 |
| 等待结构改写授权 | human gate | 保留候选与原稿；用户同意后新建 revision，拒绝则 keep current with warnings。 |
| draft 未实现 required stage | script quality defect | 回 `copywriting-draft-writer` 局部补写 / 重排，再重建 structure-bound beat map。 |
| current pointer / digest / artifact ID 不闭合 | lineage defect | 从最小 stale 对象恢复；不绕过 checker，不重跑无关 provider。 |

## 10. 最终交付页的业务表达

最终 HTML 面向用户新增三块可读信息，不把工程字段直接铺满首页：

1. **这篇怎么讲**：受众、核心承诺、观点推进和关键记忆点。
2. **哪里建议改**：精确原句、问题、对观众的影响、推荐改法和是否会改变原意。
3. **哪里用图，为什么**：按文案顺序显示生成图、证据图、共用图、故意不配图或缺素材；允许展开查看来源与 lineage。

若 `script_readiness=not_ready`，页面不得把稿件显示为“口播已就绪”；`needs_revision` 但用户已对当前 revision 接受警告时，页面可以显示“可按当前稿继续，仍有 N 项建议”，不得显示无风险通过。若 `visual_coverage_ledger` 有 unresolved beat，页面不得显示“视觉方案完整”；两者均不影响用户查看当前候选和逐项返工建议。

## 12. 编译前二义性反审计

### 12.1 审计结论

本轮按对象边界、执行顺序、producer / consumer、固定路径、条件必填、状态轴、数量轴、人类门禁、失败恢复、版本迁移和最终 HTML 投影逐项反查。发现的二义性均已在产品层给出唯一解释；当前没有需要在编译时再临场决定的业务分叉。

| 原二义性 | 唯一解释 |
|---|---|
| “短视频结构”是否就是 beat map | 不是。structure plan 定宏观路线；beat map 描述当前稿件的最小观众变化；review 检查两者是否实现。 |
| 新写稿和直供稿是否同序 | 不是。新写先结构后 draft；直供先 beat map 后结构诊断。共同下游只消费被选定的 current revisions。 |
| 直供 original draft 是否直接冒充 draft | 不。先物化语义不变的 baseline draft revision，建立稳定 draft ID / digest；之后才能映射和修订。 |
| 直供 beat map 与结构 plan 是否循环依赖 | 不循环。先写 `semantic_only` 诊断 revision；选定结构后另写 `structure_bound` revision，只有后者进入共同下游。 |
| 策略列表是否固定模板 | 不是。策略来自版本化注册表，可扩展；`custom` 有严格理由和阶段合同。 |
| 候选 / 阶段是否固定几个 | 不固定，均为内容派生 1 到 N；直供只硬要求存在 keep current，stage order 连续唯一。 |
| 结构 plan 是否同时代表质量 pass | 不是。plan 是目标；`script_design_review` 才记录实现和质量。 |
| 前置口播质检与后置联合质检是否重复 | 不重复。spoken review 决定脚本能否进视觉；copywriting quality review 检查当前文案—视觉—素材对齐并路由返工。 |
| `needs_revision` 是否阻断 | 不必然。`script_readiness` / `visual_delivery_readiness` 单独决定能否继续；warning 可按当前 revision 接受。 |
| 接受 warning 是否改写 review | 不改。review 保持只读；append-only `content_revision_decision` 记录选择，readiness 由 review + decision + digest 派生。 |
| 全文 coverage complete 是否等于图片齐全 | 不等于。coverage 证明每个 beat 已决策；素材生产、证据和人工素材由 delivery readiness 表示。 |
| 一张图多次出现算几张 | 一项内容视觉任务 / 一份母资产；每次时间位置是独立 occurrence。平台 rendition 与封面另计。 |
| 复用历史素材与复用当前任务是否相同 | 不同。历史非证据素材用 `use_existing_asset`，为本稿建立 current task；历史来源截图仍用 `use_source_evidence(reuse_verified_capture)`；`reuse_visual_task` 只是本稿 current task 的另一处覆盖 / 插入，不增加资产数。 |
| 0 张是否包含缺证据 / 等人工素材 | 不包含。只有全部 current beat 都被 `talking_head_intentional` 且没有补充任务时，才是 ready 的 0 张。 |
| AI / checker 谁能写质量 pass | checker 只写结构与引用结果；内容 / 视觉质量 pass 必须有 `codex_*_review / human_*_review` 及观察理由。 |
| 当前文件如何避免覆盖旧 revision | immutable revision + `.current.json` pointer；pointer 最后写入，是该对象的物理 current commit marker。 |

### 12.2 条件必填矩阵

| 对象 / 条件 | 必填 | 禁止 / 不适用 |
|---|---|---|
| structure `design_before_draft` | Brief、账号快照、受众、核心承诺、材料 / 证据、strategy registry、候选 / 推荐 / selected strategy、stages、selection reason | `source_draft_ref / source_beat_map_ref` 必须为空 |
| structure `diagnose_existing_draft` | 上述共同字段 + source draft / semantic-only beat map、`keep_current` 候选、transformation level、decision / authorization | 未授权时不得选择 reorder / rebuild 为可执行 current plan |
| direct baseline draft | original draft artifact / digest、normalized digest、`mode=materialize_user_baseline`、revision=1 | 不允许润色、重排、改变标点语义或主张强度 |
| structure `waiting_human` | candidates、recommended candidate、semantic impact、待授权问题、`plan_status=pending_selection` | selected strategy / stages 可以为空；不得写 `plan_status=blocked`，不得进入 draft / visual |
| structure `ready / ready_with_warnings` | selected candidate / strategy、stages、selection decision、适用的 authorization ref | 不得保留未解决的 waiting_human |
| content beat `semantic_only` | 唯一 source anchor、信息增量、观众前后状态、draft revision / digest | structure plan / stage 必须为空；只能供直供结构诊断 |
| content beat `structure_bound` | 上述共同字段 + current structure plan / stage mapping | 不能引用其他 draft revision；未绑定 stage 不得进入口播 / 视觉 |
| `generate_visual` | visual job、viewer problem、expected change、information delta、production path、presentation intent | 不得填写 source evidence verdict |
| `create_deterministic_visual` | visual job、信息单元、文字内容、布局规则、target canvas / slot、renderer ref | 不创建 provider task；文字和数据必须可确定性复算 |
| `use_source_evidence` | claim ID、source / capture / binding、证据状态、最小可读区域、`capture_mode=new_capture / reuse_verified_capture`；复用时另需 capture hash / captured_at / freshness verdict | 不得使用 Image 2 生成来源画面；不得改写成 `use_existing_asset` 绕过证据约束 |
| `use_existing_asset` | existing asset ID / hash / provenance / rights、当前语义与画布适配理由、current task ID、occurrence | 不创建 provider / source capture task，不得承担 factual evidence 角色；历史“曾通过”不能替代本稿适配审查 |
| `reuse_visual_task` | current task ID、occurrence ID、插入锚点、复用理由 | 不生成新 provider task，不增加 asset count |
| `talking_head_intentional` | 主播表达优势、加图损失 | 不生成 task，不允许空理由 |
| `evidence_blocked` | claim、阻断原因、正文处置、恢复动作 | 在 claim 未降级前不得 delivery ready |
| `manual_visual_required` | 所需素材、责任、交付条件、插入锚点 | 素材未到前不得 materialized / ready |

### 12.3 状态轴唯一语义

各 producer 不再共享一个模糊 `pass`：

```yaml
structure:
  selection_status: auto_selected / human_selected / waiting_human / blocked
  plan_status: pending_selection / ready / ready_with_warnings / blocked / stale
beat_map:
  mapping_status: ready / blocked / stale
script:
  review_status: pass / needs_revision / blocked
  decision_action: accept_current / apply_local_revision / request_rewrite / waiting_authorization / block
  script_readiness: ready / ready_with_warnings / not_ready
visual:
  coverage_completeness_status: complete / incomplete / stale
  visual_delivery_readiness: ready / ready_with_warnings / waiting_assets / waiting_authorization / blocked
alignment:
  alignment_status: pass / pass_with_warnings / needs_script_revision / needs_visual_revision / blocked / stale
```

`evidence_blocked / manual_visual_required` 可以处于 `coverage_completeness_status=complete`，因为分析已完成；前者在主张尚未降级时使 delivery blocked，后者使 delivery waiting_assets。`unresolved_beat_ids[]` 只包含缺少 / 非法主处置或引用断裂的 beat，不包含已经诚实判断为 blocked / manual 的 beat。

`waiting_human / waiting_authorization / waiting_assets` 是正常业务等待态，不属于 `blocked`。`blocked` 只表示输入、合同、追溯、来源 / 权利硬边界或 producer 无法产出合法对象；checker 和最终 HTML 必须分别展示等待原因与失败原因。

状态组合必须满足：beat map 只有 `full_coverage_status=complete` 且 `unresolved_ranges=[]` 时才可 `mapping_status=ready`；structure 只有 selection 已完成且 stages 合法时才可 plan ready；script 只有 current review、current decision 与 current draft digest 三者闭合时才可投影 readiness；alignment 只有绑定 current draft、beat、coverage 和 asset revisions 时才可 pass。任一 source digest 变化统一进入 `stale`，不能降级成普通 warning。

### 12.4 数量口径

新合同不再用一个 `derived_visual_count` 同时表达四件事：

```yaml
derived_visual_asset_count:        # 当前内容需要的唯一视觉任务 / 母资产数
materialized_visual_asset_count:   # 已得到并通过结构检查的唯一素材数
provider_generation_task_count:   # 实际需要 Image 2 基础生成的任务数
provider_generation_attempt_count:# provider 尝试记录数；重试 / 拒绝会使其大于任务数
source_capture_task_count:         # 实际需要公开来源捕获的任务数
source_capture_attempt_count:      # 捕获尝试数；与成功截图数分开
visual_insert_occurrence_count:    # 精确插入次数；同一资产可多次出现
platform_rendition_count:          # 各平台派生成品数，不算新增内容图片
cover_asset_count:                 # 独立封面任务数，不进入正文图片数
```

`derived_visual_asset_count` 包含 `generate_visual / create_deterministic_visual / use_source_evidence / use_existing_asset / manual_visual_required` 产生的唯一 current task。`generate_visual` 严格对应 Image 2 基础生成；`create_deterministic_visual` 的 provider task 为 0。`use_existing_asset` 计入 current task 和 materialized asset，但 provider / source capture task 均为 0；`reuse_visual_task` 只增加 occurrence，不增加资产数。provider task 统计唯一基础生成任务，provider attempt 统计实际尝试；确定性叠字、裁切、平台 rendition 和封面派生按既有合同分开。

编译时的基数公式固定为：

```text
derived_visual_asset_count = current accepted task IDs 中 generate / deterministic / source evidence / existing / manual 的去重数
materialized_visual_asset_count = 上述 task 中已经绑定 current accepted asset 且结构检查通过的去重数
provider_generation_task_count = current generate_visual task 的去重数
provider_generation_attempt_count = 绑定 current provider tasks 的 append-only attempt records 数
source_capture_task_count = capture_mode=new_capture 的 current use_source_evidence task 去重数
source_capture_attempt_count = 绑定上述 current capture tasks 的 append-only attempt records 数
visual_insert_occurrence_count = current occurrence IDs 的去重数
```

`capture_mode=reuse_verified_capture` 仍是 `use_source_evidence`，会计入图片需要数和已成数，但不会增加本轮 source capture task / attempt。缓存 capture 的 claim binding、来源、hash、时间与 freshness 任一不满足时，必须新捕获或进入 evidence blocked，不能改用普通 existing asset 逃逸。

`visual_insert_occurrence` 的最小合同为：

```yaml
occurrence_id:
visual_task_id:
sequence_order:
covered_beat_ids: []
insert_after_beat_id:
insert_before_beat_id:
source_anchor_digest:
recorded_time_range:       # 未录制时 null；有音视频时才填写
```

每个 occurrence 必须有非空 `covered_beat_ids[]` 和至少一个 before / after beat anchor；相邻多个 beat 可共用一次 occurrence，非相邻复用必须拆成多次 occurrence。`recorded_time_range` 只能补充定位，不能覆盖 beat anchor 或成为纯文案必填。

合法 ready 的 0 张要求：所有 current beat 的主处置均为 `talking_head_intentional`、没有补充视觉任务、没有 evidence / manual / waiting task。`zero_visual_reason` 在 v0.4 只允许作为逐 beat 决策派生的人类摘要，不再作为独立输入证明。

### 12.5 current revision 与写入顺序

每个新对象的写入顺序固定为：

```text
生成 immutable candidate revision
-> Schema / cross-reference 校验
-> 生成并校验人类 Markdown 投影
-> 写 artifact event / lineage / hash
-> 最后原子替换 {object}.current.json pointer
```

pointer 缺失、hash 不符、source draft digest 过期或下游引用的 artifact ID 不是 current 时均 blocked。修改已选计划必须产生新 revision；历史 revision、用户决策和 warning 只读保留。checker 除动态报告外不写 pointer、plan、draft、beat、coverage 或 readiness。

## 13. 产品验收样例

至少覆盖以下脱敏 fixture：

| 样例 | 必须证明 |
|---|---|
| 强观点直供稿、用户要求保留原声 | 能发现证据与叙事问题，但不自动改掉立场。 |
| 热点新写稿 | 先自动选择有理由的结构，再写 draft；执行记录不能伪造为写后倒推。 |
| 直供稿结构尚可 | `keep_current` 被选中，局部 warning 可继续，不强制套新模板。 |
| 直供 baseline | normalized body digest 与 original 一致，诊断前未发生润色或重排。 |
| 直供稿需要重排 | 未授权时 waiting_human；授权后新 draft / beat / review revision，不覆盖原稿。 |
| 自定义结构 | 内置策略均不适配且理由充分，custom stages 可实现并可审查。 |
| 固定模板误用 | 仅因“前三秒 / 三幕式”选择策略必须 fail。 |
| 结构 stage 缺失 | 实现审查标 missing，不能由 Hook 分数掩盖。 |
| 热点证据稿 | topic / event / claim / source 与节点、证据图闭合。 |
| 情绪很强但只有结论 | 标出机制 / 证据 /推进缺口，不用“有感染力”直接 pass。 |
| 段落重复、指代含糊、长句难说 | 给出精确锚点和局部修订动作。 |
| 好口播且无需插图 | 每个节点 `talking_head_intentional` 有理由，0 张合法通过。 |
| 多节点共用一张机制图 | coverage 全，accepted task 只计一张。 |
| 确定性四项结构卡 | `create_deterministic_visual` 计一项内容图片与一项 materialized asset，provider task / attempt 均为 0。 |
| 同一记忆图首尾复用 | 一项资产、两次 occurrence、一次 provider 任务。 |
| 历史已验证素材复用于新稿 | 建立一个 `use_existing_asset` current task；需要数 / 已成数各加一，provider / source capture 均为 0，并重新核对 provenance / rights / 当前语义与画布。 |
| 历史来源截图仍可复用 | 保持 `use_source_evidence(reuse_verified_capture)` 与 claim / source / freshness 绑定；需要数 / 已成数各加一，本轮 capture task / attempt 为 0。 |
| 一个节点先证据后解释 | 两个不同 visual job、两个 occurrence；不是重复堆图。 |
| 漏掉一个连接段 | completeness gate fail，不得靠全文 `zero_visual_reason` 掩盖。 |

| 核心承诺改变 | 全篇脚本审查与视觉覆盖 stale。 |
| 图与当前口播矛盾 / 重复字幕 | 对齐门禁返修视觉。 |
| 重大改写未授权 | review 如实写 needs revision，decision 写 waiting authorization，structure 为 pending selection；保留原稿且不派发图片任务，不误报 blocked。 |
| current pointer 指向旧 draft | checker blocked，不允许下游继续。 |
