# 工作流工程缺陷复盘与修订方案

> 归档状态：historical_audit_only；2026-07-18 移出当前解释热路径，不得作为 current 合同真源。

> 状态：工程复盘 / 规则修订依据  
> 主责：记录 SAMPLE-HISTORICAL-001 这轮实战暴露出的 workflow 工程缺陷，并给出后续修订方向。  
> 边界：本文只修订内容工作流和交付治理，不实现短视频 SaaS、平台发布、外部图片 API 或产品工程功能。

---

## 一、本轮暴露的问题

SAMPLE-HISTORICAL-001 证明了“热点 -> 选题 -> Brief -> 口播 -> 画中画 -> 质检 -> 平台包装 -> 交付记录”主链路可以跑通，但也暴露出四类工程缺陷：

```text
1. 后台链路完整，不等于前台交付完整。
2. 画中画提示词完整，不等于图片资产已交付。
3. 项目内 HTML 可用，不等于离开项目目录还能用。
4. manifest 能恢复 AI 链路，不等于能描述人类交付包。
```

这些问题不是内容质量问题，而是 workflow 工程边界不够清楚。

---

## 二、参考成熟做法

### 1. HTML 输出要区分“依赖外部资源”和“自包含”

成熟文档工具会区分普通 HTML 和可独立分享的自包含 HTML。Quarto 的 HTML 文档默认可能依赖同级资源目录，也支持把图片、CSS、脚本等资源嵌入 HTML，生成无外部依赖的 standalone HTML。

映射到本项目：

```text
project_local_html：项目内验收页，保留相对链接，方便 AI 追溯。
standalone_html：独立 HTML，图片和关键追溯摘要内嵌，不依赖项目目录。
```

### 2. 静态站点构建要复制资源

Jupyter Book / Sphinx 类工具在构建 HTML 时，会把需要的静态资源复制到构建目录，而不是让页面依赖源目录中的散落文件。

映射到本项目：

```text
portable_bundle：可转交目录包，HTML、assets、sources、manifest 放在同一个 export 目录。
```

### 3. 可交付包需要资源清单

Frictionless Data Package 用 `datapackage.json` 描述一组资源；BagIt 用 payload manifest 和 checksum 确认包内文件完整性。

映射到本项目：

```text
export-manifest.json：描述交付包里有哪些文件、用途、路径、来源 ID、是否必需。
manifest-sha256.txt：可选，用于核验可转交包是否缺文件或被改动。
```

## 三、修订后的交付形态

后续每轮内容至少区分三种交付形态：

| 形态 | 文件 / 目录 | 用途 | 链接策略 |
|---|---|---|---|
| `project_local` | `deliverables/final-delivery.html` | 项目内验收、AI 恢复、深度追溯 | 相对链接指向 session 内 MD 和图片 |
| `portable_bundle` | `deliverables/export/{session_id}/` | 发给别人、网盘、转交运营 | 资源复制进 export 目录，链接只指向包内文件 |
| `standalone_html` | `deliverables/export/{session_id}/final-delivery-standalone.html` | 单文件转发、邮件、临时预览 | 图片和必要摘要内嵌，后台 MD 只做摘要或附录 |

默认规则：

```text
每轮确认采用后，必须生成 project_local。
当用户要“发给别人 / 交付客户 / 上传网盘 / 不在本机看”时，必须生成 portable_bundle。
当用户明确要单文件、或者接收方不适合拿目录包时，再生成 standalone_html。
```

## 四、新增工程门禁

### 1. 交付页门禁

`final-delivery.html` 合格必须满足：

```text
能说明选题与切口。
能复制正式文案。
能预览或明确标记画中画图片。
能下载图片。
能复制各平台发布物料。
能链接回后台 MD。
页面顶部明确：已确认 / 未发布 / 不自动发布。
```

### 2. 图片资产门禁

最终交付前：

```text
required_visuals 必须生成图片，或明确 image_status = pending_external / generation_failed。
图片必须进入 assets/images/。
图片资产必须有 image-assets.md 或等价清单。
图片不得含真实手机号、微信号、车牌、真实用户头像或真实平台后台。
```

### 3. 可转交包门禁

`portable_bundle` 合格必须满足：

```text
export/ 目录里包含 HTML、assets、sources、export-manifest.json。
HTML 内所有链接不得跳回原 session 目录。
图片可下载。
关键后台材料至少复制摘要版或完整 MD 到 sources/。
export-manifest.json 列出所有文件、用途、来源、是否必需。
可选生成 manifest-sha256.txt。
```

### 4. 状态门禁

内容完成不能只写 `session_completed`。必须补充：

```text
delivery_page_mode：project_local / portable_bundle / standalone_html
final_delivery_status：html_ready / bundle_ready / standalone_ready / needs_export / blocked
image_assets_status：all_generated / partially_generated / pending_external / generation_failed / manual_required / mixed / not_required
export_status：not_requested / export_ready / export_needs_fix / export_blocked
```

## 五、修订后的完成定义

一轮内容不再只以“内容交付记录完成”为最终状态。

更准确的完成分层：

```text
content_ready：内容链路完成，等待人工确认。
delivery_confirmed：用户确认采用。
project_local_ready：项目内 HTML 验收页已生成。
portable_ready：可转交目录包已生成。
standalone_ready：单文件 HTML 已生成。
published_recorded：人工发布结果已回填。
```

默认完成到 `project_local_ready` 即可算本项目一轮工作闭环；是否继续到 `portable_ready` 或 `standalone_ready` 由用户场景决定。

## 六、暂不实现但必须预留

```text
不自动生成 ZIP，除非用户明确要可转交压缩包。
不接 Seedream API，只保留外部图片模型兼容字段。
不上传网盘、官网或平台。
不把 portable_bundle 当成已发布。
不让用户重复输入选题、文案、产品边界或提示词。
```

---

## 七、下一步建议

1. 把 `final-delivery.html` 生成逻辑沉淀为固定 skill 或平台包装后的子流程。
2. 给 `portable_bundle` 增加模板：`export-manifest.json`、`assets/`、`sources/`。
3. 给 `copywriting-quality-review` 增加“交付体验质检”维度。
4. 每跑 3-5 条真实内容后，复盘一次字段是否过重、交付包是否好用。
