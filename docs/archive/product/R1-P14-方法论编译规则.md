# R1-P14 方法论编译规则

> 归档状态：historical_audit_only；2026-07-18 移出当前产品热路径，不得作为 current 合同真源。

> 状态：已确认并已编译引用  
> 所属路线：R1 方法论没有编译成执行合同 / P14 讨论稿没有充分编译成 skill  
> 目标：定义“讨论稿、调研、方法论、复盘、用户反馈”如何进入本项目的可执行体系。  
> 边界：本文件不改 `SKILL.md`，不进入 skill 编译；确认后才作为编译依据。

## 1. 为什么需要 P14

当前问题不是缺方法论，而是方法论容易停在讨论稿里。

典型风险：

```text
讨论时说清楚了，skill 没吸收。
解释文档写得很完整，但执行时 agent 没读到。
字段词典没更新，交接物还是老字段。
合同有新规则，但 SKILL.md 没编译，真实运行仍靠 agent 扶。
同一条规则在 README / STATUS / explanation / skill 里重复出现，后面互相打架。
```

P14 要解决的是：

```text
一条方法论到底应该进哪里。
什么时候只作为解释。
什么时候必须变成字段。
什么时候必须变成合同。
什么时候才能改 SKILL.md。
什么时候必须补样例或 validator。
```

## 2. 方法论输入来源

本项目允许吸收的方法论来源分为六类：

| 来源 | 例子 | 默认处理 |
|---|---|---|
| 用户当轮反馈 | “不要让用户说继续”“最终产物应该是 HTML” | 先进入产品文档或 reference 草案 |
| 真实运行复盘 | 三篇都做导致链路混乱、画中画不稳 | 进入问题包 / 路线图 / 合同 |
| dbskill 方法论 | hook、script-flow、resonate、content | 提炼原则，不原样复制 |
| 平台 / 内容体验方法 | YouTube、TikTok、Google Helpful Content、NN/g | 提炼为字段、门禁或质检项 |
| workflow 工程方法 | Temporal、Airflow、Dagster、LangGraph | 提炼为状态、分支、资产、门禁规则 |
| 项目已有协议 | AGENTS、字段词典、文档治理规范 | 作为优先真源，不重复发明 |

## 3. 编译目标层

一条方法论进入系统时，必须选择目标层，不允许“哪里都写一点”。

| 目标层 | 放哪里 | 什么时候放这里 |
|---|---|---|
| 产品路线 | `docs/product/` | 还在设计、需要涛哥确认、涉及取舍或优先级 |
| 执行参考 | `docs/reference/` | 已经是稳定规则，AI 每轮需要按它执行 |
| 解释说明 | `docs/explanation/` | 解释为什么这么做，供人理解，不直接当执行合同 |
| 字段词典 | `交接物字段词典.md` | 影响 artifact 字段、状态值、ID、枚举 |
| skill 合同 | `skills/{skill_id}/CONTRACT.md` | 影响某个 skill 的触发、输入、输出、门禁、失败处理 |
| SKILL.md | `skills/{skill_id}/SKILL.md` | 合同已确认，要进入真实执行说明 |
| sample run | `examples/` 或 sample session | 需要证明陌生 agent 能跑通 |
| validator / checklist | `docs/reference/` 或后续脚本 | 需要可检查、可验收、可失败定位 |

## 4. 编译路由表

| 方法论内容 | 先进入 | 确认后进入 | 例子 |
|---|---|---|---|
| “为什么要这样做” | `docs/explanation/` 或 `docs/product/` | 不一定进入 skill | 为什么最终产物用 HTML |
| “以后必须这样做” | `docs/reference/` | 相关 `CONTRACT.md` | 选题确认后自动到底 |
| “某个 skill 必须接什么产什么” | `skills/{skill_id}/CONTRACT.md` | `skills/{skill_id}/SKILL.md` | Brief 通过后自动写口播 |
| “新增字段或状态” | `docs/product/` 草案 | `交接物字段词典.md` + 合同 | `hook_route`、`body_information_density_score` |
| “用户需要怎么被引导” | `docs/reference/人类引导与任务后导航规范.md` | 合同和 SKILL.md | 不问“是否继续” |
| “失败时怎么恢复” | 合同 | SKILL.md + validator | 图片缺失标 pending_external |
| “质量检查项” | 合同 / reference | quality-review SKILL + validator | Hook 承诺未兑现阻断 |
| “文件放哪里” | `docs/reference/文档治理与目录规范.md` | 合同 / SKILL | final HTML 放 deliverables |
| “开源发布边界” | `docs/product/` | `docs/reference/GitHub开源上线检查清单.md` | sample account 脱敏 |

## 5. 编译状态机

每条方法论从讨论到执行，必须经历这些状态：

```text
idea
-> product_draft
-> product_confirmed
-> contract_draft
-> contract_confirmed
-> skill_compiling
-> skill_active
-> sample_verified
-> validator_covered
```

状态含义：

| 状态 | 含义 | 允许动作 |
|---|---|---|
| `idea` | 对话中的想法或反馈 | 只能记录，不当规则 |
| `product_draft` | 产品草案，待确认 | 可继续讨论、修订 |
| `product_confirmed` | 产品方向确认 | 可进入合同 |
| `contract_draft` | skill 合同草案 | 不改 SKILL.md |
| `contract_confirmed` | 合同确认 | 可进入 skill 编译 |
| `skill_compiling` | 正在改 SKILL.md | 只能按合同改 |
| `skill_active` | SKILL.md 已吸收 | 可进入样例验证 |
| `sample_verified` | sample run 跑通 | 可评估成熟度 |
| `validator_covered` | 有检查清单或脚本覆盖 | 可进入发布候选 |

硬规则：

```text
没有 product_confirmed，不写入 SKILL.md。
没有 contract_confirmed，不写入 SKILL.md。
没有 sample_verified，不宣称 L3。
没有 validator_covered，不宣称 L4。
```

## 6. 方法论编译 DoR

一条方法论进入合同前，必须满足：

```text
[ ] 能说清它解决哪个问题 ID。
[ ] 能说清适用哪个 skill 或哪类 artifact。
[ ] 能转成字段、门禁、状态、失败处理或验收样例。
[ ] 不只是口号或偏好。
[ ] 不和 AGENTS / 字段词典 / 文档治理规范冲突。
[ ] 不要求登录平台、自动发布、外部 API 或业务代码改动。
```

不满足时，只能留在 explanation / product 草案，不能进入合同。

---

## 7. 方法论编译 DoD

一条方法论算“编译完成”，必须满足：

```text
[ ] 目标层明确。
[ ] 上游来源记录清楚。
[ ] 已写入对应合同或 reference。
[ ] 如果新增字段，字段词典有后续修订任务。
[ ] 如果影响执行，SKILL.md 有后续编译任务。
[ ] 有至少一个验收样例。
[ ] 有失败时的恢复路径。
[ ] README / PROJECT_MAP / STATUS 能追到入口。
```

## 9. 不追求完美的取舍

P14 不把所有方法论都塞进 `SKILL.md`。

取舍：

```text
SKILL.md 只放执行必需规则。
CONTRACT.md 放进入 skill 前的产品合同。
docs/product 放取舍、路线、成熟度、草案。
docs/explanation 放背景和为什么。
字段词典只放稳定字段，不放长篇方法论。
validator 只检查可检查的东西，不检查审美偏好。
```

这能避免开源用户下载后看到一个巨大的、读不完的 skill，也避免 agent 每次靠全文检索猜规则。

## 10. P14 验收标准

P14 通过的标准：

```text
新增方法论有明确目标层。
所有 R1 方法论输入都能走编译状态机。
合同和 SKILL.md 的边界清楚。
讨论稿不会直接变成执行规则。
确认前不改 SKILL.md。
确认后知道该改哪个 SKILL.md、哪个字段、哪个 validator。
```
