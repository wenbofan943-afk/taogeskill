## 1. 问题判断

当前合同草案已经把“前 5 秒 Hook”放进了 `copywriting-draft-writer` 和 `copywriting-quality-review`，但还不够。

真实风险是：

```text
Hook 有了，但 Hook 路由不清。
Hook 看起来抓人，但正文撑不起承诺。
正文有结构，但信息密度下降。
观点很多，但核心机制不清。
口播顺，但每段没有新的信息增量。
```

这不是新增第 18 个问题，而是并入 R1：

| 归属 | 原因 |
|---|---|
| P14 讨论稿没有充分编译成 skill | 内容创作方法论还没有完整进入合同 |
| P15 dbskill 式编译不足 | Hook、正文、共鸣、逻辑延续应该拆成可执行小检查 |
| P13 trace 只是记录不是检查器 | 信息密度、正文兑现、逻辑延续需要进入质检清单 |
| P01 skill 不是执行合同 | 文案 skill 的输入输出还缺“正文质量合同” |

## 2. 成熟方法论参考

### 2.0 方法论栈判断

`dbskill` 不够作为唯一底座。它应该作为本项目的方法论上层，而不是全部方法论来源。

推荐采用四层方法论栈：

```text
底层：成熟内容体验原则
中层：短视频 / 平台创意原则
上层：dbskill 创作者诊断方法
项目层：涛哥创作工作流的账号、产品、交付合同
```

各层职责：

| 层级 | 代表来源 | 负责什么 | 不负责什么 |
|---|---|---|---|
| 成熟内容体验原则 | Google Helpful Content、Nielsen Norman Group | 内容是否对人有用、信息路径是否清楚、用户能否快速判断价值 | 不提供短视频口播风格 |
| 平台创意原则 | YouTube Creator、TikTok Creative Center | 前置吸引点、节奏、观看理由、平台原生表达 | 不负责本项目账号 / 产品边界 |
| dbskill 创作者诊断 | `dbs-hook`、`dbs-script-flow`、`dbs-resonate`、`dbs-content` | Hook、逻辑延续、共鸣、文字洁癖、短视频创作者判断 | 不提供完整 workflow 状态机和开源交付规范 |
| 项目合同 | 本项目 `CONTRACT.md`、字段词典、manifest | 输入输出、路径、门禁、自动推进、失败处理、开源边界 | 不追求平台算法预测或爆款承诺 |

取舍逻辑：

```text
dbskill 负责“像人一样诊断内容”。
平台 / 内容体验原则负责“这件事是否符合更普遍的内容质量规律”。
本项目合同负责“AI 能不能稳定执行，不串账号，不断链，不误发布”。
```

因此，R1 不是把 dbskill 原样搬进来，而是把它和更通用的方法论一起压缩成字段、门禁、失败处理和验收样例。

### 2.1 官方平台和内容体验参考

| 来源 | 可吸收原则 | 本项目取舍 |
|---|---|---|
| YouTube Creator / 创作者教育 | 开头要快速建立观看理由，但视频要持续兑现承诺 | 不只评分 Hook，还要检查正文兑现 |
| TikTok Creative Center / 创意最佳实践 | 短视频需要前置吸引点、原生表达和清晰节奏 | 不套模板标题，要让 Hook 和正文节奏一致 |
| Google Helpful Content | 内容应该对人有帮助，而不是只为流量包装 | 不把“热点包装”凌驾于真实信息价值 |
| Nielsen Norman Group 信息气味 / 倒金字塔 | 用户需要快速判断内容是否值得继续读 / 看 | 口播每段都要有信息路径和可感知增量 |

参考链接：

- YouTube Creator Academy：https://www.youtube.com/creators/
- TikTok Creative Center：https://ads.tiktok.com/business/creativecenter/
- Google Search helpful content：https://developers.google.com/search/docs/fundamentals/creating-helpful-content
- Nielsen Norman Group Inverted Pyramid：https://www.nngroup.com/articles/inverted-pyramid/
- Nielsen Norman Group Information Scent：https://www.nngroup.com/articles/information-scent/

### 2.2 dbskill 本地方法论参考

| dbskill | 吸收原则 |
|---|---|
| `dbs-hook` | 好开头 = 话题 + Hook + 可信度；写不出好开头，常常是内容本身没素材 |
| `dbs-script-flow` | 观众划走常见原因是逻辑断裂、信息密度下降、口播卡壳 |
| `dbs-resonate` | 内容必须有一个核心机制，全面但不精准会稀释共鸣 |
| `dbs-content` | 内容质量不是 AI 味问题，而是是否把事情搞清楚、说清楚 |

本项目吸收这些原则，但不直接复制外部 skill。后续应把它们编译为本项目自己的字段、检查项和门禁。

## 3. 新的内容质量三层模型

后续文案链路不再只检查“前 5 秒 Hook”，而是检查三层：

```text
Hook 路由
-> 正文信息密度
-> 共鸣与兑现
```

### 3.1 Hook 路由

Hook 不是一句漂亮开头，而是一个路线选择。

每个 Hook 必须标注：

```text
hook_route：这条开头走哪种吸引路径。
content_promise：它向观众承诺后面会得到什么。
credibility_source：为什么观众信你能讲这件事。
body_payoff：正文哪一段兑现这个承诺。
```

允许的 Hook 路由：

| hook_route | 使用场景 | 必须配套 |
|---|---|---|
| result_contrast | 有结果、变化、前后对比 | 正文必须给机制，不只晒结果 |
| data_shock | 有可靠数据或公开事实 | 必须有来源和解释 |
| pain_scene | 目标人群有强代入痛点 | 正文必须回到具体场景 |
| conflict_judgment | 有明确判断或反常识 | 正文必须给推导链 |
| curiosity_gap | 有信息缺口 | 正文必须兑现答案，不能吊胃口 |
| silent_truth | 说出受众不敢说的话 | 立场必须稳定，不能变说教 |
| field_credibility | 依靠行业经验或现场感 | 必须有观察、案例或细节 |
| product_demonstration | 用产品能力解释问题 | 不能扩大产品能力或像广告 |

禁止的 Hook：

```text
只有情绪，没有正文兑现。
只有反问，没有具体话题。
只靠夸张数字，但无来源。
开头承诺 A，正文讲 B。
把产品能力包装成灰产、截流、批量获客或自动私信。
```

### 3.2 正文信息密度

正文不是把 Brief 展开成长句，而是持续给观众新的信息增量。

每段口播必须标注一个 `segment_function`：

```text
hook_payoff
fact
mechanism
example
contrast
transition
product_bridge
cta
```

每段还必须有 `information_delta`：

```text
这一段比上一段多告诉了观众什么？
如果删掉这一段，观众会不会少理解一个关键点？
这一段是否只是在换说法重复？
```

正文信息密度最低要求：

```text
每 5-8 秒至少有一个明确的信息增量。
连续两段不能承担同一个功能。
产品露出段必须服务核心观点，不能单独插广告。
抽象判断后必须跟机制、例子或场景。
```

### 3.3 共鸣与兑现

内容最终要检查两件事：

```text
共鸣：有没有站在目标人群那边，说出他们真实在意的东西。
兑现：Hook 承诺的价值，正文是否真的给到了。
```

最小检查项：

| 检查项 | 问题 |
|---|---|
| core_mechanism | 这条内容真正要讲清的机制是什么 |
| audience_silent_truth | 有没有说出目标人群心里有但说不清的话 |
| stance_consistency | 立场是否从头到尾稳定 |
| promise_payoff | Hook 承诺是否在正文兑现 |
| memory_point | 观众看完能记住哪句话 / 哪个判断 |

## 5. 门禁补充

后续 R1 编译时，`copywriting-draft-writer` 不应只看 Hook 分数。

建议门禁：

| 门禁 | 通过标准 | 不通过去哪里 |
|---|---|---|
| Hook 路由 | `hook_route` 清楚，且有正文兑现段 | 回 draft |
| 正文信息密度 | `body_information_density_score >= 7` | 回 draft |
| 核心机制 | 一句话能说清这条内容讲明白了什么 | 回 Brief 或 draft |
| 逻辑延续 | 没有明显段落跳跃 | 回 draft |
| 产品桥接 | 产品露出服务观点，不扩大能力 | 回 Brief 或 quality_review |

如果只有 Hook 强，但正文低密度：

```text
draft_status = draft_needs_body_fix
next_skill = copywriting-draft-writer
```

如果 Hook 路由和核心机制都不清：

```text
draft_status = draft_needs_brief_fix
next_skill = content-brief-compiler
```

## 7. 成熟度判断

当前 P01 合同如果不补本文件，只能算：

```text
L2.5：流程合同完整，但内容质量合同不足。
```

补入本文件并完成 R1 后，可以接近：

```text
L3：不仅知道下一步去哪，也知道文案为什么能继续往下走。
```

但仍不追求完美：

```text
不预测播放量。
不承诺爆款。
不做平台算法模拟。
不要求每条内容都有复杂结构。
不把内容创作变成重型剧本工程。
```

本项目的取舍是：

```text
用最小字段保证 Hook 有路由、正文有信息增量、质检能发现撑不起承诺的问题。
```
