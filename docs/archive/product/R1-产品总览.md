## 1. R1 要解决什么

R1 解决的是：

```text
把“agent 扶着能跑通”的内容 workflow，收敛成“skill 能按合同执行”的第一组规则。
```

具体覆盖：

```text
P01：skill 不是执行合同。
P02：依赖 agent 扶着跑。
P08 子集：选题确认后不应反复让用户说继续。
P13：execution trace 只是记录，不是检查器。
P14：讨论稿没有编译成 skill / 合同 / 字段 / 检查项。
P15：skill 粒度和入口不够 dbskill 式清楚。
内容质量子集：不能只强调前 5 秒 Hook，还要保证正文信息密度、共鸣、承诺兑现和核心机制。
```

R1 不解决：

```text
多选题 fan-out / fan-in。
旁支任务封锁。
画中画数量和图片资产合同细化。
最终 HTML 模板产品化细节和可转交包。
Seedream 等外部模型降级链路。
GitHub 开源发布包净化。
```

这些分别进入 R2 / R3 / R4。

## 1.1 SAMPLE-HISTORICAL-005 后的 R1 返修边界

SAMPLE-HISTORICAL-005 已经跑通 R1 单篇主链路，但它不是完整真实测试通过的证据。本次样本暴露的问题分为三类：

| 类别 | 是否属于 R1 | 处理方式 |
|---|---|---|
| 单篇链路自动推进、账号 / 产品前置、research_run_id 贯穿、最终 HTML、trace 自洽 | 是 | 进入 R1 产品层返修 |
| visual_plan prompt 缩水、quality_review 没拦住缩水画中画、trace 使用 imagegen 前后矛盾 | 是，但偏 skill 编译问题 | 先在 R1 产品层明确验收口径，再进入 skill 编译修订 |
| 长任务断流、多选 fan-out、图片数量合同、Seedream 旁路、portable_bundle、开源样例 | 否 | 分别归入 R2 / R3 / R4 |

因此 R1 返修只修这些点：

```text
1. R1 完成后不能直接进入完整真实测试。
2. R1 sample run 只能证明单篇主链路，不证明 R2-R4。
3. visual_plan 的完整 prompt 结构必须成为 R1 阻断检查。
4. quality_review 必须能拦住“有图但不合格”的画中画。
5. execution_trace 必须自洽，不能同时声明用了 imagegen 又说未使用。
```

R1 返修不做：

```text
图片数量通用合同。
Seedream 外部模型适配。
多篇并行生产。
断流恢复运行模型。
可转交包 / 单文件 HTML。
GitHub 开源包。
```

## 1.2 R1 的恢复证据边界

结合 SAMPLE-HISTORICAL-005 的断流问题和成熟 workflow 做法，R1 必须区分两件事：

```text
恢复证据：R1 必须有。
脚本级断点续跑：R1 不做，进入 R2。
```

R1 必须做到：

| 项 | R1 要求 |
|---|---|
| 当前状态 | `manifest.yaml` 必须记录 `current_stage`、`current_artifact`、`next_skill`、`session_status` |
| 事件历史 | `intermediate/00-execution-trace.md` 必须记录每一步输入、输出、执行来源和结果 |
| 人类决策 | 账号确认、选题选择、最终验收等必须记录 decision_type 和 state_updates |
| 外部能力 | imagegen、联网、文件系统等必须标记为 environment_capability |
| 恢复判断 | 断流后，AI 应能读 manifest + trace 判断“从哪里继续 / 是否已经完成 / 是否只需 postcheck” |

R1 不承诺：

```text
自动 resume runner。
阶段级 checkpoint 文件。
幂等重跑规则。
失败重试策略。
run lock。
多分支恢复。
结构化日志查询。
```

这些归入 R2 运行模型。R1 如果把“可人工 / AI 恢复判断”写成“脚本级断点续跑”，视为测试范围误报。

## 2. 成熟 workflow 对 R1 的启发

R1 参考成熟 workflow 的共性，不照搬重型工程系统。

| 成熟做法 | 本项目吸收 | 本阶段不做 |
|---|---|---|
| Workflow Definition | 每个 skill 有触发、输入、输出、停顿点、失败处理和下一步 | 不引入真正 workflow engine |
| Interrupt / Human Gate | 只在账号确认、选题确认、风险返工、最终验收等位置停 | 不把每个中间步骤都变成人工确认 |
| Asset / Lineage | research_run_id、topic_id、brief_id、draft_id 等贯穿链路 | R1 不细化图片资产链 |
| Validator / Scorecard | trace/check 注册表能判断断链、错停、扶跑过重 | R1 先做检查清单，不做自动脚本 |
| Small Skills | 每个 skill 小而清楚，旧入口只做兼容 | 不把所有逻辑塞回一个大入口 |
| Open Source Readability | 给新读者明确入口、真源、确认项和后续动作 | R1 不完成开源发布包 |

取舍口径：

```text
本项目是轻量内容 workflow / skill 包，不是服务端编排平台。
R1 的合理目标是“确认后可进入 skill 编译和样本验证”，不是“整个项目已经开源可上线”。
```

## 5. R1 质量评估标准

R1 不追求“文档最多”，追求五个质量结果。

| 质量项 | 合格标准 | 当前判断 |
|---|---|---|
| 入口清楚 | 新读者知道先读哪 3 个文件 | 已由本总览补齐 |
| 规则真源清楚 | 字段、门禁、检查项、合同版本各有真源 | 已补齐 |
| 编译边界清楚 | 知道确认前不能改 SKILL，确认后按 P15 顺序编译 | 已补齐 |
| 人类确认清楚 | 不是一句模糊“同意”，而是 R1-C01 到 R1-C13 的确认 | 已补齐 |
| 后续风险清楚 | 知道 R1 不解决 R2/R3/R4，不误报开源就绪 | 已补齐 |

R1 仍然不算完美，原因是：

```text
validator 还是清单，不是脚本。
真实 sample run 还没在编译后验证。
R2 / R3 / R4 还没完成。
开源示例、贡献文档、净化发布包还没做。
```

因此 R1 当前最多只能进入：

```text
product_definition_ready_for_confirmation
```

确认并编译、验证后，才可能进入：

```text
L3 candidate
```


## 7. R1 完成口径

R1 产品层完成，不等于 R1 编译完成。

| 阶段 | 可说 | 不可说 |
|---|---|---|
| 产品层已完成 | R1 可以提交确认 | R1 已经 L3 |
| 涛哥已确认 | 可以进入 skill 编译 | skill 已经稳定 |
| SKILL.md 已编译 | 可以跑样本验证 | 项目已开源可用 |
| 样本验证通过 | 核心 skill 进入 L3 候选 | R2/R3/R4 已完成 |
