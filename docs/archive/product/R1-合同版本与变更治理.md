# R1 合同版本与变更治理

> 归档状态：historical_audit_only；2026-07-18 移出当前产品热路径，不得作为 current 合同真源。

> 状态：已确认并已编译引用  
> 所属路线：R1 方法论没有编译成执行合同  
> 解决缺口：合同版本治理、变更兼容、旧入口迁移、旧 session 恢复口径  
> 边界：本文件只定义产品层版本规则；不改 `skills/*/SKILL.md`，不修改历史 session 产物。

---

## 1. 为什么 R1 需要合同版本

R1 已经有 8 个核心 skill 合同，但如果没有版本治理，会出现：

```text
今天确认了一套合同，明天又改了字段，旧 session 不知道按哪套恢复。
某个 skill 合同改了，其他合同还停在旧口径，链路断在中间。
hotspot-copywriting-research 这种旧入口不知道是兼容、废弃还是继续扩展。
R1 总验收说“可编译”，但没有说明是哪一版合同可编译。
```

成熟 workflow 的共同点不是“永远不变”，而是变更有版本、有兼容边界、有迁移说明。

---

## 2. 版本对象

R1 里有三层版本，不得混用：

| 版本对象 | 字段名 | 含义 | 示例 |
|---|---|---|---|
| 合同组版本 | `contract_set_version` | R1 核心链路合同作为一组的版本 | `r1-contract-set-v0.1` |
| 单 skill 合同版本 | `contract_version` | 某个 skill 合同自己的版本 | `0.1.0` |
| 运行样本适配版本 | `run_contract_set_version` | 某个 session 按哪套合同运行或验收 | `r1-contract-set-v0.1` |

规则：

```text
R1 是否可编译，必须绑定 contract_set_version。
单个 CONTRACT.md 可以小版本变更，但 R1 总验收必须记录合同组版本。
真实 session 的 manifest / trace 后续应记录 run_contract_set_version。
```

## 3. R1 初始合同组

当前 R1 初始合同组定义为：

```yaml
contract_set_version: r1-contract-set-v0.1
status: draft
scope:
  - propagation-router
  - hotspot-topic-research
  - content-brief-compiler
  - copywriting-draft-writer
  - talking-head-image-pip
  - copywriting-quality-review
  - platform-packaging-adapter
  - final-delivery-builder
compatibility_entry:
  - hotspot-copywriting-research
```

`r1-contract-set-v0.1` 的状态只能在涛哥确认后从 `draft` 进入：

```text
confirmed_for_compilation
```

不能直接跳到：

```text
active
```

`active` 必须等对应 `SKILL.md` 编译且样本验证后再讨论。

---

## 4. 状态机

合同组状态：

```text
draft
-> ready_for_confirmation
-> confirmed_for_compilation
-> compiling
-> compiled_pending_sample
-> sample_verified
-> active
-> deprecated
```

状态含义：

| 状态 | 含义 | 允许动作 |
|---|---|---|
| `draft` | 产品定义仍在补齐 | 只能改产品文档 / 合同草案 |
| `ready_for_confirmation` | 产品层已齐，等待涛哥确认 | 可继续微调，不改 SKILL.md |
| `confirmed_for_compilation` | 涛哥确认进入编译 | 可把 CONTRACT 状态改 confirmed |
| `compiling` | 正在改 `SKILL.md` | 只能按已确认合同改 |
| `compiled_pending_sample` | skill 已编译，待样本验证 | 可跑 sample run |
| `sample_verified` | 样本验证通过 | 可评估 L3 候选 |
| `active` | 当前推荐使用版本 | 可作为默认运行合同 |
| `deprecated` | 旧版本不再扩展 | 仅支持恢复旧 session 或迁移 |

## 5. 单 skill 合同版本规则

每个核心 `CONTRACT.md` 必须有：

```yaml
contract_version: 0.1.0
contract_set_version: r1-contract-set-v0.1
status: draft / confirmed / compiling / active / deprecated
```

版本号含义：

| 变更 | 版本变化 | 示例 |
|---|---|---|
| 文案澄清，不改变字段 / 门禁 / 输出 | patch | `0.1.0 -> 0.1.1` |
| 新增可选字段、补充失败处理 | minor | `0.1.0 -> 0.2.0` |
| 改必需字段、改自动推进、改人类门禁 | major | `0.1.0 -> 1.0.0` |

R1 阶段如果发生 major 级产品改动，必须回到 R1 总验收，不得直接编译。

---

## 6. 变更类型

| 变更类型 | 是否需要涛哥确认 | 是否影响 R1 总验收 |
|---|---|---|
| typo / 格式 / 链接修正 | 否 | 否 |
| 解释补充，不改变合同 | 否，记录即可 | 否 |
| 新增可选字段 | 视影响而定 | 可能 |
| 新增必需字段 | 是 | 是 |
| 改自动推进规则 | 是 | 是 |
| 改人类门禁 | 是 | 是 |
| 改输出路径 | 是 | 是 |
| 改旧入口兼容策略 | 是 | 是 |
| 降低安全边界 / 自动发布边界 | 不允许 | 阻断 |

## 7. 旧 session 恢复口径

R1 版本治理不强制改历史产物。

历史 session 恢复时按以下规则：

```text
如果 manifest 记录了 run_contract_set_version，按记录版本恢复。
如果没有记录，则按 session 创建时 STATUS / trace 的项目状态推断，并标记 inferred_contract_version。
如果旧 session 缺少新字段，不自动补成事实，只能标记 missing_in_legacy_run。
如果用户要求继续旧 session，优先提示“这是旧合同样本，可能需要一次迁移检查”。
```

旧 session 可以被迁移，但迁移必须生成：

```text
contract_migration_note
```

不得静默覆盖历史产物。

## 8. 旧入口治理

`hotspot-copywriting-research` 在 `r1-contract-set-v0.1` 中定位为：

```text
compatibility_entry
```

产品规则：

```text
不建立完整主链路合同。
不承载新字段。
不承载新流程。
不作为 README 主入口。
只做旧唤醒词解释和转发。
```

状态建议：

```yaml
compatibility_status: alias_only
target_entry: propagation-router
deprecation_policy: keep_until_r1_active_then_review
```

如果后续发现旧入口仍被大量使用，再进入 R4 开源入口设计，不在 R1 扩展它。
