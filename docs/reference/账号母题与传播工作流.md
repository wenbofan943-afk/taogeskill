# 账号母题与传播工作流

> 状态：传播研究草案
> 主责：定义本项目自媒体传播的账号母题、热点关联、内容创作和 dbskill 质检流程。
> 边界：本文是传播战术规划，不是产品验收包，不是工程开发任务，不直接修改客户端、服务器、数据库、积分或发版链路。

---

## 一、核心判断

当前产品能力在 v1.9.1 阶段基本固定：

```text
公开互动分析工具
-> 评论区 / 直播间公开互动整理
-> AI 分析
-> 分析结果 / 重点互动 / 互动资产库
-> 免费学习版申请制
```

传播工作不能脱离产品能力自嗨。所有内容都要从产品能力倒推：

1. 谁会用。
2. 为什么要用。
3. 原来怎么解决。
4. 我们减少了什么麻烦。
5. 哪些热点能放大这个问题。
6. 这个热点和产品能力有没有真实关系。

同时，账号母题不是热点搜索关键词。账号母题定义的是“这个账号能接住什么热点”，不是让 AI 只按母题窄搜热点。

---

## 二、账号母题

“账号母题”是账号长期反复生产内容的战术规划单元。

它不是单篇选题，也不是一句 slogan，而是：

```text
一个稳定的人群问题
+ 一个稳定的产品能力对应
+ 一组可持续展开的内容角度
+ 一套可蹭热点的关联规则
```

成熟账号不能每天临时想发什么。要先有母题，再从母题里长出选题。

## 三、当前推荐账号母题

### 母题 1：评论区里有价值，但大多数人没整理

对应人群：

1. 二手本地经营者。
2. 本地生活商家。
3. 门店老板。
4. 做账号矩阵的小团队。

用户问题：

```text
评论区每天有人问价、问库存、问服务、骂产品、提需求，但老板和运营只是偶尔翻一翻，没有系统整理。
```

对应产品能力：

```text
评论区公开互动采集 -> AI 分析 -> 重点互动 / 分析结果 / 互动资产库
```

可持续选题方向：

1. 为什么评论区是最便宜的用户调研。
2. 为什么很多商家一边说没客户，一边不看评论区。
3. 评论区里哪些话值得老板亲自看。
4. 直播间下播后，评论不应该直接消失。
5. AI 不只是写文案，也能帮你整理用户反馈。

---

### 母题 2：账号矩阵不是多发内容，而是统一看反馈

对应人群：

1. 多门店商家。
2. MCN / 账号工作室。
3. 企业内部多个品牌号 / 产品号。
4. 销售号、老板号、门店号同时运营的团队。

用户问题：

```text
账号越多，反馈越散。每个号都有人评论，但没有人把这些公开互动统一整理。
```

对应产品能力：

```text
账号池 / 授权账号管理 -> 多账号公开互动整理 -> 统一分析结果
```

可持续选题方向：

1. 账号矩阵最怕的不是没内容，是没人看反馈。
2. 老板号、销售号、门店号的评论，应该放在一起看。
3. 多账号运营不是发得多，而是反馈闭环更快。
4. 账号矩阵的真实资产，不只是粉丝数，还有评论里的问题。

---

### 母题 3：直播复盘不能只看 GMV，也要看互动

对应人群：

1. 直播运营。
2. 店播团队。
3. 本地商家直播间。
4. 本地服务直播间。

用户问题：

```text
直播间评论刷得快，下播后只看成交和在线人数，没把用户问了什么、卡在哪里、对什么感兴趣整理出来。
```

对应产品能力：

```text
直播间评论整理 -> AI 分析 -> 直播互动复盘
```

可持续选题方向：

1. 下播后，你真的知道用户在直播间问了什么吗。
2. 直播复盘不能只看成交额。
3. 直播间里没人买，不代表没人表达需求。
4. 直播评论是产品、话术和成交阻力的现场记录。

---

### 母题 4：AI 的价值不只是生成内容，也能整理现场

对应人群：

1. 小团队老板。
2. 运营人员。
3. 内容团队。
4. 售后 / 客服 / 产品负责人。

用户问题：

```text
大家习惯用 AI 写文案，却忽略 AI 可以整理已有公开互动，把评论、反馈、问题和需求变成可读结果。
```

对应产品能力：

```text
公开互动内容 -> AI 分类 / 摘要 / 重点识别 -> 可导出结果
```

可持续选题方向：

1. AI 不只会写，它还能帮你看。
2. 比起让 AI 编内容，不如先让 AI 整理真实评论。
3. 小团队最缺的不是工具，是把现场信息整理清楚。
4. 评论区和直播间，是你已经拥有的数据源。

---

### 母题 5：免费学习版不是免费白嫖，而是开放试用和真实反馈

对应人群：

1. 愿意尝鲜的个人用户。
2. 有真实账号运营需求的小团队。
3. 对 AI 工具有兴趣的运营人员。
4. 可能成为早期客户的企业用户。

用户问题：

```text
用户不知道工具是否真能用，也不知道自己的场景适不适合。直接收费推广成本高，免费学习版可以先降低试用门槛。
```

对应产品能力：

```text
/trial 申请 -> 人工审核 -> 免费学习版 license -> AI 分析额度 -> 使用观察
```

可持续选题方向：

1. 为什么我先把这个工具免费给大家试。
2. 一个工具还没被真实用户骂过，就别急着商业化。
3. 免费学习版不是无限免费，是为了验证真实场景。
4. 我想看看真实商家到底怎么用公开互动分析。

## 四、热点关联度评分

热点不是看到就蹭。先评分。

每个热点按 5 个维度各 1 分：

| 维度 | 判断问题 |
|---|---|
| 人群相关 | 这个热点和我们的目标用户有关吗 |
| 问题相关 | 它能引出评论管理、直播复盘、用户反馈、账号矩阵或 AI 分析吗 |
| 产品相关 | 我们工具能力能自然接上吗 |
| 风险可控 | 会不会显得灰产、蹭灾难、蹭争议或越界 |
| 表达空间 | 能不能形成清楚观点，而不是硬贴产品 |

评分规则：

```text
0-2 分：不蹭
3 分：轻内容，可以做评论或短观点
4 分：可做正式内容
5 分：重点内容，值得做图文 + 短视频组合
```

硬规则：

1. 不能为了蹭热点，把产品说成它做不到的东西。
2. 不能借热点强化“截留、骚扰、批量获客、灰产引流”口径。
3. 不能蹭灾难、个人隐私、违法犯罪细节。
4. 不能把公开互动分析讲成自然人身份识别或跨平台画像。

## 五、内容生产流水线

```text
账号档案
-> 三池热点雷达
   - S 池：公共大热点 / 大时代情绪
   - B 池：行业热点 / 圈层热点
   - A/C 池：目标人群现场 / 账号母题问题
-> S/B/A/C/D 桥接
-> 关联度评分
-> 文案策略判断
-> 人群痛点匹配
-> 产品能力匹配
-> 选题一句话
-> 推导链
-> 涛哥确认选题
-> 内容 Brief 编译
-> 调用 IP 资产（认知档案 / 表达工坊 / 知识原子）
-> 内容形式判断
-> 口播草案
-> 画中画视觉方案
-> 文案 + 视觉联合质检
-> 分发包装输入包
-> 多平台分发包
-> human_confirm（待涛哥最终确认）
```

v1.9.1 阶段，产品能力固定，内容自动化不改变产品本身。

真正可以自动化的是：

1. 从 S / B / A-C 三池里找候选热点。
2. 判断热点能否桥接到目标人群现场和账号母题。
3. 判断热点和产品能力是否真实相关。
4. 生成选题角度。
5. 选题确认后自动编译内容 Brief。
6. 生成口播草案。
7. 生成画中画视觉方案和 image 提示词。
8. 用 dbskill 方法论做文案 + 视觉联合质检和微调建议。
9. 质检通过后生成分发包装输入包和多平台分发包。

不能自动化的是：

1. 最终价值判断。
2. 是否发布。
3. 是否代表产品正式口径。
4. 是否改变产品承诺。

因此本项目传播工作流的最终闸门是：

```text
涛哥人工确认后再发布。
```

另外，文案创作前有一个人工中间闸门：

```text
选题 + 策略 + 推导链必须先确认。
```

内容 Brief 是选题确认后的自动编译层，不是默认人工审批节点；只有缺信息或高风险时才停下来找涛哥确认。

也就是说，默认流程不是从热点直接出文案，而是：

```text
账号档案
-> 三池热点雷达
-> 热点候选
-> S/B/A/C/D 桥接
-> 评分
-> 选题卡
-> 文案策略
-> 涛哥确认切口
-> 内容 Brief 编译
-> IP 资产调用
-> 再生成口播草案
-> 画中画视觉方案
-> 文案 + 视觉联合质检
-> 分发包装输入包
-> 多平台分发包
-> human_confirm（待涛哥最终确认）
```

只有涛哥明确说“不用中间确认，直接出文案”，才跳过这个中间闸门。

本阶段默认到 `platform_package.package_status = package_pass` 且 `next_skill = human_confirm` 结束，不默认进入发布和发布复盘。

每个选题必须回答：

1. 讲给谁。
2. 为什么他现在要听。
3. 属于哪个账号母题。
4. 和哪个热点有关；如果无热点，也要说明为什么常青。
5. 和产品哪个能力有关。
6. 会不会被误解。
7. 用哪个 dbskill 质检。
8. 发布后看什么反馈。

写文案前还必须回答：

1. 本轮调用了哪些涛哥表达资产。
2. 本轮调用了哪些知识原子或案例。
3. 哪些话是当前产品不能承诺的。
4. 哪些表达会让人误解成灰产、截流或自动骚扰。

如果没有调用到 IP 资产，允许继续写草案，但必须标注“本轮未调用 IP 资产”，避免把通用 AI 文案误当成涛哥本人表达。

## 六、dbskill 质检分工

dbskill 不负责替我们写内容，它负责把关和微调。

| 阶段 | 使用工具 | 作用 |
|---|---|---|
| 选题确认后 | `dbs-content` | 判断这个选题适合什么形式、表达是否服务产品 |
| 热点借势前 | `dbs-spread` | 判断热点背后的情绪和传播心理，避免硬蹭 |
| 文稿初稿后 | `dbs-resonate` | 检查有没有共鸣，核心是否被稀释 |
| 短视频开头 | `dbs-hook` | 检查前 5 秒是否建立话题、Hook 和可信度 |
| 小红书标题 | `dbs-xhs-title` | 做标题和封面方向 |
| 对标拆解 | `dbs-benchmark` | 拆别人怎么讲 AI 工具、数据工具、运营工具 |
| 商业化复盘 | `dbs-diagnosis` | 阶段性看免费版、申请制、付费版和服务化方向 |

## 七、第一阶段工作

先不做 100 个选题。

第一阶段只做：

1. 每个账号母题各 2 个选题。
2. 共 10 个选题。
3. 每个选题都按“人群、痛点、热点、产品能力、风险、dbskill 质检”六项记录。
4. 选出 3 个最适合先发的内容，分别做短视频脚本、图文草案和 GitHub / 官网口径联动。

目标不是一口气铺量，而是验证：

```text
哪个人群有反应
哪个母题能讲清楚
哪个表达不会跑偏
哪个内容能带来申请和试用
```
