# 文档治理与目录规范

> 状态：目录和产物治理基准
> 主责：规定涛哥创作工作流如何摆放方法论、skill、账号档案、中间产物、最终交付物和索引。
> 边界：本文件不是产品数据库 schema，不改变客户端、服务器、发布链路或平台账号行为。

---

## 一、设计依据

本项目参考成熟 workflow 的共性做法：

1. 数据 / 内容产物分层：原始输入、中间产物、最终输出要分开，避免草案被误当交付物。
2. pipeline 分层：每一层只产给下一层用的稳定交接物，避免一个文件承担所有责任。
3. 文档分型：教程、操作指南、参考契约、方法论解释分开，降低人和 AI 的阅读负担。
4. manifest 索引：每轮内容必须有机器可读索引，记录输入、输出、状态、路径和下一步。
5. 前台交付和后台追溯分离：最终给人看的页面必须好读、好复制、好下载；后台 Markdown 负责追溯和恢复。
6. 可转交包自描述：离开项目目录的交付包必须带资源清单和包内链接，不能依赖原 session 路径。
7. 人类停顿点和自动推进分离：需要人判断时给推荐动作和可直接回复的话；门禁已通过时自动进入下一环节，不让用户说“继续”。

这些原则落到本项目，就是：

```text
账号隔离
session 隔离
intermediate / deliverables 隔离
人读 README
AI 读 manifest
字段按词典
引导按规范
跨账号看 indexes
```

人类引导和任务后导航的真源是 `docs/reference/人类引导与任务后导航规范.md`。任何 skill 需要等待用户选择时，必须按它输出主推荐、原因、可回复短句和后续动作；任何 skill 门禁已通过且没有人工选择点时，不得要求用户再说“继续”。

skill 执行透明度的真源是 `docs/reference/skill执行透明度与成熟度规范.md`。每轮 session 必须记录 `intermediate/00-execution-trace.md`，区分 skill_defined、agent_orchestrated、user_decision、environment_capability 等来源，避免把 agent 扶跑误判成 skill 成熟能力。

R2 多分支、断流恢复、checkpoint、parent / child 生命周期、分支台账和状态命名空间的真源是 `docs/reference/R2-运行模型执行规范.md`。涉及“三篇都做 / 接着上次 / 活了吗 / 分支恢复 / 索引重建”的运行逻辑时，必须先按该规范判断，不得在单个 skill 里临场发明状态。

## 三、账号目录

每个账号必须是独立目录：

```text
accounts/{account_slug}/
├── README.md
├── account_profile.md
├── mother_topics.md
├── style_guide.md
├── boundaries.md
├── runs/
└── index.md
```

说明：

| 文件 | 主责 |
|---|---|
| `README.md` | 账号入口和当前状态 |
| `account_profile.md` | 账号档案，替代旧的 `accounts/{账号名}.md` |
| `mother_topics.md` | 账号母题和栏目方向 |
| `style_guide.md` | 表达风格、常用表达、禁用表达 |
| `boundaries.md` | 热点边界、产品露出边界、风险禁区 |
| `runs/` | 该账号每轮内容 workflow |
| `index.md` | 该账号内容索引 |

如果暂时没有拆出 `mother_topics.md`、`style_guide.md`、`boundaries.md`，可先让 `account_profile.md` 承载，但新增内容不得再回到平铺文件。

## 四、单轮 session 目录

每条真实内容一个 session：

```text
accounts/{account_slug}/runs/{session_id}/
├── README.md
├── manifest.yaml
├── inputs/
├── intermediate/
├── deliverables/
│   └── export/
├── assets/
│   ├── images/
│   │   ├── generation-records/
│   │   └── metadata/
│   └── source-material/
└── archive/
```

### `inputs/`

保存本轮开始时的输入快照：

```text
account_snapshot.md
product_snapshot.md
campaign_snapshot.md
source_notes.md
```

输入快照解决“后来账号档案变了，旧内容为什么这么写”的溯源问题。

### `intermediate/`

保存 AI 工作链路中的中间产物：

```text
00-execution-trace.md
01-research-run.md
02-topic-card.md
03-content-brief.md
04-draft.md
05-visual-plan.md
06-quality-review.md
07-platform-package-input.md
08-platform-package-draft.md
checkpoints/
branch-request-ledger.md
branch-summary.md
```

中间产物可以被自动覆盖、修订或返工，但必须保留 `source_research_run_id` 和上游 ID。

R2 规则：

```text
checkpoints/ 保存追加式恢复快照。
branch-request-ledger.md 只在 parent session 有分支请求时生成。
branch-summary.md 只做 fan-in 汇总，不合并多个 child 的正文。
```

### `deliverables/`

保存最终交付物：

```text
final-delivery.html
final-script.md
final-visual-plan.md
final-platform-package.md
content-delivery-record.md
export/
```

`final-delivery.html` 是人类验收入口；Markdown 文件是它的后台源材料和追溯链接。只有人工确认或进入最终确认节点后的内容，才能写入 `deliverables/`。草案不得命名为 final。

### `deliverables/export/`

保存可转交交付包。项目内验收页和可转交包必须分开：

```text
deliverables/export/{session_id}/
├── final-delivery.html
├── assets/
│   └── images/
├── sources/
├── export-manifest.json
└── manifest-sha256.txt
```

规则：

```text
project_local：deliverables/final-delivery.html，允许链接到 session 内 intermediate、assets、deliverables。
portable_bundle：deliverables/export/{session_id}/，所有链接必须指向包内文件。
standalone_html：deliverables/export/{session_id}/final-delivery-standalone.html，图片和关键摘要内嵌，尽量不依赖外部文件。
```

如果用户只是自己在项目里验收，生成 `project_local` 即可；如果用户说“发给别人 / 交付客户 / 传网盘 / 拿到别的电脑看”，必须生成 `portable_bundle` 或 `standalone_html`。

### `assets/`

保存生成图、参考图、截图、脱敏素材、导出表格示意等素材。真实用户资料必须脱敏。

每张最终画中画图片必须有稳定文件名、对应 beat_id、插入位置和生成状态。图片生成失败时不得假装已有图片，必须在最终交付页标记待外部生成或人工补图。

R3 图片资产链必须保留：

```text
assets/images/image-assets.md
assets/images/generation-records/
assets/images/metadata/
```

规则：

```text
prompt 卡不是图片资产。
每次生成、外部待生成、失败或人工上传必须有 generation record。
generated 图片必须有 metadata sidecar。
pending_external / generation_failed / manual_required 不得在 HTML 中展示为 generated。
图片重做不得覆盖旧 image_asset_id。
```

### `archive/`

保存废弃方案、返工记录和版本差异，防止主链路变脏。

## 五、manifest.yaml

每个 session 必须有 `manifest.yaml`。它是 AI 恢复工作流的首选机器可读索引。

最小字段：

```yaml
contract_set_version:
session_id:
account:
account_slug:
product_profile_id:
campaign_profile_id:
started_at:
updated_at:
current_stage:
current_artifact:
source_research_run_id:
next_skill:
session_status:
trace_check:
  trace_check_id:
  overall_result:
  blocking_count:
  warning_count:
r2_runtime:
  task_context_type:
  content_run_id:
  parent_session_id:
  branch_request_id:
  branch_request_status:
  fan_out_status:
  fan_in_status:
  run_lock:
    run_lock_status:
    lock_owner:
    lock_owner_type:
  recovery:
    last_completed_stage:
    current_stage:
    failed_stage:
    resume_from:
    resume_action:
    idempotency_policy:
  latest_checkpoint:
  state_transitions:
artifacts:
  research_run:
    id:
    path:
    status:
  topic_card:
    id:
    path:
    status:
  content_brief:
    id:
    path:
    status:
```

规则：

```text
current_artifact 必须指向 accounts/{account_slug}/runs/{session_id}/ 下的文件。
artifacts.*.path 使用 session 内相对路径。
human_deliverables 必须列出 final_delivery、image_assets、export_bundle 等人类可用产物。
media_assets 必须列出图片文件、用途、生成状态和路径。
R3 编译后的样本必须写入 image_asset_set、generation_records、metadata_sidecar 和 html_embed_manifest 摘要。
R1 编译后的新样本必须写入 `contract_set_version: r1-contract-set-v0.1`。
R1 编译后的新样本必须写入 trace_check 摘要，便于判断是否达到 L3 候选。
R2 编译后的新样本必须写入 R2 runtime 字段；状态必须使用 `run_*`、`branch_request_*`、`fan_out_*`、`fan_in_*` 命名空间。
manifest.yaml 和 工作流状态记录.md 冲突时，先读取具体产物，再修正两边索引。
```

### R1 新样本边界

R1 sample run 必须创建新的 session 目录，不能复用 R1 编译前旧 session 当验收样本。

新建前必须先读取：

```text
docs/reference/R1-sample-run产物模板.md
```

新样本目录必须满足：

```text
accounts/{account_slug}/runs/{new_session_id}/manifest.yaml
accounts/{account_slug}/runs/{new_session_id}/intermediate/00-execution-trace.md
```

manifest 必须写：

```yaml
contract_set_version: r1-contract-set-v0.1
sample_run_type: r1_single_content
legacy_session: false
```

如果只是恢复旧 session，则必须写：

```yaml
legacy_session: true
not_for_r1_acceptance: true
```

| skill | 输入 | 输出 |
|---|---|---|
| `hotspot-topic-research` | `account_profile`、`product_profile`、来源池 | `intermediate/01-research-run.md`、`intermediate/02-topic-card.md` |
| `content-brief-compiler` | 已选择 `topic_card` | `intermediate/03-content-brief.md` |
| `copywriting-draft-writer` | `content_brief` | `intermediate/04-draft.md` |
| `talking-head-image-pip` | `draft` | `intermediate/05-visual-plan.md`、`assets/images/` |
| `copywriting-quality-review` | `draft` + `visual_plan` | `intermediate/06-quality-review.md` |

| `final-delivery-builder` | `content_delivery_record` + `platform_package` + `image_assets` | `deliverables/final-delivery.html`、可选 `deliverables/export/{session_id}/` |

当 `content_delivery_record.delivery_status = delivery_confirmed` 后，再生成：

```text
deliverables/final-delivery.html
deliverables/final-script.md
deliverables/final-visual-plan.md
deliverables/final-platform-package.md
```

最终交付页必须整合选题切口、正式文案、实际画中画图片、插入位置和多平台发布物料，并链接回对应 Markdown 追溯文件。

如果生成可转交包，还必须生成：

```text
deliverables/export/{session_id}/final-delivery.html
deliverables/export/{session_id}/assets/images/
deliverables/export/{session_id}/sources/
deliverables/export/{session_id}/export-manifest.json
```

## 八、迁移与清理规则

以下旧路径只作为迁移识别对象，不得继续创建或保留为空壳入口：

```text
accounts/{账号名}.md
内容产物/{账号名}/{session_id}/
根目录完整 Brief / 文案 / 画中画正文
```

迁移到：

```text
accounts/{account_slug}/account_profile.md
accounts/{account_slug}/runs/{session_id}/intermediate/
accounts/{account_slug}/runs/{session_id}/deliverables/
```

迁移后：

```text
更新 工作流状态记录.md 的 current_artifact。
更新 session manifest.yaml。
根目录汇总表只留索引和摘要。
确认没有实际产物后，删除旧目录空壳，不在根目录保留“内容产物”入口。
如果因审计需要临时保留旧文件，必须迁入对应账号 session 的 archive/，并在 manifest.yaml 里标记 archive_artifact。
```

### 发版候选目录

公开候选包、zip、hash 和 release gate 报告不属于根目录入口文件，必须进入版本化发版目录：

```text
releases/v{version}/
├── public_release/
├── taoge-creative-workflow-{version}-public-release.zip
├── taoge-creative-workflow-{version}-public-release.zip.sha256
├── release-gate-report.md
└── release-gate-report.json
```

根目录不得保留新生成的：

```text
public_release/
*.public-release.zip
*.public-release.zip.sha256
release-gate-report.md
release-gate-report.json
```

如发现旧产物散落在根目录，应先确认它们是脚本生成物，再迁入 `releases/v{version}/` 或删除后重建。

## 十、完成定义

一轮内容产物治理合格，必须满足：

```text
账号目录存在。
session 目录存在。
manifest.yaml 存在。
current_artifact 指向 session 目录。
intermediate 和 deliverables 没混放。
source_research_run_id 没断链。
根目录汇总表不是唯一正文来源。
人能从 README 看懂状态。
AI 能从 manifest.yaml 恢复链路。
最终交付页 final-delivery.html 可供人类直接验收、复制文字和下载图片。
必要画中画图片已生成到 assets/images/，或明确标记为 pending_external / generation_failed。
generated 图片有 metadata sidecar；pending / failed / manual 图片有 generation record 或人工任务说明。
如果用户需要转交，deliverables/export/{session_id}/ 存在，且 HTML 不依赖原 session 外部路径。
manifest.yaml 已记录 delivery_page_mode、final_delivery_status、image_assets_status 和 export_status。
```
