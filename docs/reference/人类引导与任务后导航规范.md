# 人类引导与任务后导航规范

> 状态：全局规则  
> 主责：规定本 workflow 什么时候停、怎么引导、什么时候自动继续。  
> 参考：`dbskill-dontbesilent2025` 的双模式路由思想：任务前路由，任务后导航。

---

## 一、核心原则

好的 workflow 不是让用户猜下一步，而是让用户做真正需要人判断的选择。

本项目必须遵守：

```text
能自动衔接的，不向用户索要“继续”。
需要人类判断的，给推荐方案、原因和可直接回复的话。
用户已经给出明确选择的，不二次确认。
任务结束后，根据本轮结论推荐 2-3 个下一步，并说明为什么。
```

用户只需要判断内容方向、最终 HTML 是否满意、是否归档、是否转交或人工发布；不需要知道 `next_skill`、`delivery_status`、`human_gate` 这些字段。

---

## 二、什么时候必须停

只在以下情况停给用户：

| 停顿点 | 为什么必须停 | 引导重点 |
|---|---|---|
| 账号档案 P0 缺失 | 不知道“谁来说”会串账号 | 一次最多问 3 个口语化问题 |
| 换账号后账号档案待确认 | 账号实际情况可能变化，旧画像会导致选题、语气和禁区偏移 | 读档案后给摘要，让用户认可或指出变化 |
| 产品 / 活动对象不清 | 不知道“这次说什么、不能怎么说”会越界 | 让用户补对象边界或选择已有对象 |
| Topic Gate 后选题 | 选题是方向判断，必须人选 | 先给选题决策面板，再给主推荐、备选、放弃或重找 |
| Brief 不通过 | 核心观点或产品边界不稳 | 建议回到 Brief 或补字段 |
| Hook / 画面出现多方案取舍 | 审美和策略取舍需要人判断 | 推荐一个默认方向，保留 A/B 选择 |
| 质检不通过 | 发布风险或质量风险未清 | 说明最小返工路径 |
| 平台包完成 | 不是人工门禁 | 自动生成内容交付记录并进入 final-delivery-builder |
| 最终交付页完成 | 人需要看最终可读交付物是否满意 | 给认可、局部返工、导出转交、人工发布记录、归档 / 放弃选项 |
| 人工发布后补记录 | 发布结果来自人工动作 | 引导记录发布时间、平台和链接 |

除此之外，默认自动推进。

### R1 决策类型

R1 编译后，人类门禁必须把用户自然语言映射为以下 `decision_type`，不得临场自造状态：

| decision_type | 含义 | 后续动作 |
|---|---|---|
| `approve` | 用户认可当前推荐或最终交付 | 自动进入下游或写入确认状态 |
| `select` | 用户选择一个明确对象，例如 topic_id / hook 方案 | 自动流转到对应下游 |
| `edit` | 用户要求局部修改 | 回到指定上游 artifact 后再自动推进 |
| `reject` | 用户否定当前产物 | 回上游重做或重找 |
| `archive` | 用户决定本条暂不继续 | session 进入归档 |
| `export` | 用户要求转交包或单文件 HTML | 进入 final-delivery-builder 导出链路 |
| `publish_record` | 用户已人工发布，要求记录 | 只记录人工发布结果，不自动发布 |
| `cancel` | 用户取消当前链路 | session 归档或放弃 |
| `branch_request` | 用户要求多篇、多账号或旁支任务 | R1 只识别并停住，交给 R2 fan-out / branch_lock |
| `manual_proxy_test` | 用户明确要求 agent 代替人工跑测试 | 允许 agent 模拟选择，但必须写入 manifest / trace，不能冒充真实用户确认 |
| `account_onboarding` | 首次使用或账号不存在，需要创建账号档案 | 进入 account-onboarding，创建草案后让用户确认 |

R1 不把“继续”作为合法必需决策。用户明确选择后，系统自动推进。

### R1 决策恢复表

人类回复被映射为 `decision_type` 后，必须同步更新 manifest / workflow_session_record；不得只在聊天里说“好的”。

| decision_type | 必须更新的字段 | 恢复位置 |
|---|---|---|
| `approve` | `last_decision=approve`，当前 artifact 的通过状态，`next_skill` 指向下游 | 下游 skill |
| `select` | `last_decision=select`，被选对象状态改为 selected，例如 `topic_status=topic_selected_for_brief` | 被选对象的下游 skill |
| `edit` | `last_decision=edit`，`revision_path`，`current_artifact` 指向需修改 artifact，`next_skill` 指向返工 skill；若来自最终交付页，还必须写 `delivery_status=delivery_needs_fix` | 返工 skill，修完后自动回 final-delivery-builder |
| `reject` | `last_decision=reject`，当前 artifact 状态改 rejected / needs_fix，`blocked_reason` 写明原因 | 上游重做节点 |
| `archive` | `last_decision=archive`，`session_status=session_archived`，必要时 `delivery_status=delivery_archived` | done |
| `export` | `last_decision=export`，`final_delivery_status=needs_export`，`export_status=not_requested / export_needs_fix` | final-delivery-builder |
| `publish_record` | `last_decision=publish_record`，`publish_status=publish_manually_published / publish_unknown` | workflow_session_record |
| `cancel` | `last_decision=cancel`，`session_status=session_archived`，必要时 `delivery_status=delivery_discarded` | done |
| `branch_request` | `last_decision=branch_request`，`blocked_reason=needs_r2_fan_out`，`session_status=session_waiting_human` | R2 运行模型，R1 不继续 |
| `manual_proxy_test` | `last_decision=manual_proxy_test`，`confirmation_status=simulated_by_agent_for_test`，`agent_assist_level` 至少为 medium | 测试链路继续，但 checker 必须给 warn |
| `account_onboarding` | `last_decision=account_onboarding`，`onboarding_status=waiting_human_confirmation`，`current_artifact` 指向 account_profile 草案 | 用户确认后回 propagation-router |

恢复时先读：

```text
manifest.yaml
-> workflow_session_record
-> current_artifact
```

如果 `last_decision` 已处理过，不得重复执行同一个人工决定；只能根据当前 artifact 状态继续。

### R2 多分支决策类型

R2 编译后，多分支和恢复相关的人类决策必须进入 `human_interrupt` / `human_decision_payload`，不得只停留在聊天里。

| decision_type | 含义 | 后续动作 |
|---|---|---|
| `branch_request` | 用户提出多篇、多账号或旁支任务 | 进入 R2 `detect_branch_request` |
| `branch_confirm` | 用户确认按规划拆分 child session | 进入 `plan_fan_out` / `create_child_session` |
| `branch_cancel` | 用户取消本次多分支请求 | `branch_request_status=branch_request_cancelled` |
| `resume` | 用户要求从断流处继续 | 按 R2 恢复读取顺序生成 `resume_report` |
| `archive_parent_keep_children` | 归档 parent，但保留 child | 更新 parent close policy 和 state_transition |
| `archive_parent_archive_children` | 归档 parent，同时归档 child | 写入 child 状态变化，不删除产物 |

R2 状态必须使用命名空间：

```text
run_* / branch_request_* / fan_out_* / fan_in_*
```

阻断原因写入：

```text
blocked_reason
```

## 三、必须自动推进的节点

以下节点不能要求用户再说“继续”：

| 当前结果 | 自动进入 | 禁止说法 |
|---|---|---|
| 用户已选 `topic_id`，且 topic_card 完整 | `content-brief-compiler` | “是否生成 Brief？” |
| `brief_status = brief_pass` 且 `human_gate = no` | `copywriting-draft-writer` | “请回复继续写口播” |
| 口播草案通过最低门槛，且需要画中画 | `talking-head-image-pip` | “是否做画中画？” |
| `review_status = review_pass` 且无人工门禁 | `platform-packaging-adapter` | “是否继续做分发包？” |
| 平台包通过并生成 `content_delivery_record` | `final-delivery-builder` | “是否生成最终交付页 / 是否继续？” |
| 最终 HTML 完成 | 等用户最终验收 / 人工发布 / 返工 / 归档 / 转交 | “下一步怎么做？” |

如果用户说“三篇都做 / 都要 / 这三个都跑”，R1 不在同一个 session 里硬跑多篇。必须输出：

```text
decision_type：branch_request
blocked_reason：needs_r2_fan_out
session_status：session_waiting_human
```

并用人话说明：这属于多分支生产，需要进入 R2 的 fan-out / fan-in 规则，避免三篇内容串号。

R2 编译后，如果当前任务是内容生产且选题池完整，用户说“三篇都做”时，不再停在抽象说明；必须生成 `human_decision_payload`，规划 parent / child session，写入 branch_request 和 `intermediate/branch-request-ledger.md`。如果当前任务是产品开发、文档治理或开源治理，则只登记 `branch_request_deferred`，不启动内容生产。

自动推进时可以说明：

```text
这一步已经通过，我会自动进入 {下一环节}。如果你想打断，可以直接说“回到 {上一环节} 改 {具体部分}”。
```

## 四、引导语结构

凡是停给用户，都必须同时包含：

```text
当前做到哪了：一句人话。
推荐你选什么：给一个默认推荐。
为什么推荐：基于本轮结论，不说空话。
还能怎么选：2-3 个安全备选。
你可以怎么回复：给可直接复制的短句。
每个回复会发生什么：说明会自动进入哪个环节或如何回退。
```

结构化交接物里必须有：

```text
decision_type
decision_mapping
human_prompt
human_reply_examples
recommended_action
auto_next_action
task_after_navigation
state_updates
```

`recommended_action` 是给 AI 和人看的默认推荐；`auto_next_action` 只在用户选择后描述自动流转，不要求用户说“继续”。

## 五、标准引导模板

### 1. 换账号后账号档案对齐

```text
我先把“{账号名}”的账号档案和你对齐一下。这样做是为了避免账号实际情况已经变了，但我还按旧画像找热点，最后选题、语气、产品露出或禁区不符合你的预期。

我读到的当前档案是：{账号定位}；主要讲给 {目标人群}；当前目标是 {业务目标}；长期母题是 {母题摘要}；内容里产品 / 服务露出应该 {露出比例}；明确不能碰的是 {禁区摘要}。

如果这版没变化，你直接回复“认可”或“同意”，我就继续做产品对象检查和热点研究；如果有变化，直接说要改哪里，比如“目标人群改成新能源车主”“这条不要引导车商账号”“禁区加上不聊事故责任”。
```

### 1.1 首次使用 / 没有账号

```text
我先帮你建一个账号档案，这样后面找热点、写文案、做画中画时不会串号。

你不用填表，先告诉我三件事：
1. 账号叫什么？
2. 主要讲给谁听？
3. 现在最想达成什么目标？比如涨粉、引流、建立信任、卖服务。

你简单说就行，我会整理成账号档案，再给你确认。
```

### 1.2 账号档案草案完成

```text
我整理了一版账号档案：账号定位是 {账号定位}；主要讲给 {目标人群}；当前目标是 {业务目标}；主要母题是 {母题摘要}；明确不能碰的是 {禁区摘要}。

如果这版没问题，你回复“认可”或“同意”，我就继续检查这次要说的产品 / 活动对象；如果要改，直接说“目标人群改成……”“禁区加上……”。
```

### 2. 选题完成，等待选题

```text
这是本轮的 topic_selection_panel：{topic_selection_panel_id}，当前 panel_status 是 {panel_status}。

这轮我不是直接拍 3 个题，而是先做了热点探索和筛选：原始候选 {raw_candidate_count} 个，进入评分 {scored_candidate_count} 个，主推荐 {main_recommendation_count} 个，降级 {degraded_candidate_count} 个，淘汰 {rejected_candidate_count} 个。主要筛掉原因是：{filtered_reason_summary}。

我建议优先选 {recommended_topic_id}，因为 {recommendation_reason}。

这几个候选的分工是：
- {topic_id_1}：{topic_role_1}，适合 {goal_1}，代价是 {tradeoff_1}
- {topic_id_2}：{topic_role_2}，适合 {goal_2}，代价是 {tradeoff_2}
- {topic_id_3}：{topic_role_3}，适合 {goal_3}，代价是 {tradeoff_3}

这些取舍必须写入 selection_tradeoff_map，不能只靠口头解释。

你可以直接回复“选 {recommended_topic_id}”，我会自动生成 Brief、写口播、做画中画、质检、平台包装和最终 HTML；如果想换方向，可以回复“选 {backup_topic_id} / 重找一轮 / 只要行业趋势”。
```

### 3. Brief 通过

```text
Brief 已经能写稿了，我会自动进入口播草案，不需要你再回复“继续”。如果你想打断，直接说“回到 Brief 改核心观点”。
```

### 4. 质检通过

```text
质检通过，内容还没发布。我会自动做四个平台的封面标题、视频标题、发布描述和话题标签；如果你想先改，直接说“按建议改口播”或“重做首屏画中画”。
```

### 5. 平台包完成

```text
最终 HTML 验收页已经生成，文案、画面、平台物料都在一个页面里。

如果满意，回复“认可”或“记录为已确认”；如果要小修，回复“只改抖音标题”“回到口播改前 5 秒”“回到画中画改首屏图”或“平台包装重来”，我会回到对应环节修完并重新生成 HTML；如果今天不发，回复“归档今天不发”；如果要发给别人，回复“导出转交包”。
```

### 6. 最终交付页完成

```text
最终交付页已经生成，适合在本项目里验收。你可以打开 HTML 复制文案、下载图片、查看平台物料。

你可以这样回复：
认可：我会把本版记为最终确认。
只改抖音标题 / 只改小红书标题：我会回到平台包装，修完后重新生成 HTML。
回到口播改前 5 秒：我会回到口播草案，修完后自动重跑画中画、质检、平台包装和 HTML。
回到画中画改首屏图：我会回到画中画，修完后自动重跑质检、平台包装和 HTML。
导出转交包 / 导出单文件 HTML：我会生成可转交版本。
记录发布结果：只记录你人工发布的信息，不自动发布。
归档今天不发：保留资产但不继续推进。
```

## 六、禁止写法

不要把系统状态丢给用户：

```text
请确认。
请选择状态。
是否进入下一步？
是否继续？
请回复继续写口播。
等待人工确认。
下一步怎么做？
请选择 delivery_status。
```

也不要只列技能名：

```text
下一步：content-brief-compiler。
建议进入下一个环节。
```

必须改成人话：

```text
你选了 T002，我会把这张选题卡状态改成已选择，然后自动生成 Brief 和口播草案。中间如果发现边界风险，我会停下来告诉你该改哪里。
```

## 七、全流程引导检查表

每轮收口前检查：

```text
是否只有真正需要人判断的地方才停。
用户明确选择后，是否自动进入下一环节。
是否有主推荐，而不是平铺一堆选项。
是否解释了为什么推荐。
是否给出可直接回复的短句。
是否说明每个回复会触发什么。
是否避免字段名、状态名、skill 名替代人话。
是否更新工作流状态记录。
是否保持 account / session / research_run_id 不断链。
换账号时是否先做了账号档案对齐确认。
```

如果任一项不满足，先修正引导语和交接物，再继续产出。
