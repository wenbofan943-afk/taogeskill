# 交接物字段词典

> 状态：传播工作流字段基准  
> 主责：统一涛哥创作工作流中各 skill 的交接物名称、技术字段、中文展示名、状态值和别名处理。  
> 边界：本文件只约束涛哥创作工作流和 skill 编排，不是产品数据库 schema，不直接约束客户端、服务器或发布链路。

## 一、硬规则

1. 跨 skill 传递时必须使用“技术字段名”，例如 `brief_status`，不要只写“Brief 状态”。
2. 中文展示名只用于给人看，不能替代技术字段。
3. 同一个产物只能有一个标准技术名；旧叫法只作为别名解释，不得继续新增。
4. 每个交接物输出时必须带 `next_skill`；到人工确认或流程结束时，`next_skill` 写 `human_confirm` 或 `done`。
5. 每个状态字段只能使用本文件列出的允许值；需要新增值时，先修订本词典，再修订上下游 skill。
6. 用户交互位置必须同时给“状态字段”和“口语化引导语”，不能只输出字段名。

## 三、字段命名规范

| 规则 | 写法 |
|---|---|
| ID 字段 | `{artifact}_id`，例如 `topic_id`、`brief_id`、`draft_id` |
| 状态字段 | `{artifact}_status`，例如 `brief_status`、`review_status` |
| 下游 skill | `next_skill` |
| 来源 ID | `source_{artifact}_id`，例如 `source_review_id` |
| 调研来源 ID | `source_research_run_id` |
| 产物路径 | `artifact_path` 或 `artifact_paths`，必须指向 `accounts/{账号名}/runs/{session_id}/` 下的文件 |
| 布尔判断 | 用 `is_` / `has_` 前缀，或直接用状态值；避免混用 `pass` |
| 中文字段 | 可以出现在展示层，但旁边必须有标准技术字段 |

禁止新增这些模糊字段：

```text
状态
pass
是否通过
是否进入下一步
下一步
Brief 状态
draft 状态
package_pass
```

如需展示，统一写成：

```text
- brief_status：brief_pass
- next_skill：copywriting-draft-writer
- 给用户的引导语：这版 Brief 能写稿了，我会进入口播草案。
```

`topic_selection_panel` 是 Topic Gate 给用户看的“选题候选反馈面板”。它不替代 `topic_card`，而是解释本轮热点探索做到了什么、为什么筛掉一批、为什么推荐这几个，以及用户选不同题的收益和代价。


字段关系：

```text
raw_candidate_count：进入热点候选池的原始候选数。
scored_candidate_count：进入热点评分表的候选数。
main_recommendation_count：进入主推荐区的候选数。
degraded_candidate_count：降级为轻观点、行业趋势、复盘或方法论的候选数。
rejected_candidate_count：淘汰或不展示的候选数。
filtered_reason_summary：只写原因摘要，不重复整张候选池。
topic_option_ids：只引用 topic_card 的 topic_id，不复制完整选题卡。
topic_role_map：topic_id -> topic_role。
selection_tradeoff_map：topic_id -> 选择收益和代价。
recommended_topic_id：默认推荐的 topic_id，必须存在于 topic_option_ids。
```

禁止：

```text
不得只给 3 个标题而不输出 topic_selection_panel。
不得用 candidate_funnel 这类不可拆字段替代 raw/scored/main/degraded/rejected 五个计数字段。
不得把被筛掉的候选静默删除；至少要在 filtered_reason_summary 说明主因。
```

## 六、用户交互引导语字段

所有需要用户选择的位置，输出中必须包含：

```text
human_prompt
human_reply_examples
recommended_action
auto_next_action
task_after_navigation
```

其中：

```text
human_prompt：给用户看的口语化引导语，不使用字段名代替人话。
human_reply_examples：用户可以直接回复的短句。
recommended_action：本轮默认推荐动作，必须说明为什么推荐。
auto_next_action：用户选择后系统会自动做什么；不能要求用户说“继续”。
task_after_navigation：参考 dbskill 的任务后导航，根据本轮结论推荐 2-3 个下一步并解释原因。
```

示例：

```text
human_prompt：最终 HTML 验收页已经生成。你可以直接人工发布；如果要改，直接说“只改抖音标题”“回到口播改前 5 秒”“回到画中画改首屏图”或“平台包装重来”，我会回到对应环节修完并重新生成 HTML。
human_reply_examples：认可 / 只改抖音标题 / 回到口播改前 5 秒 / 回到画中画改首屏图 / 归档今天不发 / 导出转交包 / 记录发布结果
recommended_action：先查看最终 HTML；满意则认可或人工发布；不满意则说清局部返工点。
auto_next_action：选题确认且平台包装通过后自动进入 final-delivery-builder，不等待“确认采用”。
task_after_navigation：认可确认 / 人工发布记录 / 局部返工并重建 HTML / 归档今天不发 / 导出转交包。
```

收口阶段不能只说“等待确认”。必须把用户可以怎么回说清楚：

```text
human_prompt：最终 HTML 验收页已经生成。你可以直接复制文案、下载图片、拿平台物料人工发布；如果要改，直接说要改哪里，我会按 revision_path 回到对应上游修订并重新生成 HTML。
human_reply_examples：认可 / 只改小红书标题 / 回到口播改前 5 秒 / 回到画中画改首屏图 / 归档今天不发 / 导出转交包 / 记录发布结果
```

不要只写：

```text
等待确认
涛哥确认
待选择
请确认
下一步怎么做
是否继续
请回复继续写口播
```

自动推进规则以 `docs/reference/人类引导与任务后导航规范.md` 为准。Brief 通过、质检通过、平台包装通过、最终 HTML 生成前的下一环节都不得再让用户回复“继续”或“确认采用”。

## 七、字段治理规则

1. 新增 skill 前，先在本文件登记它吃什么交接物、产什么交接物。
2. 修订 skill 输出格式时，必须核对本词典，避免新增同义字段。
3. 如果发现历史文档使用旧字段，优先在词典里登记别名，再逐步修订核心 skill。
4. 传播研究资产文件可以用中文标题，但结构化块必须带技术字段。
5. 每轮“小扫地”要检查是否出现新的模糊字段：`状态`、`pass`、`下一步`、`是否通过`、`等待确认`。
