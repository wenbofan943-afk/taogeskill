# 热点文案 Skill 方法论与 SaaS 承接设计

> 状态：传播研究方法论单元  
> 主责：定义热点文案 skill 的轻量执行边界，以及未来如何迁入短视频创作 SaaS。  
> 边界：本文不是产品开发验收包，不是代码开发任务，不直接修改客户端、服务器、数据库、积分、license、构建或发版链路。

---

## 一、核心判断

热点文案能力本质上是一个 skill，不是前端 UI，也不是内容管理后台。

它应该像一个“智能脚本”：

```text
一句话唤醒
-> 读取必要上下文
-> 按固定步骤思考
-> 在关键位置停下来等涛哥确认
-> 输出稳定格式
-> 可被下次继续调用
```

它不应该在本项目里长成一套发布日历、账号后台或 SaaS 看板。

---

## 二、三项目分工

当前三套能力的关系：

| 项目 | 定位 | 负责什么 | 不负责什么 |
|---|---|---|---|
| 认知推演工作流 | 上游脑库 workflow | 认知档案、表达资产、知识原子、推演路径、文案初稿、去 AI 味 | 账号管理、发布管理、产品功能 |
| 短视频创作项目 | 中游内容生产 SaaS | IP 层、账号矩阵、母题、作品、发布观察、回流 | 当前具体产品的客户端和服务器功能 |
| 涛哥创作工作流 | 下游轻量内容工作流 | 当前产品传播材料、账号内容资产和工作流方法论 | 长期多账号内容生产中台 |

一句话：

```text
认知推演工作流负责“我怎么想”。
短视频创作项目负责“我怎么发”。
涛哥创作工作流负责“围绕某个账号和产品，今天怎么形成可发布内容”。
```

---

## 三、Skill 的轻量边界

当前热点文案能力拆成 6 个主 skill + 1 个兼容入口：

```text
propagation-router
  传播总控，只做路由、交接物检查和下一步建议。

hotspot-topic-research
  热点发现、评分、母题关联、推导链和选题卡。

content-brief-compiler
  已选题卡到内容 Brief 的上下文编译。

copywriting-draft-writer
  第一阶段只负责把通过的 Brief 写成短视频口播草案；图文、长文、朋友圈、社群、FAQ 和官网说明先保留为未来路由，不展开制作办法。

talking-head-image-pip
  热点口播画中画视觉策略、画面编排和 image 提示词。

copywriting-quality-review
  文案质检、AI 味、涛哥味、口播流畅度、产品风险和下一步建议。

platform-packaging-adapter
  质检通过后，先编译分发包装输入包，再为同一条视频生成抖音、快手、小红书、视频号的封面标题、视频标题、发布描述和话题标签。

hotspot-copywriting-research
  旧唤醒词兼容入口，只做转向，不再承载全部流程。
```

完整工作流仍是：

```text
热点发现
-> 热点评分
-> 母题关联
-> 推导链判断
-> 内容 brief
-> 短视频口播草案
-> 画中画视觉策略 / image 提示词
-> 文案 + 视觉联合质检
-> 分发包装输入包
-> 多平台分发包装
-> 涛哥人工确认
```

第一阶段主路径：

```text
涛哥选择选题
-> 自动编译 Brief
-> 自动生成短视频口播草案
-> 自动生成画中画视觉策略 / image 提示词
-> 自动做文案 + 视觉联合质检
-> 自动编译分发包装输入包
-> 自动生成多平台分发包装
-> 只在最终结果让涛哥确认
```

人工默认只卡两个点：

```text
选题确认。
最终发布前确认。
```

中间的 Brief、口播草案、画中画策略和联合质检默认自动推进；只有出现事实不稳、推导链过虚、产品承诺越界、账号定位冲突、内容形式无法判断或风险表达时，才停下来问涛哥。

任何需要用户交互的位置，必须给口语化引导语，不能只抛字段名或状态名。用户应该能直接回复“选 A”“按推荐继续”“回到 Brief”“重做首屏图”“只改抖音标题”“确认这一版”等自然语言。

五秒完播率是口播主路径的第一优先级。

本项目采用成熟短视频方法论和 dbskill 的共同定义：

```text
开头是内容的试用装。
好开头 = 话题 + Hook + 可信度。
前 1 秒停住划走动作。
前 3 秒让观众知道讲什么、为什么值得看。
前 5 秒让 Hook 承诺和正文接上。
```

口播草案必须先生成“五秒留存设计”，再写正文。五秒留存设计至少包含 3 个 Hook 方案、推荐 Hook、心理触发点、可信度来源、转场句、首屏画中画任务和五秒划走风险。

五秒留存评分满分 10 分：

```text
话题清晰度：2 分
Hook 强度：2 分
可信度 / 现场感：2 分
悬念或冲突：2 分
转场承接：1 分
口播顺滑：1 分
```

推荐 Hook 低于 7 分，不进入画中画，先回到口播草案重写开头。

当前不做：

1. 不做发布工具。
2. 不做前端页面。
3. 不做内容日历。
4. 不做账号管理。
5. 不做数据库写入。
6. 不接平台发布 API。
7. 不自动登录平台。
8. 不自动评论、私信、互动。
9. 不把草案当正式口径。

## 三点五、Skill 流转稳定性

拆 skill 的前提是交接物稳定。没有交接物，不拆。

当前固定交接物以 [交接物字段词典](./交接物字段词典.md) 为准。方法论文档只保留阶段说明，不重复维护字段清单，避免“上面叫这个、下面叫那个”。

标准链路：

```text
account_profile
-> research_run_record
-> hotspot_candidate
-> topic_card
-> content_brief
-> draft
-> visual_plan
-> quality_review
-> platform_package_input
-> platform_package
-> content_delivery_record
-> human_confirm
```

第一阶段 `content_format` 默认是 `短视频口播`。图文、长文、朋友圈、社群、FAQ 和官网说明只作为未来可扩展 `content_format` 保留，不在本阶段展开写法。

流转规则：

```text
无交接物 -> propagation-router -> hotspot-topic-research
research_status = research_done -> hotspot-topic-research，进入热点候选池和评分
topic_status = topic_selected_for_brief -> content-brief-compiler
brief_status = brief_pass -> copywriting-draft-writer，默认生成短视频口播草案
draft_status = draft_created -> talking-head-image-pip
visual_plan_status = visual_plan_pass -> copywriting-quality-review，做文案 + 视觉联合质检
review_status = review_pass -> platform-packaging-adapter，先编译 platform_package_input
input_status = input_pass -> platform-packaging-adapter，生成 platform_package
package_status = package_pass -> platform-packaging-adapter，生成 content_delivery_record
delivery_status = delivery_ready -> human_confirm，给涛哥确认采用、返工、归档或放弃的明确选项
delivery_status = delivery_confirmed / delivery_archived / delivery_discarded -> done，本轮收口
任一阶段不通过 -> 按 next_skill 回退
```

每轮任务还要有轻量状态记录：

```text
workflow_session_record
-> 记录 current_stage / current_artifact / artifact_ids / last_decision
-> 用户说“接着上次”时，先恢复状态，再路由
-> 用户确认完成、归档或放弃时，写 session_completed / session_archived
```

拆分原则：

```text
可以拆 skill，但必须先定义交接物。
每个专项 skill 只读上一步交接物，不重新解释全局。
总控 skill 只做路由，不做诊断。
旧入口只兼容旧唤醒词，不继续膨胀。
```

## 四、成熟项目经验取舍

调研成熟内容工作流后，本项目只吸收适合 skill 的部分：

| 成熟做法 | 本项目吸收方式 | 不吸收什么 |
|---|---|---|
| Content Brief | 写成文案前的内容 brief 卡 | 不做 UI 表单 |
| Human-in-the-loop | 选题和最终文案必须涛哥确认 | 不做复杂审批流 |
| Guardrails | 事实等级、产品承诺、灰产误解、AI 味质检 | 不做合规后台 |
| Editorial Calendar | 只保留未来可迁字段 | 当前不做日历 |
| Content Pillars | 使用“账号母题”概念 | 不照搬英文内容支柱体系 |
| Evals / Quality Review | 结构化 dbskill 质检表 | 不引入评测平台 |
| Research Log | 写成 `research_run_record`，记录来源计划、实际来源、核验和风险 | 不做新闻数据库 |
| Save / Restore / Report | 写成 `workflow_session_record`，支持接着上次和任务后导航 | 不做完整内容管理后台 |
| Post-task Navigation | `propagation-router` 每轮结束推荐 2-3 个下一步 | 不把推荐变成强制流程 |

核心取舍：

```text
成熟项目的方法论可以吸收。
成熟项目的系统形态不在本项目复制。
```

## 五、标准执行链

### 1. 运行前输入卡

每轮先确定：

```text
账号 / 产品：
目标人群：
内容目标：
热点范围：
是否允许大热点：
是否读取 IP 资产：
输出形式：
```

缺省值可以由 skill 自行假设；但涉及账号定位、产品承诺、敏感风险、是否发布时，必须停下来问涛哥。

### 2. 热点探索

输出 10-20 个候选热点，初筛保留 5-8 个。

每个热点至少包含：

```text
来源
时间
一句话事实
热度信号
事实核验等级
热点生命周期
可能母题
风险
```

事实核验等级：

```text
A：官方 / 权威来源确认
B：多来源交叉确认
C：单来源，需谨慎
D：传闻或不可核验，不建议做
```

热点生命周期：

```text
正在上升
高位扩散
开始衰退
已过时
适合复盘
```

### 3. 选题 + 策略 + 推导链

对通过初筛的热点进行 12 分评分。

8 分以上进入选题卡；5-7 分只保留轻观点；低于 5 分淘汰。

选题必须输出：

```text
文案策略
核心指标
产品占比
推导链
哪一跳最虚
如何补桥
是否值得继续
```

到这里必须停下，让涛哥选择。

### 4. 内容 brief

涛哥确认选题后，先生成内容 brief，再写正文。

内容 brief 是自动生成的中间产物，不是默认人工审批节点。它用来把上层资产编译成写作输入包，防止文案跑散：

```text
目标账号
目标人群
内容目标
文案策略
核心观点
事实核验等级
热点生命周期
账号母题
对应产品能力
必须调用的 IP 资产
不能承诺什么
不能怎么讲
推荐平台 / 形式
成功判断
人工确认点
```

默认规则：

```text
brief_status = brief_pass。
brief_status = brief_needs_human_confirm。
brief_status = brief_risk_high。
brief_status = brief_needs_format_change。
brief_status = brief_blocked。
```

只有目标冲突、账号定位冲突、产品露出不清、核心观点不清、事实风险高、CTA 不清、IP 资产缺失或产品承诺越界时，才停下来找涛哥确认。

### 5. IP 资产调用

正式写文案前，尽量调用上游 IP 资产：

```text
认知档案：涛哥怎么看问题
表达工坊：涛哥怎么说话
知识原子：涛哥有哪些观点、案例、判断
修改偏好：涛哥不喜欢哪些 AI 味
```

如果本轮没调用到 IP 资产，必须说明原因。

### 6. 文案 + 质检

第一阶段只输出短视频口播草案：

```text
短视频口播草案
标题备选
```

然后用 dbskill 方法论质检：

```text
Hook 是否成立
观点是否像涛哥
产品是否硬插
事实是否可靠
是否有灰产误解
是否有 AI 味
是否值得发布
```

如果事实、产品承诺或风险项不过，不输出可发布版，只输出修改建议。

### 6.5 多平台分发包装

文案 + 视觉联合质检通过后，调用 `platform-packaging-adapter`。

这一步不是重写视频，也不是自动发布，而是同一条视频的入口包装：

```text
口播草案
-> 画中画方案
-> 质检通过
-> 编译分发包装输入包
-> 抖音 / 快手 / 小红书 / 视频号入口包装
-> 涛哥人工确认
-> 人工发布
```

分发包装输入包是显式编译层，不是新的人工确认点：

```text
brief / draft / visual_plan / review
-> platform_package_input
-> platform_package
```

它必须包含：

```text
账号
内容目标
目标人群
目标平台
核心话题
核心观点
推荐 Hook
前 5 秒口播
首屏画中画任务
视频主体摘要
产品露出比例
产品边界
不可说
风险词
CTA
平台目标
```

输入包不通过时，不能生成正式平台包：

```text
Hook 不清 -> 回到口播草案。
首屏画中画任务不清 -> 回到画中画。
review 未通过 -> 回到质检。
产品边界不清 -> 回到 Brief 或找涛哥确认。
```

平台包通过后才进入最终人工确认。质检通过不等于最终确认，质检通过只代表可以进入分发包装。

最终确认引导语：

```text
这版分发包已经完成，可以人工发布。你可以回复“确认这一版”，也可以回复“只改抖音标题 / 重做小红书包装 / 回到口播改 Hook / 回到画中画改首屏图”。
```

每个平台必须输出：

```text
封面标题
视频标题
发布描述
话题标签
推荐组合
平台语气说明
风险提醒
人工发布备注
```

平台适配原则：

| 平台 | 优先目标 | 包装重点 |
|---|---|---|
| 抖音 | 停留、热点情绪、推荐流点击 | 强 Hook、冲突感、热点情绪、少量精准话题 |
| 快手 | 真实感、社区信任、垂类识别 | 接地气、真人经验、少一点营销味 |
| 小红书 | 搜索、收藏、方法感、种草 | 关键词、痛点、清单感、封面标题和标题强绑定 |
| 视频号 | 信任、转发、熟人链路 | 稳、可信、行业观察感、适合分享 |

边界：

```text
不为单个平台重剪视频。
不接平台发布 API。
不自动登录。
不抓后台数据。
不承诺流量结果。
```

### 7. 热点画中画提示词

当口播草案或关键段落已存在，且需要做短视频画中画时，调用 `talking-head-image-pip`。

这一步不是生成工程图、HTML 卡片或 PPT 说明页，而是做传播视觉：

```text
口播文字
-> 留存风险点
-> 留人任务
-> 情绪 / 冲突 / 观点
-> 画面隐喻
-> 留人任务驱动的视觉提示词卡
-> 剪辑可用素材
```

核心规则：

```text
画中画不是装饰。
画中画的目的，是在观众本来会划走的地方，让他多停 2-5 秒。
```

每个画面必须回答：

```text
观众此刻卡在哪里？
留存风险是什么？
这张图的唯一留人任务是什么？
这张图帮观众理解什么？
它是冲突、证据、隐喻、对比、场景、结果还是情绪？
如果没有这张图，口播会不会更弱？
不用这张图会损失什么？
```

画中画前置程序：

```text
拆口播：Hook / 热点事实 / 观点转折 / 抽象解释 / 案例证据 / 产品露出 / CTA。
找风险：连续讲太久、太抽象、需要证据、情绪不够、转场容易断、信息密度下降、口播卡壳、产品能力说不清。
定必要性：必须画中画 / 建议画中画 / 可不画 / 不建议画。
定留人任务：停住划走 / 补信息 / 增可信 / 放大冲突 / 解释抽象 / 节奏打断 / 顺接转场 / 强化记忆点。
再写提示词。
```

提示词工程口径：

```text
Prompt = 留人任务 + 口播语义 + 画面类型 + 主体动作 + 场景环境 + 镜头构图 + 光线情绪 + 项目风格 + 风险约束 + 负面提示 + 验收标准
```

每条提示词必须包含镜头 / 构图 / 字幕安全区 / 小屏可读性，并且通过验收：

```text
能否让观众多停 2-5 秒？
是否贴合口播当下语义？
是否解决明确留存风险？
是否会抢口播？
是否误导产品能力？
是否适合 9:16 小屏？
是否只是好看但没用？
```

默认不再使用“工程出图逻辑”作为传播画面的主方案。工程图只保留在证据演示、流程解释、结果展示等少数场景。

在当前 Codex 环境里，image 生成可以作为内置能力直接调用。当前不把 image 出图做成外部 API、CLI 服务、渲染后台、上传链路或长期工程模块。

执行边界：

```text
中间阶段先产视觉策略和 image 提示词。
最终交付前，必要画中画必须实际出图，或明确标记 pending_external / generation_failed。
出图后返回图片和使用的提示词。
图片保存到 accounts/{账号名}/runs/{session_id}/assets/images/。
最终 HTML 交付页展示图片、下载入口和插入位置。
未来可为非 Codex 用户预留 Seedream 4.0 / 5.0 等外部模型旁路，但当前不接 API、不保存 API key。
不自动发布，不自动剪辑，不上传平台。
```

热点画中画可用的主要画面类型：

| 类型 | 用途 |
|---|---|
| 冲突图 | 开头抓注意力，放大矛盾 |
| 隐喻图 | 把抽象观点转成可感知画面 |
| 证据图 | 展示工具干了什么，但不能泄露真实数据 |
